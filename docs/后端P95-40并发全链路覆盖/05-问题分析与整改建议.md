# 后端 P95-40 并发问题分析与整改建议

## 1. 当前问题的本质

当前最大的矛盾已经不是“有没有覆盖到后端所有路由”，而是：

> 为什么全链路 40 并发场景已经实现命中全覆盖，但大多数场景仍然不能成功返回，因而无法形成有效 P95 覆盖。

## 2. 当前问题分层

### 2.1 第一层：认证问题（401）

现象：

- 全链路结果中 `401 = 2860`
- 读链路专项结果中也仍有少量 `401`

说明：

- token 生命周期、复用方式、登录场景自身行为仍会影响部分结果。
- 这类问题不一定是业务接口本身慢，而可能是压测身份态不稳定。

建议：

1. 区分“获取 token 的场景”和“消费 token 的场景”。
2. 对全链路大套件，优先使用稳定 token 池，而不是混合过多即时登录行为。
3. 对登录链路单独建套件，不与大多数业务链路混跑。

### 2.2 第二层：鉴权问题（403）

现象：

- 全链路结果中 `403 = 4749`，是占比最高的失败类型。

说明：

- 这不是性能问题，而是角色权限与场景设计不匹配。
- 当前全量场景中，很多请求虽然 method/path 正确，但使用的账号并没有对应权限。

建议：

1. 按模块拆分角色，例如：用户域、工艺域、生产域、质量域、设备域。
2. 不要用单一角色去压全量链路。
3. 对写链路尤其要单独准备具备对应权限的账号或 token。
4. 将“权限不匹配”从性能问题中剥离出来，先做鉴权治理。

### 2.3 第三层：样本问题（404）

现象：

- 全链路结果中 `404 = 196`

说明：

- 这通常意味着场景使用了不存在的 `id`、资源尚未创建，或者环境数据被重置后样本失效。
- Detail 类链路最容易受到这个问题影响。

建议：

1. 建立可重复初始化的标准样本集。
2. 对 detail 场景尽量引用可重建样本，而不是依赖偶然存在的历史数据。
3. 在场景文件旁维护“样本依赖说明”。

### 2.4 第四层：接口定义或调用方式问题（405 / 422）

现象：

- `405 = 301`
- `422 = 49`

说明：

- `405` 说明 method/path 组合不正确，或者实际接口定义已变化。
- `422` 说明请求参数、query 或 body 不满足当前接口约束。

建议：

1. 定期用路由清单反查场景定义。
2. 将场景文件与接口签名演进联动维护。
3. 对写链路场景增加最小必需参数校验。

## 3. 为什么不建议继续直接扩大全链路场景数

因为当前问题不在“数量不足”，而在“质量不足”。

继续扩场景数会带来：

- 更多 `401/403/404/405/422`
- 更低的整体成功率
- 更难解释的总体 P95
- 更差的维护成本

所以后续优先级应是：

1. 先提升现有场景成功率。
2. 再根据需要扩展未覆盖链路。

## 4. 推荐整改路线

### 路线一：拆分全链路套件用途

把当前“全链路”拆成两个明确目标：

- **覆盖审计套件**：回答“有没有漏路由”。
- **性能验收套件**：回答“成功路径在 40 并发下表现如何”。

不要强行要求一个套件同时解决两类问题。

### 路线二：按角色拆权限域

建议按至少以下维度拆 token 池：

- 平台管理员
- 用户与权限管理员
- 生产业务角色
- 质量业务角色
- 设备业务角色
- 仅查看角色

这样可以显著降低 `403` 对整体结果的污染。

### 路线三：按链路类型拆套件

建议至少拆成：

1. 公共读链路套件
2. detail 读链路套件
3. 流式导出套件
4. 写链路功能套件
5. 写链路性能套件
6. 鉴权专项套件
7. 登录与会话专项套件

### 路线四：建立样本资产

建议为以下类型建立固定样本：

- 用户
- 角色
- 产品
- 工艺路线
- 工艺阶段
- 订单
- 订单工序
- 消息
- 供应商
- 设备台账/计划/执行记录

并让场景引用这些稳定样本，而不是隐式假设环境中一定有某条记录。

## 5. 当前最值得优先做的三件事

### 5.1 先把 403 压下来

因为 `403` 是最大噪声源，也是当前“命中全覆盖无法转化为有效全覆盖”的首要原因。

### 5.2 为 detail/写链路补标准样本

这样可以显著减少 `404/422`，提升场景可重复性。

### 5.3 保留 91 场景专项作为正式基线

在全链路问题未完全收敛前，不要放弃已有的可用基线。它仍然是当前最稳定、最适合持续回归的性能套件。

## 6. 推荐阶段性目标

### 阶段一

目标：让全链路套件从“命中全覆盖”提升到“多数场景有成功请求”。

衡量建议：

- `zero_success_count` 明显下降
- `403` 占比显著下降

### 阶段二

目标：让全链路套件形成“可分模块解释的有效覆盖”。

衡量建议：

- 每个模块至少有一组高成功率场景集
- `404/405/422` 只出现在少数可解释场景中

### 阶段三

目标：形成真正可用于发布门禁的后端 40 并发全链路回归体系。

衡量建议：

- 主要业务链路成功率稳定
- 总体 `p95_ms <= 500`
- 总体 `error_rate <= 5%`
- 失败项主要来自已知例外而非大面积权限问题
