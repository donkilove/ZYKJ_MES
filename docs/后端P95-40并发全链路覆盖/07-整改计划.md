# 后端 P95-40 并发整改计划

## 1. 文档目标

本文用于将“后端 P95-40 并发全链路整改”正式落成可执行计划，统一后续整改顺序、阶段目标、输入输出、验收标准与复跑检查口径。

本文特别解决以下问题：

1. 当前应该先修什么，后修什么。
2. `270` 场景与 `91` 场景分别承担什么职责。
3. 每一阶段的输入、输出、验收标准和优先级是什么。
4. 每一轮复跑应该看哪些检查项，避免整改顺序混乱。

---

## 2. 当前基线与边界

### 2.1 当前基线

- `270` 场景全链路套件：当前已经实现**命中全覆盖**，但尚未实现**有效全覆盖**。
- `91` 场景读链路专项：当前仍是**正式性能基线**，可继续承担阶段性门禁。
- 当前失败主体不是“没打到路由”，而是大量：
  - `401`
  - `403`
  - `404`
  - `405`
  - `422`

### 2.2 两套件分轨原则

必须持续区分：

1. **覆盖审计套件**
   - 对象：`270` 场景全链路套件
   - 目标：回答“有没有漏路由”“场景是否命中后端路由”
   - 当前用途：覆盖审计，不直接作为正式性能门禁

2. **性能验收套件**
   - 对象：`91` 场景读链路专项
   - 目标：回答“高成功率路径在 `40` 并发下是否满足门禁”
   - 当前用途：正式性能基线与持续回归门禁

### 2.3 总执行策略

> 先把请求做成，再把请求做快。

固定顺序如下：

1. 冻结基线与口径
2. 权限治理，优先压 `403`
3. 认证治理，压 `401`
4. 样本资产治理，压 `404`
5. 契约与场景定义校准，治理 `405/422`
6. 形成模块级有效覆盖
7. 重构正式门禁，再做 P95 冲刺

## 4. 分阶段执行说明

## 4.1 阶段 0：冻结基线与口径

### 目标

先固定“量什么、怎么解释、当前以什么为正式基线”。

### 主要动作

1. 冻结双套件职责。
2. 冻结失败分类口径。
3. 冻结当前正式性能基线为 `91` 场景。
4. 冻结当前 `270` 场景的解释口径为“覆盖审计优先”。

### 输入

- `03-现状盘点与结果解读.md`
- `04-执行说明与命令模板.md`
- `05-问题分析与整改建议.md`
- `06-证据索引.md`
- `.tmp_runtime/backend_40_e2e_combined_retry_20260411_120615.json`
- `.tmp_runtime/p95_loop_1.json`

### 输出

- 双套件职责说明
- 统一失败分类表
- 当前阶段性验收口径

### 验收标准

- 不再把 `270` 场景直接当正式性能门禁
- `91` 场景继续承担正式性能回归职责
- 汇报口径统一使用“命中覆盖 / 有效覆盖 / 门禁通过”三层概念

### 每轮复跑检查项

- 复跑命令是否误把 `270` 场景用于正式性能结论
- 汇报文案是否混用“命中全覆盖”和“有效全覆盖”
- 当前结果引用是否仍指向最新正式基线

### 依赖与并行

- 无前置依赖
- 必须最先完成

---

## 4.2 阶段 1：权限治理，优先压 403

### 目标

解决“命中了路由，但因权限不匹配直接失败”的主体问题。

### 主要动作

1. 建立角色/权限矩阵，至少覆盖：
   - 平台管理员
   - 用户与权限管理员
   - 生产业务角色
   - 质量业务角色
   - 设备业务角色
   - 只读角色
2. 为场景建立“场景 -> 角色域 -> token 池”映射。
3. 优先处理 `403` 最密集模块：
   1. `craft`
   2. `equipment`
   3. `production`
   4. `products`
   5. `quality`
4. 对 `401+403` 混合场景分桶，区分：
   - 纯权限不匹配
   - 角色错误 + token 不稳定
   - 资源域隔离冲突

### 输入

- `401/403` 失效场景清单
- 模块级场景列表
- 当前角色权限现状
- `08-角色-场景映射表.md`

### 输出

- 角色矩阵
- 场景-角色映射
- 模块 token 池方案
- 权限问题分桶清单

### 验收标准

- `403` 占比显著下降
- 五大重点模块都出现稳定成功请求样本
- 不再出现整批场景几乎全 `403` 的情况

### 每轮复跑检查项

- 各模块 `403` 数量与占比
- 各模块是否出现成功请求样本
- 场景是否使用了正确角色域 token
- 是否出现“为了过场景而临时扩权”的错误做法

### 依赖与并行

- 依赖阶段 0
- 各业务模块可并行梳理，但统一角色口径必须先定

---

## 4.3 阶段 2：认证治理，压 401

### 目标

把登录、token 获取、token 消费从混跑状态中拆开，减少认证污染。

### 主要动作

1. 拆分“登录获取 token”与“业务消费 token”。
2. 为不同角色建立稳定 token 池与刷新策略。
3. 不再让大型业务套件混跑过多即时登录行为。
4. 单独保留认证/会话专项，例如：
   - `auth-login`
   - `auth-logout`
   - `auth-me`
   - `me-profile`
   - `authz-snapshot`
   - `authz-permissions-me`
5. 检查 token 与 session 池在高并发下的复用模型。

### 输入

- 阶段 1 输出的角色矩阵与 token 池方案
- 认证相关场景清单
- 当前 token 生命周期与会话复用方式

### 输出

- 认证专项场景清单
- token 生命周期规则
- 业务 token 复用规范

### 验收标准

- `401` 明显收敛
- 当前“有成功但被 `401` 污染”的场景显著减少
- 认证问题可以在专项中单独解释，不再污染业务套件

### 每轮复跑检查项

- 认证专项是否与业务套件混跑
- token 是否在压测窗口内过期
- 相同 token 是否被不合理复用
- `401` 是否集中在可解释的登录/会话链路

### 依赖与并行

- 依赖阶段 1 的角色与 token 口径
- 可与阶段 1 收尾部分局部并行

---

## 4.4 阶段 3：样本资产治理，压 404

### 目标

让 detail 类和写链路场景依赖的数据可准备、可重建、可验证。

### 主要动作

1. 建立标准样本资产清单。
2. 建立样本初始化与回收流程。
3. 给 detail 场景绑定稳定样本。
4. 给写链路补前置依赖说明。
5. 为场景补“样本依赖说明”。

### 输入

- 阶段 1-2 收敛后的场景集
- `404` 失效模式
- `09-样本资产清单.md`

### 输出

- 样本资产清单
- 初始化顺序
- 场景样本依赖说明
- 样本回收/复位口径

### 验收标准

- `404` 显著下降
- detail 场景具备重复执行能力
- 环境重置后能恢复核心样本集

### 每轮复跑检查项

- 样本实体是否存在
- 样本状态是否符合接口前置条件
- 写链路是否破坏共享样本
- 初始化后校验点是否全部通过

### 依赖与并行

- 依赖阶段 1-2 基本收敛
- 各模块样本盘点可并行，但样本命名规范必须统一

## 4.5 阶段 4：契约与场景定义校准，治理 405 / 422

### 目标

让场景 JSON、接口定义和当前实现重新对齐。

### 主要动作

1. 逐模块核对 method/path 是否正确。
2. 逐模块核对 query/body 的最小必需参数。
3. 将场景按读链路、detail、写链路、导出链路、鉴权/会话链路重新整理。
4. 清理已过期或已偏离当前实现的场景定义。
5. 用 `10-405422差异清单模板.md` 维护修复闭环。

### 输入

- 阶段 3 的样本资产与初始化结果
- 场景 JSON
- 当前接口实现与约束
- `10-405422差异清单模板.md`

### 输出

- 契约差异清单
- 场景修正清单
- 已批准例外清单

### 验收标准

- `405/422` 明显下降
- 场景定义与当前接口契约基本一致
- 不再出现成批 method/path 错配

### 每轮复跑检查项

- method/path 是否与当前实现一致
- query/body 是否满足最小必需参数
- 差异清单是否更新到最新结果
- 例外场景是否有明确备注与责任人

### 依赖与并行

- 依赖阶段 3 的稳定样本
- `405` 与 `422` 两类修正可并行推进

## 4.6 阶段 5：形成模块级有效覆盖

### 目标

先让重点模块形成高成功率、可解释、可持续复跑的有效覆盖集合。

### 主要动作

1. 为重点模块建立高成功率子套件：
   - `craft`
   - `equipment`
   - `production`
   - `products`
   - `quality`
   - `users/authz`
2. 各模块内部按顺序推进：
   1. 先读链路
   2. 再 detail 链路
   3. 最后写链路
3. 为每个模块记录：
   - 成功率
   - 主要失败状态码
   - 样本依赖
   - 角色依赖
   - 是否适合纳入正式性能回归

### 输入

- 前四阶段整改结果
- 最新模块场景集
- 最新复跑结果

### 输出

- 模块级高成功率子套件
- 模块级有效覆盖报告
- 模块级剩余问题清单

### 验收标准

- 每个重点模块至少形成一组高成功率子套件
- `zero_success_count` 明显下降
- 不再只有少数认证相关接口成功

### 每轮复跑检查项

- 各模块成功率是否稳定
- 各模块 `zero_success_count` 是否下降
- 主要失败码是否已从 401/403 转向少量可解释异常
- 模块子套件是否具备重复执行能力

### 依赖与并行

- 依赖前四阶段输出
- 各模块之间可并行推进

## 4.7 阶段 6：重构正式门禁，再做 P95 冲刺

### 目标

形成长期可持续的“覆盖审计 + 模块回归 + 正式性能门禁”三层体系。

### 主要动作

1. 固化三层套件：
   - 覆盖审计套件
   - 模块级性能回归套件
   - 正式全链路性能门禁套件
2. `91` 场景继续承担短期正式门禁。
3. 从模块级高成功率集合中筛选未来全链路性能候选集。
4. 仅在高成功率集合上做性能优化：
   - SQL
   - 应用热点
   - 缓存
   - 连接池
5. 每轮只改一类变量，并保留前后对比证据。

### 输入

- 阶段 5 的模块级高成功率子套件
- `91` 场景当前基线
- 性能对比结果与观测指标

### 输出

- 三层套件体系说明
- 新的全链路性能候选集
- 性能优化对比证据

### 验收标准

- `91` 场景持续通过
- 新的全链路性能候选集具备高成功率与可解释性
- 正式门禁结果可复现、可回归、可审计

### 每轮复跑检查项

- `91` 场景是否持续通过
- 新候选集是否重新被 `401/403` 污染
- 性能指标变化是否来自真实成功路径
- 每轮优化是否只有单一变量变化

### 依赖与并行

- 依赖阶段 5 基本完成后才能开展
- 不能提前对混杂失败集合做性能结论

## 6. 关键风险与控制

### 风险 1：角色边界定义不清

**影响**：修一处坏一片，`403` 会反复出现。  
**控制方式**：角色矩阵版本化，模块级回归清单强制执行。

### 风险 2：样本漂移或污染

**影响**：同一场景多轮执行结果不稳定。  
**控制方式**：样本初始化、隔离、回收流程标准化。

### 风险 3：接口契约与实现继续脱节

**影响**：`405/422` 会反复回潮。  
**控制方式**：每轮场景调整后同步复核接口契约与实现。

### 风险 4：过早做性能优化

**影响**：把身份/样本/契约问题误判成性能瓶颈。  
**控制方式**：坚持“先做成，再做快”，只在高成功率集合上看 P95。
