# 后端 P95-40 并发覆盖口径与判定标准

## 1. 为什么要先统一口径

同样叫“覆盖”，在当前项目里至少有三种完全不同的含义：

1. 场景有没有打到路由。
2. 路由有没有成功返回。
3. 成功返回后，P95 和错误率是否满足门禁。

如果不区分这三层，结论很容易互相冲突。

## 2. 三个核心概念

### 2.1 命中覆盖

定义：场景请求至少触发到了目标 method/path 对应的后端路由。

特点：

- 关注“有没有打到”。
- 不要求返回 `200`。
- 可用于做“场景编排完整性”审计。

当前状态：**已达成全覆盖**。

### 2.2 有效覆盖

定义：场景请求不仅打到了路由，而且产生了成功请求，能够真实代表该链路在当前环境下的可用性与性能表现。

特点：

- 关注“是否成功”。
- 通常要排除大规模 `401/403/404/405/422` 污染。
- 可用于做“链路是否具备验收价值”的判断。

当前状态：**未达成全覆盖**。

### 2.3 门禁通过

定义：在有效请求基础上，整体指标满足门禁阈值。

当前工具的默认门禁条件是：

- `p95_ms <= 500`
- `error_rate <= 0.05`

只有同时满足，`gate_passed` 才会是 `true`。

## 3. 工具如何判定

当前门禁工具是 `tools/perf/backend_capacity_gate.py`，其输出结果至少包括：

- `overall`
- `overall_with_warmup`
- `scenarios_metrics`
- `gate_passed`

其中：

- `overall` 表示正式测量窗口内的结果。
- `overall_with_warmup` 包含预热阶段。
- `scenarios_metrics` 是逐场景明细。
- `gate_passed` 由整体 `p95_ms` 与 `error_rate` 同时判定。

## 4. 当前的两个主要套件

### 4.1 全链路命中覆盖套件

文件：`tools/perf/scenarios/combined_40_scan.json`

用途：

- 确认全后端路由是否都已被编排进场景。
- 审计“有没有漏路由”。

特点：

- 场景数 `270`
- 覆盖唯一注册路由 `253`
- 覆盖 GET 路由 `131`
- 同一路由可能存在多个样本 ID、不同 query 或不同业务状态，因此“场景数 > 路由数”是正常现象。

### 4.2 读链路专项性能套件

文件：`tools/perf/scenarios/full_89_read_40_scan.json`

配合内建场景：

- `login`
- `authz`
- `users`
- `production-orders`
- `production-stats`

组合后形成 `91` 场景专项集合。

用途：

- 做当前阶段最可用的 P95 回归。
- 做读链路优化后的性能验收。

## 5. 推荐的验收口径

| 使用场景 | 推荐口径 | 推荐套件 |
| --- | --- | --- |
| 审计是否漏测路由 | 命中覆盖 | `combined_40_scan.json` |
| 审计读链路性能回归 | 有效覆盖 + 门禁通过 | `91` 场景专项集合 |
| 分析权限、样本或接口定义问题 | 逐场景成功率与状态码 | `combined_40_scan.json` 结果明细 |
| 评估写链路容量 | 单独设计写链路门禁 | 不建议直接复用当前口径 |

## 6. 常见误区

### 6.1 “全部路由都打到了”不等于“全部路由都通过了”

这是当前最容易混淆的点。`combined_40_scan` 已经说明**场景编排完成**，但没有说明**场景执行成功**。

### 6.2 “P95 结果存在”不等于“P95 结果可信”

当 `401/403` 大规模出现时，虽然工具仍会输出 `p95_ms`，但该值只能说明“当前混合结果下的延迟”，不能直接当成业务成功路径的性能基线。

### 6.3 “专项集合通过”不等于“全链路已通过”

`91` 场景专项集合通过，说明**这套精选读链路**已具备性能回归价值；并不代表 `253` 条后端路由都已形成有效性能覆盖。
