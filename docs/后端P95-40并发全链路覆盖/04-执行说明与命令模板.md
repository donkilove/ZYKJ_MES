# 后端 P95-40 并发执行说明与命令模板

## 1. 目的

本文用于说明当前后端 P95-40 并发相关套件怎么执行、适合在什么场景下执行，以及结果应该如何保存和解释。

## 2. 当前工具入口

CLI 入口在 `tools/project_toolkit.py`，对应子命令为：

```bash
python tools/project_toolkit.py backend-capacity-gate --help
```

工具本体在：

- `tools/project_toolkit.py`
- `tools/perf/backend_capacity_gate.py`
- `tools/perf/write_gate/sample_context.py`
- `tools/perf/write_gate/sample_registry.py`

压测账号初始化脚本在：

- `backend/scripts/init_perf_capacity_users.py`
- `backend/scripts/init_perf_production_craft_samples.py`

## 3. 关键参数说明

默认重要参数包括：

- `--base-url`：目标后端地址
- `--scenarios`：场景名列表
- `--scenario-config-file`：外部 JSON 场景定义文件
- `--concurrency`：并发数，当前常用值是 `40`
- `--session-pool-size`：HTTP 会话池大小，当前常用值是 `20`
- `--token-count`：token 池大小
- `--duration-seconds`：正式测量时长
- `--warmup-seconds`：预热时长
- `--spawn-rate`：worker 拉起速度
- `--p95-ms`：P95 阈值，默认 `500`
- `--error-rate-threshold`：错误率阈值，默认 `0.05`
- `--output-json`：结果输出路径
- `--sample-context-file`：样本上下文 JSON；用于把 `{sample:key}` 占位符替换成真实样本 `id/code`

若使用 `--scenario-config-file`，当前还支持以下 JSON 配置能力：

- 顶层 `token_pools`：声明多 token 池
- 场景级 `token_pool`：显式绑定场景使用哪个 token 池
- 场景级 `role_domain`：若未显式指定 `token_pool`，默认按 `pool-<role_domain>` 绑定

这意味着阶段 1-2 已经可以在工具层把“场景 -> 角色域 -> token 池”的口径落到执行配置里，而不必继续把所有业务场景都压到单一 token 池上。

## 4. 推荐执行场景

### 4.1 做全链路命中覆盖审计

适用文件：`tools/perf/scenarios/combined_40_scan.json`

适用目标：

- 检查全后端路由是否都被编排进场景。
- 检查大场景集调度是否存在遗漏。

注意：

- 这类执行结果不能天然视为性能验收结果。
- 执行后必须重点看成功率和状态码分布。

### 4.2 做正式性能回归

适用文件：`tools/perf/scenarios/full_89_read_40_scan.json`

适用目标：

- 验证读链路在 40 并发下是否仍满足门禁。
- 作为当前阶段最稳定的 P95 回归基线。

## 5. 命令模板

### 5.1 读链路专项 91 场景门禁模板

```bash
python tools/project_toolkit.py backend-capacity-gate \
  --base-url http://127.0.0.1:8000 \
  --scenario-config-file tools/perf/scenarios/full_89_read_40_scan.json \
  --scenarios login,authz,users,production-orders,production-stats \
  --concurrency 40 \
  --session-pool-size 20 \
  --token-count 1 \
  --duration-seconds 60 \
  --warmup-seconds 5 \
  --spawn-rate 10 \
  --p95-ms 500 \
  --error-rate-threshold 0.05 \
  --request-timeout-seconds 10 \
  --output-json .tmp_runtime/p95_loop_manual_run.json
```

说明：

- 实际执行时，`--scenarios` 通常需要与外部配置中的场景共同组成完整集合。
- 若希望复现实仓库中的 `91` 场景结果，应以历史 evidence 中实际使用的场景串或自动生成逻辑为准。

### 5.3 多 token 池配置模板

若要按角色域拆 token 池，建议在场景配置文件中采用以下结构：

```json
{
  "token_pools": [
    {
      "name": "pool-production",
      "login_user_prefix": "loadtest_production_",
      "token_count": 8
    },
    {
      "name": "pool-quality",
      "login_user_prefix": "loadtest_quality_",
      "token_count": 8
    }
  ],
  "scenarios": [
    {
      "name": "production-orders-read",
      "method": "GET",
      "path": "/api/v1/production/orders",
      "token_pool": "pool-production"
    },
    {
      "name": "quality-trend-read",
      "method": "GET",
      "path": "/api/v1/quality/trend",
      "role_domain": "quality"
    }
  ]
}
```

说明：

- `token_pool` 优先级高于 `role_domain`。
- `role_domain=quality` 且未写 `token_pool` 时，默认绑定 `pool-quality`。
- 若场景引用了不存在的 token 池，工具会在启动阶段直接失败，而不是在压测过程中静默落回默认池。
- 当前仓库中的 `tools/perf/scenarios/*.json` 已嵌入标准 `token_pools` 定义，并完成首轮真实池切换。
- 当前已切到真实池的角色域包括：`auth -> pool-admin`、`user-admin -> pool-user-admin`、`production -> pool-production`、`quality -> pool-quality`、`equipment -> pool-equipment`。
- `readonly` 相关场景当前仍保留在 `default` 池，因为仓库里尚无独立只读角色。

### 5.4 压测账号初始化模板

在切到真实角色池前，建议先执行：

```bash
python backend/scripts/init_perf_capacity_users.py --password Admin@123456
```

当前默认会初始化以下压测池与短前缀：

| 池名 | 角色代码 | 用户名前缀 | 默认数量 | 说明 |
| --- | --- | --- | --- | --- |
| `pool-admin` | `system_admin` | `ltadm` | 2 | 平台级排障与兜底复核 |
| `pool-user-admin` | `system_admin` | `ltusr` | 2 | 当前仓库无单独内置用户管理员角色，先复用系统管理员 |
| `pool-production` | `production_admin` | `ltprd` | 4 | 生产域主池 |
| `pool-quality` | `quality_admin` | `ltqua` | 4 | 质量域主池 |
| `pool-equipment` | `maintenance_staff` | `ltmnt` | 4 | 设备域主池 |
| `pool-operator` | `operator` | `ltopr` | 4 | 操作员池，自动绑定首个启用工序阶段 |

说明：

- 当前用户口径要求用户名尽量短，因此默认前缀都控制在 5 个字符以内。
- `pool-readonly` 暂未默认生成，因为当前仓库没有独立内置只读角色。
- 执行脚本前会先确保内置角色与 admin 基线存在。
- 脚本已在本地成功执行过一轮，当前共初始化 `20` 个压测账号。

## 6. 结果保存建议

建议统一保存到 `.tmp_runtime/`，并在文件名中包含：

- 套件名
- 日期
- 是否 retry
- 是否 smoke

示例：

- `.tmp_runtime/backend_40_e2e_combined_retry_20260411_120615.json`
- `.tmp_runtime/p95_loop_1.json`
- `.tmp_runtime/coverage_smoke_20260411.json`

## 7. 结果查看重点

### 7.1 看整体

优先看：

- `overall.success_rate`
- `overall.error_rate`
- `overall.p95_ms`
- `overall.status_counts`
- `gate_passed`

### 7.2 看逐场景

如果整体失败，再看：

- 哪些场景 `successful_requests = 0`
- 哪些场景主要返回 `401/403`
- 哪些场景主要返回 `404/405/422`

### 7.3 看调度完整性

对于大场景集，还应确认：

- 是否存在 `zero_request_count > 0`
- 是否存在大量 `zero_success_count`
- 是否出现“前缀场景反复执行，尾部场景几乎未命中”的现象

## 8. 当前已知执行经验

### 8.1 不要手动维护超长场景名串

建议永远从 JSON 动态生成，避免：

- 漏场景
- 拼错场景名
- 场景名与配置不一致

### 8.2 大场景集必须关注 worker 起始索引打散

当前 `backend_capacity_gate.py` 已引入 `_initial_worker_scenario_index()`，用于避免所有 worker 都从场景头部起跑。

这类问题在小场景集上不明显，但在 `270` 场景规模上会直接造成“表面执行了，实际没有真正覆盖尾部场景”。

### 8.3 不要把低成功率结果直接用于性能结论

当失败主要由认证、权限、样本问题主导时，结果更适合进入“问题分析”流程，而不是直接进入“性能验收”流程。

### 8.4 角色域拆池后，先验证“有没有进对池”

当你开始使用多 token 池后，每轮复跑都应新增检查：

- 场景是否绑定到了预期的 `token_pool`
- 是否仍有大量业务场景落回默认池
- `401/403` 是否随着拆池而下降

如果拆池后 `403` 没明显改善，优先回查 `08-角色-场景映射表.md`，不要直接把问题归因到性能本身。
