# 后端 P95-40 并发执行说明与命令模板

## 1. 目的

本文用于说明当前后端 P95-40 并发相关套件怎么执行、适合在什么场景下执行，以及结果应该如何保存和解释。

## 2. 当前工具入口

CLI 入口在 `tools/project_toolkit.py`，对应子命令为：

```bash
python tools/project_toolkit.py backend-capacity-gate --help
```

工具本体在：

- `tools/project_toolkit.py`
- `tools/perf/backend_capacity_gate.py`

## 3. 关键参数说明

默认重要参数包括：

- `--base-url`：目标后端地址
- `--scenarios`：场景名列表
- `--scenario-config-file`：外部 JSON 场景定义文件
- `--concurrency`：并发数，当前常用值是 `40`
- `--session-pool-size`：HTTP 会话池大小，当前常用值是 `20`
- `--token-count`：token 池大小
- `--duration-seconds`：正式测量时长
- `--warmup-seconds`：预热时长
- `--spawn-rate`：worker 拉起速度
- `--p95-ms`：P95 阈值，默认 `500`
- `--error-rate-threshold`：错误率阈值，默认 `0.05`
- `--output-json`：结果输出路径

## 4. 推荐执行场景

### 4.1 做全链路命中覆盖审计

适用文件：`tools/perf/scenarios/combined_40_scan.json`

适用目标：

- 检查全后端路由是否都被编排进场景。
- 检查大场景集调度是否存在遗漏。

注意：

- 这类执行结果不能天然视为性能验收结果。
- 执行后必须重点看成功率和状态码分布。

### 4.2 做正式性能回归

适用文件：`tools/perf/scenarios/full_89_read_40_scan.json`

适用目标：

- 验证读链路在 40 并发下是否仍满足门禁。
- 作为当前阶段最稳定的 P95 回归基线。

## 5. 命令模板

### 5.1 读链路专项 91 场景门禁模板

```bash
python tools/project_toolkit.py backend-capacity-gate \
  --base-url http://127.0.0.1:8000 \
  --scenario-config-file tools/perf/scenarios/full_89_read_40_scan.json \
  --scenarios login,authz,users,production-orders,production-stats \
  --concurrency 40 \
  --session-pool-size 20 \
  --token-count 1 \
  --duration-seconds 60 \
  --warmup-seconds 5 \
  --spawn-rate 10 \
  --p95-ms 500 \
  --error-rate-threshold 0.05 \
  --request-timeout-seconds 10 \
  --output-json .tmp_runtime/p95_loop_manual_run.json
```

说明：

- 实际执行时，`--scenarios` 通常需要与外部配置中的场景共同组成完整集合。
- 若希望复现实仓库中的 `91` 场景结果，应以历史 evidence 中实际使用的场景串或自动生成逻辑为准。

### 5.2 全链路 270 场景审计模板

```bash
python tools/project_toolkit.py backend-capacity-gate \
  --base-url http://127.0.0.1:8000 \
  --scenario-config-file tools/perf/scenarios/combined_40_scan.json \
  --scenarios <由 combined_40_scan.json 动态生成的全量场景名列表> \
  --concurrency 40 \
  --session-pool-size 20 \
  --token-count 40 \
  --duration-seconds 20 \
  --warmup-seconds 5 \
  --spawn-rate 10 \
  --p95-ms 500 \
  --error-rate-threshold 0.05 \
  --request-timeout-seconds 10 \
  --output-json .tmp_runtime/backend_40_e2e_combined_manual.json
```

说明：

- 不能手写猜测场景名，建议从 `combined_40_scan.json` 动态生成。
- 历史上已经出现过人工拼接 `--scenarios` 导致 `unsupported scenarios` 的情况。

## 6. 结果保存建议

建议统一保存到 `.tmp_runtime/`，并在文件名中包含：

- 套件名
- 日期
- 是否 retry
- 是否 smoke

示例：

- `.tmp_runtime/backend_40_e2e_combined_retry_20260411_120615.json`
- `.tmp_runtime/p95_loop_1.json`
- `.tmp_runtime/coverage_smoke_20260411.json`

## 7. 结果查看重点

### 7.1 看整体

优先看：

- `overall.success_rate`
- `overall.error_rate`
- `overall.p95_ms`
- `overall.status_counts`
- `gate_passed`

### 7.2 看逐场景

如果整体失败，再看：

- 哪些场景 `successful_requests = 0`
- 哪些场景主要返回 `401/403`
- 哪些场景主要返回 `404/405/422`

### 7.3 看调度完整性

对于大场景集，还应确认：

- 是否存在 `zero_request_count > 0`
- 是否存在大量 `zero_success_count`
- 是否出现“前缀场景反复执行，尾部场景几乎未命中”的现象

## 8. 当前已知执行经验

### 8.1 不要手动维护超长场景名串

建议永远从 JSON 动态生成，避免：

- 漏场景
- 拼错场景名
- 场景名与配置不一致

### 8.2 大场景集必须关注 worker 起始索引打散

当前 `backend_capacity_gate.py` 已引入 `_initial_worker_scenario_index()`，用于避免所有 worker 都从场景头部起跑。

这类问题在小场景集上不明显，但在 `270` 场景规模上会直接造成“表面执行了，实际没有真正覆盖尾部场景”。

### 8.3 不要把低成功率结果直接用于性能结论

当失败主要由认证、权限、样本问题主导时，结果更适合进入“问题分析”流程，而不是直接进入“性能验收”流程。
