# 后端 P95-40 并发角色-场景映射表

## 1. 文档目标

本文用于支撑 `07-整改计划.md` 的阶段 1 与阶段 2，统一“场景 -> 角色域 -> token 池”映射口径，避免后续整改时把权限问题、认证问题和业务契约问题混在一起处理。

本文优先解决以下问题：

1. 不同模块应该优先使用哪一类角色域执行场景。
2. 读链路、detail 链路、写链路是否可以共用同一角色。
3. token 池应该如何按角色域拆分，避免 `401/403` 混杂。
4. 每轮复跑时应该核对哪些角色与权限检查项。

---

## 2. 使用原则

1. 一个场景只绑定一个主角色域，不允许为了“先跑通”临时混用多个高权限账号。
2. 认证专项与业务专项分开维护；登录拿 token 的场景，不直接混入大规模业务压测。
3. 读链路、detail 链路、写链路允许使用不同角色域，不强求同一个账号覆盖整条业务链。
4. 若场景同时出现 `401` 与 `403`，优先标记为“角色域与 token 稳定性双重待排查”，禁止直接归因成纯性能问题。
5. 本表是整改执行基线，不替代真实权限清单；若现网权限模型与本表不一致，应先记录差异再修正映射。

---

## 3. 角色域定义基线

| 角色域 | 主要职责 | 典型接口范围 | 推荐 token 池口径 | 高风险误用 |
| --- | --- | --- | --- | --- |
| 平台管理员 | 平台级配置、系统初始化、跨域兜底排障 | 全局配置、初始化、少量平台管理接口 | `pool-admin`，仅用于平台级场景与排障复核 | 误用为全量业务默认账号，导致权限问题被掩盖 |
| 用户与权限管理员 | 用户、角色、权限快照与授权配置 | `users`、`roles`、`authz`、部分 `audits` | `pool-user-admin`，与业务域 token 分开 | 误代业务角色执行生产/质量/设备写链路 |
| 生产业务角色 | 生产订单、工序、工单、工艺执行主路径 | `production`、`processes`、部分 `products/craft` | `pool-production`，优先承接生产域主链路 | 误承担用户域或质量域管理接口 |
| 质量业务角色 | 检验、质检结果、质量流程 | `quality`、部分 `messages` | `pool-quality`，仅覆盖质量域业务操作 | 误用于设备或生产写链路，造成大量 `403` |
| 设备业务角色 | 设备台账、设备计划、执行记录 | `equipment`、部分 `messages` | `pool-equipment`，独立维护设备域 token | 误用于工艺或权限管理场景 |
| 只读角色 | 公共列表、统计、看板、明细查询 | 只读 `GET`、报表、统计、看板类接口 | `pool-readonly`，不承接写链路 | 被错误用于需要写权限或提交动作的场景 |

## 4. 模块级场景映射基线

> 说明：本表给出首版整改推荐口径。若模块内部权限继续细分，可在本表基础上按子域再拆。

| 模块 | 读链路推荐角色域 | detail 链路推荐角色域 | 写链路推荐角色域 | token 池建议 | 备注 |
| --- | --- | --- | --- | --- | --- |
| `auth` / `sessions` / `me` | 按被测角色分别建池 | 按被测角色分别建池 | 不并入大业务套件 | `pool-auth-*` 独立维护 | 认证专项只回答登录、会话、身份态是否稳定 |
| `users` / `roles` / `authz` | 用户与权限管理员 | 用户与权限管理员 | 用户与权限管理员 / 平台管理员 | `pool-user-admin` | 禁止用生产、质量、设备账号直接跑权限域写链路 |
| `products` | 只读角色 | 只读角色 / 生产业务角色 | 生产业务角色 | `pool-readonly` + `pool-production` | 产品读链路可先用只读角色，变更类场景转生产域 |
| `craft` | 只读角色 / 生产业务角色 | 生产业务角色 | 生产业务角色 | `pool-production` | 工艺 detail 与写链路优先落在生产域 |
| `production` | 生产业务角色 | 生产业务角色 | 生产业务角色 | `pool-production` | 生产域是阶段 1 的重点治理对象 |
| `quality` | 只读角色 / 质量业务角色 | 质量业务角色 | 质量业务角色 | `pool-quality` | 质量链路不要借用生产域 token 兜底 |
| `equipment` | 只读角色 / 设备业务角色 | 设备业务角色 | 设备业务角色 | `pool-equipment` | 设备域是阶段 1 的重点治理对象 |
| `messages` / `audits` / `ui` | 只读角色 | 只读角色 / 对应业务角色 | 对应业务角色 | 与来源业务域共池 | 不单独兜底；按消息来源业务域回收权限 |

## 5. 场景类型到角色域的快速映射

| 场景类型 | 主角色域 | 备用角色域 | 处理原则 |
| --- | --- | --- | --- |
| 登录、登出、获取当前用户 | 按被测角色分别建池 | 无 | 不与大业务套件混跑 |
| 权限快照、权限查询、授权操作 | 用户与权限管理员 | 平台管理员 | 若需平台权限，单独标注原因 |
| 公共列表、统计、看板读取 | 只读角色 | 对应业务角色 | 优先用低权限角色验证真实可见性 |
| detail 查询 | 对应业务角色 | 只读角色 | 前提是样本实体与资源域一致 |
| 写链路创建/修改/提交 | 对应业务角色 | 平台管理员 | 禁止默认用平台管理员替代业务角色 |
| 跨模块排障复核 | 平台管理员 | 无 | 只用于复核，不纳入正式性能门禁 |

---

## 6. token 池拆分建议

| token 池 | 适用角色域 | 建议用途 | 维护要求 |
| --- | --- | --- | --- |
| `pool-auth-*` | 各角色独立认证专项 | 登录、登出、会话有效性检查 | 与业务 token 池物理分离 |
| `pool-user-admin` | 用户与权限管理员 | 用户、角色、授权链路 | 避免混入生产域请求 |
| `pool-production` | 生产业务角色 | 生产、工艺、部分产品写链路 | 重点观察 `401/403` 是否同时下降 |
| `pool-quality` | 质量业务角色 | 质量链路 | 只承接质量域业务请求 |
| `pool-equipment` | 设备业务角色 | 设备链路 | 只承接设备域业务请求 |
| `pool-readonly` | 只读角色 | 公共读链路、统计、看板 | 不参与写链路压测 |
| `pool-admin` | 平台管理员 | 平台级复核与差异定位 | 不作为业务场景默认池 |

建议首轮整改按“角色域拆池 -> 模块绑定主池 -> 场景逐项确认”的顺序推进，而不是先大规模复跑再倒推账号问题。

---

## 7. 每轮复跑检查项

1. 场景是否已经绑定唯一主角色域。
2. `401` 是否主要集中在认证专项，而不是业务主套件。
3. `403` 是否仍集中在 `craft`、`equipment`、`production`、`products`、`quality` 五大重点模块。
4. 同一 token 是否被跨域、跨模块不合理复用。
5. 是否出现“为了过场景而临时扩权”的错误做法。
6. detail 与写链路是否误用了只读角色。
7. 权限排障时是否把平台管理员结果误当作正式门禁结果。

---

## 8. 交付与维护要求

- 阶段 1 完成后，应将“推荐角色域”补齐为“已确认角色/账号/权限集”。
- 阶段 2 完成后，应将各 token 池的刷新策略与生命周期规则补回对应条目。
- 若某模块新增独立角色域，应先更新本文，再更新场景与套件配置，避免文档与执行口径脱节。
