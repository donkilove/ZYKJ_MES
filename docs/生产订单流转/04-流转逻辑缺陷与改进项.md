## 2. 主链路逻辑漏洞（优先 P0/P1）

### 2.1 缺陷品对下工序放行量的影响【已澄清，文档已补强（2026-04-27）】

涉及文档：
- `01-业务流转总览.md` §5.4、§6.7、§6.8
- `02-研发附录_状态接口与补链.md` §7、§7.1、§7.2
- `03-生产订单流转思维导图.md` §1（结束生产分支与维修单完成分支）、§3（维修回流流程图）

现状口径（代码已实现，文档已对齐）：
1. `completed_quantity` 定义为良品累计量，结束生产时只加 `quantity`，不加 `defect_quantity`
2. 下工序放行依据 `min(上工序 completed_quantity, 订单数量)`，即只放良品
3. 缺陷件在结束生产时占用本次可生产量但不沉淀进 `completed_quantity`，必须通过维修单完成回流处置：
   - 可修件 → 目标工序 `visible_quantity` 增加（目标工序需 ≤ 来源工序）
   - 跨工序回流 → 来源工序 `visible_quantity` 同步压低
   - 报废且 `scrap_replenished=True` → 首工序 `visible_quantity` 增加
   - 报废且不补料 → 不再回流
4. 订单状态在维修回流路径用 `completed_quantity < visible_quantity` 判定，会把已完成订单拉回 `in_progress`

业务确认结论（2026-04-27）：
- 口径 1（`completed_quantity` 仅含良品）：业务确认，"下一工序的可生产量等于上一工序的良品量"
- 口径 3（缺陷件占用可生产量直到维修完成才放回）：业务确认符合现场预期

衍生待办：详见 §5.21（统一两个订单状态收口函数）

### 2.2 多操作员场景下"每个操作员各自首件"的口径与漏洞【已澄清，文档+代码已对齐（2026-04-27）】

涉及文档：
- `01-业务流转总览.md` §6.4
- `02-研发附录_状态接口与补链.md` §5.3、§6.1

业务确认结论（2026-04-27）：
1. 同一订单、同一工序下，**每个操作员要生产前都必须各自做一次首件检验**——首件是按子工单（操作员）维度做的，不是工序级共享
2. **不允许"首件已通过后再加进来的新操作员"**——历史订单不再补"新操作员的零量子工单"，新增操作员需要通过创建新订单 / 改单（订单回到 pending）路线纳入

修复前的逻辑漏洞（已落代码）：
1. 首件门禁要求工序状态 ∈ {pending, partial} 且子工单 pending
2. 首件通过会立即把工序置为 in_progress
3. 工序变 partial 的前置条件是"至少有一笔报工但未达订单总量"
4. A 操作员首件通过 → 工序 in_progress；在 A 报第一笔报工之前工序不会进入 partial → 同道工序的 B 操作员发起首件时，工序仍为 in_progress → 首件门禁直接拒绝
5. 首件通过路径会调用 `_activate_visible_sub_orders` 把同工序所有可见 pending 子工单批量置为 in_progress——既覆盖了"每人各做首件"的口径，又会让没做首件的人误入执行态
6. `ensure_sub_orders_visible_quantity` 会"为后来新增的操作员补 0 数量子工单"，再在分配量提升时变 pending——等价于允许首件后新增操作员

代码层修复要点（worktree `worktree-fix-first-article-per-operator`）：
1. `production_execution_service.submit_first_article`：
   - 工序门禁放开为 `{pending, in_progress, partial}`，让其他操作员也能在工序已进入 in_progress / partial 时各自发起首件
   - 首件通过时**只在工序原本是 `pending` 的情况下**才把工序推进为 `in_progress`，避免把已 partial 的工序倒回 in_progress 丢历史
   - 删除 `_activate_visible_sub_orders`：首件只把当前操作员自己的子工单置为 in_progress，不批量激活其他人
2. `production_execution_service.end_production`：工序门禁放开为 `{in_progress, partial}`——B 操作员首件通过时，工序若已经在 partial，B 也能正常报工
3. `production_order_service.ensure_sub_orders_visible_quantity`：去掉"补新操作员零量子工单"段落，只对既有子工单做可见量同步
4. `production_order_service._backfill_operator_sub_orders` → `_backfill_historical_release_sub_orders`：保留历史放行量回填，但不再借这个入口给新操作员补子工单
5. `production_order_service._build_my_order_item` 与 assist 视角的 `can_first_article / can_end_production` 三处状态集合同步对齐到上述新口径

衍生约束（业务侧请确认）：
- 同一操作员的子工单在结束生产后若分配量未做完，会回到 pending；后续再"开始首件"算同一周期复工还是要重做首件，由产品侧最终拍板。当前实现按"sub_order pending 即可再首件"放行，未做"周期内只首件一次"的强约束

### 2.3 首件不通过后没有重试上限和聚合视图【P3】

涉及文档：`01-业务流转总览.md` §6.4、§6.5

现状只描述"留痕、不放行、操作员重新发起"。缺：
1. 重试次数上限
2. 多次失败时的升级 / 通知机制
3. 是否在订单详情页聚合展示历次失败原因

### 2.4 首件扫码会话的并发场景缺失【P3】

涉及文档：`02-研发附录_状态接口与补链.md` §6.2

现状：
- 新发起会话只把同操作员同工序下的 `pending / rejected` 改成 cancelled
- `expired` 会话保持原样
- 去重粒度是 `操作员 + 订单工序`，不区分子工单

未覆盖的场景：
1. 操作员在同一工序下被拆出多个子工单时，会话怎么挂
2. expired 会话是否要在新发起时也归档
3. 刷新会话时旧 token 是否立即作废（默认应作废，但没有口径）

### 2.5 代班授权挂死与缺撤销通道【P1】

涉及文档：
- `01-业务流转总览.md` §6.9
- `02-研发附录_状态接口与补链.md` §9

现状口径：
- 创建即生效（approved）
- 仅"结束生产"会 consume
- 同一发起人 + 同一工序 + 同一目标操作员不允许重复挂

漏洞：
1. 代班人只做了首件却再没动作，授权就一直 approved 挂着 → 目标操作员无法发起新代班
2. 没有"目标操作员撤销 / 发起人主动终止"的入口
3. 改单（订单回到 pending 重建路线）时，老授权何去何从没有规定
4. 子工单被重新均分后，老授权的目标量是否仍合理也没说

### 2.6 手工完工的副作用收口【已澄清，文档+代码已对齐（2026-04-27）】

涉及文档：
- `01-业务流转总览.md` §6.11
- `02-研发附录_状态接口与补链.md` §4.4

业务确认结论（2026-04-27，1A / 2A / 3A / 4C / 5A 组合）：

1. **首件扫码会话**：把订单下所有仍 `pending` 的会话一律置为 `cancelled`，避免会话依然能被扫码后提交
2. **代班授权**：把 `approved` 但未 consumed 的授权一律置为 `consumed`（沿用既有状态，不新增 cancelled），并写 `consumed_at`
3. **并行实例**：直接复用 `_invalidate_pipeline_instances_for_order(reason="order_completed_manual")`，把仍激活的实例失活
4. **维修单**：维修单不强制收尾，也不允许跟手工完工并存——**当订单还有 `in_repair` 维修单时，手工完工会直接被拒绝（409 Conflict）**，要求先把维修单收掉。这一条是 4C 方案，避免维修回流改 `visible_quantity` 时和手工完工互相打架
5. **事件日志**：仍然只写一条主事件 `order_completed_manual`，但 payload 里把级联收口的实体清单（`cancelled_review_session_ids` / `consumed_assist_authorization_ids` / `invalidated_pipeline_instance_count`）一并写入

代码层修复要点：
1. `production_order_service.complete_order_manually`：
   - 入口前置查 `mes_repair_order` 中 `source_order_id == order.id` 且 `status == in_repair` 的维修单——若存在则抛 `RuntimeError`，由 API 层映射为 409 并把仍在维修的维修单号回带回前端
   - 主链推进逻辑（工序/子工单/订单状态置 completed）保持不变
   - 主链处置后调用新增的 `_cascade_close_order_relations(...)` 一次性完成首件会话取消、代班授权 consumed、并行实例失活
   - 事件日志 payload 新增三类实体清单字段
2. `production_order_service._cascade_close_order_relations`（新增）：
   - 用 `update().where(... in_(ids))` 一次性把会话改 `cancelled`、授权改 `consumed`（顺带写 `consumed_at`）
   - 复用既有 `_invalidate_pipeline_instances_for_order(reason="order_completed_manual")`
   - 返回三类实体的处理结果，由 `complete_order_manually` 写进事件日志

衍生约束（业务侧后续可拍）：
- 维修单本身没有"取消"状态，所以 4C 是"维修必须先走完"。如果后续想支持"订单作废 → 维修单一起作废"，要单独补维修单 cancelled 态（与 §3.7 的订单 cancelled 一起讨论更合理）
- §6.14（事件日志清单）会因为本次 payload 的扩展，增加一组待补充字段——已记入 §6.14

## 3. 状态机本身覆盖不完整

### 3.7 订单状态只有 3 个，缺关键业务态【P1】

涉及文档：`01-业务流转总览.md` §5.1

缺失的语义：
- `cancelled / void`：客户取消、临时不做
- `paused / on_hold`：设备故障、原料断供、质量冻结
- `failed / aborted`：工艺路线本身出错且无法回退

当前唯一退出动作只有"手工完工"，被滥用为"取消"会让数据语义彻底失真。

### 3.8 工序状态缺暂停 / 停线语义【P1】

涉及文档：`01-业务流转总览.md` §5.2

现场常见的"换型中、待料、待检"全被压扁到 in_progress / partial，看板和报表无法体现真实瓶颈。

### 3.9 订单一旦开工不可改过于刚性【P1】

涉及文档：
- `01-业务流转总览.md` §6.2
- `02-研发附录_状态接口与补链.md` §4.2、§4.3

现状：进入 in_progress 后路线、操作员、并行模式都没有合法修改通道。
现场常见诉求：
- 调换班组
- 追加 / 减少数量
- 换工艺步骤

当前只能"手工完工 + 重开新单"，链路被强行切断。

## 5. 查询补链与再分配

### 5.12 visible_quantity 回填只挂在查询入口【P2】

涉及文档：`02-研发附录_状态接口与补链.md` §8.2

现状：
- "我的工单 / 代理工单" 查询前会做 `_backfill_historical_release_quantity`
- "我的代班工单" 不做
- 管理端列表、报表导出、数据迁移、定时任务等完全绕过补链

风险：
- 补链是"兜底"而不是"机制"
- 长尾下数据会越积越歪

建议：
- 把补链下沉成领域事件
- "工序累计完成量变化"或"工序操作员变化"时同步触发，而不是查询触发

### 5.13 操作员变更没有再分配的合法动作【P3】

涉及文档：`02-研发附录_状态接口与补链.md` §8.1

现状：
- 新增操作员只补 0 数量子工单
- 没有"重新均分 / 主动调拨"的接口
- 离职 / 调岗操作员名下未完成的子工单没有合法搬移路径

后果：
- 新员工长期处于"零可见量子工单"的僵尸状态
- 老员工挂着无法消化的存量

### 5.21 订单状态收口函数：普通报工 vs 维修完成的隐性不一致【P2】

涉及代码：
- `production_execution_service.py::_refresh_order_status`
- `production_repair_service.py::_recompute_order_status_by_quantities`

现状：
1. 普通报工流程在结尾调 `_refresh_order_status`，判定标准是"所有工序 `status == COMPLETED`"
2. 维修单完成流程在结尾调 `_recompute_order_status_by_quantities`，判定标准是"所有工序 `completed_quantity ≥ visible_quantity`"
3. 两条路径在普通现场结果一致，但属于隐性不一致

风险：
- 任一边动一下（例如改 `process.status` 状态机或 `visible_quantity` 赋值时机）都可能让两条路径分叉
- 维修回流后再下一次普通报工触发 `_refresh_order_status` 时，可能把维修流程刚刚标过的订单状态再覆盖回来

建议：
- 收口为单一函数（推荐数量比较版："所有工序 `completed_quantity ≥ visible_quantity` 视为完成"），普通报工流程也调用
- 增加单测断言：报工 → 维修完成 → 报工 在三阶段订单状态都正确

来源：§2.1 衍生

## 6. 可追溯与审计

### 6.14 订单事件日志没有完整事件清单【P2】

涉及文档：全部三篇

现状：文档反复出现"写订单事件日志 / 写事件留痕"，但没有一处枚举完整事件类型。

### 6.15 缺批次 / 序列号粒度【P3】

涉及文档：`01-业务流转总览.md` §4、`02-研发附录_状态接口与补链.md` §7

行业现实：
