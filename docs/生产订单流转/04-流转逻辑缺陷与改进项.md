# 生产订单流转逻辑缺陷与改进项

## 0. 文档定位

- 面向对象：业务、研发、测试共同评审
- 文档目标：基于当前 `01-业务流转总览.md`、`02-研发附录_状态接口与补链.md`、`03-生产订单流转思维导图.md` 三份口径文档，列出**流转逻辑层面**仍然不闭环、不完善或存在歧义的地方
- 口径范围：以仓库 2026-04-26 当前实现为基准
- 处理方式：后续按"一项一议"的节奏逐条澄清、修文档或落代码

## 2. 主链路逻辑漏洞（优先 P0/P1）

### 2.1 缺陷品对下工序放行量的影响【已澄清，文档已补强（2026-04-27）】

涉及文档：
- `01-业务流转总览.md` §5.4、§6.7、§6.8
- `02-研发附录_状态接口与补链.md` §7、§7.1、§7.2
- `03-生产订单流转思维导图.md` §1（结束生产分支与维修单完成分支）、§3（维修回流流程图）

现状口径（代码已实现，文档已对齐）：
1. `completed_quantity` 定义为良品累计量，结束生产时只加 `quantity`，不加 `defect_quantity`
2. 下工序放行依据 `min(上工序 completed_quantity, 订单数量)`，即只放良品
3. 缺陷件在结束生产时占用本次可生产量但不沉淀进 `completed_quantity`，必须通过维修单完成回流处置：
   - 可修件 → 目标工序 `visible_quantity` 增加（目标工序需 ≤ 来源工序）
   - 跨工序回流 → 来源工序 `visible_quantity` 同步压低
   - 报废且 `scrap_replenished=True` → 首工序 `visible_quantity` 增加
   - 报废且不补料 → 不再回流
4. 订单状态在维修回流路径用 `completed_quantity < visible_quantity` 判定，会把已完成订单拉回 `in_progress`

业务确认结论（2026-04-27）：
- 口径 1（`completed_quantity` 仅含良品）：业务确认，"下一工序的可生产量等于上一工序的良品量"
- 口径 3（缺陷件占用可生产量直到维修完成才放回）：业务确认符合现场预期

衍生待办：详见 §5.21（统一两个订单状态收口函数）

### 2.2 多操作员场景下"第二人开始首件"会被工序状态卡死【P0】

涉及文档：
- `01-业务流转总览.md` §6.4
- `02-研发附录_状态接口与补链.md` §5.3、§6.1

现状口径：
- 首件门禁要求工序状态 ∈ {pending, partial} 且 子工单 pending
- 首件通过后工序立即变 in_progress
- 工序变 partial 的前置条件是"至少有一笔报工但未达订单总量"

逻辑漏洞：
1. A 操作员首件通过 → 工序 in_progress
2. 在 A 报第一笔报工之前，工序不会进入 partial
3. 同道工序的 B 操作员发起首件时，工序仍为 in_progress → **首件门禁直接拒绝**
4. 当前模型里多人并发首件不可能同时通过

建议处理方向：
- 把首件门禁改成"按子工单维度判断"
- 工序状态不再作为首件硬约束
- 工序状态 in_progress 不应阻止其他子工单做自己的首件

### 2.3 首件不通过后没有重试上限和聚合视图【P3】

涉及文档：`01-业务流转总览.md` §6.4、§6.5

现状只描述"留痕、不放行、操作员重新发起"。缺：
1. 重试次数上限
2. 多次失败时的升级 / 通知机制
3. 是否在订单详情页聚合展示历次失败原因

### 2.4 首件扫码会话的并发场景缺失【P3】

涉及文档：`02-研发附录_状态接口与补链.md` §6.2

现状：
- 新发起会话只把同操作员同工序下的 `pending / rejected` 改成 cancelled
- `expired` 会话保持原样
- 去重粒度是 `操作员 + 订单工序`，不区分子工单

未覆盖的场景：
1. 操作员在同一工序下被拆出多个子工单时，会话怎么挂
2. expired 会话是否要在新发起时也归档
3. 刷新会话时旧 token 是否立即作废（默认应作废，但没有口径）

### 2.5 代班授权挂死与缺撤销通道【P1】

涉及文档：
- `01-业务流转总览.md` §6.9
- `02-研发附录_状态接口与补链.md` §9

现状口径：
- 创建即生效（approved）
- 仅"结束生产"会 consume
- 同一发起人 + 同一工序 + 同一目标操作员不允许重复挂

漏洞：
1. 代班人只做了首件却再没动作，授权就一直 approved 挂着 → 目标操作员无法发起新代班
2. 没有"目标操作员撤销 / 发起人主动终止"的入口
3. 改单（订单回到 pending 重建路线）时，老授权何去何从没有规定
4. 子工单被重新均分后，老授权的目标量是否仍合理也没说

### 2.6 手工完工的副作用没有收口【P0】

涉及文档：
- `01-业务流转总览.md` §6.11
- `02-研发附录_状态接口与补链.md` §4.4

现状只描述"工序、子工单、订单都置 completed"。但同一时刻可能并存：
- 有效期内的首件扫码会话（pending / approved 但还没回写正式记录）
- approved 但未 consumed 的代班授权
- 已激活的并行实例
- 该订单已挂出的维修单

未规定的级联动作：
1. 首件会话是否一律 cancelled
2. 代班授权是否一律置为 consumed 或 cancelled
3. 并行实例是否失效
4. 维修单是否随之收尾或解除关联
5. 是否补一条统一的"订单手工完工 → 关联实体收口"事件

## 3. 状态机本身覆盖不完整

### 3.7 订单状态只有 3 个，缺关键业务态【P1】

涉及文档：`01-业务流转总览.md` §5.1

缺失的语义：
- `cancelled / void`：客户取消、临时不做
- `paused / on_hold`：设备故障、原料断供、质量冻结
- `failed / aborted`：工艺路线本身出错且无法回退

当前唯一退出动作只有"手工完工"，被滥用为"取消"会让数据语义彻底失真。

### 3.8 工序状态缺暂停 / 停线语义【P1】

涉及文档：`01-业务流转总览.md` §5.2

现场常见的"换型中、待料、待检"全被压扁到 in_progress / partial，看板和报表无法体现真实瓶颈。

### 3.9 订单一旦开工不可改过于刚性【P1】

涉及文档：
- `01-业务流转总览.md` §6.2
- `02-研发附录_状态接口与补链.md` §4.2、§4.3

现状：进入 in_progress 后路线、操作员、并行模式都没有合法修改通道。
现场常见诉求：
- 调换班组
- 追加 / 减少数量
- 换工艺步骤

当前只能"手工完工 + 重开新单"，链路被强行切断。

## 4. 并行模式悬空场景

### 4.10 并行门禁的具体规则没有写【P1】

涉及文档：
- `01-业务流转总览.md` §6.10
- `02-研发附录_状态接口与补链.md` §10

文档反复说"必须满足并行开始 / 结束门禁"，但门禁的判定规则从来没明示：
1. 是否允许下游并行实例先于上游开始首件
2. 同一并行序号跨工序的实例是否必须同步推进
3. 各实例之间的数量关系

业务与研发对"到底允许多并行"目前没有共识。

### 4.11 并行模式中途变更没有过渡口径【P1】

涉及文档：
- `01-业务流转总览.md` §6.10
- `02-研发附录_状态接口与补链.md` §10

未规定的内容：
1. 已 in_progress 的实例失效后，已报工数量怎么并入主链
2. 失效再启用时，已写入历史记录的 pipeline_instance_id 是否需要回写或保留
3. 关闭并行模式时，未完成的子工单可见性如何重算

## 5. 查询补链与再分配

### 5.12 visible_quantity 回填只挂在查询入口【P2】

涉及文档：`02-研发附录_状态接口与补链.md` §8.2

现状：
- "我的工单 / 代理工单" 查询前会做 `_backfill_historical_release_quantity`
- "我的代班工单" 不做
- 管理端列表、报表导出、数据迁移、定时任务等完全绕过补链

风险：
- 补链是"兜底"而不是"机制"
- 长尾下数据会越积越歪

建议：
- 把补链下沉成领域事件
- "工序累计完成量变化"或"工序操作员变化"时同步触发，而不是查询触发

### 5.13 操作员变更没有再分配的合法动作【P3】

涉及文档：`02-研发附录_状态接口与补链.md` §8.1

现状：
- 新增操作员只补 0 数量子工单
- 没有"重新均分 / 主动调拨"的接口
- 离职 / 调岗操作员名下未完成的子工单没有合法搬移路径

后果：
- 新员工长期处于"零可见量子工单"的僵尸状态
- 老员工挂着无法消化的存量

### 5.21 订单状态收口函数：普通报工 vs 维修完成的隐性不一致【P2】

涉及代码：
- `production_execution_service.py::_refresh_order_status`
- `production_repair_service.py::_recompute_order_status_by_quantities`

现状：
1. 普通报工流程在结尾调 `_refresh_order_status`，判定标准是"所有工序 `status == COMPLETED`"
2. 维修单完成流程在结尾调 `_recompute_order_status_by_quantities`，判定标准是"所有工序 `completed_quantity ≥ visible_quantity`"
3. 两条路径在普通现场结果一致，但属于隐性不一致

风险：
- 任一边动一下（例如改 `process.status` 状态机或 `visible_quantity` 赋值时机）都可能让两条路径分叉
- 维修回流后再下一次普通报工触发 `_refresh_order_status` 时，可能把维修流程刚刚标过的订单状态再覆盖回来

建议：
- 收口为单一函数（推荐数量比较版："所有工序 `completed_quantity ≥ visible_quantity` 视为完成"），普通报工流程也调用
- 增加单测断言：报工 → 维修完成 → 报工 在三阶段订单状态都正确

来源：§2.1 衍生

## 6. 可追溯与审计

### 6.14 订单事件日志没有完整事件清单【P2】

涉及文档：全部三篇

现状：文档反复出现"写订单事件日志 / 写事件留痕"，但没有一处枚举完整事件类型。

至少应补充对照表：
- 建单 / 改单 / 删单 / 手工完工
- 首件通过 / 首件不通过 / 首件扫码会话状态
- 报工完成 / 缺陷自动建维修单
- 下工序放行（含补链触发）
- 代班授权创建 / 消耗 / 撤销
- 并行模式变更 / 实例失效 / 实例激活

否则审计排障没有抓手。

### 6.15 缺批次 / 序列号粒度【P3】

涉及文档：`01-业务流转总览.md` §4、`02-研发附录_状态接口与补链.md` §7

现状：`ProductionRecord` 只到数量级别。

行业现实：
- 化工 / 食品 / 医疗器械 / 关键件加工要求批号或序列号级追溯
- 当前模型缺乏对批次 / SN 的扩展点

## 7. 其它实操层面的不完善

### 7.16 工艺模板与运行中订单的解耦关系没写【P2】

涉及文档：`02-研发附录_状态接口与补链.md` §4.1

未明示：模板被改了，已经 in_progress 的订单是否要同步？默认应该是"不同步"，但需要在文档里说出来。

### 7.17 数量量纲与精度未声明【P3】

涉及文档：全部三篇

现状只用"数量"措辞，没说：
- 量纲（件 / 公斤 / 米）
- 是否允许小数
- 比较与累加时的精度策略

这会反向影响 `min`、累加、可生产量等核心计算的语义。

### 7.18 权限网关偏弱【P2】

涉及文档：
- `01-业务流转总览.md` §6.11
- `02-研发附录_状态接口与补链.md` §4.4

现状：
- 手工完工只校验"当前登录密码"
- 没有"角色 / 审批 / 双人复核"维度
- 删除待生产订单也没有二次确认

### 6.19 维修闭环被踢出范围，但报工又会自动建维修单【P2】

涉及文档：
- `01-业务流转总览.md` §2、§6.8
- `02-研发附录_状态接口与补链.md` §7.2

矛盾点：
- §2 把"维修单后续维修闭环"排除在文档之外
- §6.8 又把"自动建维修单"放在主链路硬动作里

读者拿到这份文档无法看完整产线。建议至少补一节"维修单与生产订单的回写边界"。

### 7.20 思维导图与正文小不一致【P3】

涉及文档：
- `03-生产订单流转思维导图.md` §1（并行模式约束）
- `01-业务流转总览.md` §6.10

不一致点：
- 思维导图：至少两个有效工序编码
- 正文：工序总数 ≥ 2 且被选并行工序 ≥ 2

需要统一表达，避免业务侧按思维导图看口径变松。
