## 3. 状态模型

| 工序 | `PROCESS_STATUS_PENDING` | `pending` |
| 工序 | `PROCESS_STATUS_IN_PROGRESS` | `in_progress` |
| 工序 | `PROCESS_STATUS_PARTIAL` | `partial` |
| 工序 | `PROCESS_STATUS_COMPLETED` | `completed` |
| 子工单 | `SUB_ORDER_STATUS_PENDING` | `pending` |
| 子工单 | `SUB_ORDER_STATUS_IN_PROGRESS` | `in_progress` |
| 子工单 | `SUB_ORDER_STATUS_DONE` | `done` |

### 3.2 前端状态映射

前端在 `frontend/lib/features/production/models/production_models.dart` 中维护对应的中文标签：

1. 订单：
   - `pending -> 待生产`
   - `in_progress -> 生产中`
   - `completed -> 生产完成`
2. 工序：
   - `pending -> 待生产`
   - `in_progress -> 进行中`
   - `partial -> 部分完成`
   - `completed -> 生产完成`
3. 子工单：
   - `pending -> 待执行`
   - `in_progress -> 执行中`
   - `done -> 已完成`

## 4. 建单、改单、删单、手工完工

### 4.1 建单

主入口：

- API：`POST /api/v1/production/orders`
- 服务：`production_order_service.py::create_order`

核心规则：

1. 产品必须存在且处于 `active`
2. 供应商必须可用
3. 工艺路线可以来自：
   - 模板
   - 手工 `process_steps`
   - 简化 `process_codes`
4. 订单创建时：
   - `status = pending`
   - `current_process_code = 首工序`
   - `pipeline_enabled = false`
5. `_build_order_process_rows()` 会创建工序行：
   - 第一工序 `visible_quantity = order.quantity`
   - 其余工序 `visible_quantity = 0`
6. `ensure_sub_orders_visible_quantity()` 会按工序操作员拆分初始子工单

### 4.2 改单

主入口：

- API：`PUT /api/v1/production/orders/{order_id}`
- 服务：`production_order_service.py::update_order`

核心规则：

1. 只有 `pending` 订单允许修改
2. 修改时会重建整条工序路线
3. 并行模式配置会被清空为关闭
4. 子工单会按新路线重新生成/同步

### 4.3 删单

主入口：

- API：`DELETE /api/v1/production/orders/{order_id}`
- 服务：`production_order_service.py::delete_order`

核心规则：

1. 只有 `pending` 订单允许删除
2. 删除前会先写事件日志，再执行删除

### 4.4 手工完工

主入口：

- API：`POST /api/v1/production/orders/{order_id}/complete`
- 服务：`production_order_service.py::complete_order_manually`

核心规则：

1. API 层先校验当前登录密码
2. 服务层会把每道工序：
   - `visible_quantity` 至少补到订单总量
   - `completed_quantity` 至少补到订单总量
   - `status = completed`
3. 每个子工单会被强制改成：
   - `assigned_quantity >= completed_quantity`
   - `completed_quantity >= assigned_quantity`
   - `status = done`
   - `is_visible = false`
4. 最终订单：
   - `status = completed`
   - `current_process_code = None`

## 5. 我的工单查询模型

### 5.1 三种视角

主入口：

- API：`GET /api/v1/production/my-orders`
- 服务：`production_order_service.py::list_my_orders`

实际由 `_collect_my_order_items()` 聚合，支持：

1. `own`
   - 当前操作员自己的工单
2. `proxy`
   - 代理目标操作员视角
3. `assist`
   - 当前用户作为代班人的已生效代班工单

### 5.2 查询项的真实来源

返回项不是直接从订单表拿，而是由 `_build_my_order_item()` 计算出来，核心字段包括：

1. `visible_quantity`
2. `user_assigned_quantity`
3. `user_completed_quantity`
4. `max_producible_quantity`
5. `can_first_article`
6. `can_end_production`
7. `can_apply_assist`
8. `can_create_manual_repair`
9. `pipeline_instance_id`
10. `pipeline_instance_no`

### 5.3 动作门禁计算

`_build_my_order_item()` 的动作判断非常关键：

#### 开始首件

需要同时满足：

1. 当前是操作员上下文
2. 子工单可见
3. 工序状态在 `{pending, in_progress, partial}`（每个操作员都可在工序已开工时各自做首件）
4. 子工单状态为 `pending`
5. `max_producible_quantity > 0`
6. 如果工序被选入并行模式，则必须拿到有效并行实例
7. 并行开始门禁允许

#### 结束生产

需要同时满足：

1. 当前是操作员上下文
2. 子工单可见
3. 工序状态在 `{in_progress, partial}`（同工序里别人已经在 partial 不会卡住自己报工）
4. 子工单状态为 `in_progress`
5. `max_producible_quantity > 0`
6. 如果工序在并行模式内，则必须通过并行结束门禁

#### 发起代班

需要同时满足：

1. 当前子工单可见
2. 当前没有挂着的代班授权
3. 子工单状态在 `pending / in_progress`
4. 满足以下任一：
   - 还有可生产数量
   - 当前子工单已在 `in_progress`
   - 并行模式下处于可开始首件的待启动状态

#### 发起送修

需要同时满足：

1. 子工单状态在 `pending / in_progress`
2. `max_producible_quantity > 0`

## 6. 首件主链路

### 6.1 传统首件提交

主入口：

- API：`POST /api/v1/production/orders/{order_id}/first-article`
- 服务：`production_execution_service.py::submit_first_article`

口径要点（2026-04-27 起）：

- 首件按子工单（操作员）维度提交，**同一订单同一工序下每个操作员都要各自做一次**
- 工序门禁放开为 `{pending, in_progress, partial}`：A 通过后工序进入 in_progress，不阻挡 B 在同工序上发起自己的首件
- 首件通过**只**改当前操作员的子工单，不再批量激活其他操作员（旧的 `_activate_visible_sub_orders` 已删除）

核心流程：

1. 锁订单和当前工序
2. 校验订单未完成、工序状态 ∈ `{pending, in_progress, partial}`
3. 解析目标操作员与代班授权
4. 锁定当前子工单（必须 pending）
5. 计算可生产数量：
   - 工序剩余量 = `visible_quantity - completed_quantity`
   - 子工单剩余量 = `assigned_quantity - completed_quantity`
   - 实际可生产量取二者最小值
6. 校验开始门禁（含并行）
7. 校验首件验证码
8. 归一化模板、内容、测试值、参与人
9. 如果首件结果是 `passed`：
   - **仅当工序原本是 `pending` 时**才将工序推进为 `in_progress`（已 in_progress / partial 保持原状）
   - 当前操作员的子工单改为 `in_progress`、`is_visible=True`
   - 订单改为 `in_progress`
10. 生成 `FirstArticleRecord`
11. 如果通过，再补一条 `ProductionRecord(record_type=first_article)`
12. 写订单事件日志

### 6.2 首件扫码复核

主入口：

1. 操作员创建会话：
   - `POST /api/v1/production/orders/{order_id}/first-article/review-sessions`
2. 操作员轮询/刷新：
   - `GET /.../status`
   - `POST /.../refresh`
3. 手机页读取详情：
   - `GET /api/v1/production/first-article/review-sessions/detail`
4. 质检员提交结果：
   - `POST /api/v1/production/first-article/review-sessions/submit`

会话服务在 `first_article_review_service.py`，核心状态包括：

1. `pending`
2. `approved`
3. `rejected`
4. `expired`
5. `cancelled`

关键规则：

1. 会话默认 5 分钟失效
2. 新发起会把同一操作员、同一订单工序下旧的 `pending/rejected` 会话改成 `cancelled`
3. 刷新会话会重发 token，并把状态重置成 `pending`
4. 质检员提交 `failed`：
   - 会话变 `rejected`
   - 不生成正式首件记录
5. 质检员提交 `passed`：
   - 内部调用 `submit_first_article(..., result="passed", skip_verification_code=True)`
   - 会话再改成 `approved`
   - 回填正式首件记录 ID

## 7. 报工与下工序放行

主入口：

- API：`POST /api/v1/production/orders/{order_id}/end-production`
- 服务：`production_execution_service.py::end_production`

核心计算顺序：

1. 锁订单、工序、子工单
2. 校验：
   - 订单未完成
   - 工序必须 ∈ `{in_progress, partial}`（同工序里别人已经在 partial 不阻挡当前操作员报工）
   - 子工单必须 `in_progress`
   - 报工数量 > 0
3. 计算：
   - `process_remaining = visible_quantity - completed_quantity`
   - `sub_remaining = assigned_quantity - completed_quantity`
   - `max_producible = min(process_remaining, sub_remaining)`
4. 如果带缺陷项：
   - `defect_quantity = sum(defect_items.quantity)`
   - `quantity + defect_quantity <= max_producible`
5. 更新累计量（**只加良品量 `quantity`，缺陷量不进 `completed_quantity`**）：
   - 工序 `completed_quantity += quantity`
   - 子工单 `completed_quantity += quantity`
6. 刷工序状态：
   - 达到订单总量 -> `completed`
   - 否则 -> `partial`
7. 刷子工单状态：
   - 达到分配量 -> `done`
   - 否则 -> 回到 `pending`

### 7.1 下工序放行规则

如果存在下一工序：

1. 计算 `released_visible_quantity = min(当前工序completed_quantity, 订单数量)`
2. 顺序工序和并行边界当前都采用"已累计良品量可放行"的思路（`completed_quantity` 严格只含良品）
3. 若 `target_visible > next_process.visible_quantity`，则提升下一工序 `visible_quantity`
4. 再调用 `ensure_sub_orders_visible_quantity()` 同步下一工序子工单

这也是查询侧历史补链的依据。

> 备注：因为 `completed_quantity` 严格等于良品累计量，下工序天然只能拿到上工序的良品量。缺陷件靠 §7.2 维修单完成回流路径处理。

### 7.2 缺陷联动与维修单完成回流

#### 7.2.1 报工建维修单

服务入口：`production_repair_service.py::create_repair_order`（`auto_created=True`）

当 `defect_items` 非空且缺陷量 > 0 时：

1. 系统基于当前报工记录自动创建维修单
2. `production_quantity = quantity + defect_quantity`，`repair_quantity = 缺陷件总数`
3. 缺陷现象逐条落入 `RepairDefectPhenomenon`，并回填本次 `ProductionRecord.id`
4. 维修单 ID、维修单号回写订单事件日志 payload
5. 维修单初始状态为 `in_repair`

注意：缺陷量**不进** `completed_quantity`，但**已占用本次报工的可生产量**（来源是 `max_producible` 上限校验）。直到维修单完成，这部分量的最终去向才被确定。

#### 7.2.2 维修单完成回流

服务入口：`production_repair_service.py::complete_repair_order`

业务侧三条输入：

1. `cause_items`：每条带 `is_scrap`，决定走维修还是报废
2. `return_allocations`：每条返工件流向哪个目标工序（仅当存在可修件时）
3. `scrap_replenished`：报废后是否从首工序补料

代码侧关键动作：

1. **校验回流目标合法性**
   - `target_rank ≤ source_rank`：返工目标必须是来源工序本身或更前工序，不允许跳到更后工序
2. **可修件回流**
   - `target_process.visible_quantity += qty`
   - 若 `target_process.status == COMPLETED` 且新的 `visible > completed`，回退为 `partial`（若 `completed_quantity > 0`）或 `pending`
   - 调用 `ensure_sub_orders_visible_quantity()` 同步目标工序子工单
3. **跨工序回流时来源工序补偿**
   - 当 `target_process.id != source_process.id`：
     - `source_process.visible_quantity = max(source.completed_quantity, source.visible_quantity - qty)`
     - 状态同样按需回退
4. **报废补料**
   - `scrap_replenished=True` 且 `scrap_quantity > 0` 时，首工序 `visible_quantity += scrap_quantity`
   - 状态按需回退、子工单同步
5. **重算订单状态**
   - 维修完成路径调用 `_recompute_order_status_by_quantities`
   - 判定标准：任意工序 `completed_quantity < visible_quantity` 即为未完成
   - 若被回流抬高 `visible_quantity` 拉回 `in_progress`，订单状态会从 `completed` 退回

#### 7.2.3 两个订单状态收口函数并存

| 函数 | 调用时机 | 判定标准 |
| --- | --- | --- |
| `production_execution_service._refresh_order_status` | 普通报工流程末尾 | 所有工序 `status == COMPLETED` |
| `production_repair_service._recompute_order_status_by_quantities` | 维修单完成流程末尾 | 所有工序 `completed_quantity ≥ visible_quantity` |

两条路径在普通流程下结果一致，但属于隐性不一致——任一边动一下都可能让两条路径分叉。建议后续统一为单一收口函数，详见 `04-流转逻辑缺陷与改进项.md` §5.21。

## 8. 子工单同步与历史补链

这部分是当前项目理解生产订单流转的关键。

### 8.1 子工单可见量同步

核心函数：

- `production_order_service.py::ensure_sub_orders_visible_quantity`

职责：

1. 若某工序还没有子工单，按操作员列表初始化子工单（**仅建单 / 改单等订单 pending 阶段会走到这一步**）
2. 若目标可见量大于当前分配总量，则把差额均匀分给现有子工单（不会新增操作员行）
3. 再根据 `assigned_quantity` 与 `completed_quantity` 刷新：
   - `status`
   - `is_visible`

口径变更（2026-04-27）：**已删除"为后来新增的操作员补 0 数量子工单"这一段**。原因：业务侧明确"首件已通过后才加进来的新操作员"是不允许的——历史订单不再借这个入口纳入新操作员，新增请走改单或新建订单路线。

### 8.2 历史放行量回填

核心函数：

- `production_order_service.py::_backfill_historical_release_quantity`
- `production_order_service.py::_backfill_historical_release_sub_orders`（原名 `_backfill_operator_sub_orders`，2026-04-27 重命名以贴近职责）

职责：

1. 找到上一道工序的 `completed_quantity`
2. 计算当前工序理论应有的 `visible_quantity`
3. 如果理论值大于当前值，则补齐
4. 再同步当前工序**已存在**子工单的可见性（不会为缺失的操作员补行）

触发点：

1. `我的工单` 查询前
2. `代理工单` 查询前

注意：

- `我的代班工单` 视角不直接做这层回填，它依赖目标操作员子工单已经可见

## 9. 代班即时生效模型

主入口：

- API：`POST /api/v1/production/orders/{order_id}/assist-authorizations`
- 服务：`assist_authorization_service.py::create_assist_authorization`

关键规则：

1. 订单未完成
2. 当前工序未完成
3. 目标操作员和代班人都必须是有效用户
4. 目标操作员子工单必须存在
5. 同一发起人、同一订单工序、同一目标操作员，不允许再挂一条 `pending/approved` 且尚未结束生产消耗的重复授权
6. 创建后状态直接写成 `approved`

消耗规则：

1. 首件使用时只记 `first_article_used_at`
2. 结束生产使用时：
   - 写 `end_production_used_at`
   - 状态改为 `consumed`
   - 写 `consumed_at`

## 10. 并行模式

主入口：

1. 查看：
   - `GET /api/v1/production/orders/{order_id}/pipeline_mode`
2. 更新：
   - `PUT /api/v1/production/orders/{order_id}/pipeline_mode`

服务入口：

- `production_order_service.py::update_order_pipeline_mode`

关键约束：

1. 只有具备权限的人能修改
2. 已完成订单不能改
3. 开启时，订单工序数必须 >= 2
4. 开启时，选中工序数必须 >= 2
5. 选中工序必须全部属于当前订单路线
6. 首个选中工序必须仍有可生产数量

状态变化：

1. 更新时会先失效旧并行实例
2. 若本次为开启，则再生成新的并行实例
3. 同时写：
   - `pipeline_mode_updated`
   - `pipeline_instances_invalidated`
   - `pipeline_instances_activated`

## 12. 当前维护时最应该盯住的点

1. 改报工逻辑时，不要只看 `completed_quantity`，一定同步检查：
   - `visible_quantity`
   - 子工单 `assigned_quantity`
   - 子工单 `is_visible`
2. 改查询逻辑时，不要绕开 `_backfill_historical_release_sub_orders()`，否则老订单可见性容易退化
3. 改代班逻辑时，先确认是否仍维持“发起即生效”口径
4. 改首件扫码复核时，要同时核对：
