# 生产订单业务流转总览

## 1. 文档定位

- 面向对象：业务、实施、测试、研发共同阅读。
- 文档目标：用业务语言说明“生产订单从创建到收尾”在当前项目中的真实流转逻辑。
- 当前口径基于仓库 2026-04-26 版本代码与历史留痕，不以外部参考项目为准。

## 2. 适用范围

本说明覆盖以下主链路：

1. 订单创建、编辑、删除、手工完工
2. 我的工单 / 代理工单 / 我的代班工单查询
3. 首件发起、首件扫码复核、首件通过/不通过
4. 结束生产（报工）
5. 下工序放行、子工单可见性同步
6. 代班即时生效
7. 并行模式与并行实例
8. 报工带缺陷时自动进入维修链路

本说明不展开以下主题的完整业务细节：

1. 维修单后续维修闭环
2. 报废统计页与质量趋势页
3. 消息中心的全部消息同步策略

## 3. 核心角色

| 角色 | 主要职责 |
| --- | --- |
| 订单管理人员 | 创建、编辑、删除待生产订单；维护并行模式；必要时手工完工 |
| 操作员 | 在“我的工单”里执行首件、结束生产、发起代班、触发送修 |
| 代理操作员 | 在“代理工单”视角代看指定操作员可执行工单 |
| 代班人 | 在“我的代班工单”视角代替目标操作员执行首件或结束生产 |
| 质检员 / 班组长 | 通过扫码复核页判定首件合格/不合格 |

## 4. 核心对象

| 对象 | 业务含义 |
| --- | --- |
| 生产订单 | 整张单据，记录产品、供应商、数量、工艺路线、当前工序等 |
| 订单工序 | 订单下每一道工序，独立记录状态、可见数量、累计完成数量 |
| 子工单 | 按操作员分配的执行份额，决定“谁当前能看到并操作这道工序” |
| 并行实例 | 并行模式下，同一并行序号跨工序的执行链路实例 |
| 代班授权 | 允许代班人替目标操作员执行首件/结束生产的即时授权 |
| 首件复核会话 | 操作员发起二维码后，供质检员扫码判定的临时会话 |
| 生产记录 | 结束生产后的正式报工记录；首件通过时也会留一条首件类型记录 |
| 订单事件日志 | 订单创建、首件、报工、并行模式变更、代班等过程留痕 |

## 5. 状态口径

### 5.1 订单状态

| 状态值 | 中文口径 | 说明 |
| --- | --- | --- |
| `pending` | 待生产 | 订单已建好，但尚未进入正式生产中 |
| `in_progress` | 生产中 | 订单至少已有一道工序进入执行中 |
| `completed` | 生产完成 | 订单已整体完成，或被手工标记完工 |

### 5.2 工序状态

| 状态值 | 中文口径 | 说明 |
| --- | --- | --- |
| `pending` | 待生产 | 工序尚未正式开始，但可能已经被放行出可生产数量 |
| `in_progress` | 进行中 | 工序已通过首件，正在执行 |
| `partial` | 部分完成 | 工序已有部分报工完成，但未做到订单总量 |
| `completed` | 生产完成 | 工序累计完成量已达到订单总量 |

### 5.3 子工单状态

| 状态值 | 中文口径 | 说明 |
| --- | --- | --- |
| `pending` | 待执行 | 该操作员已分到可执行数量，尚未开始或尚未收完 |
| `in_progress` | 执行中 | 该操作员名下子工单已通过首件并进入执行 |
| `done` | 已完成 | 该操作员名下分配量已全部完成，或当前无剩余可执行量 |

### 5.4 数量字段口径

| 字段 | 业务含义 |
| --- | --- |
| 订单 `quantity` | 整张订单的目标数量 |
| 工序 `visible_quantity` | 当前工序累计可见 / 可生产数量；上游放行 + 维修回流共同推升它，单调上升 |
| 工序 `completed_quantity` | 当前工序累计良品数量；只在结束生产时按本次报工良品量 `quantity` 累加，缺陷量不计入 |
| 子工单 `assigned_quantity` | 该操作员名下当前已分到的可生产数量 |
| 子工单 `completed_quantity` | 该操作员名下累计良品数量；累加规则与工序一致 |

关键口径：

1. `completed_quantity` 始终代表"良品累计量"。
2. 缺陷量在结束生产时只参与"`quantity + defect_quantity ≤ 当前可生产量`"上限校验，并自动建维修单，不沉淀进 `completed_quantity`。
3. 下工序放行依据是上一工序的 `completed_quantity`（即良品），缺陷件不会顺流到下工序。

## 6. 主流程

### 6.1 创建订单

1. 管理端创建订单时，必须满足：
   - 产品存在且处于可生产口径
   - 供应商可用
   - 工艺路线有效，来源可以是模板或手工工序步骤
   - 数量大于 0
2. 创建后系统会自动生成订单工序。
3. 第一工序初始 `visible_quantity` 直接等于订单数量，后续工序初始为 `0`。
4. 系统会按该工序下可执行操作员，自动拆分子工单数量与可见性。
5. 新单默认状态为：
   - 订单：`pending`
   - 第一工序：`pending`
   - 第一工序子工单：通常为 `pending`

### 6.2 待生产阶段

1. 订单处于 `pending` 时，可以在管理端继续编辑或删除。
2. 编辑只允许待生产订单，且会重建整条工艺路线和子工单分配。
3. 删除也只允许待生产订单。
4. 此阶段订单的 `current_process_code` 指向第一道未完成工序。

### 6.3 工单查询与可见性

系统当前提供 3 种执行视角：

1. `我的工单`
   - 当前登录操作员自己的可见子工单
2. `代理工单`
   - 有代理权限时，代看目标操作员的可见子工单
3. `我的代班工单`
   - 当前用户作为代班人的已生效代班任务

业务上最重要的规则是：

1. 不是所有工序一建单就能执行。
2. 一道工序能否被看到，取决于：
   - 工序 `visible_quantity`
   - 子工单 `assigned_quantity`
   - 子工单 `is_visible`
3. 查询入口会自动补一层历史修复：
   - 如果老订单本应已放行下工序，但 `visible_quantity` 没同步出来，查询时会补齐
   - 如果后来新增了某个工序操作员，系统也会把缺失的子工单补出来

### 6.4 开始首件

1. 开始首件的本质，是让当前操作员的子工单从"待生产"进入"正式开工"。
2. 关键口径：**同一订单、同一工序下，每个操作员要生产前都必须各自做一次首件检验**——首件按子工单（操作员）维度做，不是工序级共享。
3. 当前项目里，开始首件不是无条件可点，必须同时满足：
   - 订单未完成
   - 当前工序状态为 `pending` / `in_progress` / `partial`（工序已开工不阻挡其他操作员各自做首件）
   - 当前子工单状态为 `pending`
   - 当前仍有可生产数量
   - 若该工序处于并行模式内，还必须满足并行开始门禁
4. 首件有两条结果分支：

#### A. 首件通过

1. 仅当工序原本是 `pending` 时，才把工序推进为 `in_progress`；若工序已 `in_progress` 或 `partial`，**保持原状不回退**
2. 当前操作员的子工单状态改为 `in_progress`（**不会**联动其他操作员的子工单）
3. 订单状态改为 `in_progress`
4. 生成正式首件记录
5. 写入"首件通过"事件日志

#### B. 首件不通过

1. 系统仍会保留首件记录
2. 但不会把工序、子工单、订单推进到执行中
3. 也就是说，不通过只留痕，不放行正式生产

#### C. 多操作员同工序的协同口径

1. A 完成首件 → 工序 in_progress，A 的子工单 in_progress；B 的子工单仍为 pending
2. B 后续要生产时，需要自己再发起一次首件；工序处于 in_progress / partial 不阻挡
3. **不允许首件已通过后再加进来的新操作员**——历史订单不再为后来分配的操作员补 0 数量子工单；如确需新增，请通过创建新订单或改单（订单回到 `pending`，重建路线）方式纳入

### 6.5 首件扫码复核

当前项目已把首件从“手输验证码”演进为“扫码复核”模式：

1. 操作员先填写首件内容、测试值、模板、参与人等信息
2. 系统创建一条首件复核会话并生成二维码链接
3. 质检员 / 班组长扫码后，只能查看，不能改操作员填写的内容
4. 会话当前主要状态包括：
   - `pending`：待复核
   - `approved`：复核通过
   - `rejected`：复核不通过
   - `expired`：过期
   - `cancelled`：旧会话被新会话替换
5. 当前会话默认有效期是 5 分钟

复核结果分两支：

1. `passed`
   - 系统内部转为正式首件通过
   - 会生成正式首件记录
   - 订单/工序/子工单推进到执行中
2. `failed`
   - 会话标记为不通过
   - 不创建正式首件记录
   - 操作员需要修改内容后重新发起

### 6.6 结束生产

1. 结束生产对应的是正式报工。
2. 当前工序能结束生产，必须满足：
   - 当前工序状态为 `in_progress` 或 `partial`（同工序里别人正在报工不阻挡当前操作员报工）
   - 当前子工单状态为 `in_progress`
   - 报工数量大于 0
   - 报工数量 + 缺陷数量不能超过当前可生产数量
   - 若工序在并行模式中，还必须满足并行结束门禁
3. 报工后系统会：
   - 增加工序累计完成量
   - 增加子工单累计完成量
   - 生成正式生产记录
   - 写入“报工完成”事件日志

报工后的状态变化：

1. 如果工序累计完成量达到订单总量：
   - 工序变为 `completed`
2. 否则：
   - 工序变为 `partial`
3. 如果子工单累计完成量达到分配量：
   - 子工单变为 `done`
   - 不再可见
4. 否则：
   - 子工单回到 `pending`
   - 继续可见，等待下一次进入执行

### 6.7 下工序放行

结束生产后，系统会把上一道工序的累计良品量（`completed_quantity`）放行给下一道工序。

当前项目的口径不是"整单全部做完才放下一道"，而是：

1. 前工序做完一部分良品
2. 下一工序就按 `min(上工序 completed_quantity, 订单数量)` 提升 `visible_quantity`
3. 再把这个可见量同步到下一工序的子工单分配与可见性

这意味着：

1. 下一工序可以提前接单，不必等上一工序整单完工
2. 这正是 `partial` 状态和 `visible_quantity` 的实际业务含义
3. **缺陷件不会顺流到下工序**：`completed_quantity` 只含良品，下工序拿到的可生产量天然只对应上工序的良品量；缺陷件靠维修单回流（见 §6.8）

### 6.8 报工带缺陷时的维修联动

#### 6.8.1 报工建维修单

1. 结束生产支持同时提交缺陷项
2. 如果缺陷数量大于 0，系统会自动创建维修单
3. 维修单与当前报工记录建立关联，缺陷现象逐条落入维修单
4. 因此"结束生产"不是只记良品数量，它还会把异常生产自动导入后续维修链路
5. 缺陷件**不会**进入 `completed_quantity`，但**会占用本次结束生产的可生产量**——直到维修单完成才会被处理掉

#### 6.8.2 维修单完成回流

维修单进入维修状态后由维修岗位完成处置。完成时业务侧需要给出三类信息：

1. 缺陷原因（每条带"是否报废"标记）
2. 良品回流目标（`return_allocations`）：每条返工件回到哪一道工序重做
3. 报废是否补料（`scrap_replenished`）

系统据此推进的关键动作：

1. **可修件回流**
   - 每条 `return_allocations` 把返工数量加到目标工序的 `visible_quantity` 上
   - 目标工序必须是来源工序本身或更前工序，不允许跳到更后工序
   - 回流写的是 `visible_quantity` 而不是 `completed_quantity`：业务含义是"返修件需要在目标工序及之后再做一遍"
   - 若目标工序之前已经 `completed`，会因 `visible > completed` 退回 `partial` 或 `pending`
2. **跨工序回流的来源工序补偿**
   - 若目标工序不是来源工序本身，来源工序的 `visible_quantity` 会被压低（不低于 `completed_quantity`）
   - 业务含义：缺陷件已被勾回更前工序，来源工序对应的可见量同步收回
3. **报废补料**
   - 当 `scrap_replenished = True` 且报废量 > 0 时，首工序的 `visible_quantity` 增加报废量
   - 业务含义："这批料整张订单要重新从头开做"，下游工序通过普通放行机制再被推上去
4. **订单状态重算**
   - 维修完成路径用"任意工序 `completed_quantity < visible_quantity`"判定订单未完成
   - 一旦回流让某工序 `visible > completed`，订单会从 `completed` 退回 `in_progress`

完整的良品 / 缺陷 / 报废流转关系：

| 流向 | 影响字段 | 业务含义 |
| --- | --- | --- |
| 良品 → 下工序 | 下工序 `visible_quantity` 提升、本工序 `completed_quantity` 累加 | 正常流转 |
| 缺陷 → 维修单 | 本工序可生产量被占用 | 等维修判定 |
| 维修可修件 → 目标工序 | 目标工序 `visible_quantity` 增加；若来源 ≠ 目标，来源 `visible_quantity` 同步压低 | 返工重做 |
| 维修报废件 + 补料 | 首工序 `visible_quantity` 增加 | 整张单从头补做 |
| 维修报废件 + 不补料 | 不再回到生产链 | 直接报废结案 |

### 6.9 代班

当前项目中的代班已经不是审批流，而是“发起即生效”：

1. 发起代班时，只要订单未完成、工序未完成、目标操作员子工单存在，就可以创建
2. 创建后代班授权直接写成 `approved`
3. 代班记录页更多是查看记录，不再是待审批工作台

代班生效后：

1. 代班人在“我的代班工单”视角下看到任务
2. 可以替目标操作员执行：
   - 首件
   - 结束生产
3. 首件代班只允许按授权规则使用
4. 结束生产一旦使用，授权状态会转为 `consumed`

### 6.10 并行模式

并行模式不是所有订单都能开，当前限制比较明确：

1. 订单不能已完成
2. 工序总数必须至少 2 道
3. 被选中的并行工序必须至少 2 道
4. 被选工序必须都存在于当前订单路线中
5. 首个被选工序当前必须还有可生产数量

开启并行模式后：

1. 系统会为选中工序生成并行实例
2. 查询页和详情页会显示并行实例编号
3. 首件和结束生产都要额外受并行开始/结束门禁控制

关闭或重配并行模式后：

1. 原并行实例会被失效
2. 如需启用新的并行组合，会生成新的并行实例

### 6.11 手工完工

管理端支持"手工结束订单"，这是管理员收口动作，不是正常报工动作。

前置门禁：

- **若该订单仍有 `in_repair` 维修单，手工完工会被直接拒绝**——必须先把维修单全部收尾（在维修岗位完成处置）后再来手工完工。这避免维修回流改 `visible_quantity` 与手工完工同时改订单状态时互相打架。

当前行为是：

1. 先校验当前登录密码
2. 校验该订单没有 `in_repair` 维修单（否则报错并提示仍在维修中的维修单号）
3. 然后把所有工序强制补到完成
4. 把所有子工单强制补到完成并隐藏
5. 把订单状态直接置为 `completed`
6. **级联收口该订单的关联实体**：
   - 把订单下仍 `pending` 的首件扫码会话一律置为 `cancelled`
   - 把订单下 `approved` 但未 consumed 的代班授权一律置为 `consumed`（同时写 `consumed_at`）
   - 把订单下仍激活的并行实例一律失活，原因记为 `order_completed_manual`
7. 写入"订单手工完工"事件日志，payload 中带回级联收口的实体清单（首件会话 ID 列表、代班授权 ID 列表、并行实例失活条数）

这条链路的业务含义是：

- 维修单收完之后，可以人工把整张订单直接收尾，并保证关联的首件会话、代班授权、并行实例都不会"挂死"

## 7. 业务上最容易误解的点

1. 订单没有 `partial` 状态，只有工序有 `partial`
2. 工序是 `pending`，不代表它一定不可做；如果 `visible_quantity` 已放行，它已经具备“待开工”能力
3. 首件不通过会留记录，但不会推进执行状态
4. 查询页能看到的“我的工单”，不是简单按订单分配，而是按子工单可见性和数量计算出来的
5. 代班当前不是审批通过后再生效，而是创建即生效
6. 手工完工会直接把整张单据强制收口，和正常报工不是一回事
