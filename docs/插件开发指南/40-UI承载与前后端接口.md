# UI承载与前后端接口

## 1. 目的

- 本文档说明插件 UI 如何被宿主内嵌承载。
- 本文档说明插件进程启动后，需要向宿主暴露哪些页面与心跳接口。

## 2. 适用范围

适用对象：

1. 维护插件 Web UI 的开发者
2. 维护宿主承载层的人
3. 需要排查“页面打不开”“插件启动成功但工作区空白”等问题的人

## 3. 与其他分册关系

- 本文档建立在 `20-插件包结构与清单规范.md` 与 `30-Python入口与依赖加载.md` 之上。
- 本文档只解释“宿主如何打开插件页面”和“插件需要暴露什么接口”。
- 插件前端资源本身如何组织，可结合 `60-串口助手示例拆解.md` 一起看。

## 4. 当前承载模式

当前体系采用：

1. 插件自带 Web UI
2. 插件启动本地 HTTP 服务
3. 宿主通过内嵌 WebView 打开插件返回的 `entry_url`

这意味着：

1. 插件必须自己提供页面资源
2. 宿主不理解插件内部页面结构
3. 宿主不负责渲染插件业务表单

## 5. 当前接口要求

插件启动后，至少应对宿主提供：

1. `entry_url`
   - 指向插件页面入口
2. `heartbeat_url`
   - 用于宿主做存活检查

当前样板的 `ready` 消息形态如下：

```json
{
  "event": "ready",
  "pid": 12345,
  "entry_url": "http://127.0.0.1:43125/index.html",
  "heartbeat_url": "http://127.0.0.1:43125/__heartbeat__"
}
```

## 6. 当前页面入口约定

当前样板插件的页面入口是：

```text
http://127.0.0.1:<dynamic_port>/index.html
```

约定说明：

1. 页面入口应由插件自己返回给宿主
2. 端口应由插件在启动时动态获取
3. 页面资源默认从插件自己的 `web/` 目录提供

## 7. 当前心跳约定

当前样板插件使用：

```text
/__heartbeat__
```

宿主的期望是：

1. 心跳地址可被正常访问
2. 返回 `200 OK`
3. 页面运行期间可持续响应

当前样板返回：

```json
{"status":"ok"}
```

## 8. 当前宿主与插件边界

宿主负责：

1. 启动插件进程
2. 读取 `ready` 消息
3. 保存 `entry_url`
4. 保存 `heartbeat_url`
5. 在工作区中内嵌页面
6. 在插件关闭时停止进程
7. 提供宿主级工作区动作，例如：
   - `重启插件`
   - `全屏 / 退出全屏`
   - `关闭插件`

## 9.1 当前全屏态行为

当前宿主已支持“插件工作区全屏”模式。

进入全屏后：

1. 主壳左侧导航隐藏
2. 插件中心内部插件选择栏隐藏
3. 插件工作区尽量吃满主窗口可用空间

退出全屏后：

1. 主壳左侧导航恢复
2. 插件中心内部插件选择栏恢复
3. 工作区恢复普通卡片布局

插件负责：

1. 启动自己的本地 HTTP 服务
2. 暴露页面入口
3. 暴露心跳接口
4. 提供前端静态资源
5. 提供业务 API

## 10. 当前前端页面建议

插件页面应遵守以下建议：

1. 页面入口通常为 `web/index.html`
2. 页面逻辑通常放在 `web/app.js`
3. 静态资源引用应使用插件自己服务下能解析的相对路径
4. 不要引用宿主本地文件路径
5. 页面应假定自己运行在宿主内嵌环境，而不是外部浏览器标签页

## 11. 示例

### 10.1 样板插件的心跳与静态资源服务

当前 `serial_assistant` 在 `app/server.py` 中：

1. 监听 `127.0.0.1`
2. 暴露 `/__heartbeat__`
3. 暴露 `/api/ports`、`/api/open`、`/api/send`、`/api/close`
4. 把 `web/` 目录中的 `index.html` 与 `app.js` 提供给宿主页面

### 10.2 样板插件的页面结构

当前 `serial_assistant/web/` 下：

1. `index.html` 负责页面骨架
2. `app.js` 负责页面交互
3. 页面通过 `fetch("/api/...")` 调用本地插件服务

## 12. 常见错误

1. 只返回 `entry_url`，不返回 `heartbeat_url`
2. 页面入口存在，但引用的脚本路径错误
3. 插件服务监听到了随机端口，却把固定端口写进 `entry_url`
4. 把页面资源路径写成宿主路径，而不是插件服务路径
5. 页面打开后只显示空白，但插件其实没有真正提供 `index.html`
6. 插件页面使用固定宽度布局，导致在宿主全屏态下仍然浪费大量空间
