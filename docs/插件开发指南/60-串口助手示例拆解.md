# 串口助手示例拆解

## 1. 目的

- 本文档用当前仓库中的 `serial_assistant` 作为标准样板，解释一套完整插件是如何组装起来的。
- 读完本文档后，开发者应该能按相同模式创建一个新的插件目录。

## 2. 适用范围

适用对象：

1. 第一次开发插件的人
2. 需要理解样板插件各文件职责的人
3. 需要从样板复制出新插件骨架的人

## 3. 与其他分册关系

- 本文档是对前面分册的真实样板补充。
- 如果只看规则还不够直观，应把本文档和 `20`、`30`、`40` 三篇一起看。

## 4. 样板目录

当前样板目录如下：

```text
plugins/serial_assistant/
  manifest.json
  launcher.py
  app/
    server.py
    serial_bridge.py
  web/
    index.html
    app.js
  vendor/
    serial/
  tests/
    test_launcher.py
    test_serial_bridge.py
```

## 5. 文件职责拆解

### 5.1 `manifest.json`

职责：

1. 告诉宿主插件的基础信息
2. 声明入口脚本
3. 声明 UI 类型和运行时要求
4. 声明依赖路径、权限和生命周期参数

### 5.2 `launcher.py`

职责：

1. 读取宿主环境变量
2. 组装 `sys.path`
3. 启动本地 HTTP 服务
4. 输出 `ready` 消息
5. 保持进程存活

### 5.3 `app/server.py`

职责：

1. 选择空闲端口
2. 暴露心跳接口
3. 暴露串口业务 API
4. 提供 `web/` 静态资源

### 5.4 `app/serial_bridge.py`

职责：

1. 封装串口列表查询
2. 封装打开、发送、读取、关闭等操作
3. 让 HTTP 层不直接处理串口细节

### 5.5 `web/index.html`

职责：

1. 提供页面骨架
2. 渲染串口选择、发送区和日志区

### 5.6 `web/app.js`

职责：

1. 绑定按钮事件
2. 调用本地 `/api/*` 接口
3. 更新状态和日志显示

### 5.7 `vendor/serial/`

职责：

1. 存放插件自带的 `pyserial` 依赖
2. 确保插件不依赖宿主公共 Python 库

### 5.8 `tests/`

职责：

1. 验证启动链是否可用
2. 验证关键业务逻辑是否正常

## 6. 最小启动链路

当前样板的最小启动链路如下：

1. 宿主扫描 `plugins/serial_assistant/manifest.json`
2. 宿主调用 `plugins/runtime/python312/python.exe launcher.py`
3. `launcher.py` 组装 `sys.path`
4. `launcher.py` 导入 `app.server.start_server`
5. `server.py` 启动本地服务
6. `launcher.py` 输出 `ready` JSON
7. 宿主读取 `entry_url`
8. 宿主在工作区内嵌 `index.html`

## 7. 样板里的关键实现点

### 7.1 环境变量读取

样板优先读取：

1. `MES_PLUGIN_DIR`
2. `MES_PLUGIN_VENDOR_DIR`
3. `MES_PLUGIN_APP_DIR`

### 7.2 页面入口

样板当前返回：

```text
http://127.0.0.1:<port>/index.html
```

### 7.3 心跳入口

样板当前返回：

```text
http://127.0.0.1:<port>/__heartbeat__
```

### 7.4 API 入口

样板当前暴露：

1. `/api/ports`
2. `/api/open`
3. `/api/send`
4. `/api/read`
5. `/api/close`

## 8. 如何基于样板做新插件

推荐做法：

1. 复制 `plugins/serial_assistant/`
2. 把目录名改成新的插件 `id`
3. 修改 `manifest.json` 中的名称、权限和业务说明
4. 保留 `launcher.py` 的宿主接入模式
5. 替换 `app/` 中的业务逻辑
6. 替换 `web/` 中的页面
7. 根据新业务调整 `vendor/` 中的依赖
8. 保留并重写 `tests/`

## 9. 常见错误

1. 复制样板后只改了显示名称，没有改目录名和 `id`
2. 重写业务逻辑时把 `ready` 消息格式改坏
3. 只替换了页面，没有替换对应的 API
4. 忘了把新的第三方依赖带进 `vendor/`
5. 删除了测试目录，导致样板失去最基本的可回归能力
