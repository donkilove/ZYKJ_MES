# 开发调试测试与交付

## 1. 目的

- 本文档说明开发一个插件时，推荐采用什么流程。
- 本文档说明插件应该怎么自测、怎么和宿主联调，以及交付前需要检查什么。

## 2. 适用范围

适用对象：

1. 新建插件的开发者
2. 维护插件宿主联调链路的人
3. 需要判断插件能否进入仓库交付的人

## 3. 与其他分册关系

- 本文档默认你已经读过：
  - `20-插件包结构与清单规范.md`
  - `30-Python入口与依赖加载.md`
  - `40-UI承载与前后端接口.md`
- 本文档不重新解释协议本身，而是说明如何验证这些协议是否真的可用。

## 4. 推荐开发流程

建议按以下顺序开发新插件：

1. 复制现有样板插件目录
2. 修改 `manifest.json`
3. 编写或调整 `launcher.py`
4. 在 `app/` 中实现业务逻辑与本地 HTTP 服务
5. 在 `web/` 中实现前端页面
6. 把第三方依赖收进 `vendor/`
7. 先做插件自测
8. 再做宿主联调

## 5. 插件自测要点

插件在进入宿主联调前，至少应验证：

1. `launcher.py` 能独立启动
2. `ready` 消息结构正确
3. `entry_url` 可访问
4. `heartbeat_url` 可访问
5. `vendor/` 中的依赖可以正常导入
6. 页面能调用自己的本地 API

## 6. 当前仓库中的样板测试

当前 `serial_assistant` 已有以下测试可参考：

1. `plugins/serial_assistant/tests/test_launcher.py`
   - 验证插件能输出合法 `ready` 消息
2. `plugins/serial_assistant/tests/test_serial_bridge.py`
   - 验证串口桥接逻辑

这说明：

1. 插件自身测试应该放在插件目录内
2. 插件测试可以直接用宿主提供的固定解释器运行

## 7. 宿主联调要点

插件进入宿主联调时，应重点确认：

1. 插件中心能发现该插件
2. 点击“打开插件”后能成功启动
3. 宿主能拿到 `entry_url`
4. 页面能在主程序内部显示
5. 点击 `全屏` 后，主壳左侧导航和插件中心内部选择栏会隐藏
6. 点击 `退出全屏` 后，宿主布局会恢复
7. 宿主关闭插件时进程能结束

## 8. 推荐命令

### 8.1 使用宿主固定解释器跑插件测试

```powershell
& "plugins\runtime\python312\python.exe" -m pytest plugins\serial_assistant\tests -q
```

### 8.2 直接启动前端进行联调

```powershell
python start_frontend.py
```

### 8.3 只验证插件目录是否存在

```powershell
Get-ChildItem plugins
```

## 9. 日志与问题定位

当插件无法正常启动时，建议按以下顺序排查：

1. 先看 `manifest.json` 是否能被宿主扫描
2. 再看 `launcher.py` 是否输出了正确的 `ready` 消息
3. 再看 `entry_url` 指向的页面资源是否存在
4. 再看 `heartbeat_url` 是否能响应
5. 最后看插件业务 API 是否真的可用

## 10. 交付前检查清单

插件交付前，至少应确认：

1. `manifest.json` 与目录名一致
2. `launcher.py` 没有写死宿主绝对路径
3. `vendor/` 已包含所需第三方依赖
4. `web/` 中页面资源可被插件自己的服务正常提供
5. 不提交无意义缓存和临时文件
6. 插件自测通过
7. 宿主联调通过
8. 全屏态与普通态切换正常

## 11. 推荐随包提交内容

当前插件应随目录一并提交：

1. `manifest.json`
2. `launcher.py`
3. `app/`
4. `web/`
5. `vendor/`
6. `tests/`

不应提交：

1. `__pycache__/`
2. 临时日志
3. 本地临时调试文件

## 12. 常见错误

1. 只在宿主里点开页面，却从未单独验证 `launcher.py`
2. 依赖装在开发者电脑里，没放进 `vendor/`
3. 页面能在浏览器本地打开，但在宿主内嵌环境下资源路径失效
4. 交付目录里保留了大量无关缓存
5. 测试只验证业务 API，没有验证 `ready` 消息和页面入口
