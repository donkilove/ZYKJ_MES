# 指挥官模式与工作流

## 1. 指挥官模式

### 1.1 触发条件

以下任一条件成立时，必须按指挥官模式执行：

1. 用户明确要求使用“指挥官模式”。
2. 任务需要拆成两个及以上可独立验收的原子任务。
3. 任务要求执行与验证角色显式分离。
4. 任务涉及规则整改、流程重构、跨模块收敛或多轮失败重试闭环。

### 1.2 角色定义

- 主 agent：拆解目标、定义验收、维护任务队列、派发子 agent、汇总结果、判定通过或不通过。
- 调研子 agent：补齐上下文、依赖、范围边界、风险与现有实现。
- 执行子 agent：在明确边界内完成实现、修复、整理或补充。
- 验证子 agent：独立复核并真实执行验证，给出通过或不通过结论。

### 1.3 强制门禁

- 主 agent 不直接承担业务实现，不直接兼任最终验证。
- 每个原子任务至少经过“执行子 agent -> 独立验证子 agent”闭环。
- 若受更高优先级工具约束无法派发子 agent，必须采用分阶段独立验证补偿，并在 `evidence/` 中明确原因、替代动作、影响范围与残余风险。
- 验证失败后，必须重新派发执行子 agent 修复，再由新的验证子 agent 复检；若无法派发子 agent，则按“修复阶段与验证阶段显式分离”的补偿方式执行。
- 子 agent 完成当前原子任务或回执被判废后，必须立即关闭，不得空闲常驻。
- 若子 agent 回执出现越位口吻、角色冒用或面向用户的成稿式交付，主 agent 必须丢弃或重派。

### 1.4 连续执行规则

- 默认不中断，不因命名偏好、局部方案选择或普通歧义停下等待用户。
- 仅在破坏性操作、权限或凭证缺失、外部环境不可用、目标本身无法判定时，才允许收口为硬阻塞。
- 若某一原子任务阻塞但其他任务可推进，应先清空可推进队列，再统一汇总阻塞影响。

## 2. 标准工作流

### 2.1 最小闭环

1. Research：完成背景检索、边界确认、约束识别。
2. Plan：维护计划、原子任务、验收标准、失败重试口径。
3. Implement：由执行角色在边界内完成修改。
4. Verify：由独立验证角色真实执行检查并出具结论。
5. Deliver：汇总变更、验证、阻塞、风险与迁移说明。

### 2.2 启动阶段要求

主 agent 在启动阶段必须完成：

1. 提炼目标、范围、非目标与成功标准。
2. 盘点相关代码、文档、测试、依赖与已有证据。
3. 判断是否存在前置硬阻塞。
4. 拆分为若干可独立验收的原子任务。
5. 为每个原子任务定义输入、输出、验证方式与完成判定。

### 2.3 原子任务拆分标准

- 单一目标：一个原子任务只解决一个明确问题。
- 单一验收：必须存在清晰通过标准。
- 单一主要输出物：聚焦一组紧密相关文件或一个独立行为闭环。
- 可独立验证：可以单独测试、检查或证明完成。
- 可定位失败：失败时能清晰回溯到具体原子任务。

### 2.4 收尾阶段要求

全部原子任务完成后，主 agent 统一输出：

- 完成内容
- 影响范围
- 验证清单与结果
- 失败重试与收口情况
- 阻塞项与影响
- 迁移说明；若无迁移需求，写“无迁移，直接替换”
