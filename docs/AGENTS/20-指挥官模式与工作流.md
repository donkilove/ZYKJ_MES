# 轻量工作流

## 1. 废止说明

- 指挥官模式已彻底废止，不再作为本项目的默认或强制流程。
- 不再要求主 agent、调研子 agent、执行子 agent、验证子 agent 的角色分离。
- 不再要求每个任务都派发子 agent、维护重型门禁或写完整闭环模板。
- 本文件保留原文件名，仅用于避免历史链接断裂；现行语义以“轻量工作流”为准。

## 2. 默认工作流

1. 理解目标：读用户当前指令、相关代码、规则和必要历史记录。
2. 判断范围：区分小改动、跨模块改动、高风险改动和规则变更。
3. 实施修改：沿用现有架构、命名、测试和工具链，控制无关扩散。
4. 真实验证：运行与改动匹配的检查；不能验证时说明原因和风险。
5. 简洁交付：说明改了什么、验证了什么、还有什么限制。

## 3. 何时加重流程

以下场景建议使用计划工具、书面拆解或更完整留痕：

- 跨后端、前端、数据库、接口契约等多个边界。
- 规则文档、提交策略、项目记忆或工具链本身发生变化。
- 涉及数据迁移、权限、安全、删除、批量改写或不可逆操作。
- 排障超过 15 分钟，或已经出现一次以上失败重试。
- 用户明确要求计划、审计、复盘、提交前核对或完整记录。

## 4. 拆分标准

需要拆分任务时，优先按以下口径拆：

- 一个任务解决一个清晰问题。
- 每个任务有可验证结果。
- 失败时能定位到具体文件、接口或流程。
- 拆分服务于推进和验证，不为了形式增加流程。

## 5. 收口要求

交付时按任务复杂度说明：

- 完成内容。
- 验证命令或替代检查。
- 未验证原因、阻塞或残余风险。
- 涉及迁移或兼容性取舍时，说明迁移口径。
