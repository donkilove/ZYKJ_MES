# 质量交付与留痕

## 1. 质量门槛

- 构建、编译、静态检查需零报错。
- 单元、集成、契约、E2E、性能与回归验证应覆盖关键路径与异常分支。
- 关键任务应生成覆盖率、扫描摘要、性能结果或等效质量证据。
- 构建与交付流程需可重复、可审计、可回滚。

## 2. 交付要求

- 明确完成项、未完成项、验证结果、残余风险、阻塞与迁移口径。
- 不交付占位方案、空模板或 `NotImplemented`。
- 若存在迁移：给出切换与回滚步骤。
- 若无迁移：统一写“无迁移，直接替换”。

## 3. evidence 要求

### 3.1 必留痕内容

- 用户原始目标与关键约束
- 任务拆分与排序依据
- 原子任务输入、输出与验收标准
- 子 agent 输出摘要
- 验证命令与验证结果
- 失败轮次与修复记录
- 降级、硬阻塞与代偿措施
- 最终交付结论与迁移说明

### 3.2 证据字段

证据记录至少包含：

- 证据编号
- 来源
- 形成时间
- 适用结论
- 记录责任

### 3.3 evidence 代记

- 若只读调研子 agent 无法直接写入 `evidence/`，允许由主 agent 代记。
- 代记时必须注明：代记责任人、代记时间、原始来源、适用结论。

## 4. 交付收口要求

- 最终交付必须能串起“触发 -> 执行 -> 验证 -> 重试 -> 收口”闭环。
- 若存在工具降级，必须同时说明不可用工具、影响范围、替代动作与残余风险。
- 若存在硬阻塞，必须说明已尝试动作、当前影响与建议下一步。
- 若本轮修改规则文档本身，需额外说明入口变化、分册变化或引用变化。
