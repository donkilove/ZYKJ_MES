# 质量交付与留痕

## 1. 质量门槛

- 被改动触发的构建、编译、静态检查需零报错。
- 单元、集成、契约、E2E、性能与回归验证按风险覆盖关键路径与异常分支。
- 关键任务可生成覆盖率、扫描摘要、性能结果或等效质量证据。
- 发布、迁移和高风险交付流程需可重复、可审计、可回滚。

## 2. 交付要求

- 按任务复杂度说明完成项、验证结果、残余风险、阻塞与迁移口径。
- 不交付占位方案、空模板或 `NotImplemented`。
- 若存在迁移：给出切换与回滚步骤。
- 若无迁移：可简写为“无迁移”或不单列迁移说明。

## 3. 留痕触发条件

小任务不强制写 `evidence/`。以下场景需要更新 `evidence/`、`progress.md` 或等效项目日志：

- 规则文档、工具链、项目记忆或提交策略发生变化。
- 跨后端、前端、数据库、接口契约等多个边界。
- 涉及数据迁移、权限、安全、删除、批量改写或不可逆操作。
- 排障超过 15 分钟，或出现失败重试。
- 提交前需要保留验证口径、范围说明或残余风险。
- 用户明确要求留痕、审计、复盘或交接。

### 3.1 建议记录内容

- 用户目标与关键约束。
- 影响范围与主要文件。
- 关键判断、失败重试和修复动作。
- 验证命令与验证结果。
- 工具降级、硬阻塞与残余风险。
- 涉及 Graphify 或记忆类工具时，记录读取、写入、刷新、查询对象与验证结果；形成长期复用价值的稳定项目事实时，记录是否已写入 `docs/project-memory/`、是否已刷新图谱或标记待语义更新。不得记录密钥、令牌、内部敏感链接或一次性调试噪音。

### 3.2 可选证据字段

需要结构化审计时，可记录：

- 证据编号。
- 来源。
- 形成时间。
- 适用结论。
- 记录责任。

### 3.3 evidence 代记

- 若协作工具或子任务无法直接写入 `evidence/`，允许由当前执行者代记。
- 代记时建议注明：代记责任人、代记时间、原始来源、适用结论。

## 4. 交付收口要求

- 复杂任务交付应能串起“触发 -> 执行 -> 验证 -> 重试 -> 收口”闭环。
- 若存在影响结论可靠性的工具降级，需说明不可用工具、影响范围、替代动作与残余风险。
- 若存在硬阻塞，需说明已尝试动作、当前影响与建议下一步。
- 若本轮修改规则文档本身，需额外说明入口变化、分册变化或引用变化。
