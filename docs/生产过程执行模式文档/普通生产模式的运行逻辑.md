# 普通生产模式的运行逻辑

## 一、什么是普通生产模式

普通生产模式是系统的默认生产执行方式。订单的 `pipeline_enabled = false`（或不设置）时即为普通模式。

核心特征：**基于良品流转的串行推进**——上一道工序有良品产出并流入下一道工序，下一工序即可开工（进行首件检验），不要求上一道工序全部完成。

用一个三层工序 A→B→C 的例子：

```
A 首件开工 → A 报工 partial → A 释放可见量 → B 首件开工 → B 报工 partial → B 释放可见量 → C 首件开工 → C 报工 → 订单完工
  ↑ A partial/completed 后，B 可见量>0，B 可以开工
              ↑ B partial/completed 后，C 可见量>0，C 可以开工
```

---

## 二、核心概念：无流水线实例

普通模式下，系统**不创建任何流水线实例**（`ProcessPipelineInstance`）。

`pipeline_enabled = false` 时，所有流水线相关逻辑均不生效：
- `is_pipeline_process_selected_for_order` → `False`
- `is_pipeline_parallel_edge_for_processes` → `False`
- 首件开工不分配实例，报工不校验实例绑定

---

## 三、可见量初始分配

订单创建时，各工序的 `visible_quantity` 初始分配如下：

| 工序 | visible_quantity 初始值 |
|------|:----------------------:|
| 第一道工序（process_order = 1） | `= order.quantity`（可直接开工） |
| 其他工序 | `= 0`（等待上游释放） |

下游工序必须等待上游报工释放可见量后才能开工。

---

## 四、完整执行流程

### 4.1 A 首件开工

**条件**：
- 工序状态为 `pending`
- A 的 `visible_quantity > 0`（新订单默认为 `order.quantity`）
- 操作员在 A 上有工序绑定
- 今日核验码匹配

门禁判断：流水线未启用 → `_is_start_gate_allowed` 直接返回 `True`，不卡任何工序。

系统不分配流水线实例。A 进入 `in_progress`。

### 4.2 A 收工报工

**条件**：
- 工序状态为 `in_progress` 或 `partial`
- 子订单状态为 `in_progress`
- 操作员有可用产能（`max_producible > 0`）

门禁判断：流水线未启用 → `_is_end_gate_allowed` 直接返回 `True`，不卡任何工序。

A 报工后，`completed_quantity` 增加。系统判断 A 是否达到订单总量：

| 情况 | A 的状态 | 对下游可见量的影响 |
|------|:-------:|-----------------|
| `completed_quantity < order.quantity` | `partial` | 释放当前 `completed_quantity` 给下游 |
| `completed_quantity >= order.quantity` | `completed` | 释放全部 `order.quantity` 给下游 |

> **状态说明**：实际生产订单数量通常较大（数百至上千件），单次报工极少恰好等于订单总量。因此 `partial` 是常态，`completed` 仅在最后一批报工填满订单总量时触发。

**关键规则**：普通模式下可见量**只在当前工序 partial 或 completed 时才向下传递**，不是实时流转。如果 A 只做了一部分就停止，B 也能看到这部分数量。

### 4.3 B 首件开工

此时 B 的 `visible_quantity` 已由 A 释放（≥ 0），B 可以正常开工。

门禁不卡（流水线未启用），只需满足：工序 pending + 有可见量 + 今日核验码。

### 4.4 B 收工报工

与 4.2 相同逻辑。完成后可见量释放给 C。

### 4.5 C 收工报工

C 完成后，检查是否所有工序均已达 visible_quantity：

- 若**所有工序 completed_quantity >= visible_quantity** → 订单状态更新为 `completed`
- 否则 → 订单保持 `in_progress`，`current_process_code` 指向第一个未完成的工序

## 五、门禁规则

### 5.1 首件开工门禁

流水分支代码：`_is_start_gate_allowed`

| 条件 | 结果 |
|------|:--:|
| 流水线未启用 | **直接放行**（不检查上游状态） |
| 第一道工序（没有上游） | 直接放行 |
| 有上游，且流水线启用（非并行边） | 上游 `status == completed` → 放行，否则阻挡 |
| 有上游，且流水线启用（并行边） | 上游 `status in {in_progress, partial, completed}` → 放行 |

**普通模式**：流水线未启用 = 第一行，直接放行，不卡工序。

### 5.2 收工报工门禁

流水分支代码：`_is_end_gate_allowed`

| 条件 | 结果 |
|------|:--:|
| 流水线未启用 | **直接放行**（不检查上游报工状态） |
| 第一道工序 | 直接放行 |
| 有上游，且流水线启用（非并行边） | 直接放行（开工时已卡过） |
| 有上游，且流水线启用（并行边） | 上游 `completed_quantity > 0` → 放行，否则阻挡 |

**普通模式**：流水线未启用 = 第一行，直接放行，不卡工序。

### 5.3 产能余量检查

```
pool_remaining > 0 → 可开工
pool_remaining <= 0 → 阻挡（普通模式无例外）
```

普通模式没有流水线并行边的产能豁免，必须有实际可见量才能开工。

## 六、可见量释放规则

当前工序报工后，计算下游可见量：

| 当前工序状态 | 下游 visible_quantity |
|------------|:--------------------:|
| `partial` 或 `completed` | `=` 当前工序 `completed_quantity`（全部释放） |
| `in_progress`（未达 visible 量） | 保持原值不变 |

**普通模式没有实时流转**：下游只有在看到上游 partial/completed 时才获得可见量。上游做完多少，下游就能做多少。

**子订单可见量扩展点说明**：`ensure_sub_orders_visible_quantity` 函数
（`production_order_service.py:609`）为预留扩展点，当前返回 `False`，不实际更新子订单
层面的 `visible_quantity`。当前生产报工的数量校验以**工序级别**的 `visible_quantity`
为准，子订单级别的可见量校验尚未完全实装。此函数在工序可见量更新时被调用（见
`end_production` line 988-992），待将来引入子订单级产能管控时需补全其内部逻辑。

---

## 七、订单完工判断

系统通过 `_refresh_order_status` 判断订单是否完工：

```
遍历所有工序：
  找到第一个 completed_quantity < visible_quantity 的工序 → 订单 in_progress，current_process_code = 该工序
  所有工序均 completed_quantity >= visible_quantity → 订单 completed
```

注意：`visible_quantity` 由上游报工释放决定，不是订单总量。因此"所有工序都 partial/completed 且下游均已释放"才代表真正完工。

---

## 八、与代班（assist）的交互

普通模式下代班功能**完整可用**，不经过流水线实例路径，无任何限制。

完整流程：

```
1. 主管在代班授权页面创建授权（指定帮工人、目标操作员、订单、工序）
2. 代班授权立即生效（无需确认）
3. 帮工人以目标操作员身份执行首件开工或收工报工
4. 报工完成后授权被消费（status → consumed），一次授权只能用一次
```

代班执行时系统自动：
- 用 `effective_operator_user_id`（目标操作员）绑定子订单
- 在事件日志中同时记录帮工人和目标操作员

---

## 九、与流水线模式的对比

| 维度 | 普通模式 | 流水线模式 |
|------|:-------:|----------:|
| 开工门禁 | 流水线未启用，直接放行 | 上游 pending 时阻挡（并行边） |
| 收工门禁 | 流水线未启用，直接放行 | 上游未报工时阻挡（并行边） |
| 下游可见量 | 上游 partial/completed 才释放 | 并行边实时流转（每报一次传一次） |
| 流水线实例 | 无实例 | 动态分配，通过 link_id 串联 |
| 代班支持 | 完整可用 | 不建议使用 |
| 适用场景 | 大多数标准工序链 | 多工序连续小批量生产 |

---

## 十、已知边界

1. **下游工序必须等上游 partial/completed**：如果 A 只报了一部分工（仍是 in_progress 状态），B 的可见量来自 A 的 `completed_quantity`，而非实时增量。

2. **普通模式不支持跨工序并行开工**：无法让 A 和 B 同时开工，必须等 A 报完工后再开 B。如果需要并行开工能力，应使用流水线模式。

3. **报工超量自动截断**：`end_production` 中 `completed_quantity` 达到 `order.quantity` 后会截断，多报的量不会溢出到下游。

4. **改单对工序可见量的影响**：修改订单工艺路线时，无历史的工序 visible_quantity 重置为 `order.quantity`（首道）或 `0`（其他），有历史的工序保留已有数量。
