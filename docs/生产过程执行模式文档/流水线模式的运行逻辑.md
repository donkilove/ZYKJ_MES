# 流水线模式的运行逻辑

## 一、概述

ZYKJ_MES 的生产过程有两种执行模式：**普通模式**和**流水线模式**。

普通模式是串行执行的：每道工序必须"首件→报工"全部完成后，下一道工序才能开工。任何时候只有一个人在干活。

流水线模式是并行启动、顺序收工的：选中参与流水线的多道工序可以**连续启动首件**（全部进入"进行中"状态），但**报工必须严格按工序顺序来**（上游报完下游才能报）。

两种模式通过订单上的 `pipeline_enabled` 开关字段控制，可以在生产过程中随时切换（有限制条件）。

---

## 二、普通模式（串行执行）

### 2.1 执行流程

以三道工序 A→B→C 为例：

```
A 首件开工 → A 报工 → B 首件开工 → B 报工 → C 首件开工 → C 报工
```

| 步骤 | 操作 | 前置条件 |
|------|------|---------|
| 1 | A 首件开工 | A 有可生产数量 |
| 2 | A 收工报工 | A 处于 in_progress |
| 3 | B 首件开工 | A 已释放可见量到 B（即 A 处于 partial 或 completed），且 B 有可生产数量 |
| 4 | B 收工报工 | B 处于 in_progress |
| 5 | C 首件开工 | B 已释放可见量到 C |
| 6 | C 收工报工 | C 处于 in_progress |

### 2.2 规则

- 任何时候只有**一道工序**处于 `in_progress` 状态
- 下游开工的前提是：上游已报工，且下游有可生产数量（`pool_remaining > 0`）
- 完成量逐道传递：A 报 5 件 → B 可见量 +5，B 报 5 件 → C 可见量 +5

---

## 三、流水线模式（并行启动，顺序收工）

### 3.1 开启流水线

管理员通过接口将 A、B、C 标记为流水线工序。

系统做三件事：
1. 将订单的 `pipeline_enabled` 设为 `true`
2. 将 `pipeline_process_codes` 设为 `"A,B,C"`
3. 将之前所有活跃的流水线实例（`ProcessPipelineInstance`）全部标记为失效

### 3.2 完整执行流程

```
            A 首件 ──────────────── A 报工(5件) ──────
               ↓                         ↓
               B 首件 ────────── B 报工(5件) ──────
                  ↓                    ↓
                  C 首件 ──── C 报工(5件) ──────
```

| 步骤 | 操作 | 前置条件 |
|------|------|---------|
| 1 | A 首件开工 | 第一道工序没有上游，**且 A 有可生产数量** |
| 2 | B 首件开工 | A 已首件开工（A 状态为 in_progress），B 在流水线选中列表里，A→B 是并行边 |
| 3 | C 首件开工 | B 已首件开工，C 在流水线选中列表里，B→C 是并行边 |
| 4 | A 收工报工 | A 处于 in_progress，无下游限制 |
| 5 | B 收工报工 | A 已报工（A 的 `completed_quantity > 0`），B→C 是并行边 |
| 6 | C 收工报工 | B 已报工 |

### 3.3 规则

**首件开工规则**：

| 条件 | 能否开工 |
|------|:--:|
| 流水线关闭时（任何工序） | 直接放行，不管上游状态 |
| 第一道工序，有可生产数量 | ✅ |
| 第一道工序，无可生产数量 | ❌ |
| 并行边的下游，上游状态为 in_progress / partial / completed | ✅ |
| 并行边的下游，上游状态为 pending（还没开工） | ❌ |
| 普通边的下游，上游状态为 completed | ✅ |
| 普通边的下游，上游状态不是 completed | ❌ |

**收工报工规则**：

| 条件 | 能否报工 |
|------|:--:|
| 流水线关闭时（任何工序） | 直接放行 |
| 第一道工序 | 直接放行 |
| 并行边的下游，上游 `completed_quantity > 0`（上游报过工） | ✅ |
| 并行边的下游，上游 `completed_quantity == 0` | ❌ |
| 普通边的下游 | 直接放行（开工时已经卡过条件了） |

### 3.4 可见量释放规则

当前工序报工后，计算下游工序的可见量：

| 当前工序与下游的关系 | 释放规则 |
|------|------|
| 并行边 | 当前工序累计完成量 → 赋值给下游 `visible_quantity` |
| 普通边且当前工序状态为 partial / completed | 同上 |
| 其他 | 保持下游原有 `visible_quantity` 不变 |

并行边的设计意图是"实时流转"：上游做多少，下游就能看到多少。

## 四、两种模式对比

| 维度 | 普通模式 | 流水线模式 |
|------|---------|-----------|
| 多工序同时进行 | ❌ 不允许 | ✅ 允许多道工序同时 in_progress |
| 下游首件条件 | 上游报完工 + 下游有可生产数量 | 上游首件开工即可 |
| 下游报工条件 | 上游报完工 | 按工序顺序，上游报完下游才能报 |
| 首件顺序 | 严格串行 A→A完→B→B完 | 连续启动 A→B→C，不交叉 |
| 报工顺序 | 严格串行 | 严格按工序顺序 A→B→C |
| 切换限制 | 无 | 关闭时有未完成工序则拒绝 |

## 五、数据模型

### 5.1 订单表字段

| 字段 | 类型 | 默认值 | 说明 |
|------|------|:---:|------|
| `pipeline_enabled` | Boolean | `false` | 流水线开关 |
| `pipeline_process_codes` | Text | `""` | 逗号分隔的选中工序编码，如 `"A,B,C"` |

### 5.2 流水线实例表

`mes_process_pipeline_instance`：

| 字段 | 说明 |
|------|------|
| `order_id` | 所属订单 |
| `order_process_id` | 所属工序 |
| `process_code` | 工序编码（冗余） |
| `pipeline_seq` | 并行序号（同一工序内递增） |
| `pipeline_link_id` | 链路标识（上下游同一流水线共享） |
| `pipeline_instance_no` | 唯一编号 `P{订单号}-{工序序号}-{并行序号}-{随机码}` |
| `is_active` | 是否有效 |
| `invalid_reason` | 失效原因 |

### 5.3 流水线实例生命周期

```
首件开工 → 系统分配实例（is_active=true）
  ↓
收工报工 → 实例保持活跃
  ↓
关闭流水线 或 订单完工 → 全部活跃实例失效（is_active=false, 记录原因和失效时间）
```

## 六、API 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/orders/{id}/pipeline-mode` | 查看当前流水线设置 |
| `PUT` | `/orders/{id}/pipeline-mode` | 开启/关闭流水线，设置选中工序 |
| `GET` | `/pipeline-instances` | 查询流水线实例列表 |

### 6.1 开启流水线

```json
PUT /orders/123/pipeline-mode
{
    "enabled": true,
    "process_codes": ["A", "B", "C"]
}
```

### 6.2 关闭流水线

```json
PUT /orders/123/pipeline-mode
{
    "enabled": false,
    "process_codes": []
}
```

### 6.3 开启条件

- 订单至少有 2 道工序
- 选中的有效工序码至少 2 个
- 第一个选中工序有可生产数量（`visible_quantity - completed_quantity > 0`）

### 6.4 关闭条件

- 所有选中工序的状态均为 `completed`，否则拒绝关闭

## 七、与代班（assist）的交互

代班执行不经过正常的流水线实例分配路径。代班人（helper）通过 `assist_authorization_id` + `effective_operator_user_id` 执行生产时，流水线实例不会被正确创建或继承。

这意味着：如果订单开启了流水线，应避免通过代班方式执行。代班更适合普通模式下的辅助操作。

---

## 八、已知边界

1. **关闭流水线后门禁放宽**（已修复）：流水线关闭时，`_is_start_gate_allowed` 和 `_is_end_gate_allowed` 直接返回 `true`，不再检查上游工序状态。关闭前由 `update_order_pipeline_mode` 卡住未完成工序。

2. **开启/切换时旧实例失效**：每次修改流水线设置都会把旧实例全部失效，需要工人重新首件开工来分配新实例。

3. **实例不预创建**：`_create_pipeline_instances_for_order` 当前是空实现（返回 0），实例只在工人实际首件开工时才动态分配。
