# 流水线模式的运行逻辑

## 一、什么是流水线模式

流水线模式允许选中参与的多道工序**连续启动首件**（全部进入"进行中"状态），但**报工必须严格按工序顺序来**（上游报完下游才能报）。

用一个三层工序 A→B→C 的例子：

```
开启流水线（选中 A、B、C）
  ↓
A 首件 → B 首件 → C 首件     ← 连续开工，全部 in_progress
  ↓
A 报工 → B 报工 → C 报工     ← 严格顺序收工，A 报完 B 才能报
```

模式通过订单上的 `pipeline_enabled` 开关字段控制。

---

## 二、开启与关闭

### 2.1 开启条件

- 订单至少有 2 道工序
- 选中的有效工序码至少 2 个
- 第一个选中工序有可生产数量（`visible_quantity - completed_quantity > 0`）

### 2.2 开启时系统做什么

```
1. pipeline_enabled = true
2. pipeline_process_codes = "A,B,C"（逗号分隔）
3. 所有旧的活跃流水线实例 → 标记失效
```

### 2.3 关闭条件

- 所有选中工序的状态均为 `completed`，否则拒绝关闭

### 2.4 关闭时系统做什么

```
1. pipeline_enabled = false
2. pipeline_process_codes = ""
3. 所有活跃流水线实例 → 标记失效（reason: pipeline_disabled）
```

---

## 三、核心概念：并行边

相邻两道工序如果**都在**流水线选中列表里，它们之间就是一条"并行边"。

```
选中: A, B, C

A→B: 并行边（都在选中集）
B→C: 并行边（都在选中集）
```

```
选中: A, C

A→B: 普通边（B 不在选中集）
B→C: 普通边（B 不在选中集）
```

并行边的核心含义：上游不需要完全做完，下游就能开工。普通边则回到串行逻辑：上游必须完成，下游才能动。

## 四、完整执行流程

### 4.1 A 首件开工

**条件**：第一道工序，没有上游，**且 A 有可生产数量**。

门禁判断：`previous_process is None` → 直接放行。产能检查：`pool_remaining > 0` → 通过。

系统分配一个流水线实例，A 进入 `in_progress`。

### 4.2 B 首件开工

**条件**：A 已首件开工（A 状态为 in_progress）。

门禁判断：A→B 是并行边 → A 状态为 in_progress（在允许范围内）→ 放行。系统分配实例，通过 `pipeline_link_id` 与 A 的实例串联。B 进入 `in_progress`。

### 4.3 C 首件开工

**条件**：B 已首件开工。

门禁判断：B→C 是并行边 → B 状态为 in_progress → 放行。系统分配实例并串联。C 进入 `in_progress`。

此时 A、B、C 三道工序都处于 `in_progress`，首件阶段完成。

### 4.4 A 收工报工

A 报工 5 件。A 是第一道工序，无下游限制，直接放行。

A 的完成量释放给 B：并行边上，A 做了多少 B 的可见量就变成多少（实时流转）。

### 4.5 B 收工报工

B 报工 5 件。门禁检查：A→B 是并行边 → A 的 `completed_quantity > 0` → 放行。

B 的完成量释放给 C。

### 4.6 C 收工报工

B 已报工，C 的门禁通过。C 报工完成。订单完工，所有流水线实例失效。

## 五、门禁规则

### 5.1 首件开工门禁

流水分支代码：`_is_start_gate_allowed`

| 条件 | 结果 |
|------|:--:|
| 第一道工序（没有上游） | 直接放行 |
| 流水线关闭了 | 直接放行 |
| 并行边的下游，上游状态为 in_progress / partial / completed | 放行 |
| 并行边的下游，上游状态为 pending（还没开工） | **阻挡** |
| 普通边的下游，上游状态为 completed | 放行 |
| 普通边的下游，上游状态不是 completed | **阻挡** |

### 5.2 收工报工门禁

流水分支代码：`_is_end_gate_allowed`

| 条件 | 结果 |
|------|:--:|
| 第一道工序 | 直接放行 |
| 流水线关闭了 | 直接放行 |
| 并行边的下游，上游 `completed_quantity > 0` | 放行 |
| 并行边的下游，上游 `completed_quantity == 0` | **阻挡** |
| 普通边的下游 | 直接放行（开工时已卡过） |

### 5.3 产能余量检查

```
pool_remaining > 0                    → 可开工
pool_remaining <= 0 但并行边 + 有效实例 → 也允许（上游还在产出）
```

## 六、可见量释放规则

当前工序报工后，计算下游可见量：

| 当前工序与下游关系 | 下游 visible_quantity |
|------|------|
| 并行边 | `=` 当前工序累计完成量 |
| 不是并行边，但当前工序已 partial 或 completed | `=` 当前工序累计完成量 |
| 其他 | 保持原值不变 |

并行边上的实时流转：上游每做一件，下游就能看到一件。

## 七、流水线实例

### 7.1 是什么

`ProcessPipelineInstance` 表，记录每次并行执行的"身份牌"。

| 字段 | 说明 |
|------|------|
| `order_id` | 所属订单 |
| `order_process_id` | 所属工序 |
| `pipeline_seq` | 并行序号 |
| `pipeline_link_id` | 链路标识（上下游相同流水线共享） |
| `pipeline_instance_no` | 唯一编号 `P{订单号}-{工序序号}-{序号}-{随机码}` |
| `is_active` | 是否有效 |

### 7.2 生命周期

```
首件开工 → 分配实例（is_active=true, 分配 pipeline_seq 和 pipeline_link_id）
  ↓
收工报工 → 实例保持活跃
  ↓
关闭流水线或订单完工 → 所有活跃实例失效（is_active=false）
```

### 7.3 链路串联

下游首件开工时，如果与上游是并行边，系统会去上游找相同 `pipeline_seq` 的实例，把它的 `pipeline_link_id` 拿过来，这样上下游就串在同一条流水线上了。

---

## 八、数据模型

订单表字段：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|:---:|------|
| `pipeline_enabled` | Boolean | `false` | 流水线开关 |
| `pipeline_process_codes` | Text | `""` | 逗号分隔选中工序编码 |

| `GET` | `/orders/{id}/pipeline-mode` | 查看流水线设置 |

## 十、与代班（assist）的交互

代班执行不经过正常的流水线实例分配路径。代班人通过 `assist_authorization_id` + `effective_operator_user_id` 执行生产时，流水线实例不会被正确创建或继承。

如果订单开启了流水线，应避免通过代班方式执行。

---

## 十一、已知边界

1. **关闭流水线后门禁不再卡工序**（已修复）：关闭时 `_is_start_gate_allowed` 和 `_is_end_gate_allowed` 直接返回 `true`。

2. **每次修改设置旧实例全部失效**：需要工人重新首件开工分配新实例。

3. **实例不预创建**：`_create_pipeline_instances_for_order` 当前是空实现，实例在工人首件开工时才动态分配。
