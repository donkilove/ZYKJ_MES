# 旧版生产域全量迁移蓝图总览

## 1. 文档目的
本专题用于指导将上一版系统的生产域能力迁移到当前 FastAPI + Flutter 版本。  
本轮输出仅为“可执行蓝图”，不直接实施开发。

## 2. 迁移目标
- 迁移范围按“全量生产域”执行。
- 不迁移旧历史业务数据，仅迁移能力与增量数据结构。
- 首批优先迁移“子订单分配 + 代班审批”。
- 消息中心/消息投递能力不纳入本轮，生产事件统一落到订单事件日志。
- 分批顺序固定：A -> B -> C -> D，不并行打乱。

## 3. 范围边界
### 3.1 纳入范围
- 订单管理与订单查询增强（含子订单可见量、代班审批、并行生产控制）。
- 生产数据查询增强（今日实时、未完工进度、手动筛选、导出）。
- 报废统计与维修闭环（含维修完成回流分配）。

### 3.2 不纳入范围
- 旧版消息中心页面与消息投递链路。
- 旧版数据库历史数据清洗与搬迁。
- 与生产域无直接关系的用户/设备/工艺基础能力重构。

## 4. 当前架构落点
- 后端：`backend/app/api/v1/endpoints/production.py` + 生产域 service 分层。
- 前端：`mes_client/lib/pages/production_page.dart` 下三 Tab（订单管理、订单查询、生产数据）。
- 可见性：`backend/app/core/page_catalog.py` + 页面可见性配置体系。

## 5. 分批策略概览
- A 批（第一优先）：子订单分配 + 代班审批。
- B 批：并行生产（Pipeline）模式。
- C 批：生产数据查询增强与导出。
- D 批：报废统计与维修闭环。

每批都要求：
- 独立 Alembic 迁移。
- 独立后端接口与前端页面改造。
- 独立测试验收与回归报告。

## 6. 兼容与发布原则
- 现有 V1 已上线接口保持兼容，不改已有字段语义。
- 新能力统一走 `/api/v1/production/...` 前缀。
- 每批功能在完成“后端 + 前端 + 测试 + 文档”后再进入下一批。
- 支持逐批上线，不要求全量功能一次性交付。

## 7. 交付清单
- `00_迁移总览.md`：总目标、范围、边界、顺序。
- `01_旧新功能差异矩阵.md`：旧新版能力映射与缺口。
- `02_分批实施计划.md`：A/B/C/D 可执行实施清单。
- `03_API与数据模型设计.md`：新增接口、权限、数据结构草案。
- `04_测试与验收标准.md`：测试方案、验收门槛、回归清单。
