# 用户模块整改细化

## 1. 目标

- 将 `docs/功能规划V1深度审查-20260318/10-用户模块.md` 中的审查结论，继续细化到“页面改造点、接口改造点、数据表改造点、实施顺序”。
- 本文只给整改设计，不直接改动代码。
- 设计原则：以需求说明为准，优先闭合 P0 业务规则；无兼容需求时，建议直接替换旧契约。

## 2. 总体策略

### 2.1 推荐改造顺序

1. P0：单角色闭环 + 自定义角色可分配 + 用户模块去工序化。
2. P0：系统管理员权限配置保底规则。
3. P1：功能权限配置粒度重构。
4. P1：注册审批历史、导出筛选一致性。
5. P2：页面展示、文案、会话信息对齐。

### 2.2 迁移策略

- 用户模块接口建议直接替换，不保留 `role_codes` / `process_codes` 旧输入契约。
- `sys_user_role` 建议增加“每个用户仅一条角色绑定”的数据库约束，并先清洗历史多角色数据。
- 权限粒度重构不一定需要新增表，但会涉及 `sys_permission_catalog`、`sys_role_permission_grant` 数据重建或重算。

## 4. 详细改造点

### 4.1 U-01 单角色闭环 + 自定义角色可分配

#### 4.1.1 问题说明

- 当前角色创建走 `sys_role`，允许新增自定义角色。证据：`backend/app/services/role_service.py:74`
- 但用户创建、编辑、注册审批的角色校验仍固定依赖 `VALID_ROLE_CODES`。证据：`backend/app/services/user_service.py:64`
- 前端又会把全部角色都展示到单选列表中。证据：`frontend/lib/pages/user_management_page.dart:436`、`frontend/lib/pages/registration_approval_page.dart:393`
- 结果是“角色能创建，不能分配”，业务闭环断裂。

#### 4.1.2 页面改造点

- `frontend/lib/pages/user_management_page.dart`
  - 新建用户弹窗继续保留单选角色，但角色数据源改为“启用中且未删除角色”。
  - 编辑用户弹窗同样只允许选择一个角色，不再展示历史多角色兼容文案为长期方案；历史多角色应在数据迁移时一次性清洗。
  - 角色项文案建议展示：`角色名称 + 角色类型 + 状态`，避免同名/自定义角色识别不清。
- `frontend/lib/pages/registration_approval_page.dart`
  - 审批通过弹窗改为同一套角色单选源，不再假设只有 5 个内置角色可用。

#### 4.1.3 接口改造点

- `backend/app/schemas/user.py`
  - `UserCreate.role_codes -> role_code`
  - `UserUpdate.role_codes -> role_code`
  - `UserItem.role_codes / role_names -> role_code / role_name`
- `backend/app/schemas/auth.py`
  - `ApproveRegistrationRequest.role_codes -> role_code`
  - `RegistrationActionResult.role_codes -> role_code`
- `backend/app/services/user_service.py`
  - 删除 `_validate_role_codes()` 对 `VALID_ROLE_CODES` 的固定枚举依赖。
  - `_resolve_roles()` 改为只接受单个 `role_code`，并按 `sys_role` 查询真实角色；校验条件改为：角色存在、未删除、已启用。
  - 新建、编辑、审批通过全部统一走同一单角色解析逻辑。
- `backend/app/api/v1/endpoints/users.py`
  - 审计字段改为记录 `role_code`、`role_name` 单值，而不是列表。
- `backend/app/api/v1/endpoints/auth.py`
  - 审批通过接口同步切到 `role_code` 单值。
- `frontend/lib/services/user_service.dart`
  - `createUser()`、`updateUser()`、`approveRegistrationRequest()` 全部从 `roleCodes: List<String>` 改为 `roleCode: String`。
- `frontend/lib/models/user_models.dart`
  - 用户模型改成单角色字段，避免前端继续保留“多角色”心智。

#### 4.1.4 数据表改造点

- 现状：`sys_user_role` 是多对多中间表。证据：`backend/app/models/associations.py:6`
- 推荐改造：保留中间表结构，但新增唯一约束 `uq_sys_user_role_user_id(user_id)`，从数据库层强制“一用户仅一个角色”。
- 数据迁移步骤：
  1. 先执行现有 `normalize_users_to_single_role()` 逻辑思想，按优先级收敛历史多角色数据。证据：`backend/app/services/user_service.py:333`
  2. 清理完成后再给 `sys_user_role.user_id` 加唯一约束。
- 说明：这里不建议第一阶段直接把 `sys_user_role` 改造成 `sys_user.role_id`，因为影响面会外溢到权限、登录、生产等多个模块；先用唯一约束完成单角色收敛，变更边界更小。

#### 4.1.5 验收标准

- 自定义角色创建后，可直接在用户新建、编辑、注册审批里选择并保存成功。
- 任一用户在数据库层最多只存在 1 条角色绑定记录。
- 前端与接口文档中不再出现“角色列表”表达。

### 4.2 U-02 用户模块对外契约去工序化

#### 4.2.1 问题说明

- 当前用户创建、编辑、注册审批接口都要求传 `process_codes`。证据：`backend/app/schemas/user.py:15`、`backend/app/schemas/auth.py:68`
- 前端在选择工段后自动把该工段下全部工序编码塞回接口。证据：`frontend/lib/pages/user_management_page.dart:476`、`frontend/lib/pages/registration_approval_page.dart:434`
- 需求说明中，用户模块对“工段”只做引用，不要求在用户模块维护工序。

#### 4.2.2 页面改造点

- `frontend/lib/pages/user_management_page.dart`
  - 删除 `_getProcessCodesByStage()`、`_getStageIdFromProcessCodes()` 等用户页内部工序推导逻辑。
  - 新建/编辑弹窗仅展示“工段”单选。
  - 列表列继续展示工段，不再展示 `stageNames` 这种由工序反推出来的多工段集合。
- `frontend/lib/pages/registration_approval_page.dart`
  - 删除审批通过弹窗里的工序派生逻辑，只保留工段选择。
- `frontend/lib/models/user_models.dart`
  - 删除 `processCodes`、`processNames`、`stageNames` 这些只服务于用户模块页面的字段。

#### 4.2.3 接口改造点

- `backend/app/schemas/user.py`
  - 从 `UserCreate`、`UserUpdate`、`UserItem` 中删除 `process_codes`、`process_names`、`stage_names`。
- `backend/app/schemas/auth.py`
  - 从 `ApproveRegistrationRequest`、`RegistrationActionResult` 删除 `process_codes`。
- `backend/app/services/user_service.py`
  - `_resolve_processes()` 保留为内部派生函数，但不再接受来自前端的 `process_codes` 输入；改为只基于 `stage_id` 自动推导启用工序。
  - 对非操作员仍自动清空 `stage_id` 和派生工序。
- `frontend/lib/services/user_service.dart`
  - 删除 `processCodes` 入参和序列化逻辑。

#### 4.2.4 数据表改造点

- `sys_user_process` 第一阶段不做 DDL 变更。证据：`backend/app/models/associations.py:14`
- 处理策略：
  - 将其降级为“后端派生投影表”，由 `stage_id` 自动维护；
  - 用户模块页面和接口不再感知该表；
  - 生产模块仍可暂时复用该投影，避免把用户模块整改扩散成生产模块重构。
- 后续若生产执行授权改为“按工段”而非“按工序”，再考虑删除该表。

#### 4.2.5 验收标准

- 用户模块前端、接口契约只保留“角色 + 工段”。
- 生产等下游模块不需要改用户录入动作，也能继续读取到后端派生的工序映射。

### 4.3 U-03 系统管理员权限配置保底

#### 4.3.1 问题说明

- 当前只保证“至少保留一个激活的系统管理员账号”。证据：`backend/app/services/user_service.py:307`
- 当前并未保证系统管理员角色必然拥有 `功能权限配置页面` 的查看/编辑权限。证据：`backend/app/services/authz_service.py:60`、`backend/app/services/authz_service.py:2050`
- 这会导致理论上存在“还有系统管理员账号，但已经没人能进入权限配置页面”的失控场景。

#### 4.3.2 页面改造点

- `frontend/lib/pages/function_permission_config_page.dart`
  - 当选中 `system_admin` 且模块为 `system` 时，模块开关与“管理功能权限配置”能力必须显示为锁定开启状态。
  - 非系统管理员查看 `system` 模块时继续保持只读，但要增加明显提示：`该模块仅系统管理员可配置`。
  - 保存前若本次变更会导致系统失去权限配置入口，前端可先提示，但最终以后端校验为准。

#### 4.3.3 接口改造点

- `backend/app/services/authz_service.py`
  - 在 `update_capability_pack_role_config()` 与 `apply_capability_pack_role_configs()` 提交前增加保底校验：
    1. `system_admin` 角色的 `page.function_permission_config.view` 不允许移除；
    2. `authz.role_permissions.update` 不允许移除；
    3. 若当前系统仅剩一个激活系统管理员账号，则禁止使其失去上述权限。
  - 建议抽出统一函数，如 `ensure_function_permission_guardrail()`，由用户停用/删除与权限保存共用。
- `backend/app/services/user_service.py`
  - 在 `ensure_can_deactivate_user()`、`delete_user()` 路径中追加校验：若该用户是最后一个“具备权限配置入口的激活系统管理员”，则禁止停用/删除。
- `backend/app/api/v1/endpoints/authz.py`
  - 后端冲突时返回 400，并给出可读错误，如：`必须至少保留一个可进入功能权限配置页面的系统管理员账号`。

#### 4.3.4 数据表改造点

- 无需新增表或字段。
- 直接使用现有表完成校验：
  - `sys_user`
  - `sys_user_role`
  - `sys_role_permission_grant`
  - `sys_permission_catalog`
- 其中核心判断条件基于权限码：
  - `page.function_permission_config.view`
  - `authz.role_permissions.update`
  - `feature.system.role_permissions.manage`

#### 4.3.5 验收标准

- 任意一次权限保存、停用用户、删除用户操作，都不能把系统带到“无人可配置权限”的状态。
- `system_admin` 角色始终可进入功能权限配置页面并保存配置。

### 4.4 U-04 权限配置粒度重构

#### 4.4.1 问题说明

- 当前配置页采用“能力包”开关。证据：`frontend/lib/pages/function_permission_config_page.dart:589`
- 需求要求的是更接近“页面查看 / 新建 / 编辑 / 删除 / 审批 / 配置”的粒度。
- 当前用户模块里如“维护用户信息”把创建、编辑、删除、启停、重置密码、导出都打包在同一能力下。证据：`backend/app/core/authz_hierarchy_catalog.py:76`

#### 4.4.2 页面改造点

- `frontend/lib/pages/function_permission_config_page.dart`
  - 左侧角色列表建议改为顶部 `模块下拉 + 角色下拉`，与需求表述一致。
  - 主区从 `SwitchListTile` 改为“页面维度 + 动作勾选矩阵”：
    - 用户管理页：查看、新建、编辑、停用/启用、重置密码、删除、导出
    - 注册审批页：查看、通过、驳回
    - 角色管理页：查看、新建、编辑、停用/启用、删除
    - 审计日志页：查看
    - 个人中心页：查看、修改本人密码、查看当前会话
    - 登录日志/在线会话页：查看登录日志、查看在线会话、强制下线
  - 保存结果区应提示“本次新增/移除的动作权限”。

#### 4.4.3 接口改造点

- 推荐直接替换现有 `capability-packs` 写接口，不再用“能力包”命名承载细粒度动作。
- 建议新增接口：
  - `GET /authz/page-action-matrix?module_code=user&role_code=system_admin`
  - `PUT /authz/page-action-matrix`
- 若暂不新开接口，也至少要：
  - 重切 `backend/app/core/authz_hierarchy_catalog.py` 中用户模块 feature 定义；
  - 把当前粗粒度 capability pack 拆成更细的 feature/action 组合；
  - 同步调整前端映射模型。

#### 4.4.4 数据表改造点

- 无需新增物理表，沿用：
  - `sys_permission_catalog`
  - `sys_role_permission_grant`
- 但这是一次“权限目录数据重建”：
  - 用户模块旧权限码需要直接替换；
  - 已有角色授权需按新码重算，不建议继续兼容旧 capability code。

#### 4.4.5 验收标准

- 能单独控制“用户管理-导出”和“用户管理-删除”等动作，不再被同一开关绑定。
- 页面表现与需求文档中的粒度表述一致。

### 4.5 U-05 注册审批历史与筛选闭环

#### 4.5.1 问题说明

- 后端已支持 `status` 过滤。证据：`backend/app/api/v1/endpoints/auth.py:252`
- 前端固定只传 `pending`，因此审批页只能看待审批列表。证据：`frontend/lib/pages/registration_approval_page.dart:156`

#### 4.5.2 页面改造点

- `frontend/lib/pages/registration_approval_page.dart`
  - 顶部增加状态筛选：`待审批 / 已通过 / 已驳回 / 全部`
  - 增加关键字搜索框，复用后端 `keyword`
  - 列表增加 `驳回原因` 列，仅在驳回记录或详情中展示
  - “待审批数量”文案需随当前状态切换为“当前列表总数”或拆成统计卡片

#### 4.5.3 接口改造点

- 后端接口已具备，不需要新增接口。
- `frontend/lib/services/user_service.dart`
  - 保持 `listRegistrationRequests()` 的 `status` 可空透传，页面不再固定写死 `pending`。

#### 4.5.4 数据表改造点

- 无变更；`sys_registration_request` 已具备 `status`、`rejected_reason`、`reviewed_by_user_id`、`reviewed_at`。证据：`backend/app/models/registration_request.py:15`

### 4.6 U-06 导出与筛选一致性

#### 4.6.1 问题说明

- 用户页支持在线状态筛选。证据：`frontend/lib/pages/user_management_page.dart:1205`
- 但导出接口与前端导出调用均未传 `is_online`。证据：`frontend/lib/services/user_service.dart:63`、`backend/app/api/v1/endpoints/users.py:150`

#### 4.6.2 页面改造点

- `frontend/lib/pages/user_management_page.dart`
  - 导出时直接复用当前筛选状态，不再单独组装“部分筛选项”。

#### 4.6.3 接口改造点

- `backend/app/api/v1/endpoints/users.py`
  - `GET /users/export` 新增 `is_online` 参数，并复用 `list_users()` 的过滤能力。
- `frontend/lib/services/user_service.dart`
  - `exportUsers()` 新增 `isOnline` 参数透传。

#### 4.6.4 数据表改造点

- 无变更，在线状态仍基于 `sys_user_session` 聚合计算。

### 4.7 U-07 页面展示与文案对齐

#### 4.7.1 页面改造点

- `frontend/lib/pages/user_management_page.dart`
  - 删除确认文案从“不可恢复”改为“逻辑删除后默认不在常规列表展示，不可登录，可按需恢复或转停用策略”。
- `frontend/lib/pages/role_management_page.dart`
  - 从卡片列表改为 `DataTable`，补齐字段：角色名称、角色说明、角色类型、关联用户数、状态、操作。现状证据：`frontend/lib/pages/role_management_page.dart:413`
- `frontend/lib/pages/login_session_page.dart`
  - 登录日志和在线会话改为表头式表格。
  - 在线会话页把 `active` 显式映射为“在线”。
- `frontend/lib/pages/account_settings_page.dart`
  - 移除“会话令牌”展示。现状证据：`frontend/lib/pages/account_settings_page.dart:299`

#### 4.7.2 接口改造点

- 会话接口可选优化：`OnlineSessionItem` 新增 `status_label`，让前端不直接展示底层枚举值。现状证据：`backend/app/schemas/session.py:22`
- 若不想改接口，也至少在前端做 `active -> 在线`、`forced_offline -> 已强制下线`、`expired -> 已过期` 映射。

#### 4.7.3 数据表改造点

- 无变更；`sys_user_session.status` 现有字段足够。证据：`backend/app/models/user_session.py:19`

## 7. 最终建议

- 用户模块不要继续在“现有实现上打补丁”处理自定义角色和工序字段，建议直接统一成“单角色 + 单工段”的清晰契约。
- 第一阶段不必扩大到重构生产模块授权模型，但必须把 `process_codes` 从用户模块前端与公开接口中清掉，降为后端派生逻辑。
- 权限配置页若不先加保底规则，任何后续权限改造都存在把系统锁死的风险，应与 U-01 同批处理。
- 无迁移，直接替换的部分：前端表单契约、审批契约、权限配置页面交互。
- 需要迁移的部分：`sys_user_role` 单角色约束、权限目录重切后的角色授权数据重算。
