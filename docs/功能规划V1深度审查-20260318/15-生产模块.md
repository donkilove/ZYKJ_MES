# 生产模块深度检查

## 1. 总体判断

- 结论：主链路整体可用，但仍有多个关键规则未达需求要求。
- 核心风险：报工数量与异常数量口径不闭环；删除订单的事件日志最终无法保留；维修完成前端不支持多回流分配。

## 2. 已满足项

- 订单管理已支持订单号/产品名、状态、并行模式、开始日期、交期筛选，以及新建、编辑、删除、手工完工、并行模式设置、详情查看、导出。证据：`backend/app/api/v1/endpoints/production.py:381`、`frontend/lib/pages/production_order_management_page.dart:281`
- 订单新建/编辑已支持订单号、产品、数量、开始日期、交期、备注、模板选择、手工工艺步骤、保存为模板、设为默认模板。证据：`frontend/lib/pages/production_order_form_page.dart:647`、`backend/app/schemas/production.py:16`
- 后端已校验订单号唯一、产品启用、数量大于 0、必须存在工艺路线、步骤顺序不可重复、工序与工段必须匹配。证据：`backend/app/services/production_order_service.py:179`、`backend/app/services/production_order_service.py:922`
- 订单详情和工单执行详情均已存在，包含基础信息、工序、子订单、记录、事件等区域。证据：`frontend/lib/pages/production_order_detail_page.dart:163`、`frontend/lib/pages/production_order_query_detail_page.dart:188`
- 订单查询已支持“我的工单 / 我的代班工单 / 代理视角”，并支持状态筛选、当前工序筛选、代理操作员选择和定时刷新。证据：`frontend/lib/pages/production_order_query_page.dart:278`、`backend/app/services/production_order_service.py:1282`
- 首件、报工、代班申请、手工送修建单均已具备前后端实现。证据：`backend/app/services/production_execution_service.py:221`、`backend/app/services/assist_authorization_service.py:88`
- 并行实例页面与接口均已存在，订单管理页和详情页也能跳转查看。证据：`backend/app/api/v1/endpoints/production.py:1378`、`frontend/lib/pages/production_pipeline_instances_page.dart:94`
- 生产数据页已实现总览、今日实时、未完工进度、手动筛选导出、工序统计、人员统计。证据：`backend/app/services/production_data_query_service.py:254`、`frontend/lib/pages/production_data_page.dart:796`
- 报废统计与维修订单基础链路已存在，事件日志也较完整。证据：`backend/app/services/production_repair_service.py:322`、`backend/app/services/production_event_log_service.py:12`

## 3. 部分满足项

- 订单导出虽已接后端，但前端只提示成功和字节数，没有形成完整的文件导出闭环。证据：`frontend/lib/pages/production_order_management_page.dart:154`
- 并行实例追踪页已存在，但筛选字段和展示字段与需求不完全一致，且缺少更强的订单详情联动。证据：`frontend/lib/pages/production_pipeline_instances_page.dart:152`
- 报废详情和维修订单详情都是弹窗实现，不是独立子页面。证据：`frontend/lib/pages/production_scrap_statistics_page.dart:236`、`frontend/lib/pages/production_repair_orders_page.dart:136`
- 完成维修时，后端支持多条 `return_allocations`，但前端只允许单个回流目标。证据：`frontend/lib/pages/production_repair_orders_page.dart:510`、`backend/app/schemas/production.py:516`
- 维修订单导出支持时间范围，但前端导出时未把时间范围带上，可能与当前筛选条件不一致。证据：`frontend/lib/pages/production_repair_orders_page.dart:339`、`frontend/lib/services/production_service.dart:863`

## 4. 未满足项

- 删除订单虽然会写 `order_deleted` 事件日志，但日志外键对订单使用级联删除，最终无法满足“删除也要可追溯”的需求。证据：`backend/app/services/production_order_service.py:1223`、`backend/app/models/order_event_log.py:11`

## 5. 明显偏差

- 报工时只校验 `quantity <= max_producible`，未把异常数量一并计入；随后维修建单又使用 `quantity + defect_quantity`，导致数量口径不闭环。证据：`backend/app/services/production_execution_service.py:384`、`backend/app/services/production_execution_service.py:438`
- 代班审批状态前端把 `approved` 映射为“已生效”，与需求中的“已审批”表述不一致。证据：`frontend/lib/models/production_models.dart:53`
- 报废统计筛选项文案写成“待申请 / 已申请”，而需求口径是“待处理 / 已处理”。证据：`frontend/lib/pages/production_scrap_statistics_page.dart:387`、`frontend/lib/models/production_models.dart:79`

## 6. 优先级建议

1. 先修复报工数量口径与删除追溯问题，这是当前最影响验收的两项。
2. 再补前端多回流分配、导出闭环和若干文案/字段一致性问题。
