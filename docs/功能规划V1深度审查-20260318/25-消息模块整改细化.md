# 消息模块整改细化

## 1. 目标

- 将 `docs/功能规划V1深度审查-20260318/16-消息模块.md` 中的审查结论，细化为“分批次整改计划 + 数据表/接口/页面影响范围 + 验收标准”。
- 本文先作为消息模块本轮整改的执行蓝图，后续按批次逐项落地。
- 设计原则：
  - 以需求说明为准；
  - 优先打通跳转链路、未读与列表一致性、已读状态实时同步；
  - 在未新增公告/规则完整页面的前提下，先闭合消息中心核心体验与最小管理能力。

## 2. 当前核心缺口

### 2.1 P0 缺口

1. 业务跳转只支持一级页面，不支持页签和目标上下文，且部分后端 `target_page_code` 实际上传的是页签代码。
2. 未读角标与消息列表数据口径不一致，WS 不可用时只轮询未读数，不轮询消息列表。
3. 已读状态变化未向同用户其他在线端同步推送。
4. 消息中心缺少批量标记已读与“仅看待处理”筛选。

### 2.2 P1 缺口

1. 顶部概览区中的待处理、紧急未读是近似值，不是稳定统计。
2. 缺少页面内固定详情预览区和更完整的消息详情信息。
3. 公告发布与规则配置页面仍缺失，但本期可先补最小管理闭环。
4. 缺少消息模块前后端测试资产。

## 3. 总体策略

### 3.1 推荐分批次顺序

1. 批次 A：跳转链路、未读/列表一致性与轮询兜底。
2. 批次 B：详情预览、批量已读、概要统计与交互收口。
3. 批次 C：公告/规则保留策略范围内的最小闭环与文案收口。
4. 批次 D：补测试、联调回归、日志更新并提交。

## 4. 分批次总表

| 编号 | 批次 | 优先级 | 目标 | 核心结果 |
| --- | --- | --- | --- | --- |
| M-01 | A | P0 | 业务跳转闭环 | 支持 `target_page_code + target_tab_code` 导航，并兼容后端传页签代码的旧数据 |
| M-02 | A | P0 | 未读/列表一致性 | 列表与未读数统一以 active 消息为口径，WS 降级时轮询列表与摘要 |
| M-03 | A | P0 | 已读状态同步 | 单条已读、全部已读、批量已读都向同用户在线端推送状态变化 |
| M-04 | B | P1 | 详情预览区 | 页面内固定详情预览区与更完整详情字段 |
| M-05 | B | P1 | 批量已读与仅看待处理 | 列表多选、批量操作、待处理快速筛选 |
| M-06 | B | P1 | 概览统计稳定化 | 后端提供消息摘要统计接口，前端不再用近似统计 |
| M-07 | C | P1 | 公告/规则最小闭环 | 在当前范围内补最小公告发布入口或明确范围外裁剪说明 |
| M-08 | C | P1 | 失效与提示语义 | 失效、无权限、来源删除等提示文案统一 |
| M-09 | D | P1 | 测试与提交 | 新增前后端回归测试、联调验证、日志更新与提交 |

## 5. 批次 A：跳转链路、未读/列表一致性与轮询兜底

### 5.1 后端改造点

- `message_service.py`
  - 消息列表默认只返回 `status=active` 消息，与未读统计保持一致；
  - 支持 `todo_only` 查询。
- `messages.py`
  - 新增消息摘要统计接口；
  - 已读状态接口在提交后回推 WS 事件。
- 来源模块消息生成处：
  - 将明显错误的 `target_page_code` 修正为真实一级页面 code；
  - 具体页签通过 `target_tab_code` 传递。

### 5.2 前端改造点

- `main_shell_page.dart`
  - 支持把消息中的页签 code 解析到对应一级页面，并把目标页签传给模块页；
- 各模块页容器（用户/产品/设备/生产/品质/工艺）
  - 接受并响应 `preferredTabCode`；
- `message_center_page.dart`
  - 增加列表轮询兜底，不只轮询未读角标；
  - “仅看待处理”作为独立筛选条件。

### 5.3 批次 A 验收标准

- 消息能正确跳转到来源一级页面与页签。
- WS 断开后，消息列表与未读角标仍能在轮询下保持一致。

## 6. 批次 B：详情预览、批量已读、概要统计与交互收口

### 6.1 后端改造点

- 新增 `MessageSummaryResult` 摘要统计接口。
- 新增批量已读接口。

### 6.2 前端改造点

- `message_center_page.dart`
  - 列表多选 + 批量标记已读；
  - 页面内固定详情预览区；
  - 顶部概览区改为使用后端摘要统计接口。

### 6.3 批次 B 验收标准

- 待处理、紧急未读统计不再依赖前端近似推算。
- 批量已读可正常工作，详情预览区可直接查看消息主体信息。

## 7. 批次 C：公告/规则最小闭环与文案收口

### 7.1 目标

- 在不扩大过多范围的前提下，补最小公告/提示语义闭环。

### 7.2 核心任务

- 消息详情与列表中补充：来源失效、无权限访问、对象不存在等明确提示。
- 若不实现完整公告/规则页面，则至少保证公告类消息的展示、筛选、详情与已读语义完整。

### 7.3 批次 C 验收标准

- 用户能清楚区分“无权限访问”和“来源对象失效/删除”。
- 公告消息在消息中心内具备完整查看和已读管理能力。

## 8. 批次 D：测试、联调、清理、提交

### 8.1 核心任务

1. 新增后端集成测试，至少覆盖：
   - 消息摘要统计；
   - 批量已读；
   - 列表 active 口径；
   - 跳转字段契约。
2. 新增前端回归测试，至少覆盖：
   - 消息中心批量已读；
   - 仅看待处理筛选；
   - 跳转页签承接；
   - 详情预览区展示。
3. 开发库联调：
   - 触发生产/品质/设备消息 -> 消息中心接收 -> 标记已读 -> 跳转来源页签 -> 断线后轮询补拉。
4. 更新 `docs` 日志并提交消息模块改动。

## 10. 最终建议

- 消息模块本轮最关键的是“跳转链路、未读与列表一致性、已读实时同步、详情与批量操作闭环”四条主链。
- 在这四条主链完成前，不建议将消息模块判定为可正式验收。
- 下一步应按本计划直接进入批次 A 编码，再连续推进到批次 D，最后统一联调、提交。
