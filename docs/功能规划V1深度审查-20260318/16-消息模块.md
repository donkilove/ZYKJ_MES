# 消息模块深度检查

## 1. 总体判断

- 结论：基础版消息中心已具备，但距离需求说明中的完整验收目标仍有明显差距。
- 核心风险：业务跳转链路未真正打通；实时同步只覆盖角标不覆盖列表与已读状态；公告与规则配置缺失。

## 2. 已满足项

- 已有独立一级侧边栏“消息”入口，并支持未读角标。证据：`frontend/lib/models/page_catalog_models.dart:620`、`frontend/lib/pages/main_shell_page.dart:813`
- 登录成功后会建立消息 WebSocket 连接；应用恢复前台时也会重连并刷新未读数。证据：`frontend/lib/pages/main_shell_page.dart:141`、`frontend/lib/pages/main_shell_page.dart:159`
- 消息中心页已支持关键字、状态、分类、优先级、来源模块、时间范围筛选，并支持刷新、重置筛选、全部已读。证据：`frontend/lib/pages/message_center_page.dart:254`、`frontend/lib/pages/message_center_page.dart:329`
- 已实现单条已读、全部已读。证据：`backend/app/api/v1/endpoints/messages.py:71`、`frontend/lib/services/message_service.dart:59`
- 已实现 WebSocket 建连、token 鉴权、未读数初始推送、断线重连和 HTTP 未读数轮询兜底。证据：`backend/app/api/v1/endpoints/messages.py:110`、`backend/app/services/message_connection_manager.py:12`、`frontend/lib/services/message_ws_service.dart:29`
- 消息模型已覆盖分类、优先级、来源模块、来源对象、目标页面、去重键等基础字段。证据：`backend/app/models/message.py:17`

## 3. 部分满足项

- 概览区已有“未读消息 / 待处理 / 紧急未读 / 全部消息”卡片，但其中待处理和紧急未读是基于额外拉取的近似计算，不是可靠全量统计。证据：`frontend/lib/pages/message_center_page.dart:104`
- 已有详情弹窗，并在点开消息时尝试自动标记已读，但并非页面内固定详情预览区，也缺少跳转按钮、触发时间、处理建议等字段。证据：`frontend/lib/pages/message_center_page.dart:178`
- 已有业务跳转字段建模，但前端壳层目前只支持切到一级菜单页，不支持按 `target_tab_code` 或 `target_route_payload_json` 精确进入目标上下文。证据：`backend/app/models/message.py:29`、`frontend/lib/pages/main_shell_page.dart:617`
- 已有公告类型展示与筛选，但没有真正的公告发布、停用、范围配置能力。证据：`frontend/lib/pages/message_center_page.dart:357`、`backend/app/models/message.py:17`

## 4. 未满足项

- 未实现批量标记已读与“仅看待处理”筛选。证据：`frontend/lib/pages/message_center_page.dart:254`、`frontend/lib/pages/message_center_page.dart:329`
- 未实现公告发布页面、公告范围控制、公告权限点。证据：`backend/app/core/authz_catalog.py:354`、`backend/app/core/authz_hierarchy_catalog.py:603`
- 未实现消息规则 / 订阅配置页面、接口与权限。证据：`backend/app/core/authz_catalog.py:354`
- 未实现完整的详情页或页面内详情预览区。证据：`frontend/lib/pages/message_center_page.dart:178`
- WS 不可用时只轮询未读数，不轮询消息列表，无法满足列表与角标长期一致性的要求。证据：`frontend/lib/pages/main_shell_page.dart:199`
- 未实现来源对象“无权限访问”和“对象已删除/归档”两类原因的细分提示。证据：`frontend/lib/pages/message_center_page.dart:594`、`frontend/lib/pages/main_shell_page.dart:619`
- 虽有 `status`、`expires_at` 字段，但未见完整的失效判定、保留期限清理与未读统计同步策略。证据：`backend/app/models/message.py:40`、`backend/app/services/message_service.py:177`

## 5. 明显偏差

- 后端写入的 `target_page_code` 有多处实际上是页签代码，前端导航却只识别侧边栏页面，导致消息跳转无法正确进入具体业务页签。证据：`backend/app/services/assist_authorization_service.py:308`、`backend/app/services/production_repair_service.py:558`、`frontend/lib/pages/main_shell_page.dart:259`
- 质量模块消息把 `target_page_code` 写成 `quality_first_articles`，但页面目录实际代码并非该值，属于确定性错误。证据：`backend/app/api/v1/endpoints/quality.py:178`、`backend/app/core/page_catalog.py:43`
- 未读角标与消息列表可能不一致：未读数只统计 `active` 消息，而列表查询未限制 `active`。证据：`backend/app/services/message_service.py:103`、`backend/app/services/message_service.py:54`
- 已读状态变化没有通过 WS 回推给同用户其他在线端，更多依赖轮询。证据：`backend/app/api/v1/endpoints/messages.py:71`、`backend/app/services/message_push_service.py:8`

## 6. 优先级建议

1. 优先打通“消息 -> 目标页面/页签/上下文”的跳转链路，并修正错误的页面编码。
2. 然后补已读状态实时同步、列表级轮询兜底，再补公告发布与规则配置能力。
