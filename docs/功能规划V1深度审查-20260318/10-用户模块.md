# 用户模块深度检查

## 1. 总体判断

- 结论：主体功能已具备，但关键规则未闭环。
- 核心风险：自定义角色无法真正分配给用户；权限配置缺少系统管理员保底规则；部分导出与筛选、会话展示与需求存在偏差。

## 2. 已满足项

- 需求中的 7 个页面均已落地：用户管理、注册审批、角色管理、功能权限配置、审计日志、个人中心、登录日志/在线会话。证据：`frontend/lib/pages/user_management_page.dart`、`frontend/lib/pages/registration_approval_page.dart`、`frontend/lib/pages/role_management_page.dart`、`frontend/lib/pages/function_permission_config_page.dart`、`frontend/lib/pages/audit_log_page.dart`、`frontend/lib/pages/account_settings_page.dart`、`frontend/lib/pages/login_session_page.dart`
- 用户管理主链路较完整，已支持搜索、筛选、新建、编辑、启停、重置密码、逻辑删除、导出。证据：`backend/app/services/user_service.py:90`、`backend/app/services/user_service.py:434`、`backend/app/api/v1/endpoints/users.py:286`
- 操作员必须绑定工段、非操作员不可编辑工段、用户名唯一等关键规则已实现。证据：`backend/app/services/user_service.py:215`、`backend/app/services/user_service.py:446`、`backend/app/services/user_service.py:680`
- 注册审批已支持通过、驳回、驳回原因，并在通过时由管理员设置初始密码。证据：`backend/app/api/v1/endpoints/auth.py:252`、`frontend/lib/pages/registration_approval_page.dart:316`
- 首次登录强制改密已打通，新建用户、审批通过、重置密码后均会要求改密。证据：`backend/app/services/user_service.py:463`、`backend/app/services/user_service.py:599`、`backend/app/services/user_service.py:749`、`frontend/lib/pages/force_change_password_page.dart:89`
- 角色管理已覆盖内置角色、自定义角色、禁删禁改名、绑定用户不可删等要求。证据：`backend/app/core/rbac.py:1`、`backend/app/services/role_service.py:74`
- 审计日志、本人改密、强制下线、登录日志与在线会话均已有前后端实现。证据：`backend/app/models/audit_log.py:9`、`backend/app/api/v1/endpoints/sessions.py:117`、`frontend/lib/pages/login_session_page.dart:488`

## 3. 部分满足项

- 工段“实时查询”只做到页面初始化加载与提交时后端校验，前端下拉项本身不是每次打开弹窗即时拉取。证据：`frontend/lib/pages/user_management_page.dart:202`、`frontend/lib/pages/registration_approval_page.dart:155`
- 用户导出未带在线状态筛选，导出结果可能与当前页面筛选不一致。证据：`frontend/lib/pages/user_management_page.dart:998`、`frontend/lib/services/user_service.dart:63`、`backend/app/api/v1/endpoints/users.py:150`
- 功能权限配置已支持按模块、按角色保存，但粒度更接近“能力包”而非需求中列出的“页面查看 / 新建 / 编辑 / 删除 / 审批 / 配置”。证据：`backend/app/core/authz_hierarchy_catalog.py:62`、`frontend/lib/pages/function_permission_config_page.dart:625`
- 注册审批页前端固定只查看 `pending` 状态，无法在同页查看已通过或已驳回历史。证据：`frontend/lib/pages/registration_approval_page.dart:156`、`backend/app/services/user_service.py:501`
- 角色管理、登录日志、在线会话更偏卡片/列表形式，未完全符合需求中的“表头式列表”描述。证据：`frontend/lib/pages/role_management_page.dart:413`、`frontend/lib/pages/login_session_page.dart:354`

## 4. 未满足项

- 自定义角色可创建、可配置权限，但无法在用户新建、编辑、注册审批时真正分配，这是当前模块最严重的闭环缺口。证据：`backend/app/services/role_service.py:74`、`backend/app/services/user_service.py:64`、`frontend/lib/pages/user_management_page.dart:436`
- 未实现“至少保留一个可进入功能权限配置页面的系统管理员账号”的保底规则。证据：`backend/app/services/user_service.py:307`、`backend/app/services/authz_service.py:2050`
- 也未强制保证“系统管理员固定拥有功能权限配置页面访问/编辑权限”。证据：`backend/app/services/authz_service.py:60`、`backend/app/services/authz_service.py:1860`

## 5. 明显偏差

- 前端删除用户文案提示“不可恢复”，但后端实际是逻辑删除/停用。证据：`frontend/lib/pages/user_management_page.dart:831`、`backend/app/services/user_service.py:790`
- 当前实现把 `process_codes` 引入到用户创建、编辑、审批流程中，超出了需求文档仅要求“角色 + 工段”的范围。证据：`backend/app/schemas/user.py:14`、`frontend/lib/pages/user_management_page.dart:476`
- 在线会话页直接显示原始状态值 `active`，而不是需求中的“在线/离线”；且接口只返回在线会话，不提供离线视图。证据：`backend/app/services/session_service.py:145`、`frontend/lib/pages/login_session_page.dart:455`

## 6. 优先级建议

1. 先修复“自定义角色无法分配”与“系统管理员权限保底”两项规则，否则用户模块无法按需求验收。
2. 再处理权限粒度、导出筛选一致性、会话状态文案与页面展示形式。
