# 工艺模块整改细化

## 1. 目标

- 将 `docs/功能规划V1深度审查-20260318/12-工艺模块.md` 中的审查结论，细化为“分批次整改计划 + 数据表/接口/页面影响范围 + 验收标准”。
- 本文先作为工艺模块本轮整改的执行蓝图，后续按批次逐项落地。
- 设计原则：
  - 以需求说明为准；
  - 优先闭合历史引用、发布约束和跨模块复用能力；
  - 默认直接替换旧交互和旧契约，不保留向后兼容壳层。

## 2. 当前核心缺口

### 2.1 P0 缺口

1. 未提供工段/工序轻量查询接口，难以支撑用户、设备、生产模块标准复用。
2. 历史版本引用未完整进入删除保护与引用分析，导致“已完成订单 / 历史版本”场景下仍可能误删工段、工序、模板。
3. 已发布模板仍可直接编辑当前实体，不符合“已发布模板不允许直接覆盖编辑”的要求。
4. 模板复制、系统母版套版、跨产品复制未记录来源元数据，变更追溯不完整。

### 2.2 P1 缺口

1. 模板/版本导出能力不完整，缺少单模板导出、模板版本导出。
2. 引用分析页面虽然展示了 `jump_target`，但前端没有真正跳转。
3. 步骤扩展字段 `标准工时`、`关键工序标识`、`步骤说明` 尚未落地。
4. 工艺模块几乎没有测试资产，缺少后端集成测试与前端回归测试。

## 3. 总体策略

### 3.1 推荐分批次顺序

1. 批次 A：轻量查询接口、步骤扩展字段、模板来源留痕。
2. 批次 B：已发布模板编辑约束、删除保护与历史版本引用补齐。
3. 批次 C：导出、引用分析跳转、版本展示与前端一致性收口。
4. 批次 D：补测试、联调回归、清理遗留旧逻辑并提交。

### 3.2 执行原则

- 工艺模块中的“当前模板行”仍可承载草稿态，但已发布态不得直接进入可编辑状态；必须先创建草稿或基于已发布版本生成新草稿。
- 历史版本引用需要与当前模板引用同级对待，尤其是删除和引用分析场景。
- 对外轻量查询接口必须稳定、只返回最小必要字段，且默认仅返回启用对象。

## 5. 批次 A：轻量查询接口、步骤扩展字段、来源留痕

### 5.1 目标

- 为跨模块复用打基础，同时补齐模板步骤主数据缺失字段。

### 5.2 后端改造点

#### C-01 轻量查询接口

- 新增接口建议：
  - `GET /craft/stages/light`
  - `GET /craft/processes/light`
- 返回字段最小集：
  - 工段：`id/code/name/sort_order/is_enabled`
  - 工序：`id/code/name/stage_id/stage_code/stage_name/is_enabled`
- 默认仅返回启用项，支持 `enabled` 显式透传。

#### C-02 步骤扩展字段

- 需扩展表：
  - `mes_product_process_template_step`
  - `mes_product_process_template_revision_step`
  - `sys_craft_system_master_template_step`
  - `sys_craft_system_master_template_revision_step`
- 新增字段建议：
  - `standard_minutes`：整数，默认 0
  - `is_key_process`：布尔，默认 false
  - `step_remark`：字符串，默认空

#### C-03 来源留痕

- 需扩展表：`mes_product_process_template`
- 新增字段建议：
  - `source_type`：`manual/system_master/template/cross_product_template`
  - `source_template_id`
  - `source_template_name`
  - `source_product_id`
  - `source_system_master_version`

### 5.3 前端改造点

- `frontend/lib/models/craft_models.dart`
  - 步骤模型补 `standardMinutes`、`isKeyProcess`、`stepRemark`
  - 模板模型补来源字段
- `frontend/lib/pages/process_configuration_page.dart`
  - 系统母版步骤编辑、新建模板、编辑模板、复制模板都支持维护上述步骤扩展字段
  - 模板详情和版本详情展示来源信息与步骤扩展字段
- `frontend/lib/services/craft_service.dart`
  - 修复现有可疑语法写法
  - 新增轻量查询与新字段序列化/反序列化

### 5.4 批次 A 验收标准

- 外部模块可以通过轻量接口获取启用工段/工序。
- 模板步骤和系统母版步骤可维护标准工时、关键工序标识、步骤说明。
- 新建复制模板后，可以追溯其来源模板或来源母版。

## 6. 批次 B：已发布只读、删除保护与历史版本引用补齐

### 6.1 目标

- 封住工艺模块当前最危险的两个点：已发布模板可直接编辑、历史引用未纳入保护。

### 6.2 后端改造点

#### C-04 已发布只读

- `update_template()` 不再允许直接修改 `published/archived` 模板。
- 推荐策略：
  - 若模板当前为 `draft`，允许直接编辑；
  - 若模板当前为 `published/archived`，必须通过“创建草稿版本”能力或复制生成草稿后再编辑。
- 可新增接口：
  - `POST /craft/templates/{template_id}/draft-from-published`

#### C-05 删除保护补齐

- 工段删除保护需补：
  - `mes_product_process_template_revision_step`
  - `sys_craft_system_master_template_revision_step`
  - 已完成订单 `mes_order_process`
  - 设备/质量/维修等历史引用
- 工序删除保护同理。
- 模板删除保护需补：
  - 所有状态订单引用，而非仅未完成订单
  - 历史版本引用保留场景

#### C-06 引用分析补齐

- 引用分析结果中应增加“历史版本引用”项：
  - `system_master_revision`
  - `template_revision`
  - `completed_order`
  - `historical_quality_reference`

### 6.3 前端改造点

- `frontend/lib/pages/process_configuration_page.dart`
  - 已发布/已归档模板不再显示“编辑”入口，改为“创建草稿”或“复制模板”
  - 删除提示框中展示引用分析摘要
- `frontend/lib/pages/process_management_page.dart`
  - 删除工段/工序前可直接跳入引用分析详情，且风险说明包含历史版本引用来源

### 6.4 批次 B 验收标准

- 已发布模板不能被直接覆盖编辑。
- 历史版本、已完成订单、质量/维修历史引用均会阻止误删。
- 引用分析结果能区分当前引用与历史引用风险。

## 7. 批次 C：导出、跳转、展示一致性收口

### 7.1 目标

- 补齐用户在页面上能直接感知的缺口，让工艺模块可真正验收。

### 7.2 核心任务

#### C-07 导出闭环

- 补充能力：
  - 单模板导出
  - 模板版本导出
  - 系统母版当前版本导出
  - 系统母版历史版本导出
- 导出内容需包含步骤扩展字段与来源留痕。

#### C-08 跳转闭环

- `frontend/lib/pages/craft_reference_analysis_page.dart` 不再只弹 SnackBar。
- 需要实现到以下模块/页面的跳转承接：
  - 工艺：工段/工序管理页、模板配置页
  - 用户：用户管理页
  - 产品：产品管理页
  - 生产：工单页/查询页
  - 设备：保养计划/工单页

#### C-09 展示一致性

- 模板列表排序改为更新时间倒序。
- 版本文案统一为用户可读格式。
- 系统母版区补完整步骤列表展示。
- 回滚前也展示影响分析摘要，而不只是确认框。

### 7.3 批次 C 验收标准

- 所有导出能力可直接从页面触发并落地。
- 引用分析页点击可真正跳转。
- 模板列表、详情、版本对比、导出中的字段和文案保持一致。

## 8. 批次 D：测试、联调、清理、提交

### 8.1 目标

- 用测试和联调把工艺模块本轮整改真正封口。

### 8.2 核心任务

1. 新增后端集成测试，至少覆盖：
   - 轻量查询接口
   - 已发布模板禁止直接编辑
   - 创建草稿/复制模板来源留痕
   - 工段/工序删除被历史版本引用阻止
   - 模板版本导出内容正确
2. 新增前端回归测试，至少覆盖：
   - 模板页已发布模板编辑限制
   - 引用分析页跳转承接
   - 步骤扩展字段保存与展示
3. 开发库联调：
   - 工段/工序 -> 模板配置 -> 发布 -> 回滚 -> 引用分析 -> 导出 全链路验证
4. 更新 `docs` 日志并提交工艺模块改动。

### 8.3 批次 D 验收标准

- 后端集成测试与前端回归测试通过。
- 真实接口联调通过，测试数据清理完成。
- 工艺模块所有本轮整改项进入可提交状态。

## 10. 最终建议

- 工艺模块最重要的不是再加一个页面，而是让“已发布模板不可误改、历史引用不可误删、引用分析可真实跳转、导出与版本可真实追溯”四条主链闭环。
- 在这四条闭环完成前，不建议把工艺模块判定为可正式验收。
- 下一步应按本计划直接进入批次 A 编码，然后连续推进到批次 D，最后统一联调、提交。
