# 生产模块整改细化

## 1. 目标

- 将 `docs/功能规划V1深度审查-20260318/15-生产模块.md` 中的审查结论，细化为“分批次整改计划 + 数据表/接口/页面影响范围 + 验收标准”。
- 本文先作为生产模块本轮整改的执行蓝图，后续按批次逐项落地。
- 设计原则：
  - 以需求说明为准；
  - 优先闭合数量口径、删除追溯、维修回流和导出一致性；
  - 默认直接替换旧文案和旧契约，不长期保留错误语义。

## 2. 当前核心缺口

### 2.1 P0 缺口

1. 报工时只校验报工数量，未把异常数量一并纳入可生产数量校验。
2. 删除订单会写 `order_deleted` 事件，但事件日志随订单级联删除，无法保留历史。
3. 维修完成前端只支持单回流目标，而后端已支持多回流分配。
4. 导出和页面筛选存在不一致，维修订单导出未完整带时间范围，订单导出只提示字节数不形成闭环。

### 2.2 P1 缺口

1. 代班状态文案为“已生效”，与需求“已审批”不一致。
2. 报废统计筛选文案为“待申请 / 已申请”，与需求“待处理 / 已处理”不一致。
3. 删除保护与事件详情对历史追溯字段不够明确。
4. 后端测试资产不足，需要补生产模块回归测试。

## 3. 总体策略

### 3.1 推荐分批次顺序

1. 批次 A：数量口径、订单删除追溯、基础契约收敛。
2. 批次 B：执行/代班/并行/维修关键规则闭环。
3. 批次 C：查询/导出/详情/事件日志一致性收口。
4. 批次 D：补测试、联调回归、清理遗留逻辑并提交。

### 3.2 执行原则

- 订单、工序、维修、报废相关数量都必须围绕统一数量口径计算。
- 删除订单不允许再让审计/事件记录被级联删掉，必须保留快照型历史。
- 维修回流以前端和后端都支持“多条回流分配”为准，不保留单回流假设。

## 4. 分批次总表

| 编号 | 批次 | 优先级 | 目标 | 核心结果 |
| --- | --- | --- | --- | --- |
| P-01 | A | P0 | 报工数量口径统一 | 报工时把异常数量一并纳入可生产数量校验 |
| P-02 | A | P0 | 删除订单事件可追溯 | 订单删除后事件日志仍保留快照 |
| P-03 | A | P1 | 基础文案/状态收敛 | 代班、报废文案与契约统一 |
| P-04 | B | P0 | 多回流分配前端闭环 | 维修完成支持多条回流分配 |
| P-05 | B | P0 | 维修/报废链路一致 | 维修完成、报废补充、回流更新与事件记录一致 |
| P-06 | B | P1 | 并行与代班细节收口 | 并行实例、代班视图、审批文案与操作链路一致 |
| P-07 | C | P1 | 导出与详情一致 | 订单、维修、报废导出都跟随筛选并形成完整导出闭环 |
| P-08 | C | P1 | 详情与事件一致 | 订单详情、执行详情、事件日志展示和快照字段统一 |
| P-09 | D | P1 | 测试与提交 | 新增回归测试、联调、日志更新、提交 |

## 5. 批次 A：数量口径、订单删除追溯、基础契约收敛

### 5.1 目标

- 封住最影响验收的两个问题：报工数量口径错误、删除订单无法追溯。

### 5.2 后端改造点

#### P-01 报工数量口径统一

- `production_execution_service.end_production()` 改为校验：
  - `quantity > 0`
  - `defect_quantity >= 0`
  - `quantity + defect_quantity <= max_producible`
- 维修建单 `production_quantity` 与报工记录、事件日志口径保持一致。

#### P-02 删除订单事件可追溯

- `mes_order_event_log.order_id` 不再对订单使用 `CASCADE` 删除。
- 推荐方案：
  - 改为 `SET NULL`；
  - 并在日志表补 `order_code_snapshot`、`order_status_snapshot`、`product_name_snapshot` 等快照字段。
- 删除订单时，先写快照日志，再删除订单主体。

#### P-03 基础文案/状态收敛

- 代班状态 `approved` 统一展示为“已审批”。
- 报废统计进度统一展示为“待处理 / 已处理”。

### 5.3 前端改造点

- `frontend/lib/models/production_models.dart`
  - 收敛代班与报废状态文案。
- `frontend/lib/pages/production_order_query_detail_page.dart`
  - 报工前端校验与报错信息同步后端新口径。

### 5.4 批次 A 验收标准

- 报工时若“报工数 + 异常数”超过可生产数量，会被拒绝。
- 删除待生产订单后，事件日志仍可查到 `order_deleted` 历史。

## 6. 批次 B：执行/代班/并行/维修关键规则闭环

### 6.1 目标

- 把执行侧和异常维修主链路彻底打通。

### 6.2 核心任务

#### P-04 多回流分配前端闭环

- `frontend/lib/pages/production_repair_orders_page.dart`
  - 完成维修弹窗支持多条 `return_allocations`
  - 支持按回流目标工序拆分数量
  - 前端校验总回流数量必须等于可回流数量

#### P-05 维修/报废链路一致

- 维修完成后：
  - 回流、报废补充、事件日志、维修详情快照保持一致
  - 维修订单详情中展示多回流分配结果

#### P-06 并行与代班细节收口

- 并行实例页面补齐与需求更一致的筛选和展示字段
- 代班审批状态与审批动作文案统一

### 6.3 批次 B 验收标准

- 维修完成支持多条回流分配且总量校验正确。
- 代班和并行链路无文案歧义，关键页面行为一致。

## 7. 批次 C：查询/导出/详情/事件日志一致性收口

### 7.1 目标

- 让生产模块用户侧能看到完整、一致、可导出的数据。

### 7.2 核心任务

#### P-07 导出与详情一致

- 订单导出：前端不再只提示字节数，应形成完整导出闭环。
- 维修订单导出：时间范围、状态、关键词都必须跟当前筛选一致。
- 报废统计导出：字段、进度文案、筛选条件与页面一致。

#### P-08 详情与事件一致

- 订单详情、执行详情、维修详情、报废详情统一使用快照字段。
- 事件日志中关键动作的 `payload_json` 需补关键数量和快照信息。

### 7.3 批次 C 验收标准

- 导出内容与当前页面筛选一致。
- 详情页与事件日志在关键字段上保持一致，不依赖已删除主对象才能阅读。

## 8. 批次 D：测试、联调、清理、提交

### 8.1 目标

- 用测试和联调把生产模块本轮整改封口。

### 8.2 核心任务

1. 新增后端集成测试，至少覆盖：
   - 报工数量口径校验
   - 删除订单后事件日志保留
   - 维修完成多回流分配
   - 导出筛选一致性
2. 新增前端回归测试，至少覆盖：
   - 维修多回流弹窗
   - 报废/代班文案
   - 订单导出闭环
3. 开发库联调：
   - 建单 -> 首件 -> 报工 -> 送修 -> 维修完成 -> 删除订单/查看日志 全链路验证
4. 更新 `docs` 日志并提交生产模块改动。

### 8.3 批次 D 验收标准

- 前后端测试通过。
- 真实接口联调通过，测试数据清理完成。
- 生产模块进入可提交状态。

## 10. 最终建议

- 生产模块最关键的是“数量口径闭环、删除追溯保留、多回流分配落地、导出与详情一致”四条主链。
- 在这四条主链完成前，不建议将生产模块判定为可正式验收。
- 下一步应按本计划直接进入批次 A 编码，再连续推进到批次 D，最后统一联调、提交。
