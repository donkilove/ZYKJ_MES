# 品质模块深度检查

## 1. 总体判断

- 结论：P0 业务骨架较多已落地，但处置闭环、趋势分析和统计口径一致性仍存在明显问题。
- 核心风险：报废导出会改处理进度；趋势与产品统计的聚合口径未严格跟随筛选条件；首件处置仍不完整。

## 2. 已满足项

- 每日首件已支持按日期、关键字、结果、产品、工序、操作员筛选，并支持刷新、分页、空状态和导出。证据：`frontend/lib/pages/daily_first_article_page.dart:186`、`backend/app/services/quality_service.py:77`
- 每日首件还已展示查询日期、当日校验码、校验码来源和总数。证据：`frontend/lib/pages/daily_first_article_page.dart:304`
- 品质数据页已实现总览、工序统计、人员统计、产品统计，并做了开始结束日期校验。证据：`frontend/lib/pages/quality_data_page.dart:410`、`backend/app/services/quality_service.py:233`
- 报废统计已支持列表、导出、详情查看，且详情中可看到关联维修单摘要。证据：`frontend/lib/pages/production_scrap_statistics_page.dart:236`、`backend/app/api/v1/endpoints/production.py:1415`
- 维修订单已支持列表、现象汇总、完成维修、导出、详情查看，后端也校验了数量守恒和回流合法性。证据：`frontend/lib/pages/production_repair_orders_page.dart:472`、`backend/app/services/production_repair_service.py:322`
- 已存在独立的质量趋势页和不良分析页。证据：`frontend/lib/pages/quality_trend_page.dart:83`、`frontend/lib/pages/quality_defect_analysis_page.dart:65`

## 3. 部分满足项

- 首件处置已有查看基础信息、录入处置意见、记录复检结果、记录最终判定的基础能力，但仍是弹窗式实现，没有独立页面，也没有处置日志列表。证据：`frontend/lib/pages/first_article_disposition_page.dart:54`、`backend/app/services/quality_service.py:818`
- 品质数据页已完成总览/工序/人员/产品统计，但未在同一页面内实现趋势分析区。证据：`frontend/lib/pages/quality_data_page.dart:467`
- 报废统计有详情能力，但没有产品筛选、工序筛选等更精确条件，详情也不是独立子页面。证据：`frontend/lib/pages/production_scrap_statistics_page.dart:361`
- 维修订单有详情能力，但同样以弹窗代替独立详情页，且前端导出未把时间范围一并传给后端。证据：`frontend/lib/pages/production_repair_orders_page.dart:136`、`frontend/lib/services/production_service.dart:863`
- 不良分析与质量趋势页已能查部分维度，但仍缺操作员筛选、不良类型筛选和导出接入。证据：`frontend/lib/pages/quality_defect_analysis_page.dart:136`、`frontend/lib/pages/quality_trend_page.dart:171`

## 4. 未满足项

- 品质数据页未按需求实现“趋势分析区”，只能通过独立趋势页查看。证据：`frontend/lib/pages/quality_data_page.dart:470`
- 首件处置未实现处置日志查看，无法满足完整留痕要求。证据：`backend/app/services/quality_service.py:418`、`frontend/lib/pages/first_article_disposition_page.dart:174`
- 不良分析缺少操作员、不良类型等关键筛选维度，也未实现 Top 缺陷原因排行、报废趋势、首件通过率趋势等需求项。证据：`backend/app/services/quality_service.py:900`、`frontend/lib/pages/quality_defect_analysis_page.dart:261`
- 质量趋势页没有导出能力；不良分析后端虽有导出接口，前端未接入。证据：`backend/app/api/v1/endpoints/quality.py:360`、`frontend/lib/services/quality_service.dart:439`

## 5. 明显偏差

- 报废“处理进度”的实现语义与需求不一致，且导出报废统计会直接把 `pending_apply` 改成 `applied`，等于“导出即处理”。证据：`backend/app/models/production_scrap_statistics.py:11`、`backend/app/services/production_repair_service.py:786`
- 首件详情入口被错误绑定到“处置权限”，只有处置权限的人才能看详情，这与独立的详情权限不一致。证据：`backend/app/api/v1/endpoints/quality.py:98`、`frontend/lib/pages/daily_first_article_page.dart:438`
- 首件处置未限制为“不通过记录”，当前看起来所有首件记录都可打开处置。证据：`backend/app/services/quality_service.py:818`、`frontend/lib/pages/daily_first_article_page.dart:441`
- 质量趋势中的报废数、维修数聚合未严格应用产品/工序/操作员筛选，口径与首件趋势不一致。证据：`backend/app/services/quality_service.py:623`、`backend/app/services/quality_service.py:677`
- 产品统计中的报废数、维修单数也未完整沿用工序、操作员、结果等筛选条件。证据：`backend/app/services/quality_service.py:515`、`backend/app/services/quality_service.py:552`

## 6. 优先级建议

1. 先修复报废进度语义和导出副作用，再修正质量趋势与产品统计的口径一致性。
2. 然后补齐首件处置日志、趋势分析区、不良分析筛选维度与导出闭环。
