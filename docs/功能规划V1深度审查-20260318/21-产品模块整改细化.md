# 产品模块整改细化

## 1. 目标

- 将 `docs/功能规划V1深度审查-20260318/11-产品模块.md` 中的审查结论，继续细化为“分批次整改计划 + 页面/接口/数据表影响范围 + 实施顺序”。
- 本文只给整改设计，不直接改动产品模块业务代码。
- 设计原则：以需求说明为准，优先修正错误主模型；无兼容要求时，建议直接替换旧契约，而不是继续保留 `product_id -> 当前参数表` 旧路径。

## 2. 现状核心问题

### 2.1 当前阻断点

1. 参数仍绑定 `product_id`，而不是绑定具体版本。证据：`backend/app/models/product_parameter.py:7`
2. 产品版本复制会先覆盖当前参数表，再创建草稿快照，导致“草稿未生效却污染当前参数”。证据：`backend/app/services/product_service.py:1023`
3. 生效版本参数没有真正只读约束，前端和后端仍可直接改动。证据：`frontend/lib/pages/product_parameter_management_page.dart:159`、`backend/app/services/product_service.py:786`
4. 参数查询页、详情页、导出接口多处写“生效版本参数”，但底层并不保证读取的是生效版本快照。证据：`docs/功能规划V1深度审查-20260318/11-产品模块.md:28`

### 2.2 结论

- 产品模块不能从页面细节开始补，必须先修正“产品版本 - 参数”的主数据模型。
- 在主模型重构完成前，版本、生效、查询、导出、历史追溯都只能算“表面可用”。

## 3. 总体策略

### 3.1 推荐分批次顺序

1. 批次 A：版本参数主模型与公开契约替换。
2. 批次 B：版本生命周期与并发生效规则闭环。
3. 批次 C：查询、导出、详情、历史追溯一致性收口。
4. 批次 D：旧表清理、补测、联调与回归验收。

### 3.2 推荐迁移策略

- 建议直接废弃“`mes_product_parameter` 作为当前参数主表”的公开含义，改为“版本参数表”作为唯一业务真源。
- 旧接口建议直接替换，不再保留“只传 `product_id` 即维护当前参数”的兼容模式。
- 迁移期间允许保留 `snapshot_json` 作为审计快照，但不应继续让它与业务真源双向互写。

## 4. 分批次总表

| 编号 | 批次 | 优先级 | 目标 | 核心结果 |
| --- | --- | --- | --- | --- |
| P-01 | A | P0 | 参数绑定版本 | 建立版本参数主表，所有参数维护改为按版本操作 |
| P-02 | A | P0 | 公开契约替换 | 参数管理/查询/导出接口显式带版本信息 |
| P-03 | B | P0 | 生效与复制闭环 | 复制不污染当前参数，生效只切换目标版本 |
| P-04 | B | P0 | 生效只读与并发保护 | 已生效版本只读，并发抢先生效可检测 |
| P-05 | C | P1 | 页面与查询统一 | 详情、查询、导出都严格读取目标版本参数 |
| P-06 | C | P1 | 历史与日志闭环 | 参数历史覆盖新增/编辑/删除/复制/生效/停用 |
| P-07 | C | P1 | 删除保护扩展 | 产品与版本删除覆盖工单、生产记录、质检记录等引用 |
| P-08 | D | P1 | 清理与验收 | 清理旧逻辑、补测试、联调与回归通过 |

## 5. 批次 A：版本参数主模型与公开契约替换

### 5.1 批次目标

- 让“参数属于版本”成为数据库层、接口层、页面层的一致事实。
- 从这一批次开始，`product_id` 只能表达产品身份，不能再直接代表“当前参数集合”。

### 5.2 数据表改造点

#### P-01 建立版本参数主表

- 推荐新增表：`mes_product_revision_parameter`
- 推荐字段：
  - `id`
  - `product_id`
  - `revision_id`
  - `version`
  - `param_key`
  - `param_category`
  - `param_type`
  - `param_value`
  - `param_description`
  - `sort_order`
  - `is_preset`
  - 时间戳字段
- 关键约束：
  - `uq_mes_product_revision_parameter_revision_id_param_key`
  - `revision_id` 外键指向 `mes_product_revision.id`
  - `product_id` 与 `revision_id` 必须一致，可用应用层校验或数据库约束保证

#### P-02 调整历史表归属

- 当前 `mes_product_parameter_history` 只围绕产品当前参数表工作，无法准确表达“版本内变更”。
- 推荐方案二选一：
  1. 扩展现有历史表，补 `revision_id`、`version`、`change_action`；
  2. 新增 `mes_product_revision_parameter_history`，后续完全迁移。
- 推荐直接采用方案 2，减少历史语义混乱。

#### P-03 旧表处理策略

- `mes_product_parameter` 不建议继续作为业务真源。
- 迁移策略：
  - 先把当前产品参数回填到各自的“当前可维护版本”；
  - 切换接口与前端后，将其降级为过渡表；
  - 批次 D 再决定删除或保留只读映射。

### 5.3 接口改造点

- 重点接口：`backend/app/api/v1/endpoints/products.py`
- 重点服务：`backend/app/services/product_service.py`
- 重点 Schema：`backend/app/schemas/product.py`

#### 推荐直接替换的接口语义

1. 参数列表查询：从“按产品取当前参数”改为“按产品版本取版本参数”。
2. 参数更新：从“覆盖产品当前参数”改为“仅允许更新草稿版本参数”。
3. 参数历史：从“产品级历史”改为“版本级历史，可带产品聚合视图”。
4. 参数导出：必须明确导出目标版本；若是参数查询页导出，则固定导出生效版本。

#### 推荐新增或重构的接口形状

- `GET /products/{product_id}/versions/{version}/parameters`
- `PUT /products/{product_id}/versions/{version}/parameters`
- `GET /products/{product_id}/versions/{version}/parameter-history`
- `GET /products/{product_id}/versions/{version}/parameters/export`
- `GET /products/{product_id}/effective-parameters`

### 5.4 页面改造点

- `frontend/lib/pages/product_parameter_management_page.dart`
  - 入口必须带“目标版本”；
  - 页面顶部明确展示“当前编辑版本”；
  - 非草稿版本时进入只读模式。
- `frontend/lib/pages/product_parameter_query_page.dart`
  - 固定读取生效版本；
  - 页面标题与导出都必须展示具体生效版本号。
- `frontend/lib/pages/product_management_page.dart`
  - “查看参数 / 编辑参数”动作需区分“生效版本只读”和“当前可维护版本编辑”。

### 5.5 批次 A 验收标准

- 任意一条参数都能明确追溯到唯一版本。
- 草稿参数编辑不会影响生效版本参数。
- 所有参数相关接口都不再接受“只给产品 ID 就覆盖当前参数”的旧模式。

## 6. 批次 B：版本生命周期与并发生效规则闭环

### 6.1 批次目标

- 让“新建版本、复制版本、立即生效、停用版本、删除版本”的行为真正围绕版本参数主表工作。

### 6.2 核心任务

#### P-03 复制版本不再污染当前参数

- 当前问题：`copy_product_version()` 会先恢复来源快照到 `mes_product_parameter`。证据：`backend/app/services/product_service.py:1023`
- 目标行为：
  - 复制版本时，只复制来源版本参数到新草稿版本参数表；
  - 不再改动其他版本参数；
  - 不再通过“覆盖当前参数 + 重新生成快照”实现复制。

#### P-04 生效操作原子切换

- 立即生效必须改成：
  - 校验产品启用；
  - 校验目标版本是草稿；
  - 校验目标版本参数非空；
  - 原子更新旧生效版本 -> 已失效、新草稿 -> 已生效、产品 `effective_version/effective_at`；
  - 不再依赖“当前参数表已是什么状态”。

#### P-05 并发抢先生效保护

- 推荐引入以下任一机制：
  1. `expected_effective_version` 条件更新；
  2. 基于 `product.updated_at` 的乐观锁；
  3. 对产品行做事务级锁。
- 推荐优先使用“条件更新 + 事务”，实现简单且提示清晰。

#### P-06 已生效版本只读

- 后端：非草稿版本参数更新接口直接拒绝。
- 前端：
  - 生效/失效/停用版本参数页全部只读；
  - 操作入口替换为“复制版本后编辑”。

### 6.3 页面影响

- `frontend/lib/pages/product_version_management_page.dart`
  - 版本列表操作菜单要与状态规则一致；
  - 增加“最近一次生效结果”查看区或弹窗。
- `frontend/lib/pages/product_parameter_management_page.dart`
  - 草稿版本显示“可编辑”；
  - 已生效版本显示“只读，需复制新版本后调整”。

### 6.4 批次 B 验收标准

- 复制版本后，生效版本参数完全不变。
- 已生效版本参数无法直接修改。
- 两个用户并发点“立即生效”时，后者会收到明确冲突提示，而不是静默覆盖。

## 7. 批次 C：查询、导出、详情、历史追溯一致性收口

### 7.1 批次目标

- 让“产品详情、参数查询、导出、历史记录、版本展示”全部围绕同一版本事实工作。

### 7.2 核心任务

#### P-05 详情与查询统一读取逻辑

- `frontend/lib/pages/product_management_page.dart`
  - 当前生效参数快照区必须固定读取生效版本参数；
  - 版本概览区的“当前版本 / 生效版本 / 当前版本状态”需按需求重新计算。
- `frontend/lib/pages/product_parameter_query_page.dart`
  - 只读查询必须固定走“生效版本参数接口”。

#### P-06 导出与页面一致

- 产品列表导出、版本参数导出、参数查询导出都要显示明确版本号。
- 同一条版本记录在列表、详情、导出中不得出现不同版本文案规则。

#### P-07 历史记录补齐动作谱系

- 参数历史至少覆盖：
  - 新增参数
  - 编辑参数
  - 删除参数
  - 复制版本
  - 立即生效
  - 停用版本
  - 删除草稿版本
- 建议把“参数历史”和“版本事件”区分成两个流：
  - 参数级历史：字段级变化
  - 版本级事件：复制/生效/停用/删除/回滚

#### P-08 删除保护扩展

- 当前删除保护主要看未完成工单。证据：`docs/功能规划V1深度审查-20260318/11-产品模块.md:24`
- 需要扩展到：
  - 生产记录
  - 质检记录
  - 维修记录/报废统计
  - 其他依赖产品或生效版本的业务单据

### 7.3 页面影响

- `frontend/lib/pages/product_management_page.dart`
- `frontend/lib/pages/product_version_management_page.dart`
- `frontend/lib/pages/product_parameter_management_page.dart`
- `frontend/lib/pages/product_parameter_query_page.dart`
- `frontend/lib/services/product_service.dart`
- `frontend/lib/models/product_models.dart`

### 7.4 批次 C 验收标准

- “详情页当前生效参数”“参数查询页”“导出结果”三者数据完全一致。
- 变更历史能清楚回答“谁在什么时间改了哪个版本的哪些参数”。
- 删除、停用、生效等关键操作的审计记录可追溯。

## 8. 批次 D：旧逻辑清理、补测、联调与回归验收

### 8.1 目标

- 清掉旧语义残留，避免产品模块再次出现“双真源”。

### 8.2 核心任务

1. 清理旧 `product_id -> 当前参数表` 读写分支。
2. 评估 `snapshot_json` 是否保留为只读审计快照；若保留，要明确“不能回写业务主数据”。
3. 补后端最小测试集：
   - 新建产品自动生成草稿版本
   - 复制版本不污染生效版本参数
   - 非草稿版本参数不可编辑
   - 并发生效冲突
   - 参数查询固定读取生效版本
4. 补前端最小测试集：
   - 参数页版本切换只读/可编辑状态
   - 详情页与查询页版本展示一致
   - 导出带版本号
5. 完成联调：
   - 产品管理 -> 版本管理 -> 参数管理 -> 生效 -> 查询 -> 导出 全链路回归。

### 8.3 批次 D 验收标准

- 代码中不再存在“产品当前参数表”作为业务真源的写路径。
- 前后端测试与联调通过后，再进入产品模块正式验收。

## 10. 推荐执行顺序

### 第一步：先做批次 A

- 原因：这是全模块阻断项；若不先改主模型，后续每一项修复都会继续建立在错误前提上。

### 第二步：紧接批次 B

- 原因：主模型切换后，版本复制、生效、只读和并发保护必须马上闭环，否则上线风险很高。

### 第三步：再做批次 C

- 原因：页面、查询、导出、历史的一致性要建立在已稳定的版本真源之上。

### 第四步：最后做批次 D

- 原因：清理旧逻辑与补测试应放在主链路稳定后完成，避免重复返工。

## 11. 最终建议

- 产品模块不要沿用“当前参数表 + 版本快照”双真源模式，建议直接切为“版本参数表为唯一真源，快照只做审计或彻底移除”。
- 建议默认采用破坏性替换：旧参数接口、旧前端模型、旧导出逻辑直接切掉，不保留兼容层。
- 在真正开始写代码前，建议先确认“版本参数表”与“版本参数历史表”的最终物理设计，这将决定后续改造成本。
- 若你确认按本方案推进，下一步应直接进入产品模块批次 A 设计落地，而不是先修页面细节。
