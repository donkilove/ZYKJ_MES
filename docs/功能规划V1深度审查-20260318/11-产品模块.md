# 产品模块深度检查

## 1. 总体判断

- 结论：页面和接口不少，但核心模型偏离需求，当前不建议按需求说明直接验收。
- 核心风险：参数并未绑定具体版本；已生效版本参数可被直接改动；查询与导出展示的也不一定是真正生效版本参数。

## 2. 已满足项

- 产品管理、版本管理、产品参数管理、产品参数查询 4 个页面均已存在，并由产品模块页统一承载。证据：`frontend/lib/pages/product_page.dart:13`
- 产品管理已支持名称、分类、状态、生效版本状态、更新时间范围筛选，以及新建、编辑、启停、删除、版本管理、参数查看、参数编辑、导出。证据：`frontend/lib/pages/product_management_page.dart:1866`、`backend/app/services/product_service.py:294`
- 产品名称唯一、分类固定枚举、新建自动生成 `V1.0` 草稿版本等规则已实现。证据：`backend/app/schemas/product.py:14`、`backend/app/services/product_service.py:481`
- 版本管理已支持新建版本、复制版本、立即生效、停用版本、删除版本、版本参数导出。证据：`backend/app/api/v1/endpoints/products.py:432`、`frontend/lib/pages/product_version_management_page.dart:534`
- 已实现“同一产品仅允许一个草稿版本”“生效时产品需启用且版本必须有参数”等关键校验。证据：`backend/app/services/product_service.py:977`、`backend/app/services/product_service.py:1064`
- 参数管理已支持新增、编辑、删除、保存修改原因、查看历史，`Link` 类型格式也有后端校验。证据：`frontend/lib/pages/product_parameter_management_page.dart:623`、`backend/app/services/product_service.py:106`
- 关键操作已写审计日志。证据：`backend/app/api/v1/endpoints/products.py:151`、`backend/app/api/v1/endpoints/products.py:544`

## 3. 部分满足项

- 产品详情页已有基础信息、版本概览、参数搜索与部分变更记录，但“变更记录区”主要接了参数历史，未完整覆盖产品编辑、启停、生效等全量变更。证据：`frontend/lib/pages/product_management_page.dart:1652`
- 新增/编辑产品表单缺少显式“默认状态”字段，当前是后端默认启用。证据：`frontend/lib/pages/product_management_page.dart:1231`、`backend/app/services/product_service.py:490`
- 版本管理只做了“编辑备注”式的轻量编辑，未见独立“查看最近一次生效结果”能力。证据：`frontend/lib/pages/product_version_management_page.dart:344`
- 参数管理页的筛选更接近按产品筛，不是真正按“产品版本参数”做查询。证据：`frontend/lib/pages/product_parameter_management_page.dart:276`
- 删除保护目前主要覆盖未完成工单，对需求中提到的质检记录、生产记录等更广泛引用场景覆盖不足。证据：`backend/app/services/product_service.py:658`

## 4. 未满足项

- `ProductParameter` 模型只有 `product_id`，没有 `version_id` 或 `revision_id`，这与“参数必须归属于具体产品版本”直接冲突。证据：`backend/app/models/product_parameter.py:7`
- 参数查询、更新、导出都只按 `product_id` 操作，而不是按版本操作。证据：`backend/app/api/v1/endpoints/products.py:250`、`backend/app/services/product_service.py:649`
- 已生效版本参数并非只读，前端可直接编辑，后端也会直接覆盖当前参数表。证据：`frontend/lib/pages/product_parameter_management_page.dart:159`、`backend/app/services/product_service.py:786`
- 复制版本时会先把来源版本快照恢复到当前参数表，草稿还未生效就已污染当前参数。证据：`backend/app/services/product_service.py:1023`
- 参数查询页与产品详情页标题写“生效版本参数”，但实际读取的是当前参数表，不保证真的是生效版本快照。证据：`frontend/lib/pages/product_parameter_query_page.dart:329`、`frontend/lib/pages/product_management_page.dart:1729`
- 版本生效缺少并发防覆盖保护，未体现需求中的抢先生效冲突校验。证据：`backend/app/services/product_service.py:1056`
- 参数变更历史未完整覆盖“新增 / 编辑 / 删除 / 生效 / 停用”的动作谱系。证据：`backend/app/schemas/product.py:103`、`backend/app/services/product_service.py:806`

## 5. 明显偏差

- 不同页面和导出对版本号的展示规则不一致，同一数据会出现 `V1.0`、`V1.1` 等不同文案。证据：`frontend/lib/pages/product_management_page.dart:141`、`frontend/lib/pages/product_parameter_query_page.dart:355`、`backend/app/api/v1/endpoints/products.py:804`
- 版本页把 `updatedAt` 当成“生效时间”展示，备注编辑后也会污染该时间含义。证据：`frontend/lib/pages/product_version_management_page.dart:618`、`backend/app/models/base.py:28`
- 产品详情对当前版本状态的判断过于粗糙，只用“是否等于生效版本”区分，无法正确表达已失效/已停用。证据：`frontend/lib/pages/product_management_page.dart:1713`

## 6. 优先级建议

1. 先重构“产品版本 - 参数”数据模型，再谈版本、生效、查询、导出和追溯整改。
2. 在模型整改完成前，当前产品模块只能视为页面骨架与部分流程存在，不能视为满足 V1 需求说明。
