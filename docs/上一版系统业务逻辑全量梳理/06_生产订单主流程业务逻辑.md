## 2) 业务对象与状态定义

### 2.1 核心业务对象

| 对象 | 表 | 作用 |
|---|---|---|
| 订单 | `orders` | 承载订单主信息与全局状态 |

## 4) 端到端主流程（建单→首件→开工→报工→流转→完工）

以下每条规则统一按  
`触发条件 → 判定逻辑 → 状态变化 → 数据写入 → UI结果 → 通知结果 → 代码定位`  
描述。

### 4.1 规则 A：创建订单（带工序）

- 触发条件：生产管理员在新增订单窗口提交订单，且包含工序列表。
- 判定逻辑：校验订单号/数量/日期/产品存在与唯一性；工序按提交顺序落库。
- 状态变化：订单初始为 `待生产`；全部工序初始 `待生产`；第一工序 `visible_quantity=订单总数`。
- 数据写入：写入 `orders`、`order_processes`、`process_status`、`order_history`；随后创建首工序子订单。
- UI结果：生产订单管理页可见新订单；生产订单查询页由角色可见性规则决定显示。
- 通知结果：第一工序角色收到“新订单创建”通知；首工序角色用户收到“新工序可生产”通知。
- 代码定位：
  - `src/ui/mini_page/add_order_window.py:611`

### 4.2 规则 B：创建订单（无工序）

- 触发条件：控制器新增订单时 `processes` 为空。
- 判定逻辑：仅创建订单主数据。
- 状态变化：订单进入 `待生产`，无工序流转能力。
- 数据写入：写入 `orders`、`order_history`。
- UI结果：订单管理页可见，查询页是否可操作取决于工序与角色逻辑。
- 通知结果：通知所有管理员“无工序订单已创建”。

  - `src/service/order_service.py:91`

### 4.3 规则 C：编辑订单

- 触发条件：订单管理页点击“编辑”。
- 判定逻辑：UI层先限制仅 `待生产` 可编辑；服务层再次校验订单、产品、编号唯一性。
- 状态变化：订单主信息与工序重建；工序状态重置为 `待生产`。
- 数据写入：更新 `orders`，删除并重建 `order_processes`/`process_status`，写 `order_history`。
- UI结果：编辑成功后列表刷新。
- 通知结果：无专门通知。
- 代码定位：
  - `src/ui/son_page/production_order_management_page.py:278`

### 4.4 规则 D：删除订单

- 触发条件：订单管理页点击“删除订单”并在删除弹窗确认。
- 判定逻辑：仅 `生产管理员` 可删；需密码验证通过。
- 状态变化：订单及大部分关联数据物理删除。
- 数据写入：事务内删除子订单、生产记录、工序状态、工序、历史、订单等。
- UI结果：删除成功后管理页刷新。
- 通知结果：无专门通知。
- 代码定位：

### 4.5 规则 E：首件检验完成并开工

- 触发条件：查询页点击“首件检验/开始首件”，在首件窗口提交。
- 判定逻辑：校验码先校验当日正确性，再执行“工序可用校验”（当前实现允许同日重复）。
- 状态变化：首件记录落库后，调用开工逻辑；工序状态变 `进行中`；订单状态变 `生产中`；`ongoing_users` 加入操作员。
- 数据写入：`first_article_records`；`order_processes`/`orders`/`process_status` 更新；尝试更新用户子订单为 `生产中`。
- UI结果：首件窗口成功提示并关闭，父页面延迟刷新。
- 通知结果：管理员收到“首件开始”通知。

### 4.6 规则 F：结束生产（订单分配模式）

- 触发条件：查询页“结束生产”打开结束生产窗口并确认数量。
- 判定逻辑：
  - 数量必须 `>0` 且 `<= 用户可生产数量`；
  - `用户可生产数量 = max(工序总可生产量 - 除本人外生产中用户数, 0)`；
  - 若当前用户子订单处于 `生产中` 且工序总可生产量 `>0`，则用户可输入量至少为 `1`（避免进行中用户被并发占满后无法结束本轮）。
- 状态变化：
  - 工序累计完成增加；
  - 若达到上工序可供给量则该工序 `生产完成`，否则保持 `生产中`；
  - 订单状态在“未最后工序”场景回到 `待生产`；仅当“最后工序累计完成量=订单总量”时订单进入 `生产完成`；
  - 子订单根据剩余可生产量在 `待生产/已完成` 间切换。
- 数据写入：`production_records` 新增；`process_status`、`order_processes`、`orders`、`order_sub_orders` 更新。
- UI结果：弹窗提示“订单完成/已分配下一工序/生产结束”，并刷新查询页。
- 通知结果：可能向下一工序角色与接收人发通知。
- 代码定位：
  - `src/ui/mini_page/end_production_window.py:276`
  - `src/controller/order_controller.py:743`
  - `src/service/order_service.py:748`
  - `src/impl/order_impl.py:1259`

### 4.7 规则 G：工序流转与下一工序可见数量传递

- 触发条件：当前工序结束生产且非最后工序。
- 判定逻辑：下一工序可见量取当前工序累计完成量；若下一工序无子订单则按角色全员创建。
- 状态变化：下一工序从“不可生产”转入可生产；当前工序按完成情况保持 `生产中` 或 `生产完成`。
- 数据写入：`order_processes.visible_quantity` 更新；`order_sub_orders` 创建或更新 `assigned_quantity`。
- UI结果：下一工序角色在查询页可见子订单并出现“开始首件”入口（满足可生产数量条件）。
- 通知结果：触发下一工序通知（角色通知 + 指定接收通知）。
- 代码定位：
  - `src/impl/order_impl.py:1470`
  - `src/impl/order_impl.py:1492`
  - `src/impl/order_impl.py:1504`
  - `src/impl/order_impl.py:2543`

### 4.8 规则 H：订单自动完工

- 触发条件：最后工序累计完成数量达到订单总数量。
- 判定逻辑：最后工序 `product_quantity == orders.quantity`。
- 状态变化：订单置 `生产完成`；全部工序与工序状态同步置 `生产完成`。
- 数据写入：更新 `orders`、`order_processes`、`process_status`。
- UI结果：查询页不再显示该订单给生产角色；管理页显示生产完成。
- 通知结果：在服务流转路径中可发管理员/下一工序通知（取决于调用入口）。

### 5.2 订单级可见性（按角色）

- 订单状态 `待生产`：
  - 当前角色对应工序状态为 `待生产` 或 `生产中` 才可见；
  - 第一工序显示总量，其他工序显示 `visible_quantity`；

  - 当前角色对应工序状态为 `进行中` 或 `生产中` 可见；
  - 若工序已 `生产完成/已完成`，对该角色隐藏。
- 订单状态 `生产完成`：
  - 对生产角色不显示。

## 6) 子订单分配与并发生产规则

### 6.1 初始分配（创建订单后）

- 为第一工序角色下所有用户创建子订单，每人一条。
- `assigned_quantity=订单总数`，`completed_quantity=0`，`status=待生产`。

代码定位：`src/impl/order_impl.py:2250`

### 6.2 下一工序分配

- 当前工序报工后，按下一工序角色分配：
  - 已存在下一工序子订单：更新 `assigned_quantity`，并可重新显示隐藏完成单；
  - 不存在：按角色全员创建。

代码定位：`src/impl/order_impl.py:2444`

### 6.3 用户可生产数量

- 工序总可生产量与用户可输入量是两个概念：
  - 工序总可生产量：本工序在当前时刻还能生产的总量；
  - 用户可输入量：当前用户本轮可填写的最大数量。
- 基础规则：`用户可输入量 = max(工序总可生产量 - 除自己外生产中用户数, 0)`。
- 保护规则：若当前用户子订单状态为 `生产中` 且工序总可生产量 `>0`，用户可输入量至少为 `1`。

代码定位：`src/impl/order_impl.py:2320`, `src/impl/order_impl.py:2388`

### 6.4 并发跟踪字段

- `process_status.ongoing_users` 保存当前进行中的用户列表（逗号分隔）。
- 开工时加入，报工结束移除；未完成场景不再自动加回，避免影响其他用户并发可输入量。

代码定位：`src/impl/order_impl.py:1585`, `src/impl/order_impl.py:1376`

### 6.5 显隐恢复机制

- `is_visible=False && status=已完成` 的子订单，在后续有可生产量时可恢复显示并置 `待生产`。

代码定位：`src/impl/order_impl.py:2711`

## 7) 数量与可生产数计算公式

### 7.1 工序总可生产量

记：

- `Q_order`：订单总数量
- `Q_prev`：上工序累计完成量
- `Q_curr`：本工序累计完成量

则：

- 第一工序：`Q_available_process = max(Q_order - Q_curr, 0)`
- 非第一工序：`Q_available_process = max(Q_prev - Q_curr, 0)`

代码定位：`src/impl/order_impl.py:2320`

### 7.2 用户可输入量

记：

- `N_other_producing`：除本人外、同工序处于 `status='生产中'` 的用户子订单数量

则：

- `Q_available_user = max(Q_available_process - N_other_producing, 0)`
- 若当前用户处于 `生产中` 且 `Q_available_process > 0`，则 `Q_available_user >= 1`

代码定位：`src/impl/order_impl.py:2388`

### 7.3 本次报工合法性

- 必须满足：`0 < Q_input <= Q_available_user`
- 还必须满足：`Q_curr + Q_input <= Q_prev_supply`

代码定位：

- `src/service/order_service.py:748`
- `src/impl/order_impl.py:1349`

### 7.4 工序完成判定

- `Q_curr_new >= Q_prev_supply` 时，该工序判定 `生产完成`，否则 `生产中`。

代码定位：`src/impl/order_impl.py:1361`

### 7.5 订单完成判定

- 仅最后工序参与判定：`Q_last_process == Q_order`。
- 非最后工序的“工序完成”判定依然有效：`Q_curr_new >= Q_prev_supply` 时该工序可进入 `生产完成`。

代码定位：`src/impl/order_impl.py:1517`, `src/impl/order_impl.py:2076`, `src/impl/order_impl.py:1361`

### 12.2 仍待处理风险

#### R-04 校验码“已使用”约束未真正落地

- 现象：`validate_verification_code_for_process` 当前主要做“当天校验码正确”判断，未按注释执行用户级重复限制。
- 影响：同日重复使用场景控制弱于注释描述。
- 定位：`src/impl/inspection_impl.py:467`

#### R-06 页面可见与可操作角色配置不一致

- 现象：`production_order_query` 页面可见角色与页面内 `ALLOWED_ROLES` 不完全一致。
- 影响：某些可见角色进入页面但无操作按钮。
- 定位：
  - 页面可见配置：`src/impl/user_impl.py:297`
  - 按钮允许角色：`src/ui/son_page/production_order_query_page.py:38`

#### R-07 状态语义分层较多，维护成本高（暂不修复，给出计划）

- 现象：同链路并存 `进行中`、`生产中`、`生产完成`、`已完成` 多状态口径。
- 影响：新功能扩展时易出现分支遗漏。
- 定位：
  - `src/impl/order_impl.py:1366`
  - `src/impl/order_impl.py:1428`
  - `src/impl/order_impl.py:1451`
  - `src/impl/order_impl.py:466`

优化计划：
1. 先定义统一状态字典（订单状态、工序状态、子订单状态）和允许转换矩阵。
2. 在 `service` 层增加状态变更入口，逐步收敛 `impl/ui` 直接写状态字符串。
3. 对 UI 按钮显隐改为“状态机谓词函数”，减少散落分支。
4. 补充流程回归用例：首件重试、并发报工、跨工序流转、自动完工。
5. 完成收敛后再删除兼容口径（如 `已完成` 分支兼容判断）。

#### R-08 占位符风格混用（可读性风险）

- 现象：存在 `%s` 与 `?` 混用写法。
- 影响：运行期可被适配层转换，但增加阅读与迁移成本。
- 定位：
  - 混用示例：`src/impl/order_impl.py:1186`
  - 转换逻辑：`src/utils/postgresql_adapter.py:655`
