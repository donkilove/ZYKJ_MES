### 二、总目标

请在现有仓库基础上，按优先级继续完成 V1 功能缺口，要求如下：

- 先解决阻断可用性、稳定性和高风险权限问题
- 再补齐最影响验收口径的页面、字段、筛选、导出、详情和规则闭环
- 再补跨模块联动、审计留痕和分析增强项
- 所有改动必须尽量最小、可验证、可维护，不做与任务无关的重构

### 三、优先级与执行顺序

请严格按以下顺序推进，除非你在阅读代码后确认存在更高优先级的真实阻断：

#### 第一阶段：先修阻断项

1. 用户模块
   - 修复 `backend/app/api/v1/endpoints/auth.py` 语法错误
   - 收紧注册审批“通过时必须设置初始密码”的规则
   - 补齐首次登录强制改密前端闭环
   - 收敛系统管理员相关权限分配规则，避免越权

2. 设备模块
   - 修复自动生成调度返回值解包问题
   - 回归验证自动生成、消息推送、异常分支日志

3. 消息模块
   - 统一 WebSocket 实时推送协议命名
   - 确保消息创建后实时推送与未读数刷新链路一致

#### 第二阶段：补验收缺口

4. 产品模块
   - 改为逻辑删除
   - 补齐产品列表导出、版本参数导出、参数查询导出
   - 补“查看详情/详情抽屉”
   - 打通参数说明写入链路
   - 增加 Link 类型后端格式校验

5. 生产模块
   - 补订单管理筛选项（开始日期范围、交期范围）
   - 前端补产品版本字段展示
   - 新增并行实例追踪页与入口
   - 统一代班审批语义、接口和页面文案

6. 品质模块
   - 在品质入口放开报废导出、维修导出和完成维修能力
   - 补齐首件处置不可无痕覆盖的审计链路
   - 补每日首件、品质数据的多维筛选
   - 视改动范围补报废/维修详情追溯字段

7. 工艺模块
   - 补模板/产品维度引用分析
   - 落实同工段下工序名称唯一规则
   - 增强模板复制/套版范围
   - 补必要的筛选、导出和审计留痕

#### 第三阶段：补跨模块联动与一致性

8. 接入消息模块的生产、品质关键事件来源
9. 补齐产品/工艺/品质/设备关键动作审计日志
10. 对筛选、导出、详情、追溯字段做一次跨模块一致性收口

### 四、执行方式要求

1. 不要平均撒网式改 7 个模块，必须按优先级逐步完成。
2. 每次开始一个模块前，先读该模块需求说明和最新检查结果。
3. 每完成一个模块后：
   - 更新该模块最新检查结果文档
   - 如模块状态变化，回写 `docs/V1模块聚合检查-20260314.md`
4. 不要只把问题改成“部分满足”的文案，必须优先交付实际代码修复。
5. 不要覆盖或回退与当前任务无关的现有修改。
6. 所有新增或修改的文本文件必须保持 `UTF-8` 无 `BOM`、`LF` 行尾。
7. 新增文案、注释、文档默认使用中文。

### 五、实现原则

请严格遵守以下原则：

1. 优先最小且有效的改动，不做无关重构。
2. 优先复用现有模型、接口、页面结构、权限快照和模块目录机制。
3. 修改公共接口、数据模型、配置项、脚本入口或构建行为时，必须同步检查受影响调用方。
4. 不能用占位实现、伪逻辑或 `TODO` 作为最终交付。
5. 不能降低现有安全基线，尤其不能放松权限规则和审计要求。
6. 遇到需求与代码现状冲突时，以“最小可运行、可验证、可维护”为准，但必须在最终汇报中说明偏差。

### 六、各模块最低交付要求

在推进过程中，请确保每个模块至少满足以下交付口径：

#### 1. 用户模块

- 后端接口可正常加载运行
- 注册审批通过必须由管理员设置初始密码
- 首次登录强制改密必须有前后端闭环
- 系统管理员能力不能被普通管理员错误分配

#### 2. 设备模块

- 自动生成调度不再因返回值解包异常失败
- 自动生成后的消息/日志链路可运行
- 不破坏现有设备保养主流程

#### 3. 消息模块

- WebSocket 事件协议统一
- 未读角标、列表、单条已读、全部已读实时性一致
- 首批生产、品质来源至少接入一部分关键事件
- 跳转失败时提供明确提示

#### 4. 产品模块

- 删除语义改为逻辑删除
- 三类导出能力可用
- 产品详情入口与详情视图可用
- 参数说明字段真正写入并可回显
- Link 类型格式校验有效

#### 5. 生产模块

- 订单管理页筛选字段补齐
- 产品版本字段可见
- 可从管理页或详情页进入并行实例追踪
- 代班审批语义与前后端文案一致

#### 6. 品质模块

- 品质入口下具备实际执行能力，不是只读壳子
- 首件处置可追溯，不允许无痕覆盖
- 每日首件和品质数据支持更完整筛选

#### 7. 工艺模块

- 引用分析不只看工段/工序，还能覆盖模板/产品维度
- 工序唯一性规则明确落地
- 模板复制/套版能力更接近需求口径

### 七、建议的工作节奏

请按“一个模块一个模块”推进：

1. 先阅读模块文档
2. 再定位相关前后端代码与测试
3. 再实现该模块的 P0 / P1 缺口
4. 再执行验证
5. 再更新该模块检查结果和聚合文档
6. 再进入下一个模块

如果上下文过长，优先保证：

- 先把用户模块做完并汇报
- 再继续设备模块
- 再继续消息模块

### 八、测试与验证要求

每次发生代码改动后，你必须执行并报告以下命令结果：

- `python backend/scripts/check_chinese_mojibake.py`
- `python backend/scripts/check_frontend_chinese_mojibake.py`
- `python -m pytest test/backend/test_chinese_mojibake_check.py test/backend/test_frontend_chinese_mojibake_check.py -q`

如果系统 Python 缺少 `pytest`，请改用虚拟环境解释器执行，例如：

- `".venv\\Scripts\\python.exe" -m pytest test/backend/test_chinese_mojibake_check.py test/backend/test_frontend_chinese_mojibake_check.py -q`

除上述强制检查外，请根据改动范围补充执行：

- 相关后端服务/接口 pytest
- 相关 Flutter test 或 `flutter analyze`
- 必要的端到端联调验证
- 关键页面手动路径验证

### 九、文档更新要求

你不是只改代码，还要维护检查文档的一致性：

1. 每完成一个模块，更新该模块的最新检查结果文档

完成后请用中文汇报，且必须包含：
