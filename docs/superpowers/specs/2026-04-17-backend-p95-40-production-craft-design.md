# 后端全链路 40 并发 `P95 < 500 ms` 第一批收敛设计（production + craft）

## 1. 背景

当前仓库已经形成两条不同用途的性能口径：

1. `91` 场景读链路专项：当前可作为正式性能门禁基线，历史结果已达到 `P95 = 438.11 ms`。
2. `270` 场景全链路套件：当前已实现路由级命中全覆盖，但仍不是有效全覆盖，不能直接作为正式性能门禁。

现阶段的核心矛盾不是“是否打到所有路由”，而是：

> `270` 场景全链路套件中，仍有大量场景被 `401/403/404/405/422` 等非性能噪声污染，导致成功率不足、结果不可解释、P95 不可信。

因此，本轮不直接把“`270` 场景整体 `P95 < 500 ms`”作为第一目标，而是先推进第一批模块收敛，把 `production + craft` 从“主要噪声源”改造成“高成功率、可解释、可复跑”的模块级有效覆盖子套件。

## 2. 目标

### 2.1 本轮目标

1. 为 `production + craft` 建立稳定、可重建的样本资产模型。
2. 校准这两个模块在全链路场景中的 `405/422` 契约偏差，并同步治理与样本相关的 `404`。
3. 形成 `production + craft` 的模块级高成功率子套件，覆盖读链路、detail 链路和写链路。
4. 将第一批收敛结果回灌到 `270` 场景全链路套件，验证其对全链路有效覆盖质量的真实提升。

### 2.2 非目标

1. 本轮不宣称 `270` 场景全链路已整体通过正式性能门禁。
2. 本轮不放弃现有 `91` 场景正式性能基线。
3. 本轮不一次性扩展所有业务模块，而是先打穿 `production + craft`。
4. 本轮不把纯 SQL、缓存、连接池优化作为首要动作；只有在高成功率集合建立后才进入真正的 `P95` 冲刺。

## 3. 设计总览

本轮采用“阶段 `3 -> 4 -> 5`”推进路径，对应三类核心问题：

1. 阶段 `3`：样本资产治理，压 `404`
2. 阶段 `4`：契约与场景定义校准，治理 `405/422`
3. 阶段 `5`：形成模块级有效覆盖

这三步完成后，再做一次回灌复跑，用于回答：

- `production + craft` 是否已经形成高成功率子套件？
- `270` 场景中这两个模块是否不再是大面积 `404/405/422` 的主要来源？
- 后续阶段 `6` 是否具备在该批高成功率集合上继续冲击 `P95` 的条件？

## 4. 第一批范围

第一批范围固定为 `production + craft`，且**一次性纳入三类链路**：

1. 读链路
2. detail 链路
3. 写链路

不采用“先只做读链路、后续再补写链路”的保守拆法。原因是：

- 这两个模块的样本依赖是共享的，若只做读链路，后续写链路仍会把样本契约重新打散。
- detail 和写链路所暴露的 `404/422`，本身就是当前全链路不可解释结果的重要来源。
- 一次性把三层链路纳入，能更早形成完整的模块级有效覆盖闭环。

## 5. 样本资产设计

### 5.1 样本模型

本轮采用“双层样本模型”：

1. 稳定主样本
2. 一次性写样本

#### 稳定主样本

稳定主样本用于承载读链路和 detail 链路，要求长期存在、可按业务编码稳定反查、可在环境重置后重建。建议至少包括：

- 产品：`PERF-PRODUCT-STD-01`
- 工艺路线：`PERF-ROUTE-STD-01`
- 工段：`PERF-STAGE-STD-01`
- 工序：`PERF-PROCESS-STD-01`、`PERF-PROCESS-STD-02`
- 供应商：`PERF-SUPPLIER-STD-01`
- 生产订单：`PERF-ORDER-OPEN-01`

必要的派生样本，例如工艺模板、生产执行记录、首件模板、辅助授权等，也应从这组主样本派生，而不是依赖历史脏数据。

#### 一次性写样本

一次性写样本用于承载写链路，避免写入共享主样本导致多轮复跑相互污染。规则如下：

- 每轮使用带 `PERF-RUN-<run_id>` 前缀的临时样本编码。
- 写链路执行后，要么显式复位，要么直接删除。
- 不允许写链路直接把稳定主样本永久改脏。

### 5.2 初始化顺序

初始化顺序固定如下：

1. 准备 `pool-production` 角色与 token
2. 初始化产品、工艺路线、工段工序、供应商
3. 初始化稳定生产订单与 detail 依赖样本
4. 初始化写链路所需的一次性样本
5. 执行最小 smoke，确认样本与接口前置状态已满足

### 5.3 样本合同

每条 detail 或写链路场景都必须显式声明以下样本合同字段：

- `layer`
- `sample_contract`
- `依赖样本编码`
- `restore_strategy`

这些信息必须从文档口径下沉到可执行配置或脚本中，不能只停留在说明文字。

## 6. 场景分层设计

### 6.1 分层原则

`production + craft` 场景统一按三层拆分，不再混跑：

1. `read`
2. `detail`
3. `write`

### 6.2 各层职责

#### `read`

覆盖列表、统计、reference、options、看板类接口。  
目标是验证主样本可被稳定读取，且不因写链路副作用污染结果。

#### `detail`

覆盖订单详情、工艺模板详情、工序详情、首件详情等依赖具体实体存在的接口。  
目标是将 detail 类 `404` 从“环境中没有数据”收敛为“样本合同是否满足”的显式判断。

#### `write`

覆盖创建、提交、回滚、状态流转、首件提交等修改状态的接口。  
目标是明确写链路的前置样本、执行副作用和复位策略，使写链路具备重复执行能力。

### 6.3 模块级场景文件

本轮建议产出以下模块级场景文件：

- `tools/perf/scenarios/production_craft_read_40_scan.json`
- `tools/perf/scenarios/production_craft_detail_40_scan.json`
- `tools/perf/scenarios/production_craft_write_40_scan.json`

命名可在实现计划阶段再微调，但结构必须先固定：每类链路一组独立可复跑场景集。

## 7. 验证路径设计

本轮验证采用 4 段式闭环，而不是一上来直接压 `270` 场景：

### 7.1 样本初始化验证

目标：证明稳定主样本与一次性写样本都能按合同生成。  
证据：

- 样本编码反查成功
- 关键实体状态符合前置条件
- 初始化后最小 smoke 可通过

### 7.2 模块级子套件验证

目标：先在模块内部跑通 `read/detail/write` 三层。  
证据：

- 分层结果文件
- 每层成功率
- `404/405/422` 分布
- `zero_success_count`

### 7.3 模块级 `40` 并发回归

目标：证明这批不是“串行能跑，压测就碎”。  
证据：

- 模块级 `40` 并发结果 JSON
- `P95/P99`
- 错误率
- 状态码分布

### 7.4 回灌 `270` 场景

目标：衡量第一批整改对全链路成功率和可解释性的真实贡献。  
证据：

- 回灌前后对比结果
- `production + craft` 相关场景成功率变化
- `zero_success_count` 变化
- `404/405/422` 是否明显下降

## 8. 实现切分

本轮实现不按“改完所有内容再统一复测”的方式推进，而按 4 个独立批次切分：

### 批次 1：样本资产落地

目标：将稳定主样本与一次性写样本落成可执行脚本与配置口径。  
主要输出：

- `backend/scripts/` 下的样本初始化与复位脚本
- `tools/perf/` 或 `tools/perf/write_gate/` 下的样本解析与恢复策略支撑
- `docs/后端P95-40并发全链路覆盖/09-样本资产清单.md` 的实际编码与顺序补充
- 初始化 smoke evidence

### 批次 2：场景契约校准

目标：修正 `production + craft` 在全链路场景中的 `405/422` 偏差。  
主要输出：

- `tools/perf/scenarios/combined_40_scan.json` 修订
- 新增或拆分出的模块级场景文件
- `405/422` 差异清单与复检记录
- 必要的接口校验或场景参数修正

### 批次 3：模块级高成功率子套件

目标：形成 `production + craft` 的 `read/detail/write` 高成功率子套件。  
主要输出：

- 三组模块级场景文件
- 模块级结果 JSON
- 模块级有效覆盖 evidence

### 批次 4：回灌与阶段结论

目标：将前 3 批成果回灌到 `270` 场景，用于评估第一批收敛效果。  
主要输出：

- 回灌后的 `270` 场景结果
- 前后对比 evidence
- 更新后的阶段结论与下一步入口

## 9. 完成判定

本轮完成不等同于“全链路已通过正式性能门禁”，而应满足以下条件：

1. `production + craft` 样本可稳定初始化、校验、复位。
2. 两个模块的 `404/405/422` 从“成批出现”降到“少量且可解释”。
3. `production + craft` 形成可重复执行、可解释的模块级高成功率子套件。
4. 回灌 `270` 场景后，这两个模块不再是大面积 `404/405/422` 的主要来源。
5. 为后续阶段 `6` 的真正 `P95` 冲刺建立高成功率候选集合。

## 10. 风险与控制

### 风险 1：写链路污染共享样本

控制方式：稳定主样本与一次性写样本分层；写链路必须有 `restore_strategy`。

### 风险 2：detail 场景继续依赖历史脏数据

控制方式：只允许通过稳定编码反查样本，不允许长期依赖硬编码历史 `id`。

### 风险 3：把契约/样本问题误判成性能问题

控制方式：坚持先完成样本与契约收敛，再看模块级高成功率集合上的性能指标。

### 风险 4：模块级成功不回灌全链路

控制方式：第一批必须以一次 `270` 场景回灌复跑作为收口动作，而不是只停留在局部子套件结果。

## 11. 本轮结论

本轮推荐设计为：

1. 第一批只打穿 `production + craft`
2. 但一次性纳入读链路、detail 链路和写链路
3. 先完成样本资产与契约收敛，再形成模块级高成功率子套件
4. 最后回灌 `270` 场景，验证对全链路有效覆盖质量的真实改善
