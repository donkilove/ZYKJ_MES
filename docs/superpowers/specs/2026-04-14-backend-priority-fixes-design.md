# 后端三项优先改造设计

## 目标

本轮只做三件高优先级改造：

1. 安全硬化：禁止危险默认 JWT 密钥、默认管理员密码、默认首件验证码直接进入运行态。
2. 事务边界统一：消除 service 已提交、API 再补审计并再次提交的非原子链路。
3. 首批模块拆分：对 `authz_service.py` 抽出首批独立职责边界，降低继续扩展风险。

## 实施顺序

采用以下顺序：

1. 安全硬化
2. 事务边界统一
3. `authz_service` 首批拆分

这样可以先降低底层风险，再处理事务，再在更稳定的边界上做结构拆分。

## 设计要点

### 安全硬化

- `backend/app/core/config.py` 中的弱默认值不得继续悄悄生效。
- 运行态应提供显式 fail-fast 校验。
- `backend/app/core/security.py` 不能在占位密钥下继续签发/解码 token。
- 首件验证码缺失时不得继续回退到弱默认值。

### 事务边界统一

- 本轮采用“API 层统一提交”的口径。
- service 层负责构造业务对象，不再自行 `commit()`。
- API 层负责审计写入与最终 `commit()/rollback()`。
- 首批修正链路：产品创建、角色创建。

### authz 首批拆分

- 本轮不做全面重构。
- 先从 `authz_service.py` 抽出两个方向的职责：
  - 缓存 / revision
  - 权限读取 / permission code 计算
- 对外接口保持兼容，优先做低耦合抽离。

## 验收标准

1. 危险默认值不再进入正常运行链路。
2. 产品创建与角色创建链路不再出现双层提交。
3. `authz_service.py` 至少拆出 1-2 个独立职责文件。
4. 所有完成声明必须附带真实验证结果；无法验证的部分要明确说明边界。
