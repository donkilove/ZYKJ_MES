# Python 插件宿主设计

- 日期：2026-04-24
- 主题：Flutter Windows 主程序内嵌独立 Python 插件应用
- 状态：待用户审阅

## 1. 背景

当前前端是 Flutter Windows 桌面应用，后续希望为产线补充一批独立小工具，例如串口助手、文件转换器、设备调试工具。

用户已明确以下约束：

1. 插件语言使用 Python。
2. 插件需要完全独立，自带 UI 和程序逻辑。
3. 主程序只负责扫描插件、展示入口、启动插件和承载插件界面。
4. 插件界面仍需显示在主程序内部，而不是弹出独立外部窗口。
5. 目标机器不要求单独安装 Python。
6. 主程序只提供 Python 解释器，不统一提供第三方包。
7. 第三方依赖由插件自身随包携带，并在启动时注入。
8. 解释器既可放在主程序目录，也可放在宿主约定的外部私有目录。

本设计的核心是把插件能力定义为“主程序内嵌的独立应用”，而不是“由宿主渲染的业务页面”。

## 2. 目标与非目标

### 2.1 目标

1. 主程序能够扫描插件目录并稳定展示插件入口。
2. 主程序能够在内部打开插件自带 UI，而不理解插件页面结构。
3. 插件能够在宿主提供的 Python 解释器上独立运行。
4. 插件第三方依赖能够在插件目录内自带并隔离加载。
5. 第一版具备最小生命周期管理：启动、心跳、异常提示、关闭、重启。
6. 以“串口助手”作为首个 PoC 验证完整链路。

### 2.2 非目标

1. 本轮不做插件市场、在线下载或热更新。
2. 本轮不做插件间通信。
3. 本轮不做多实例与后台保活。
4. 本轮不让插件注入 Flutter 组件或宿主业务 UI。
5. 本轮不做强安全沙箱，只做声明式权限展示。
6. 本轮不支持除 Windows 以外的平台运行。

## 3. 方案选择

本轮采用“宿主 + 独立 Python 插件 + 内嵌 Web UI”方案。

### 3.1 采用方案

1. 主程序作为插件宿主，负责扫描、启动、停止、状态显示和界面承载。
2. 每个插件是独立 Python 程序包，自带业务逻辑和 Web UI。
3. 插件启动后在本地 `127.0.0.1` 上提供私有 Web 服务。
4. 主程序内部通过 Windows WebView 容器打开插件的本地 `entry_url`。

### 3.2 不采用方案

1. 不采用“宿主渲染协议式 UI”方案。
   - 这会让宿主理解插件页面结构，不符合“插件自带 UI”的目标。
2. 不采用“内嵌原生窗口句柄”方案。
   - 焦点、缩放、输入法、窗口销毁和宿主兼容性风险过高。
3. 不采用“远程画布/流式界面”方案。
   - 首期复杂度明显过高，不适合 PoC。

## 4. 总体架构

系统分为三层：

1. 宿主层
2. 插件运行层
3. 生命周期协议层

### 4.1 宿主层

宿主职责固定为：

1. 扫描 `plugins/` 目录并读取 `manifest.json`
2. 展示插件列表与状态
3. 启动/停止插件进程
4. 获取插件 `entry_url`
5. 在主程序内部创建 WebView 工作区承载插件 UI
6. 执行心跳检查、超时判断和统一错误提示

宿主不负责：

1. 理解插件业务页面
2. 渲染插件业务表单
3. 管理插件第三方库
4. 执行插件内部业务逻辑

### 4.2 插件运行层

每个插件是一个独立程序包：

1. 自带启动脚本
2. 自带第三方依赖
3. 自带 Web UI 与静态资源
4. 自带业务逻辑与日志输出

插件对宿主只暴露：

1. 元数据
2. 启动入口
3. 就绪信号
4. `entry_url`
5. 心跳能力

### 4.3 协议层

宿主与插件只靠固定协议协作：

1. `manifest` 元数据协议
2. 启动参数协议
3. `ready` 消息协议
4. 心跳协议
5. 关闭协议

宿主与插件之间不共享 UI 组件和业务状态对象。

## 5. 插件中心 UI

宿主 UI 采用“两栏一工作区”的稳定外壳。

### 5.1 左侧插件导航栏

展示内容仅限宿主关心的信息：

1. 插件图标
2. 插件名称
3. 版本
4. 状态
5. 固定动作：打开、重启、停止、查看日志

状态统一为：

1. 未启动
2. 启动中
3. 运行中
4. 异常
5. 已停止

### 5.2 右侧插件工作区

1. 默认显示空态，引导选择插件。
2. 打开插件后，右侧工作区创建页签。
3. 每个页签内部是一个 WebView 容器。
4. WebView 只负责打开插件自带 `entry_url`。
5. 宿主不参与插件内部路由、表单和业务逻辑。

### 5.3 宿主工具条

每个插件工作区顶部仅保留宿主级动作：

1. 刷新容器
2. 重连
3. 强制关闭
4. 打开日志
5. 打开插件目录

### 5.4 错误态

无论插件内部如何实现，宿主只显示统一错误页：

1. 插件启动失败
2. 插件超时未就绪
3. 心跳丢失
4. WebView 无法打开 `entry_url`

## 6. 插件包结构

建议每个插件采用如下目录结构：

```text
plugins/
  serial_assistant/
    manifest.json
    launcher.py
    app/
    vendor/
    web/
    logs/
```

说明：

1. `manifest.json`：宿主唯一必读入口
2. `launcher.py`：插件启动脚本
3. `app/`：插件业务代码
4. `vendor/`：插件私有第三方依赖
5. `web/`：插件 UI 静态资源
6. `logs/`：插件本地日志

## 7. Manifest 设计

宿主只解析最小元数据。

建议字段如下：

```json
{
  "id": "serial_assistant",
  "name": "串口助手",
  "version": "0.1.0",
  "entry": {
    "type": "python",
    "script": "launcher.py"
  },
  "ui": {
    "type": "web",
    "mode": "embedded"
  },
  "runtime": {
    "python": "3.14",
    "arch": "win_amd64"
  },
  "dependencies": {
    "mode": "plugin_local",
    "paths": ["vendor", "app"]
  },
  "permissions": ["serial", "filesystem"],
  "lifecycle": {
    "startup_timeout_sec": 15,
    "heartbeat_interval_sec": 5
  }
}
```

设计原则：

1. 插件只声明入口和需求。
2. 插件不声明解释器绝对路径。
3. 解释器路径始终由宿主策略决定。

## 8. 启动协议

### 8.1 扫描阶段

宿主扫描 `plugins/*/manifest.json`，校验：

1. 必填字段是否存在
2. `entry.script` 是否存在
3. 运行时版本是否兼容
4. 架构是否兼容

不通过时插件标记为异常，不允许打开。

### 8.2 启动阶段

用户点击“打开插件”后：

1. 宿主解析插件目录
2. 宿主选择解释器路径
3. 宿主注入环境变量
4. 宿主启动插件进程
5. 插件输出 `ready` 消息
6. 宿主拿到 `entry_url` 后在 WebView 内打开

宿主统一拼接实际命令：

```text
<python.exe> <plugin>/launcher.py
```

### 8.3 注入环境

宿主在进程启动前注入：

1. `PYTHONHOME=<runtime_root>`
2. `PYTHONPATH=<plugin>/vendor;<plugin>/app`
3. `MES_PLUGIN_ID`
4. `MES_PLUGIN_DIR`
5. `MES_RUNTIME_DIR`
6. `MES_HOST_SESSION_ID`

插件依赖通过环境与 `sys.path` 注入，不通过宿主统一安装。

### 8.4 Ready 协议

插件启动完成后，通过 `stdout` 输出一条结构化 `ready` 消息，至少包含：

1. `status`
2. `pid`
3. `entry_url`
4. `heartbeat_url` 或等价心跳令牌

若宿主在超时时间内未收到合法 `ready`，则判定启动失败。

### 8.5 心跳与关闭

第一版采用简单策略：

1. 宿主按固定周期做心跳检查
2. 心跳失败则切换为异常状态
3. 用户关闭页签时，宿主直接停止插件进程
4. 第一版不做后台保活

## 9. Python 运行时策略

### 9.1 运行时边界

宿主只提供 Python 解释器，不提供公共第三方库。

第三方依赖全部由插件自带：

1. 宿主不维护公共 `site-packages`
2. 宿主不执行运行时 `pip install`
3. 插件依赖全部存放于插件目录内

### 9.2 解释器位置

宿主支持两种模式：

1. `bundled`
   - 解释器位于主程序目录
2. `external_managed`
   - 解释器位于宿主约定的外部私有目录

无论采用哪种模式，解释器路径都由宿主配置决定，而不是插件决定。

### 9.3 依赖加载

每个插件通过私有 `vendor/` 目录自带依赖。

推荐加载方式：

1. 宿主注入 `PYTHONPATH`
2. 插件在 `launcher.py` 内补充 `sys.path`

这样可以保证：

1. 插件间依赖不互相污染
2. 不依赖用户机器环境变量
3. 不要求目标机器安装 Python 或 pip

## 10. Windows WebView 承载策略

插件 UI 承载采用“本地 Web 服务 + 宿主内嵌 WebView”路线。

时效性结论：

1. 当前官方 `webview_flutter` 包页面未直接列出 Windows 支持。
2. Windows 侧更适合采用基于 WebView2 的实现路径。
3. 宿主应封装一层 WebView 适配，不把插件系统绑定到单一包。

因此实现时应把“插件承载接口”和“具体 WebView 包”分离。

## 11. 异常处理与权限边界

### 11.1 宿主统一异常

宿主只负责以下统一异常：

1. `manifest` 缺失或格式错误
2. 插件进程拉起失败
3. 插件超时未就绪
4. 运行中断或心跳丢失
5. WebView 打开 `entry_url` 失败

宿主统一提供：

1. 状态标记
2. 错误页
3. 重试
4. 重启
5. 查看日志

### 11.2 权限边界

第一版只做声明式权限，不承诺强沙箱。

宿主可展示的权限示例：

1. `serial`
2. `filesystem`
3. `network`
4. `subprocess`

宿主负责提示风险，但不承诺隔离用户权限范围内的本地资源访问。

## 12. 第一版范围裁剪

第一版只保留最小闭环：

1. 扫描插件
2. 展示插件
3. 启动插件
4. 内嵌显示
5. 心跳检查
6. 异常提示
7. 手动重启
8. 手动关闭

第一版明确不做：

1. 插件市场
2. 在线下载
3. 热更新
4. 多实例
5. 后台保活
6. 插件间通信
7. 强安全沙箱
8. 复杂双向 RPC
9. 插件注入 Flutter UI

## 13. 测试策略

测试分为三层。

### 13.1 宿主层测试

覆盖内容：

1. 扫描插件目录识别 `manifest`
2. 缺字段、入口脚本缺失、版本不兼容时标红
3. 启动插件后收到 `ready`
4. 超时未收到 `ready` 时显示错误态
5. 关闭页签后插件进程被停止

### 13.2 协议层测试

覆盖内容：

1. 宿主注入的环境变量完整
2. 插件 `ready` 消息格式非法时被拒绝
3. 心跳丢失后状态切换为异常
4. 重启后能建立新的干净会话

### 13.3 插件自身测试

覆盖内容：

1. `vendor/` 私有依赖可正常加载
2. 插件 Web UI 可独立运行
3. 业务逻辑在插件内部自测通过

## 14. 串口助手 PoC 边界

首个插件 PoC 采用“串口助手”。

### 14.1 最小功能

1. 枚举本机串口
2. 选择波特率
3. 打开串口
4. 关闭串口
5. 发送文本
6. 显示接收区
7. 显示连接状态和错误信息

### 14.2 宿主侧最小能力

1. 插件中心能看到“串口助手”
2. 点击后能启动插件
3. 能在主程序内部看到插件界面
4. 插件异常退出后宿主能提示并允许重启

### 14.3 第一版不做

1. 十六进制模式
2. 自动重连
3. 多串口并开
4. 数据脚本编排
5. 指令模板库
6. 权限细粒度控制

### 14.4 PoC 通过标准

以下 5 条同时成立即可视为 PoC 通过：

1. 主程序不依赖系统 Python 也能拉起插件
2. 插件依赖完全来自自己的目录
3. 插件 UI 能稳定显示在主程序内部
4. 关闭插件页后插件进程能干净退出
5. 串口助手能真实完成一次打开、发送、接收、关闭

## 15. 风险与取舍

1. 内嵌 WebView 路线实现难度显著低于原生窗口承载，但要接受插件 UI 技术栈偏向 Web。
2. 第一版不做强沙箱，意味着必须在安装/启用阶段明确风险提示。
3. 关闭页签即停止插件会牺牲后台保活体验，但换来更简单稳定的生命周期。
4. 依赖由插件私有携带会增加插件包体积，但能换来更强隔离和版本灵活性。
5. 宿主不统一装第三方依赖，后续插件作者必须自行管理依赖一致性。
