# 首页工作台业务化设计

- 日期：2026-04-12
- 设计对象：`frontend/lib/features/shell/presentation/home_page.dart`
- 设计结论：采用“待办中枢型”首页工作台，统一框架按权限分发卡片，首屏优先呈现待办，不做纵向滚动门户页。
- 迁移口径：无迁移，直接替换。

## 1. 背景与目标

当前首页仅承载欢迎信息、日期、角色身份与快速跳转，能作为登录后的落点，但还不能承担“工作台”的职责。用户进入系统后仍需自己判断“先处理什么”，这与制造执行系统高频、时效、异常驱动的使用场景不匹配。

本次改造目标是将首页升级为真正的业务工作台，并满足以下结论：

1. 首页采用统一框架，不为不同角色分裂成多套页面结构。
2. 首页首屏优先呈现“我今天要处理什么”，而不是先呈现大盘图表。
3. 首页允许补一个后端聚合接口，前端用单份摘要数据驱动卡片展示。
4. 首页必须贴合现有前端视觉语言：浅色 `Material 3`、卡片化、轻绿色欢迎区、简洁操作节奏。
5. 首页在 `1920x1080` 屏幕下默认首屏完整可见，不依赖纵向滚动。
6. 后续实现必须补齐 40 并发 P95 验证。

## 2. 设计原则

### 2.1 工作台而非门户

首页只负责三件事：

1. 告诉用户当前是谁、处于什么角色和工段。
2. 告诉用户此刻最优先要处理的事项。
3. 给出少量风险和经营摘要，辅助决策但不抢主焦点。

首页不负责：

1. 展开完整明细列表。
2. 直接执行复杂业务操作。
3. 承担多层图表看板职责。

### 2.2 首屏无滚动

首页在标准桌面大屏下应完整展示，不能靠下拉继续发现核心信息。为满足这一点，首页采取以下约束：

1. 主待办仅展示最高优先级的 4 条。
2. 风险卡与 KPI 卡只显示摘要值，不显示长列表。
3. 顶部欢迎区压缩为一行主标题 + 一行副信息。
4. 其余待办通过“查看全部待办”进入对应模块处理。

### 2.3 与现有前端风格一致

首页遵循现有前端的真实视觉语言：

1. 使用浅色 `Material 3` 气质，而不是深色可视化稿风格。
2. 延续现有首页欢迎区的轻绿色渐变氛围。
3. 页头节奏参考 `CrudPageHeader` 的标题 + 刷新动作形式。
4. 卡片面采用白底或浅灰绿底，圆角与阴影保持克制。
5. 文案与数字强调通过层级、间距和色块完成，不引入脱离当前系统的新视觉体系。

## 3. 信息架构

首页首屏固定为四个区域：

### 3.1 顶部状态条

职责：

1. 呈现当前用户身份和上下文。
2. 承载整页刷新入口。
3. 提供进入个人中心的直接入口。

展示字段：

1. `display_name`
2. `role_name`
3. `stage_name`
4. `today_notice_count`

交互：

1. `刷新数据`：刷新首页摘要数据。
2. `个人中心`：跳转 `account_settings`。

### 3.2 主卡：我的待办队列

职责：

1. 成为首页第一视觉焦点。
2. 用统一排序规则给出最该处理的事项。
3. 作为首页的主分发中心。

展示内容：

1. 顶部摘要数：
   - `待审批数`
   - `高优消息数`
   - `异常待处理数`
   - `超时/临近超时数`
2. 主体列表：
   - 仅显示 `4` 条最高优先级待办
   - 每条待办显示：
     - 标题
     - 分类标签，例如 `审批`、`生产`、`质量`、`设备`
     - 优先级或时效标签，例如 `高优`、`超时`
3. 底部入口：
   - `查看全部待办`

交互：

1. 点击单条待办：跳到目标页面/页签/详情。
2. 点击“查看全部待办”：跳到 `message` 模块的待办过滤视图。

排序规则：

1. `是否超时` 优先。
2. `优先级` 次之。
3. `发布时间/发生时间` 再次排序。

### 3.3 右侧卡 1：异常与风险

职责：

1. 告知当前最值得关注的业务风险。
2. 作为待办主卡之外的第二关注区。

展示内容：

1. `生产异常`
2. `质量预警`
3. `设备逾期保养`
4. `待确认高优消息`

每项仅显示名称与数量。

交互：

1. `生产异常`：跳 `production_order_query` 的异常过滤态。
2. `质量预警`：跳 `quality_data_query` 或 `quality_trend` 的预警过滤态。
3. `设备逾期保养`：跳 `maintenance_execution`。
4. `待确认高优消息`：跳 `message`。

### 3.4 右侧卡 2：次级业务指标

职责：

1. 给管理岗提供足够的经营状态感知。
2. 但不让首页退化成经营大盘。

展示指标：

1. `在制订单`
2. `今日产量`
3. `首件通过率`
4. `报废数`

交互：

1. `在制订单`、`今日产量`：跳 `production_data_query`
2. `首件通过率`、`报废数`：跳 `quality_data_query` 或 `quality_trend`

## 4. 布局与尺寸约束

### 4.1 首屏布局

首页采用两列布局：

1. 左列为主区，宽度约为右列的 `1.7` 倍。
2. 右列为摘要区，堆叠显示风险卡与 KPI 卡。

版式要求：

1. 顶部状态条固定高度，保持轻量。
2. 左列主待办卡吃满主要高度。
3. 右列两张卡按固定比例分配高度。
4. 默认桌面视口下不出现纵向滚动条。

### 4.2 信息密度约束

1. 主待办首屏只展示 `4` 条。
2. 右侧每张卡最多 `4` 个摘要项。
3. 顶部状态条不放多段公告正文。
4. 首页不放趋势图和大表格。

## 5. 数据与接口设计

### 5.1 路由位置

新增接口建议挂在现有 `UI` 路由下，与 `page-catalog` 保持同一语义层级：

- `GET /api/v1/ui/home-dashboard`

### 5.2 接口返回结构

建议新增以下后端 schema：

- `HomeDashboardResult`
- `HomeDashboardTodoSummary`
- `HomeDashboardTodoItem`
- `HomeDashboardMetricItem`
- `HomeDashboardDegradedBlock`

### 5.3 数据来源策略

#### 待办与风险

待办与风险以消息系统为主源，原因如下：

1. 消息域已经具备 `message_type`、`priority`、`source_module`、`target_page_code`、`target_tab_code`、`target_route_payload_json`。
2. 首页点击后可直接复用消息跳转能力。
3. 待办与风险本质上都属于“需要处理或关注的事件”，消息域语义天然匹配。

首页聚合服务应直接在服务层调用消息相关逻辑，不应通过 HTTP 再请求 `/messages`。

#### KPI 指标

KPI 直接复用现有统计服务：

1. 生产：
   - `production/stats/overview`
   - `production/data/today-realtime`
2. 质量：
   - `quality/stats/overview`

KPI 只取首页首屏必需字段，不引入首页专用重型图表数据。

### 5.4 当前用户信息

首页顶部状态条继续使用主壳现有的当前用户数据，不通过首页聚合接口重复返回：

1. `display_name`
2. `role_name`
3. `stage_name`

首页接口仅补 `notice_count` 等首页独有摘要值。

## 6. 权限与裁剪规则

### 6.1 统一框架

首页统一使用同一框架，不根据角色切换页面结构。

### 6.2 内容裁剪

卡片内容按权限与可见模块裁剪：

1. 主待办卡始终显示。
2. 风险项和 KPI 项按权限和模块可见性过滤。
3. 无权限的项不展示。
4. 不建议因为单个条目缺失而让整张卡消失。

### 6.3 空态规则

1. 主待办无数据：显示“当前没有待处理事项”。
2. 风险卡无数据：各项显示 `0` 或空摘要。
3. KPI 卡无数据：显示 `--`。

## 7. 前端实现建议

### 7.1 新增模型与服务

建议新增：

1. `frontend/lib/features/shell/models/home_dashboard_models.dart`
2. `frontend/lib/features/shell/services/home_dashboard_service.dart`

### 7.2 首页组件拆分

建议将首页拆分为下列组件，避免 `home_page.dart` 继续膨胀：

1. `home_dashboard_header.dart`
2. `home_dashboard_todo_card.dart`
3. `home_dashboard_risk_card.dart`
4. `home_dashboard_kpi_card.dart`

### 7.3 主壳协同

`main_shell_page.dart` 只负责：

1. 当前用户加载
2. 模块可见性与导航
3. 首页刷新入口
4. 首页可见时的轻量刷新触发

## 8. 刷新与状态策略

### 8.1 刷新时机

首页摘要只在以下场景刷新：

1. 登录进入首页时首次加载
2. 用户点击刷新按钮
3. 当前正处于首页，且消息 WebSocket 事件表明未读/待办有变化时，做一次防抖刷新

### 8.2 不做的事情

1. 首页隐藏时不轮询
2. 不跟随壳层所有轮询一起刷新
3. 不允许同一时间并发发起多次首页摘要请求

### 8.3 防抖与并发控制

1. 刷新中的重复点击直接忽略
2. WebSocket 驱动刷新做短防抖
3. 页面切离首页后不刷新

## 9. 异常与降级策略

首页接口应支持部分降级，而不是全页失败：

1. 主待办失败：主卡显示局部错误态
2. 风险失败：风险卡内部显示 `--`
3. KPI 失败：KPI 卡内部显示 `--`
4. 后端返回 `degraded_blocks`，前端按卡片粒度提示

首页不弹整页阻断错误，不因为单块失败而让工作台整体不可用。

### 10.1 前端验证

至少补齐：

1. `flutter test`
2. 首页 widget test：
   - 顶部状态条展示
   - 主待办 4 条展示
   - 风险卡展示
   - KPI 卡展示
   - 点击待办/指标跳转
   - 空态与局部失败态
3. `integration_test`：
   - 登录进入首页工作台
   - 刷新首页
   - 待办点击跳转到目标页
   - 风险与 KPI 点击跳转

### 10.2 后端验证

至少补齐：

1. `pytest`
2. 首页聚合 service 单元测试：
   - 待办排序
   - 权限裁剪
   - 风险与 KPI 裁剪
   - 局部降级
3. 首页聚合接口集成测试：
   - 返回结构
   - 不同角色
   - 空数据
   - 部分模块失败

### 10.3 性能门禁

新增首页接口后，必须纳入现有 40 并发 P95 门禁：

1. 将 `GET /api/v1/ui/home-dashboard` 加入现有容量验证场景。
2. 口径沿用当前项目主线：
   - `concurrency = 40`
   - `p95 <= 500ms`
   - `error_rate <= 5%`
3. 验证结果必须落 `evidence/`。

### 10.4 性能实现约束

为确保首页接口能通过 40 并发门禁，后端需遵守：

1. 首页聚合服务内部只做服务层直接聚合，不做 HTTP 回调。
2. 待办与风险尽量用少量查询完成，不为首页重复扫描大表。
3. KPI 只取首页必要字段，不加载图表明细。
4. 当前用户信息不在首页接口中重复计算和返回。

## 11. 实施边界

本设计只覆盖首页工作台业务化，不扩展到：

1. 消息中心整体重构
2. 生产/质量/设备模块明细页改版
3. 角色自定义首页布局
4. 首页内联审批或已读操作

第一版首页只负责摘要与跳转，不在首页直接执行复杂业务动作。

## 12. 最终结论

首页工作台采用“待办中枢型”方案，使用现有前端视觉语言重做，首屏无纵向滚动，左侧主区承载待办分发，右侧显示风险与 KPI 摘要。后端新增 `UI` 层聚合接口统一下发摘要数据，前端通过新增首页模型与服务消费。实现必须同步补齐前端测试、后端测试、`integration_test` 与 40 并发 P95 证据。
