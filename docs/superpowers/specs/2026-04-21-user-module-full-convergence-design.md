# 用户模块完整收口设计

## 1. 背景

前端 `core/ui` 基础件体系第一轮已经落地，用户模块也已经完成过一轮第二波迁移试点，当前已完成的重点包括：

1. `user_management`
2. `registration_approval`
3. 用户模块共享展示层

这意味着用户模块已经具备继续向完整收口推进的基础能力，但它当前仍处于“部分页面已进入新模式、总页壳层与剩余页签仍处于历史过渡态”的阶段。

与产品模块相比，用户模块的差异不在于“有没有页面可用”，而在于：

1. `UserPage` 总页壳层仍是旧式 `TabBar + TabBarView` 装配
2. 已完成迁移的业务页与尚未统一的支持页、管理页并存
3. 模块级验证尚未形成和产品模块相近的统一口径

因此这轮工作的目标不是局部修补，而是把用户模块推进到“总页与全部子页都能按统一口径被理解、被验证、被继续演进”的完整收口状态。

## 2. 当前现状

### 2.1 UserPage 总页壳层仍是历史过渡态

`frontend/lib/features/user/presentation/user_page.dart` 当前仍直接以 `TabBar + TabBarView` 组织全部页签，虽然功能可用，但它仍主要承担：

1. 页签顺序拼装
2. 可见性判断
3. 偏好页签处理
4. 部分路由载荷分发
5. 子页直接装配

它目前还不是一个清晰、稳定、可持续演进的“模块总控壳层”。

### 2.2 用户模块仍存在明显代际差异

当前用户模块内至少存在三类状态：

1. 已完成第二波迁移的主业务页
   - `user_management`
   - `registration_approval`
2. 仍按旧式写法装配的支持页 / 管理页
   - `role_management`
   - `audit_log`
   - `account_settings`
   - `login_session`
   - `function_permission_config`
3. 作为壳层存在但尚未按新口径收口的 `UserPage`

这意味着当前用户模块不能被视为“模块级完成”，只能视为“局部页签完成”。

### 2.3 当前验证方式仍偏零散

仓库中已存在：

1. `frontend/test/widgets/user_page_test.dart`
2. `frontend/test/widgets/user_management_page_test.dart`
3. `frontend/test/widgets/registration_approval_page_test.dart`
4. 其他支持页相关测试

这些测试足以证明单页可用，但还不足以证明“用户模块已经完整收口”。缺少的是：

1. 总页壳层与页签整体关系的模块级标准
2. 批次间独立收口的清晰验收口径
3. 一套可以支撑后续继续演进的统一模块叙述

## 3. 总体路线

本轮采用：

1. **一个总 spec 覆盖整个用户模块完整收口**
2. **后续实施拆成 3 批推进**

不采用“一次实现到底”的原因很直接：

1. 范围已经覆盖 1 个总页壳层和 7 个页签
2. 若把它压成单个实现批次，计划、TDD、验证和留痕成本都会明显升高
3. 当前工作流更适合“一个总方向 + 多个高内聚实现批次”

因此，这轮要解决的是“方向统一”，不是“实现一次吞完”。

## 4. 目标

1. 将 `UserPage` 推进为稳定的用户模块总控壳层。
2. 让 `user_management` 与 `registration_approval` 在总模块视角下完成真正收口。
3. 将 `role_management`、`audit_log`、`account_settings`、`login_session`、`function_permission_config` 推进到与总页壳层一致的模块口径。
4. 让用户模块形成统一的验证与留痕闭环。

## 5. 非目标

1. 本轮不改后端接口契约。
2. 本轮不同时引入新的全局基础件体系。
3. 本轮不把全部用户页强行做成同一种 CRUD 页面。
4. 本轮不把用户模块一次性压成单个实现批次。
5. 本轮不将其他模块一并纳入。

## 6. 完整收口后的目标结构状态

用户模块完整收口后，应达到以下状态：

### 6.1 总页壳层统一

`UserPage` 不再只是“能切页签”的容器，而是稳定的用户模块总控壳层。它至少应做到：

1. 页签顺序、可见性、默认落点、跳转载荷处理都有明确口径
2. 子页装配方式稳定，不再是历史遗留页与新页并置的过渡态
3. 后续新增或改用户页签时，有明确的壳层接入方式

### 6.2 主业务页完成模块级收口

以下页面应从“局部已迁移”升级为“在总模块语境下也已收口”：

1. `user_management`
2. `registration_approval`

### 6.3 剩余页签达到同一口径

以下页签不要求长得完全一样，但应达到同一套判断标准：

1. `role_management`
2. `audit_log`
3. `account_settings`
4. `login_session`
5. `function_permission_config`

统一标准包括：

1. 页面装配方式清晰
2. 关键状态与反馈出口明确
3. 与 `UserPage` 壳层的关系稳定
4. 不再依赖历史过渡写法才能工作

### 6.4 模块级验证方式统一

完整收口后，用户模块应形成：

1. 总页壳层层面的稳定测试
2. 子页层面的页面级回归
3. 至少一组模块级主路径验证
4. 对应的 `evidence` 留痕闭环

## 7. 分层方式

### 7.1 总页壳层

`UserPage` 单独作为一层，只负责：

1. 页签顺序与可见性
2. 默认页签与偏好页签切换
3. 路由载荷分发
4. 子页装配与页签壳层稳定性

它不应继续承担具体业务页规则判断。

### 7.2 主业务子页层

这一层包括：

1. `user_management`
2. `registration_approval`
3. `role_management`
4. `audit_log`

这些页面都偏“管理 / 列表 / 审批 / 审计”类型，应尽量按统一思路处理：

1. 页头
2. 筛选区
3. 反馈区
4. 列表区
5. 分页或主内容区

### 7.3 支持页签层

这一层包括：

1. `account_settings`
2. `login_session`
3. `function_permission_config`

它们不一定走同一种 CRUD 骨架，但应满足：

1. 装配方式明确
2. 反馈出口明确
3. 与 `UserPage` 的接口关系明确
4. 后续可以继续按统一模块口径演进

## 8. 三批实施边界

### 第 1 批：总页壳层与主业务页

覆盖：

1. `UserPage`
2. `user_management`
3. `registration_approval`

这一批要解决两个核心问题：

1. 用户总页壳层是否稳定、是否能长期承载子页
2. 主业务页是否真正进入第二波模式，而不是局部完成

### 第 2 批：组织与审计页

覆盖：

1. `role_management`
2. `audit_log`

这一批要解决的是“管理 / 审计类页面是否与主业务页站到同一口径”。

### 第 3 批：个人与配置页

覆盖：

1. `account_settings`
2. `login_session`
3. `function_permission_config`

这一批要解决的是“支持页签是否稳定接入总壳层，且不再依赖历史过渡写法”。

## 9. 完成标准

以下条件全部满足，才算“用户模块完整收口”：

1. `UserPage` 总页壳层完成收口
2. 7 个页签全部进入统一口径
3. 三批之间不存在“半收口挂账”
4. 形成总页壳层测试、页面级回归、模块级主路径验证和 `evidence` 闭环

## 10. 风险与控制方式

### 风险 1：范围滑成“用户模块全量重写”

控制方式：

1. 每一批只解决该批定义的问题
2. 不顺手重写不在本批目标中的页面

### 风险 2：总页壳层与子页一起改，导致回归面过大

控制方式：

1. 第 1 批只优先解决 `UserPage + user_management + registration_approval`
2. 其余页签不提前混入第 1 批

### 风险 3：支持页签被错误套成 CRUD 页

控制方式：

1. 对 `account_settings`、`login_session`、`function_permission_config` 只要求“稳定接入 + 结构清晰 + 反馈明确”
2. 不强行套统一 CRUD 页面形态

## 11. 预期结果

本轮完成后，用户模块会从“部分页已迁、部分页仍在过渡”进入“总页与全部子页都能按统一口径被理解、被验证、被继续演进”的状态。

也就是说：

1. `UserPage` 变成稳定的总控壳层
2. 主业务页、管理页和支持页都完成模块级收口
3. 后续继续维护用户模块时，不再需要区分“这是历史页”还是“这是新页”
