# Docker 后端自动化回归与后续治理设计

## 1. 背景

当前仓库已经完成一轮后端高优先级治理：

1. 安全硬化：危险默认值已被 fail-fast 拦截。
2. 事务边界统一：已清理多条 service/API 双层提交链路。
3. `authz_service` 已完成读路径与部分写路径拆分。

同时，Docker 口径的后端服务已经能够在不占用宿主 `5432` 的前提下旁路部署：

- 宿主 PostgreSQL：继续占用 `5432`
- Docker PostgreSQL：映射到宿主 `5433`
- Docker backend-web：映射到宿主 `8000`

当前最大的缺口不再是“单点代码问题”，而是“回归验证尚未沉淀为仓库内可重复执行入口”。因此本轮任务目标是：

1. 将 Docker 后端烟测固化为仓库内可执行脚本
2. 在该自动化中补上更长的业务链路
3. 在回归兜底下继续扩大事务边界统一与 `authz` 写路径拆分

## 2. 目标与非目标

### 2.1 目标

1. 提供一个仓库内可重复执行的 Docker 后端自动化回归入口。
2. 覆盖当前已验证过的关键烟测：
   - `/health`
   - 管理员登录
   - `authz` 能力包目录读取
   - 角色创建
   - 用户创建
   - 产品创建
3. 在同一套自动化里继续补首件、生产订单、导出链路。
4. 在该自动化保护下，继续清理剩余模块的事务边界问题。
5. 继续拆 `authz_service` 写路径，优先抽出角色权限矩阵更新、能力包批量应用、change log 记录。

### 2.2 非目标

1. 不在本轮改造前端或 Flutter。
2. 不改变宿主本地 PostgreSQL 的运行方式。
3. 不一次性完成全部后端模块重构。

## 3. 方案对比

### 方案 A：单个 Python 自动化脚本

- 入口示例：`tools/docker_backend_smoke.py`
- 优点：
  - 跨平台
  - HTTP 断言、超时、错误信息都容易控制
  - 未来接 CI 最顺
- 缺点：
  - 单文件会逐步变长

### 方案 B：PowerShell 编排脚本

- 入口示例：`tools/docker_backend_smoke.ps1`
- 优点：
  - 很贴合当前 Windows 环境
  - 对 `docker compose` 操作直接
- 缺点：
  - 跨平台成本更高

### 方案 C：多文件分层脚本

- 例如 `PowerShell` 负责 Docker 生命周期，`Python` 负责 HTTP 断言
- 优点：
  - 职责边界清晰
- 缺点：
  - 维护成本更高
  - 当前阶段会增加文件数量和理解成本

### 结论

采用方案 A：单个 Python 自动化脚本为主，必要时再拆辅助模块。

## 4. 设计

### 4.1 自动化脚本设计

建议新增：

- `tools/docker_backend_smoke.py`

脚本职责：

1. 设置 Docker 部署所需环境变量。
2. 调用 `docker compose up -d --build` 启动：
   - `postgres`
   - `redis`
   - `backend-web`
   - `backend-worker`
3. 等待：
   - Docker PostgreSQL 健康
   - Redis 健康
   - `backend-web` 健康
4. 执行 HTTP 级烟测与长链路断言。
5. 失败时输出：
   - 哪一步失败
   - HTTP 状态码
   - 响应正文
   - 必要时补充 `docker compose logs --tail`

### 4.2 Docker 环境口径

脚本内部统一使用以下默认值：

- `POSTGRES_HOST_PORT=5433`
- `BACKEND_WEB_HOST_PORT=8000`
- `JWT_SECRET_KEY=docker-local-jwt-secret-20260414`
- `BOOTSTRAP_ADMIN_PASSWORD=Admin_Local_20260414!`
- `PRODUCTION_DEFAULT_VERIFICATION_CODE=FA20260414`

约束：

1. 不修改宿主 `5432` 监听状态。
2. 所有后端验证以容器化后端为准。
3. 若脚本检测到 `8000` 被占用，必须明确失败并报出冲突，而不是默默换端口。

### 4.3 第一批固化的烟测

脚本第一阶段必须覆盖：

1. `/health`
2. `/api/v1/auth/login`
3. `/api/v1/authz/capability-packs/catalog?module=user`
4. `/api/v1/roles`
5. `/api/v1/users`
6. `/api/v1/products`

输出要求：

1. 每一步打印清晰步骤名
2. 成功时输出状态码
3. 失败时输出响应正文并终止

### 4.4 第二批补充的长链路

在同一脚本中继续补三类链路：

1. 首件提交流程
2. 生产订单主链路
3. 导出链路

设计原则：

1. 每类链路至少覆盖一条真实成功路径
2. 测试数据由脚本自身创建并在结束时清理
3. 不依赖人工预置中间数据

### 4.5 事务边界继续扩展

在自动化保护下，继续扫描并治理剩余双层提交链路，优先顺序：

1. `message`
2. `equipment`
3. `craft`
4. `quality`

治理原则保持不变：

1. service 负责对象构建与必要 `flush()`
2. API 负责最终 `commit()/rollback()`
3. 每次新增一批链路时，先补红灯测试，再做实现

### 4.6 authz 写路径继续拆分

本轮继续从 `authz_service.py` 中抽出低耦合写路径职责，优先：

1. 角色权限矩阵更新
2. 能力包批量应用
3. change log 记录

原则：

1. 对外接口不变
2. 先抽 helper，再考虑进一步下沉 orchestrator
3. 每次拆分后都跑现有 `authz` 单测

## 5. 验收标准

### 5.1 自动化回归验收

1. 仓库内存在单一入口脚本。
2. 脚本可以在当前机器上直接拉起 Docker 后端并完成第一批烟测。
3. 失败时能输出足够定位信息。

### 5.2 长链路验收

1. 首件、生产订单、导出三类链路至少各有一条真实成功路径。
2. 链路断言不是只看 `200`，还要校验关键返回字段或副作用。

### 5.3 后端治理验收

1. 新一批事务边界统一链路有红绿验证。
2. `authz` 写路径拆分后现有单测保持通过。

## 6. 风险与应对

### 风险

1. Docker 脚本与本地环境耦合，容易被端口冲突或宿主残留状态影响。
2. 长链路脚本如果写得太大，会变成新的维护负担。
3. 继续扩事务边界和 `authz` 写路径时，可能让本轮范围失控。

### 应对

1. 自动化脚本先固化最小成功路径，再逐批扩充。
2. 每加一类长链路，都保持清晰函数边界。
3. 若某一治理项超出本轮可控范围，优先保证自动化回归入口完成。
