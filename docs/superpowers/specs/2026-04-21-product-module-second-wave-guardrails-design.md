# 产品模块第二波迁移防回退治理设计

## 1. 背景

前端 `core/ui` 基础件体系第一轮已经落地，且产品模块第二波迁移已经连续完成以下 4 个页面：

1. `ProductVersionManagementPage`
2. `ProductManagementPage`
3. `ProductParameterManagementPage`
4. `ProductParameterQueryPage`

这意味着产品模块已经不再处于“单页试点”阶段，而是形成了完整的第二波迁移样板：既有主列表 + 工作区页，也有 CRUD + 侧栏 / 弹窗页，还有复杂参数工作台页和轻量查询页。

当前最现实的风险已经不是“迁不过去”，而是后续正常开发中把这些页面慢慢改回旧模式，例如：

1. 把统一骨架改回手工拼装
2. 把弹窗 / 侧栏 / 摘要区拆回主页面
3. 把既定查询口径回退到旧接口
4. 把动作入口位置和结构锚点改散

因此下一步最合适的工作不是立即切到别的业务模块，而是先补一层轻量但有效的产品模块防回退治理。

## 2. 当前现状

### 2.1 产品模块第二波迁移已经形成完整闭环

从最近的 `spec / plan / evidence` 可以确认，产品模块第二波迁移已形成完整闭环，最近新增的设计、计划和实施留痕均集中在产品模块。

这说明：

1. 当前页面模式已经相对稳定
2. 当前上下文仍然很热
3. 现在补治理的收益最高

### 2.2 现有回归较强，但职责还没有分层

当前已经存在较强的产品模块回归，尤其是：

1. `frontend/test/widgets/product_module_issue_regression_test.dart`

它已经覆盖了多条高价值行为链路，但它的职责更偏向：

1. 历史 issue 回归
2. 业务口径回归
3. 交互 bug 防回归

它还不是专门为“第二波迁移成果防回退”设计的结构门禁文件。

### 2.3 当前缺少一份显式的产品模块迁移共识

产品模块的页面结构、页面骨架、弹窗层级、工作区模式和摘要区模式，当前主要隐含在代码与测试里，而不是一份短小明确的约束文档里。

这意味着：

1. 熟悉上下文的人知道哪些模式已经固定
2. 不熟悉上下文的人很难快速判断“什么改动算回退”

因此仅有测试门禁还不够，还需要一份简短、明确、能与测试一一对应的治理说明。

## 3. 已确认路线

本轮采用方案 B：轻量双轨治理。

### 3.1 方案 A：测试门禁优先

只补产品模块第二波迁移的防回退测试，不补显式约束文档。

优点：

1. 见效最快
2. 约束最硬

缺点：

1. 共识只存在于测试里
2. 新进入上下文的人不容易快速理解边界

### 3.2 方案 B：轻量双轨治理

同时补两类产物：

1. 一份独立的产品模块防回退测试门禁
2. 一份短小明确的产品模块治理说明

优点：

1. 既有硬门禁，也有显式约束
2. 最适合当前“刚完成四页迁移、上下文仍很热”的阶段
3. 范围仍然可控，不会滑向大治理工程

缺点：

1. 比纯测试方案多一层整理成本

### 3.3 方案 C：深治理

在测试和文档之外，再继续引入更重的脚本化审计、golden 或更多全局抽象。

优点：

1. 长期门禁更强

缺点：

1. 会把“防回退治理”扩大成新的工程建设
2. 当前阶段过重

### 最终选择

本轮明确采用方案 B，不采用方案 A 和方案 C。

## 4. 目标

1. 为产品模块补一份独立的第二波迁移防回退测试门禁。
2. 为产品模块补一份短小、明确、可执行的第二波迁移治理说明。
3. 锁定 4 个已迁移页面的关键结构、关键交互入口与关键接口口径。
4. 将“哪些回退不允许”从隐性经验转为显式约束。

## 5. 非目标

1. 本轮不改后端。
2. 本轮不继续做页面视觉优化。
3. 本轮不切换到质量、设备等其他业务模块。
4. 本轮不重做任何一个产品页面。
5. 本轮不顺手新增新的全局基础件。
6. 本轮不引入重量级脚本化审计或 golden 体系。
7. 本轮不试图用新治理文件替代现有历史 issue 回归。

## 6. 治理总体设计

### 6.1 双轨结构

本轮治理采用两层结构：

1. 硬门禁层：独立的产品模块第二波迁移防回退测试
2. 软约束层：短小明确的产品模块治理说明

### 6.2 双轨职责

硬门禁层负责回答：

1. 这些页面的第二波迁移成果是否还在
2. 是否有人把结构、入口或接口口径改回旧模式

软约束层负责回答：

1. 当前哪些页面模式已经固定
2. 哪些变化属于不可接受的回退
3. 后续改产品页前应先复用哪些既有模式

### 6.3 门禁原则

本轮治理只锁最容易被悄悄改坏的内容：

1. 页面骨架
2. 稳定结构锚点
3. 关键交互入口
4. 关键接口口径

本轮不把治理测试写成第二套业务功能大全。

## 7. 落地文件

建议新增以下核心文件：

1. `frontend/test/widgets/product_module_second_wave_guard_test.dart`
2. `docs/frontend/product-module-second-wave-guardrails.md`

### 7.1 product_module_second_wave_guard_test.dart

职责：

1. 专门承担产品模块第二波迁移成果防回退门禁
2. 按页面分组，锁定结构、入口与接口口径
3. 与 `product_module_issue_regression_test.dart` 职责分离

### 7.2 product-module-second-wave-guardrails.md

职责：

1. 写清已固定的页面模式
2. 写清不允许的回退点
3. 写清后续改产品页时的最小自查清单

文档应保持短小，不写长篇设计理念。

## 8. 各页面防回退清单

### 8.1 ProductVersionManagementPage

建议锁定：

1. 页面仍是“产品选择面板 + 版本工作区”结构
2. 版本页头、反馈区、版本列表区、工具条这些稳定锚点仍在
3. 版本动作入口仍保留：复制、导出参数、编辑说明、立即生效
4. 版本相关操作不回退成旧布局或散落按钮

### 8.2 ProductManagementPage

建议锁定：

1. 页面仍走 `MesCrudPageScaffold`
2. 筛选区、反馈区、列表区稳定锚点仍在
3. 详情仍是页内侧栏，不回退成旧弹窗或整页跳转
4. 版本管理仍是弹窗工作区，不回退成旧式堆叠结构

### 8.3 ProductParameterManagementPage

建议锁定：

1. 列表态仍是统一骨架
2. 编辑态仍是独立工作区结构，不回退成大页内混排
3. 历史弹窗与快照弹窗入口仍在
4. 版本绑定、历史绑定、保存绑定这些关键口径不回退

### 8.4 ProductParameterQueryPage

建议锁定：

1. 查询页仍是统一骨架
2. 参数查看仍走独立弹窗展示层
3. 顶部摘要区仍在，且空态时也保留
4. 查询页仍走只读查询接口，不回退到产品管理列表接口

## 9. 测试边界

### 9.1 要测的内容

1. 页面骨架是否仍在
2. 稳定结构锚点是否仍在
3. 关键弹窗 / 侧栏 / 摘要区是否仍在
4. 关键动作入口是否仍在
5. 关键接口口径是否仍未回退

### 9.2 不测的内容

1. 后端数据正确性
2. 复杂业务流程全链路
3. 像素级视觉一致性
4. 与产品模块无关的跨模块行为

### 9.3 与既有回归的关系

`product_module_issue_regression_test.dart` 继续保留，职责不变。

新增的 `product_module_second_wave_guard_test.dart` 仅负责：

1. 第二波迁移成果的结构门禁
2. 不允许回退的关键口径门禁

二者互补，而不是互相替代。

## 10. 治理说明设计

治理说明建议采用三个部分：

### 10.1 已固定页面模式

列出 4 个产品页当前的既定模式，例如：

1. 哪些页面必须继续走 `MesCrudPageScaffold`
2. 哪些页面采用“主列表 + 侧栏 / 弹窗”
3. 哪些页面采用“列表态 + 编辑工作区”
4. 哪些页面必须保留摘要区

### 10.2 禁止回退清单

明确写出哪些改动属于回退，例如：

1. 把统一骨架改回手工拼装
2. 把独立弹窗 / 侧栏重新塞回主页面
3. 把只读查询接口改回旧列表接口
4. 把固定动作入口改散

### 10.3 后续改造 checklist

要求后续改产品模块页面前至少自查：

1. 是否复用了既有页面模式
2. 是否影响稳定锚点
3. 是否触发防回退门禁
4. 是否改变了既定接口口径

## 11. 实施顺序建议

建议按以下顺序执行：

1. 先写治理说明，先把“什么叫回退”说死
2. 再写独立防回退测试，把文档共识变成硬门禁
3. 最后只做少量必要补丁
   - 稳定 `ValueKey`
   - 少量锚点文案
   - 少量测试桩

本轮不应顺手扩展成页面重构或基础件建设。

## 12. 完成标准

以下条件全部满足，才算本轮完成：

1. `product_module_second_wave_guard_test.dart` 已落地且可独立跑通
2. 治理测试至少覆盖 4 组页面清单
3. 既有 `product_module_issue_regression_test.dart` 继续保留且不被替代
4. 治理说明已经写明“固定模式 / 禁止回退 / 后续自查”
5. 为满足治理测试只做了少量必要补丁，没有滑向大范围改造

## 13. 风险与控制方式

### 风险 1：治理测试写成第二套业务测试

控制方式：

1. 只锁结构、入口、口径
2. 不写全流程业务覆盖
3. 不替代现有 issue 回归

### 风险 2：治理文档写成空泛规范

控制方式：

1. 文档中的规则必须能在测试里找到对应门禁
2. 所有条目都与 4 个页面之一直接关联
3. 避免抽象到全局框架层

### 风险 3：治理范围向新改造扩散

控制方式：

1. 限制补丁类型为 `ValueKey`、锚点文案、测试桩
2. 若出现“顺手抽新基础件 / 顺手继续重构页面”，应视为超范围并停止

## 14. 预期结果

本轮完成后，产品模块将新增一层轻量但有效的防回退治理：

1. 第二波迁移成果被独立测试文件显式锁定
2. 页面模式与禁止回退点被短文档显式写明
3. 后续继续维护产品模块时，能更快判断哪些改动会破坏既定样板

这会让产品模块从“迁移刚完成”进入“样板已固定、可持续维护”的状态。
