# 时间同步功能设计

## 1. 背景

当前局域网内终端电脑经常出现本机时间不准确的问题，这会直接影响前端应用中的时间显示、业务提交时间、超时判断和与服务端时间相关的流程判断。

项目现状上：

1. 前端已经具备应用级软件设置入口，适合承载“时间同步”这类软件级能力。
2. 当前软件设置能力已覆盖登录页、主壳层和全局主题，因此时间同步也应当走应用级能力，而不是挂在业务模块内部。
3. 后端当前只有 `/health` 健康检查接口，尚无明确返回“服务器当前时间”的专用接口。

本轮目标是在前端新增“时间同步”能力，并默认启用，使软件在启动时能以后端服务器时间为权威时间源，自动检查本机时间是否偏移；当偏差超过阈值时，主程序自动尝试提权修正 Windows 系统时间；若失败，则明确告警并退化为“仅软件内时间校准”。

## 2. 已确认决策

以下决策已由用户确认：

1. 权威时间源：以后端服务器时间为准。
2. 功能目标：不只是软件内显示校准，而是尝试直接修正 Windows 系统时间。
3. 默认策略：前端默认启用时间同步。
4. 触发时机：应用启动后自动检查。
5. 自动修正阈值：偏差超过 `30 秒` 才自动尝试修正系统时间。
6. 权限模型：接受 UAC / 管理员权限。
7. 失败策略：若无权限、用户取消 UAC 或改时失败，则明确告警，并退化为“仅软件内时间校准”。
8. 实现路线：主程序自己提权并直接改时间，不引入独立本地助手或常驻 Windows 服务。

## 3. 目标

1. 新增应用级时间同步能力，并在默认配置下自动启用。
2. 在启动时自动检测本机时间和服务器时间偏差。
3. 当偏差超过 `30 秒` 时，自动尝试提权并修正 Windows 系统时间。
4. 当系统改时失败时，自动退化为软件内时间校准，保证软件内业务时间仍尽量与服务器保持一致。
5. 在软件设置页提供时间同步分区、状态展示和手动操作入口。

## 4. 非目标

1. 本轮不接入外部 NTP 服务或局域网内其他时间服务器。
2. 本轮不引入常驻 Windows 服务。
3. 本轮不新增单独的本地提权助手程序。
4. 本轮不做用户可配置阈值、可配置重试策略等高级设置；首版阈值固定为 `30 秒`。
5. 本轮不改变服务端各业务模块内部时间存储标准，后端仍以既有 UTC / 服务端时间逻辑为准。

## 5. 整体方案

首版采用“服务器时间接口 + 前端时间同步协调器 + 主程序自提权改时 + 软件内时间校准兜底”的四层结构。

### 5.1 后端时间接口

后端新增一个无登录态依赖的时间接口，专门返回服务器当前时间快照，作为前端时间同步的权威时间源。

### 5.2 前端时间同步协调器

前端新增应用级时间同步协调器，与现有 `SoftwareSettingsController` 处于同层级。该协调器负责：

1. 启动时触发自动检查。
2. 请求服务器时间接口。
3. 估算本机与服务器时间偏差。
4. 判断是否超过阈值。
5. 触发系统改时或退化到软件内时间校准。
6. 维护设置页展示所需的时间同步状态。

### 5.3 主程序自提权改时

首版不引入单独辅助程序，而是让主程序支持一个受控的“命令模式”：

1. 正常启动时，主进程检测偏差是否超过阈值。
2. 若超过阈值，则主进程使用 Windows `runas` / UAC 机制重新拉起自身，并带上受控命令参数，例如：
   - `--sync-system-time`
   - `--target-utc-iso=<timestamp>`
3. 被提权拉起的子进程只执行单一职责：
   - 解析目标时间
   - 调用受控的 Windows 改时逻辑
   - 输出标准化结果码
   - 立即退出
4. 原进程接收结果码，更新时间同步状态并决定是否退化。

这仍属于“主程序自己提权改时间”，但把系统级操作压缩在一个受控命令路径内，避免把改时逻辑扩散到普通 UI 流程。

### 5.4 软件内时间校准兜底

若系统改时失败，但已成功获取服务器时间，则前端进入“软件内时间校准”模式。

该模式下，前端继续保留一个服务器时间偏移量，并通过统一的 `EffectiveClock` 对业务显示时间和关键时间判断做校准。

## 6. 启动流程

### 6.1 应用启动主链路

1. 应用启动。
2. 读取软件设置。
3. 若 `时间同步` 开关关闭，则跳过检查。
4. 若开关开启，则异步请求服务器时间接口。
5. 计算本机时间与服务器时间偏差。
6. 若偏差 `<= 30 秒`：
   - 不触发 UAC
   - 状态记为“系统时间正常”
7. 若偏差 `> 30 秒`：
   - 触发一次 UAC 提权
   - 尝试修正 Windows 系统时间
8. 若改时成功：
   - 更新状态为“已自动修正”
9. 若用户取消 UAC、无权限或改时失败：
   - 弹明确告警
   - 切换到“软件内时间校准”模式

### 6.2 启动阻塞策略

首版建议采用“首屏不阻塞，检查异步启动”的方式：

1. 应用正常进入登录页或主界面。
2. 时间同步在应用早期异步执行。
3. 若需要 UAC，则由协调器在检测到超阈值后立即触发。

这样能避免把应用启动完全绑死在时间同步上，同时保留“启动即自动检查”的体验。

## 7. 软件设置与用户提示

### 7.1 设置页分区

在现有软件设置页左侧分区中新增：

1. `外观`
2. `布局偏好`
3. `时间同步`

### 7.2 持久化设置项

首版建议只新增一个持久化设置项：

- `timeSyncEnabled: bool`
  - 默认值：`true`

首版不开放阈值配置、自动重试次数配置等高级选项。

### 7.3 时间同步分区结构

时间同步分区右侧内容包含：

1. `总开关`
   - 名称：`启用时间同步`
   - 默认开启
2. `同步策略摘要`
   - 权威时间源：后端服务器时间
   - 自动修正阈值：30 秒
   - 修正方式：偏差超阈值时自动尝试提权修正 Windows 时间
   - 失败处理：明确告警并退化为软件内时间校准
3. `当前状态卡片`
   - 服务器时间
   - 本机时间
   - 当前偏差
   - 最近同步结果
   - 最近同步时间
   - 当前模式：系统时间同步 / 软件内校准 / 未启用
4. `手动操作区`
   - `立即检查并同步`
   - `仅重新校准软件内时间`

### 7.4 用户可见状态

至少覆盖以下状态语义：

1. `系统时间正常`
2. `检测到时间偏差 47 秒，已自动修正`
3. `系统改时失败，当前已退化为软件内时间校准`
4. `用户取消管理员授权，当前已退化为软件内时间校准`
5. `无法连接服务器时间接口，暂未完成同步`

### 7.5 失败提示文案

失败告警必须同时说明两件事：

1. Windows 系统时间未成功修改
2. 软件已切换为软件内时间校准，业务时间仍尽量与服务器对齐

推荐语义：

- `未能修改 Windows 系统时间，当前已切换为软件内时间校准，软件内业务时间仍按服务器时间对齐`
- `你已取消管理员授权，系统时间未修改，当前已切换为软件内时间校准`

## 8. 后端时间接口设计

### 8.1 路径与鉴权

建议新增：

- `GET /api/v1/system/time`

鉴权策略：

1. 首版不要求登录态。
2. 该接口用于应用启动期自动检查，必须在登录前可访问。

### 8.2 返回结构

继续沿用项目现有 `ApiResponse<T>` 包装。

`data` 建议包含：

1. `server_utc_iso: string`
   - 服务器当前 UTC 时间，ISO 8601 格式
2. `server_timezone_offset_minutes: int`
   - 服务器时区偏移，仅用于展示
3. `sampled_at_epoch_ms: int`
   - 服务端采样毫秒时间戳，供前端估算延迟

### 8.3 接口职责

该接口只负责返回当前服务器时间快照，不承担任何改时行为。

## 9. 前端时间同步协调器设计

### 9.1 职责边界

协调器负责：

1. 读取时间同步开关
2. 拉取服务器时间
3. 计算时间偏差
4. 判断阈值
5. 调用系统改时命令
6. 在失败时切换到软件内时间校准
7. 向设置页暴露状态

不负责：

1. 直接渲染界面
2. 承担业务模块的时间格式化细节
3. 维护服务端持久化状态

### 9.2 状态模型

建议对外统一暴露以下结果码：

1. `success`
2. `skipped_within_threshold`
3. `cancelled_by_user`
4. `permission_denied`
5. `sync_failed`
6. `server_time_unavailable`

建议同时维护一个运行时模式：

1. `disabled`
2. `system_time_ok`
3. `system_time_corrected`
4. `software_time_calibrated`
5. `unavailable`

### 9.3 偏差计算

前端请求服务器时间时，记录：

1. 请求发起时间
2. 响应接收时间
3. 服务器采样时间

首版允许使用简化估算：

- `estimatedServerNow = serverSample + roundTrip / 2`

随后计算：

- `drift = localNow - estimatedServerNow`

### 9.4 重试策略

首版建议：

1. 每次启动自动检查只做 `1 次`
2. 失败后本次启动不重复自动弹 UAC
3. 下次应用启动时重新检查
4. 用户可通过设置页手动触发再次同步

## 10. 软件内时间校准设计

### 10.1 时间源分层

建议引入两层时间源：

1. `SystemClock`
   - 直接读取本机系统时间
2. `EffectiveClock`
   - 业务实际使用时间源
   - 正常模式下：等于系统时间
   - 降级模式下：等于系统时间 + 服务器偏移量

### 10.2 使用原则

在“软件内时间校准”模式下，以下前端逻辑应优先使用 `EffectiveClock`：

1. 时间展示
2. “刚刚 / x 分钟前”类相对时间
3. 客户端侧时间比较逻辑
4. 与超时、有效期有关的前端判断

首版不要求一次性改造仓库中所有 `DateTime.now()`，但要明确把关键业务路径收敛到统一时间入口，避免新旧口径混用。

## 11. Windows 改时执行设计

### 11.1 执行方式

正常主进程仅在判断“偏差超阈值”后，才以受控参数重新拉起自身并请求 UAC。

被提权拉起的命令模式：

1. 不进入普通 UI 路径
2. 只解析目标 UTC 时间
3. 执行系统改时
4. 返回标准化结果码
5. 退出进程

### 11.2 安全边界

必须限制该命令模式的参数范围：

1. 只允许传入单个目标时间参数
2. 不接受任意命令拼接
3. 不允许复用为通用管理员命令入口

### 11.3 失败处理

以下情形一律视为失败并退化：

1. 用户取消 UAC
2. 系统策略禁止改时
3. 改时命令执行失败
4. 返回结果不可解析

## 12. 失败降级策略

### 12.1 降级触发

只要满足以下条件：

1. 已成功拿到服务器时间
2. 系统改时未成功

则立即切换到“软件内时间校准”。

### 12.2 不可降级情形

若服务器时间接口本身不可用，则无法建立偏移量，这种情况下：

1. 不进入软件内校准
2. 状态记为 `server_time_unavailable`
3. 设置页明确提示当前未完成同步

## 13. 测试策略

### 13.1 后端测试

至少覆盖：

1. 时间接口返回结构正确
2. UTC 时间格式稳定
3. 无登录态可访问

### 13.2 前端单测

至少覆盖：

1. 偏差计算逻辑
2. `30 秒` 阈值判断
3. 不同结果码到运行时状态的映射
4. 降级后 `EffectiveClock` 的偏移逻辑
5. 时间同步开关开启 / 关闭行为

### 13.3 Widget 测试

至少覆盖：

1. 时间同步分区渲染
2. 正常、已修正、失败降级、服务器不可用等状态展示
3. 手动操作按钮
4. 告警提示语义

### 13.4 Windows 集成验证

至少覆盖：

1. 偏差未超阈值时不触发提权
2. 偏差超阈值时触发改时流程
3. 用户取消 UAC 后进入软件内校准
4. 改时失败后业务时间仍按服务器偏移量工作

## 14. 风险与控制

### 风险 1：UAC 打扰用户

- 控制：
  - 只在偏差超过 `30 秒` 时触发
  - 每次启动自动尝试仅一次

### 风险 2：系统改时失败导致用户误以为功能不可用

- 控制：
  - 明确告知已退化为软件内时间校准
  - 在设置页持续展示当前模式

### 风险 3：前端时间口径混用

- 控制：
  - 引入统一 `EffectiveClock`
  - 关键路径优先收敛到统一时间入口

### 风险 4：主程序提权路径被滥用

- 控制：
  - 受控命令参数
  - 单一职责命令模式
  - 不暴露通用管理员能力
