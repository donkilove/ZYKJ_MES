# 首件扫码复核手机端 7 天免登录设计

- 日期：2026-04-26
- 状态：待实现
- 适用范围：首件扫码复核手机 Web 页面、认证口径、会话时效
- 迁移说明：无迁移，直接替换。

## 1. 背景与目标

当前首件扫码复核手机页每次扫码都会回到登录表单。即使同一台手机、同一个微信会话连续复核多张单据，也需要重复输入账号密码，现场体验偏差。

本次目标是把手机扫码复核调整为“同一台手机 7 天免登录”：

1. 用户首次在手机扫码页登录成功后，系统在本地保存扫码复核专用登录态。
2. 在 7 天有效期内，同一手机再次扫码时优先复用该登录态，直接进入复核详情。
3. 当登录态失效、账号停用、密码变更、权限收回或用户主动切换账号时，再回到登录页。
4. 该长效登录只作用于首件扫码复核手机页，不改变桌面端和主系统的默认登录时效。

## 2. 非目标

1. 不把全系统 JWT 有效期统一改成 7 天。
2. 不为手机扫码页引入刷新 token 体系。
3. 不改变现有扫码复核业务流程、权限模型和二维码时效。
4. 不改造桌面端登录页或主壳登录体验。

## 3. 方案对比

### 3.1 方案 A：扫码页专用 7 天登录态

手机扫码页登录成功后，把扫码复核专用 token 与过期时间保存到浏览器本地存储。后端为该入口签发 7 天有效的专用访问 token。页面初始化时优先尝试复用本地 token，请求复核详情成功则直接进入详情页。

优点：

1. 改动聚焦在扫码页，不影响桌面端和主系统。
2. 体验符合现场诉求，同一手机 7 天内基本无需重复登录。
3. 失败回退清晰，token 失效时自动回到登录页。

缺点：

1. 需要在后端区分“扫码页专用登录”和常规登录的 token 时效。
2. 需要新增本地持久化和自动恢复逻辑。

### 3.2 方案 B：全系统 token 统一改成 7 天

直接把现有登录 token 时效从 120 分钟提高到 7 天，手机页把 token 保存在本地复用。

优点：

1. 改动少。
2. 无需新增扫码页专用认证口径。

缺点：

1. 会影响桌面端和整个系统的会话安全边界。
2. 与本次“只优化扫码页体验”的范围不匹配。

### 3.3 方案 C：短 token + 刷新 token

引入刷新 token 或会话续期机制，手机页保存刷新 token 并定期换取新 access token。

优点：

1. 长期体验最好。
2. 安全和体验平衡更细。

缺点：

1. 复杂度明显高于当前需求。
2. 需要新增额外接口、存储和失效处理规则。

### 3.4 结论

采用方案 A。它能满足“7 天免登录”的体验目标，同时把影响范围控制在扫码复核手机页，不把主系统的默认登录边界一起拉长。

## 4. 架构设计

### 4.1 总体思路

后端新增“扫码复核手机页登录”口径，本质仍复用现有账号密码校验，但签发专用的 7 天 token。前端手机页首次登录成功后，将该 token、过期时间和最近登录账号保存到本地存储。之后页面初始化先尝试自动恢复：

1. 从本地存储读取 token 与过期时间。
2. 未过期时直接用该 token 请求首件复核详情。
3. 成功则跳过登录表单，直接显示详情。
4. 失败则清理本地登录态并回到登录页。

### 4.2 登录态边界

该登录态仅用于手机扫码复核入口：

1. 只在 `/first-article-review` 页面内读写本地 token。
2. 不影响桌面端登录页和主壳中的 `AuthService.login` 默认行为。
3. 不要求桌面端共享该 token。

### 4.3 时效

1. 扫码复核专用 token 默认有效期 7 天。
2. 本地存储的过期时间以服务端返回值为准。
3. 若浏览器本地时间异常，只要服务端返回 401 或会话失效，也必须立即清除本地登录态。

## 5. 后端设计

### 5.1 配置

在后端配置中新增扫码页专用 token 时效配置，建议为：

- `mobile_scan_review_jwt_expire_minutes: int = 10080`

该配置只用于扫码复核手机页登录，不替代现有 `jwt_expire_minutes`。

### 5.2 登录接口

新增一个扫码复核手机页专用登录接口，建议路径：

- `POST /api/v1/auth/mobile-scan-review-login`

输入与现有 `/api/v1/auth/login` 一致：

- `username`
- `password`

输出保持与现有登录接口兼容：

- `access_token`
- `token_type`
- `expires_in`
- `must_change_password`

差异仅在于：

1. token 有效期使用 `mobile_scan_review_jwt_expire_minutes`。
2. 仍记录登录日志、用户会话和最后登录终端。
3. 仍遵守现有账号状态、密码、审批与停用校验。

### 5.3 会话失效行为

以下情况返回现有认证错误码或权限错误：

1. token 过期
2. 用户被停用
3. 用户密码被强制修改且后端需要重新登录
4. 用户不具备扫码复核权限

前端不新增复杂区分逻辑，只要自动恢复失败就清本地 token 并显示登录表单；若是业务错误，再显示当前错误信息。

## 6. 前端设计

### 6.1 本地存储内容

手机扫码页新增一组独立 key，建议至少包含：

- `firstArticleScanReview.accessToken`
- `firstArticleScanReview.expiresAt`
- `firstArticleScanReview.username`

不与桌面端或其他功能共用 key。

### 6.2 页面初始化

当前页面 `initState` 只检查二维码 token 是否为空，后续改为：

1. 先读取本地扫码复核登录态。
2. 若没有本地 token，显示登录表单。
3. 若本地 token 已过期，清理后显示登录表单。
4. 若本地 token 未过期，尝试直接加载复核详情。
5. 加载成功则进入详情页。
6. 若接口返回 401/403 或其他认证失败，清理本地登录态并回到登录页。

### 6.3 登录成功

当前登录成功后只把 token 放到内存变量里；改造后应：

1. 保存 token
2. 保存服务端返回的过期时间
3. 保存最近登录账号
4. 立刻加载复核详情

### 6.4 提交复核

提交复核逻辑不变，仍使用当前内存中的 access token。若提交时返回 401，则：

1. 清理本地登录态
2. 回到登录页
3. 提示“登录已失效，请重新登录后再提交”

### 6.5 切换账号

在手机页详情区补一个轻量“切换账号”按钮。行为：

1. 清理本地登录态
2. 清空内存中的 token 与详情状态
3. 回到登录页

不额外提供“记住我”开关，默认即为 7 天免登录。

## 7. 错误处理

### 7.1 自动恢复失败

若本地 token 存在但自动加载详情失败：

1. 清除本地 token
2. 保留当前扫码 token
3. 回到登录页
4. 若错误是认证类，提示“登录已失效，请重新登录”
5. 若错误是业务类，例如二维码失效，则保留业务提示

### 7.2 账号密码错误

登录表单应继续显示明确的账号密码错误文案，而不是笼统“请求失败”。

### 7.3 二维码失效

即使用户处于免登录状态，扫码 token 仍可能失效。此时应直接显示：

- “二维码已失效，请操作员刷新后重新扫码”

不应把二维码失效误判成登录失效。

## 8. 测试设计

### 8.1 后端

新增或扩展单测，覆盖：

1. 扫码复核专用登录接口返回 7 天 `expires_in`
2. 常规登录接口仍保持原有时效
3. 手机专用登录的账号状态、密码错误、停用校验不变

### 8.2 前端

新增或扩展 widget test，覆盖：

1. 无本地登录态时显示登录表单
2. 有未过期本地 token 时自动加载详情并跳过登录表单
3. 自动恢复遇到 401 时清理本地 token 并回到登录表单
4. 提交复核遇到 401 时清理登录态并提示重新登录
5. 点击“切换账号”后清理本地登录态

### 8.3 运行态验证

至少做一次真实验证：

1. 手机首次扫码登录
2. 关闭微信页后重新扫码
3. 7 天窗口内无需再次输入账号密码即可直接进入复核页

## 9. 风险与约束

1. 微信内置浏览器对本地存储策略相对严格，但 `localStorage` 仍是当前最轻方案；若后续发现被清理频繁，再升级为 cookie 或刷新 token。
2. 该方案默认“同一台手机、同一浏览器容器”记住登录；更换微信容器、清浏览器缓存或系统清理数据后仍需重新登录。
3. 若团队后续要求更细安全边界，再把专用 7 天 token 收敛为刷新 token 体系。

## 10. 实施结论

本次按“扫码页专用 7 天 token + 本地持久化 + 失效自动回退”的最小方案实施。

这能满足现场“就登录一次然后能用很久”的目标，同时把风险和影响范围控制在首件扫码复核手机页内，不扩散到整个 MES 主系统。
