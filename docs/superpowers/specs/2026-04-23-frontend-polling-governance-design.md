# 前端轮询治理设计

- 日期：2026-04-23
- 主题：主壳全局轮询与页面级轮询统一治理
- 状态：待用户审阅

## 1. 背景

当前前端同时存在两类轮询：

1. 主壳全局轮询：
   - 权限可见性刷新
   - 未读数刷新
2. 页面级轮询：
   - 消息中心
   - 生产工单查询
   - 个人中心会话状态

现状问题有三类：

1. 应用退到后台后，部分轮询仍可能继续运行，产生无意义请求。
2. 用户切离当前模块或当前子页后，页面级 `Timer` 仍可能保持常驻。
3. 消息中心已经有主壳级 WebSocket/刷新 tick 驱动，但页面内部仍保留重复拉取，导致刷新链路重复。

本设计的目标是把主壳与页面的轮询行为统一收束到“前台 + 可见 + 活跃”的语义下，减少重复请求，同时不破坏当前业务页的刷新体验。

## 2. 目标与非目标

### 2.1 目标

1. 主壳全局轮询只在应用前台活跃时运行。
2. 页面级轮询只在对应模块和子页处于当前可见态时运行。
3. 页面恢复活跃时立即补拉一次数据，再恢复周期轮询。
4. 收敛消息中心的重复请求链，避免列表拉取后再紧跟一次重复摘要拉取。
5. 为轮询治理补齐可回归的自动化测试。

### 2.2 非目标

1. 本轮不移除所有轮询，也不把所有刷新重写成纯事件驱动。
2. 本轮不修改后端接口。
3. 本轮不做与轮询无关的 UI 重构。

## 3. 推荐方案

推荐采用“统一活跃态门禁”方案：

1. 主壳维护“应用是否允许轮询”的全局状态。
2. 模块容器维护“当前模块是否活跃”的状态。
3. 具体业务页维护“当前子页是否活跃”的状态。
4. 所有实际 `Timer` 都统一受 `pollingEnabled` 风格的输入控制。

不采用“单纯拉长轮询周期”的保守方案，因为那只能减轻负载，不能从结构上解决后台轮询与隐藏页轮询的问题。

不采用“完全事件驱动替代轮询”的激进方案，因为当前范围会显著扩大，回归风险过高。

## 4. 架构设计

## 4.1 主壳层

主壳负责两类全局轮询：

1. 权限可见性刷新
2. 未读数刷新

主壳将新增统一的轮询门禁：

1. 应用处于 `resumed` 时允许全局轮询。
2. 应用处于 `inactive`、`paused`、`detached` 时暂停全局轮询。
3. 从非活跃切回活跃时，立即执行一次补拉，再恢复周期轮询。

主壳不会在本轮删除全局轮询本身，只会把运行条件从“启动后长期常驻”改为“前台活跃时运行”。

## 4.2 模块层

主壳将“当前模块是否活跃”的状态继续传给模块页：

1. 当前选中模块收到 `moduleActive = true`
2. 其他模块收到 `moduleActive = false`

模块页再根据当前选中的 tab，把“当前子页是否活跃”的状态传给具体业务页：

1. 当前 tab 页收到 `pollingEnabled = true`
2. 同模块内未选中的 tab 页收到 `pollingEnabled = false`

这样能保证即使模块页本身仍然保留 `TabBarView` 结构，真正有轮询的子页也不会继续空转。

## 4.3 业务页层

本轮涉及的业务页包括：

1. 消息中心
2. 生产工单查询页
3. 个人中心

这些页面统一改成：

1. 不在 `initState` 中无条件永久启动轮询。
2. 当 `pollingEnabled` 从 `false` 变为 `true` 时：
   - 先立即补拉一次
   - 再启动周期轮询
3. 当 `pollingEnabled` 从 `true` 变为 `false` 时：
   - 立即取消现有定时器
   - 不再继续发请求

## 5. 消息中心专项收口

消息中心当前同时受到三种刷新来源影响：

1. 页面级 30 秒轮询
2. 主壳 WS 事件驱动的 `refreshTick`
3. 列表拉取后额外调用一次摘要刷新

本轮的收口原则是：

1. 保留页面级轮询，但只在页面活跃时运行。
2. 保留主壳 WS 驱动的 `refreshTick`，消息页只有在活跃时才把它转化为真实拉取。
3. 去掉列表刷新后的重复摘要请求，避免一次 `_load()` 后再紧跟一次独立统计拉取。

这样做后，消息中心的刷新来源会更清晰：

1. 页面活跃时的定时兜底
2. 主壳 WS 事件驱动的即时刷新
3. 用户手动刷新

## 6. 数据流

统一数据流如下：

1. `WidgetsBindingObserver` 感知应用前后台切换。
2. 主壳更新全局轮询允许状态。
3. 主壳把当前选中模块映射为模块级活跃态。
4. 模块页把当前 tab 映射为页面级 `pollingEnabled`。
5. 页面根据 `pollingEnabled` 控制定时器生命周期。

状态切换规则：

1. 后台化：
   - 主壳停止权限/未读轮询
   - 页面级轮询全部停止
2. 切换模块或 tab：
   - 离开的页面停止轮询
   - 新页面立即补拉并恢复周期轮询
3. 恢复前台：
   - 主壳立即补拉一次
   - 当前活跃业务页立即补拉一次

## 7. 测试策略

本轮按 TDD 执行，先写失败测试，再写实现。

### 7.1 主壳协调测试

覆盖内容：

1. 主壳进入后台后停止全局轮询。
2. 主壳恢复前台时立即补拉权限与未读。
3. 非活跃状态下不继续触发周期刷新。

### 7.2 页面测试

覆盖内容：

1. 消息中心 `pollingEnabled = false` 时不启动轮询。
2. 消息中心从 `false -> true` 时立即补拉一次并启动轮询。
3. 生产工单查询页失活后停止轮询，恢复时补拉。
4. 个人中心失活后停止会话轮询，恢复时补拉。

### 7.3 集成式 widget 测试

覆盖内容：

1. 切换模块后，前一个模块页面不再继续轮询。
2. 当前模块恢复活跃后能够重新刷新。

## 8. 验收标准

1. 应用进入后台后，主壳不再继续执行权限/未读轮询。
2. 应用恢复前台时，主壳会立即补拉一次必要数据。
3. 消息中心、生产工单查询、个人中心在页面失活后停止各自轮询。
4. 上述页面恢复活跃时，会立即补拉一次并恢复周期轮询。
5. 消息中心不再出现“列表刷新后又紧跟一次重复摘要拉取”的明显重复请求链。
6. 新增或调整的轮询治理测试全部通过。
7. 最终 `flutter analyze` 通过。

## 9. 风险与取舍

1. 本轮默认做破坏性收敛，不保留“页面隐藏时继续后台轮询”的旧行为。
2. 主壳全局轮询本身不删除，只调整其运行条件，避免一次性改动过猛。
3. 消息中心不在本轮完全重写为纯 WS 模型，只收重复拉取与活跃态门禁。
4. 页面恢复活跃时会多一次立即补拉，这是用少量可控请求换掉长期无效轮询的有意取舍。
