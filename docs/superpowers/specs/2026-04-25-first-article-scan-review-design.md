# 首件扫码复核设计

- 日期：2026-04-25
- 状态：待实现
- 适用范围：生产订单首件检验流程、品质首件复核权限、手机端极简 Web 扫码页
- 迁移说明：无迁移，直接替换现有“手输首件检验码”的交互入口；历史首件记录保留原样。

## 1. 背景与目标

现有首件检验由操作员在生产订单中填写首件内容、测试值和首件检验码后提交。该方式依赖人工获取与输入检验码，现场体验不够顺，且质检员/班组长的复核身份与判定动作没有形成独立闭环。

本次目标是把首件检验调整为“操作员填写实测内容，质检员扫码复核判定”的流程：

1. 操作员在首件录入页填写检验内容、检验值、模板与参与人。
2. 操作员点击“发起扫码复核”，系统生成 5 分钟有效二维码。
3. 质检员或被授权人员用手机扫码，进入极简 Web 登录与复核页。
4. 质检员核对订单、工序、参数、首件内容和检验值，只能查看，不能修改。
5. 质检员选择合格/不合格并可填写备注。
6. 合格自动生成正式首件记录；不合格返回操作员端，保留填写内容和质检备注，允许修改后重新发起扫码。

## 2. 非目标

1. 不单独拆分扫码微服务。
2. 不让手机端进入完整 MES 主壳。
3. 不按固定角色写死扫码权限。
4. 不允许质检员修改操作员填写的首件内容或检验值。
5. 不重写历史首件数据。

## 3. 架构

采用“轻量扫码复核服务内嵌现有后端”的方案。

后端新增首件扫码复核能力，负责创建扫码会话、生成一次性 token、校验 5 分钟有效期、校验能力包权限、保存复核结果，并在合格时创建正式首件记录。认证复用现有 `/auth/login` 与 Bearer token 机制。

前端分为两块：

1. 操作员端：仍在现有 Flutter 生产订单首件录入页内完成，替换原“首件检验码”输入框为扫码复核状态区。
2. 手机扫码端：新增独立极简 Web 页面，只包含登录、复核详情、结果提交、完成/错误提示，不进入主系统、不加载完整主壳。

权限使用现有“功能权限配置”。新增能力码：

- 能力码：`quality.first_articles.scan_review`
- 名称：扫码复核首件
- 建议入口：每日首件 / 首件扫码复核
- 说明：允许通过扫码页面查看待复核首件草稿，并提交合格/不合格结论。

后端扫码提交接口必须校验该能力码。前端是否显示入口不能替代后端权限判断。

## 4. 业务流程

### 4.1 操作员发起

操作员进入生产订单的“开始首件”页面，填写：

- 首件模板
- 首件内容
- 首件测试值
- 参与操作员
- 其他已有首件草稿信息

填写完成后点击“发起扫码复核”。前端调用后端创建扫码会话接口，后端保存草稿快照并返回：

- 会话 ID
- 二维码 URL
- 过期时间
- 当前状态

操作员端显示二维码、倒计时、等待质检扫码提示和“刷新二维码”按钮。

### 4.2 质检员扫码复核

质检员扫码打开极简手机 Web 页。

若未登录，页面展示账号、密码、登录按钮，登录成功后保留当前二维码 token 并进入复核详情页。手机端不跳转主系统。

复核详情页展示：

- 订单号
- 产品
- 工序
- 操作员
- 模板
- 产品参数或首件参数
- 首件内容
- 首件测试值
- 参与操作员

质检员只能查看上述字段，不能修改。质检员选择“合格”或“不合格”，可填写备注并提交。

### 4.3 结果处理

合格：

- 后端创建正式首件记录。
- 扫码会话标记为 `approved`。
- 记录质检员、复核时间、备注、正式首件记录 ID。
- 操作员端显示“首件已通过”，刷新订单状态。

不合格：

- 后端不创建正式首件记录。
- 扫码会话标记为 `rejected`。
- 记录质检员、复核时间和备注。
- 操作员端显示不合格结果和质检备注，保留原填写内容，允许操作员修改后重新发起扫码。

## 5. 状态机

扫码会话状态：

- `pending`：已生成二维码，等待质检员扫码复核。
- `approved`：质检员判定合格，已生成正式首件记录。
- `rejected`：质检员判定不合格，返回操作员修改。
- `expired`：超过 5 分钟未提交，二维码失效。
- `cancelled`：操作员刷新二维码或退出当前草稿时，旧会话作废。

刷新规则：

1. 操作员点击“刷新二维码”。
2. 后端将当前 `pending` 会话标记为 `cancelled`。
3. 后端创建新会话并返回新二维码。
4. 旧二维码再次扫码时提示“二维码已刷新，请重新扫码”。

过期规则：

1. 二维码有效期固定为 5 分钟。
2. 操作员端倒计时结束后显示“二维码已失效”和“刷新二维码”按钮。
3. 手机端访问过期 token 时提示“二维码已失效，请操作员刷新后重新扫码”。

## 6. 数据模型

新增表建议命名为 `mes_first_article_review_session`。

核心字段：

- `id`
- `token_hash`
- `status`
- `expires_at`
- `order_id`
- `order_process_id`
- `pipeline_instance_id`
- `operator_user_id`
- `assist_authorization_id`
- `template_id`
- `check_content`
- `test_value`
- `participant_user_ids`
- `reviewer_user_id`
- `review_result`
- `review_remark`
- `reviewed_at`
- `first_article_record_id`
- `created_at`
- `updated_at`

token 只在创建时明文返回给前端生成二维码，数据库只保存 `token_hash`。手机端后续携带 token 访问接口，后端哈希后匹配。

正式首件记录仍沿用现有 `FirstArticleRecord`。若现有正式记录无法保存质检复核人、复核时间与复核备注，应补充字段，确保历史审计能回答“谁复核、何时复核、判定是什么、备注是什么”。

## 7. 后端接口

建议新增接口组挂在现有生产或质量 API 下，具体路径可在实现时按现有路由风格收敛。

### 7.1 创建扫码会话

用途：操作员端发起扫码复核。

输入：

- `order_id`
- `order_process_id`
- `pipeline_instance_id`
- `template_id`
- `check_content`
- `test_value`
- `participant_user_ids`
- `assist_authorization_id`

输出：

- `session_id`
- `review_url`
- `expires_at`
- `status`

校验：

- 操作员仍具备当前订单/工序的首件发起权限。
- 当前工序仍允许首件。
- 检验内容与检验值非空。

### 7.2 查询扫码会话

用途：手机端展示复核详情，操作员端轮询状态。

输入：

- `token` 或 `session_id`

输出：

- 状态
- 过期时间
- 订单、产品、工序、操作员
- 参数与首件草稿内容
- 质检结果与备注

手机端按 token 查询时，后端需校验 token 有效性。操作员端按 session 查询时，后端需校验发起人与当前登录态。

### 7.3 提交扫码复核结果

用途：质检员提交合格/不合格。

输入：

- `token`
- `review_result`：`passed` 或 `failed`
- `review_remark`

校验：

- token 存在。
- token 未过期。
- 会话状态为 `pending`。
- 当前登录用户具备 `quality.first_articles.scan_review`。
- 订单与工序仍处于允许首件复核的状态。

输出：

- 最终状态
- 正式首件记录 ID，合格时返回
- 质检员、复核时间、备注

### 7.4 刷新二维码

用途：操作员端在二维码过期或需要重发时刷新。

输入：

- `session_id`

处理：

- 原 `pending` 会话标记为 `cancelled`。
- 创建新会话，保留当前操作员端最新草稿内容。
- 返回新二维码 URL 与过期时间。

## 8. 前端设计

### 8.1 操作员端

现有“首件录入”页调整为两阶段。

第一阶段：草稿填写。

- 保留首件模板、查看参数、添加操作员、首件内容、首件测试值、备注等已有控件。
- 移除原“首件检验码”输入框。
- 底部按钮改为“发起扫码复核”。

第二阶段：扫码等待。

- 显示二维码。
- 显示倒计时。
- 显示会话状态。
- 提供“刷新二维码”。
- 提供“取消首件”。

状态反馈：

- `pending`：等待质检扫码复核。
- `expired`：二维码已失效，可刷新。
- `rejected`：显示质检员、时间、备注，回到可编辑状态。
- `approved`：显示质检员、时间，自动完成并刷新订单状态。

### 8.2 手机扫码端

手机端页面使用极简 Web，不进入主壳。

页面：

1. 登录页：账号、密码、登录按钮。
2. 复核详情页：只读展示订单、工序、参数和操作员填写内容。
3. 判定区：合格/不合格二选一，备注输入框，提交按钮。
4. 结果页：成功、过期、取消、已使用、无权限等结果提示。

手机端不提供全局导航、不展示完整菜单、不进入首页。

## 9. 异常处理

手机端：

- token 过期：提示“二维码已失效，请操作员刷新后重新扫码”。
- token 已取消：提示“二维码已刷新，请重新扫码”。
- token 已使用：提示“本次首件复核已完成”。
- 无权限：提示“当前账号无扫码复核首件权限”。
- 草稿不存在或订单状态变化：提示“首件申请已失效，请操作员重新发起”。

操作员端：

- 过期：二维码区域显示已失效，并显示“刷新二维码”按钮。
- 不合格：显示质检员、判定时间和备注，保留原填写内容，允许修改后重新发起。
- 合格：显示质检员、判定时间，自动完成首件流程并刷新订单状态。
- 网络失败：保留草稿内容，允许重试查询或刷新二维码。

## 10. 审计与事件

业务事件建议记录：

- 发起首件扫码复核。
- 刷新首件扫码二维码。
- 首件扫码复核通过。
- 首件扫码复核驳回。
- 首件扫码复核取消或过期。

安全/权限审计建议记录：

- 手机端扫码登录。
- 无权限访问扫码复核。
- 过期 token 访问。
- 重复提交。

审计记录至少能关联：

- 订单
- 工序
- 操作员
- 质检员
- 会话 ID
- 正式首件记录 ID
- 时间
- 结果

## 11. 测试范围

后端测试：

1. 创建扫码会话成功，token 只存哈希。
2. 二维码 5 分钟过期。
3. 刷新二维码会取消旧会话。
4. 无 `quality.first_articles.scan_review` 权限时无法提交复核。
5. 合格时创建正式首件记录并绑定会话。
6. 不合格时不创建正式首件记录，并返回操作员可修改状态。
7. 重复提交被拒绝。
8. 过期、取消、已使用 token 均返回明确错误。
9. 订单或工序状态变化时拒绝提交。

前端测试：

1. 操作员端填写后可生成二维码。
2. 操作员端倒计时过期后显示刷新按钮。
3. 不合格返回后保留填写内容并显示质检备注。
4. 合格后刷新订单状态。
5. 手机端未登录时展示极简登录页。
6. 手机端登录后展示只读复核详情。
7. 手机端提交合格/不合格成功。
8. 手机端展示过期、取消、已使用、无权限错误页。

集成测试：

1. 操作员发起扫码 -> 质检员登录 -> 合格 -> 正式首件记录生成。
2. 操作员发起扫码 -> 质检员登录 -> 不合格 -> 操作员修改 -> 重新扫码 -> 合格。
3. 旧二维码刷新后不可再用。

## 12. 自检结论

- 无未决字段。
- 架构与用户确认一致：内嵌现有后端，手机端极简 Web，权限走能力包。
- 状态机覆盖合格、不合格、过期、取消和重复提交。
- 范围聚焦首件扫码复核，不包含独立微服务、不包含完整手机主系统。
