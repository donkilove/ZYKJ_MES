# 插件开发指南文档设计

- 日期：2026-04-25
- 主题：在 `docs/插件开发指南/` 下建立一套面向宿主维护者与插件开发者的插件文档体系
- 状态：待用户审阅

## 1. 背景

当前仓库中的插件体系已经形成了可落地的第一版闭环，至少包含以下事实：

1. 插件根目录固定为仓库根下的 `plugins/`
2. Python 运行时固定为 `plugins/runtime/python312/python.exe`
3. 插件以独立 Python 程序包存在，并自带业务逻辑与 Web UI
4. 宿主负责扫描插件、启动插件、停止插件、检查状态，并在主程序内部承载插件页面
5. 插件私有第三方依赖由插件自己携带，默认放在各插件目录的 `vendor/` 下
6. 当前已有一个完整样板插件：`plugins/serial_assistant/`

这些约束已经分散存在于以下材料中：

1. `docs/superpowers/specs/2026-04-24-python-plugin-host-design.md`
2. `docs/superpowers/specs/2026-04-25-plugin-runtime-redirection-design.md`
3. `evidence/2026-04-25_插件目录与解释器部署方式重定向.md`
4. `plugins/serial_assistant/`
5. 宿主运行时代码与测试

当前缺失的问题是：

1. 没有一套给新插件开发者直接使用的统一指南
2. 没有一套把“宿主约定”和“插件开发要求”同时讲清楚的文档入口
3. 现有规则散落在设计稿、evidence 和代码实现中，阅读成本高

因此，本轮需要补一套正式文档，放在 `docs/插件开发指南/` 下，作为当前插件体系的统一开发口径。

## 2. 目标与非目标

### 2.1 目标

1. 在 `docs/` 下建立专门的“插件开发指南”目录
2. 同时服务两类读者：
   - 宿主维护者
   - 插件开发者
3. 用“总览入口 + 分册正文”的方式组织文档
4. 把当前插件开发要求、宿主约定、运行时边界、调试与交付要求写清楚
5. 所有文档内容以当前仓库已经采用的实现为准，而不是未来愿景
6. 让开发者可以参考现有样板插件，从零创建一个能跑的插件

### 2.2 非目标

1. 本轮不修改插件协议本身
2. 本轮不新增插件宿主功能
3. 本轮不设计插件市场、安装器或热更新机制
4. 本轮不把旧 spec 或 evidence 整体迁移到新目录
5. 本轮不编写面向最终用户的“如何使用插件中心”说明书

## 3. 目标读者与阅读路径

本套文档同时服务两类读者。

### 3.1 宿主维护者

主要关注：

1. 宿主如何扫描插件
2. 解释器和插件目录如何定位
3. 宿主与插件之间有哪些固定协议
4. 宿主在调试、验证和交付时需要检查哪些边界

建议阅读顺序：

1. `00-总览与阅读顺序.md`
2. `10-宿主约定与目录结构.md`
3. `40-UI承载与前后端接口.md`
4. `50-开发调试测试与交付.md`

### 3.2 插件开发者

主要关注：

1. 新插件的目录如何组织
2. `manifest.json` 应该怎么写
3. `launcher.py` 如何和宿主约定对齐
4. 第三方依赖如何打包到 `vendor/`
5. Web UI 如何暴露给宿主承载
6. 怎样本地调试和自测

建议阅读顺序：

1. `00-总览与阅读顺序.md`
2. `20-插件包结构与清单规范.md`
3. `30-Python入口与依赖加载.md`
4. `40-UI承载与前后端接口.md`
5. `50-开发调试测试与交付.md`
6. `60-串口助手示例拆解.md`

## 4. 文档组织方案

本轮采用“总览入口 + 分册正文”方案。

不采用以下替代方案：

1. 单文档大手册
   - 前期看似简单，但随着插件协议和样板增多，单文件会迅速膨胀
2. 按角色拆成两套独立文档
   - 共同规则会重复，容易造成宿主口径和插件口径漂移

采用当前方案的原因：

1. 既能给新读者一个统一入口
2. 也能把宿主规则、插件规则、调试验证拆成独立分册
3. 后续新增样板插件或协议字段时，只需增量维护对应分册

## 5. 目录与分册设计

本轮将在 `docs/插件开发指南/` 下创建以下文件：

```text
docs/
  插件开发指南/
    00-总览与阅读顺序.md
    10-宿主约定与目录结构.md
    20-插件包结构与清单规范.md
    30-Python入口与依赖加载.md
    40-UI承载与前后端接口.md
    50-开发调试测试与交付.md
    60-串口助手示例拆解.md
```

### 5.1 `00-总览与阅读顺序.md`

职责：

1. 说明这套文档是做什么的
2. 说明谁应该读这套文档
3. 说明当前插件体系的总边界
4. 给出按角色区分的阅读顺序
5. 指向各分册的适用场景

### 5.2 `10-宿主约定与目录结构.md`

职责：

1. 说明仓库中的插件根目录结构
2. 说明 Python 运行时固定位置
3. 说明宿主扫描插件、定位解释器的规则
4. 说明环境变量覆盖的定位与边界
5. 说明哪些路径是正式交付口径，哪些只是调试覆盖

### 5.3 `20-插件包结构与清单规范.md`

职责：

1. 说明单个插件目录的标准结构
2. 说明 `manifest.json` 字段与含义
3. 说明必填字段、推荐字段与命名约束
4. 说明 `id`、版本、依赖路径、权限声明等规则
5. 给出最小合法样例

### 5.4 `30-Python入口与依赖加载.md`

职责：

1. 说明 `launcher.py` 的职责
2. 说明宿主会提供哪些环境变量
3. 说明插件如何组装 `sys.path`
4. 说明 `vendor/`、`app/` 与插件根目录如何参与导入
5. 说明启动时应如何输出 `ready` 消息
6. 说明哪些行为是允许的，哪些行为是禁止的

### 5.5 `40-UI承载与前后端接口.md`

职责：

1. 说明插件 UI 的承载模式是“插件自带 Web UI + 宿主内嵌显示”
2. 说明 `entry_url` 与 `heartbeat_url` 的约定
3. 说明宿主与插件之间的边界
4. 说明插件前端页面应遵守的基本规则
5. 说明常见错误，如端口没监听、路径不对、页面资源引用错误等

### 5.6 `50-开发调试测试与交付.md`

职责：

1. 说明本地调试路径
2. 说明插件自测与宿主联调方式
3. 说明日志、测试、目录检查、交付前检查清单
4. 说明推荐的开发流程
5. 说明插件交付时必须随包提交哪些内容

### 5.7 `60-串口助手示例拆解.md`

职责：

1. 以 `serial_assistant` 作为当前标准样板
2. 对照真实目录说明每个文件是做什么的
3. 解释 `manifest.json`、`launcher.py`、`app/server.py`、`web/index.html` 的对应关系
4. 给出“照着这个样板做新插件”的最小操作路径

## 6. 固定口径

为了防止文档与当前实现偏离，本套文档必须统一采用以下固定口径：

1. 插件根目录固定为仓库根下 `plugins/`
2. 主解释器固定为 `plugins/runtime/python312/python.exe`
3. `MES_PLUGIN_ROOT` 和 `MES_PYTHON_RUNTIME_DIR` 只作为调试覆盖，不是正式交付口径
4. 插件第三方依赖由插件自带，默认放在插件目录内的 `vendor/`
5. 插件必须自带业务逻辑与 UI，宿主不负责渲染插件业务页面
6. 插件默认入口是 `launcher.py`
7. 插件必须自行完成 `sys.path` 组装
8. 插件启动完成后必须输出结构化 `ready` 消息
9. 宿主主要负责扫描、启动、停止、状态判断、工作区承载和错误提示

## 7. 写作约束

本套文档应遵守以下写作约束：

1. 全部使用中文
2. 以当前仓库实现为准，不创造与代码不一致的新协议
3. 每篇文档先写“目的 / 适用范围 / 与其他分册关系”
4. 每篇正文按“规则 -> 示例 -> 常见错误”的顺序组织
5. 对强约束使用“必须”
6. 对推荐性做法使用“应该”
7. 对可选能力使用“可以”
8. 避免写成纯概念介绍，必须给可执行示例
9. 示例优先引用：
   - `plugins/serial_assistant/manifest.json`
   - `plugins/serial_assistant/launcher.py`
   - 宿主插件相关实现

## 8. 资料来源与一致性基线

正文编写时应以以下资料为一致性基线：

1. `docs/superpowers/specs/2026-04-24-python-plugin-host-design.md`
2. `docs/superpowers/specs/2026-04-25-plugin-runtime-redirection-design.md`
3. `frontend/lib/features/plugin_host/services/plugin_runtime_locator.dart`
4. `plugins/serial_assistant/manifest.json`
5. `plugins/serial_assistant/launcher.py`
6. `plugins/serial_assistant/app/`
7. `plugins/serial_assistant/web/`

若文档与上述实现冲突，以当前实现为准，并在文档中采用当前实现口径。

## 9. 维护规则

后续维护本套文档时，应遵守：

1. 若插件目录结构、运行时位置或宿主协议变更，必须同步更新对应分册
2. 若新增新的标准样板插件，可以在 `60-*` 分册中追加示例，或新开示例分册
3. 不应把设计过程文档原样搬入本目录
4. 本目录是“当前开发指南”，不是历史归档目录
5. 若未来有面向最终用户的“插件使用手册”，应另建目录，不与开发指南混写

## 10. 验收标准

以下条件同时满足，即视为本轮文档设计完成：

1. `docs/插件开发指南/` 目录结构明确
2. 各分册职责边界清晰，没有明显重叠
3. 文档同时覆盖宿主维护者与插件开发者
4. 固定口径与当前代码实现一致
5. 能直接支撑后续正文编写
