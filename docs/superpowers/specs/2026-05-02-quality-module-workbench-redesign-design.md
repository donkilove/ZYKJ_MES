# 质量模块统一工作台改版设计

## 1. 背景

质量模块当前在 `QualityPage` 下承载多组不同类型的页面，包括：

1. `每日首件`
2. `质量数据`
3. `报废统计`
4. `维修订单`
5. `质量趋势`
6. `不良分析`
7. `供应商管理`

其中，`供应商管理` 更偏管理支撑页，本轮不纳入统一工作台第一阶段改版范围；其余 6 个页签都属于质量业务一线会频繁切换使用的分析/处理页面。

当前这组页面的主要问题已经不再是“功能有没有”，而是：

1. 切页后筛选区位置、主操作区位置、结果区节奏不一致。
2. 有的页面先看到大段筛选，有的页面先看到零散卡片，有的页面首屏又几乎没有结论。
3. 页签之间视觉层级割裂，用户每切一次都要重新适应页面结构。
4. 不同类型页面虽然业务不同，但缺少统一的工作台骨架，导致可操作性和维护性都偏弱。

因此，本轮目标不是单页修补，而是先为质量模块建立一套“统一工作台骨架”，再按页面类型分批落地。

## 2. 目标

1. 将 6 个质量业务页签统一收敛到同一套工作台骨架。
2. 降低用户在页签间切换时的学习成本和操作迁移成本。
3. 提升首屏空间利用率，让页面优先展示结论与高频操作。
4. 让任务处理型、数据汇总型、分析看板型页面在同一视觉语言下保留各自业务主体。
5. 分批推进，先用代表页稳定骨架，再平移到剩余页面。

## 3. 非目标

1. 不修改后端接口、统计口径、权限语义和导出契约。
2. 不重构 `QualityPage` 的页签权限装配逻辑。
3. 不把 6 个页签做成完全相同的页面。
4. 不在本轮顺手改造 `供应商管理` 或其他质量管理支撑页。
5. 不把本轮扩散成全局 UI 基础件体系重构。

## 4. 现状问题

### 4.1 跨页骨架不稳定

当前 6 个页签虽然都挂在 `QualityPage` 下，但页面结构缺少统一骨架。具体表现为：

1. 筛选区有的松散，有的拥挤，有的与主操作区分离。
2. 摘要信息的出现顺序不一致。
3. 图表页、表格页和任务页没有共享统一的“先看什么、再做什么”的阅读顺序。

### 4.2 首屏优先级混乱

对用户来说，质量模块中的高频动作包括：

1. 快速筛选时间范围和状态
2. 看当前是否有异常、是否有待处理任务
3. 进入表格或图表做进一步定位

但现有页面经常先把大量控件或松散内容推到首屏，真正高价值的结论被压到后面。

### 4.3 操作入口和结果脱节

尤其在 `每日首件`、`维修订单`、`报废统计` 这类带执行属性的页面里，用户经常需要：

1. 先找到结果
2. 再回头去找操作按钮

这会增加扫视和切换成本，影响可操作性。

### 4.4 同一模块内页面代际差异明显

当前模块内部实际上已经混合了几种不同代际的页面表达方式：

1. 传统筛选 + 表格页
2. 带少量摘要卡的汇总页
3. 图表驱动的分析页

它们没有被统一整理到一个稳定的工作台语言中。

## 5. 已确认方向

本轮方向已经与用户确认如下：

1. 采用 `统一工作台` 路线，而不是 6 页各自独立改。
2. 采用 `同骨架，不同主体`，而不是把所有页面做成同一模板。
3. 页面按业务特性分成：
   - 任务处理型
   - 数据汇总型
   - 分析看板型
4. 实施上分批推进，第一批先落 `质量数据 + 质量趋势`。

因此，本轮 spec 的核心不是某一页怎么排，而是如何先建立质量模块统一工作台骨架，并明确分批边界。

## 6. 页面分组

### 6.1 任务处理型

包括：

1. `每日首件`
2. `维修订单`

这一类页面的核心需求是：

1. 快速看到待处理状态
2. 明确操作入口
3. 列表与动作靠近

### 6.2 数据汇总型

包括：

1. `质量数据`
2. `报废统计`

这一类页面的核心需求是：

1. 筛选效率高
2. 摘要信息清楚
3. 汇总卡、趋势区、表格区关系明确

### 6.3 分析看板型

包括：

1. `质量趋势`
2. `不良分析`

这一类页面的核心需求是：

1. 首屏先给异常与趋势
2. 图表是主视觉
3. 对比、分布和明细之间有稳定阅读顺序

## 7. 统一工作台骨架

6 个业务页签统一采用三层结构：

1. 顶部控制台
2. 中部结论带
3. 下部主体区

### 7.1 顶部控制台

顶部控制台是跨页统一的操作骨架。

必须统一的内容：

1. 日期类筛选优先靠前
2. 核心筛选集中在左侧
3. 刷新、导出和主操作稳定放在右侧或同一控制区
4. 控制台整体作为一块完整面板呈现，而不是散落按钮和输入框

它的目的不是“塞满控件”，而是让用户切页后知道：

1. 去哪里调范围
2. 去哪里做主操作
3. 哪些筛选是当前页真正重要的

### 7.2 中部结论带

每页都需要在主体区前给出当前最重要的结论，但不同类型页面的结论表达不同：

1. 任务处理型：待办数、异常数、待处置数
2. 数据汇总型：总量、通过率、报废量、覆盖范围
3. 分析看板型：异常趋势、主要问题项、核心分布

这里不要求每页都堆很多卡片，但要求“先给结论，再给明细”。

### 7.3 下部主体区

主体区保持业务差异化，不强行同构：

1. 任务处理型：列表为主，动作紧贴结果
2. 数据汇总型：概览 + 表格为主
3. 分析看板型：图表为主，列表/表格为辅

统一的是骨架和节奏，不是内容呈现形式。

## 8. 各类型页面设计原则

### 8.1 任务处理型原则

适用于：

1. `每日首件`
2. `维修订单`

原则：

1. 首屏突出当前待处理状态。
2. 行内动作、状态标签和结果信息距离尽量近。
3. 不把首屏空间浪费在长筛选区和弱价值说明上。
4. 强调“处理什么”与“怎么处理”。

### 8.2 数据汇总型原则

适用于：

1. `质量数据`
2. `报废统计`

原则：

1. 控制台筛选效率优先。
2. 摘要与趋势、表格之间的层级清晰。
3. 首屏至少先看到汇总结论，而不是直接面对大块表格。
4. 让汇总页具备“先回答问题，再给证据”的节奏。

### 8.3 分析看板型原则

适用于：

1. `质量趋势`
2. `不良分析`

原则：

1. 图表和异常总结优先于筛选。
2. 控制台收紧，不喧宾夺主。
3. 首屏先看到主要趋势或主要问题集中区。
4. 图表、分布和明细之间形成固定阅读路径。

## 9. 统一交互规则

### 9.1 控制台规则统一

这 6 页统一遵守：

1. 日期筛选优先靠前。
2. 导出按钮位置固定。
3. 刷新入口位置固定。
4. 主操作入口不在页面中漂移。

### 9.2 首屏先给结论

各页必须至少满足：

1. 进入页面后能快速看到当前状态或关键结果。
2. 用户不需要先扫完整页控件才能知道页面核心结论。

### 9.3 操作靠近结果

对带执行属性的页面，动作应尽量贴近结果本身，而不是让用户在“结果区域”和“按钮区域”之间来回跳。

### 9.4 空态与错误态统一

统一要求：

1. 空态要说明当前为什么没有数据。
2. 错误态要提供明确的重试语义。
3. 页面状态切换时尽量保持骨架稳定，不突然塌缩。

## 10. 响应式策略

### 10.1 宽桌面

适用范围：约 `>= 1280`

要求：

1. 控制台尽量单块完整展示。
2. 摘要卡优先单排。
3. 图表或表格主体拥有清楚的主视觉区。

### 10.2 中等桌面

适用范围：约 `980 - 1279`

要求：

1. 控制台允许拆成上下两层，但仍保持单面板。
2. 摘要卡可以换行，但节奏不能散。
3. 主体区在不牺牲可读性的前提下降级布局。

### 10.3 窄桌面

适用范围：约 `<= 979`

要求：

1. 控制台纵向堆叠但不乱。
2. 主体区改为更适合纵向阅读的结构。
3. 不追求和宽桌面完全同形，但必须保持清晰与可点。

## 11. 分批策略

### 11.1 第一批：统一骨架 + 代表页

覆盖：

1. `质量数据`
2. `质量趋势`

目的：

1. 用一个数据汇总型页面和一个分析看板型页面，把统一工作台骨架先打稳。
2. 验证控制台、结论带和主体区的统一语言是否成立。

### 11.2 第二批：任务处理型页面

覆盖：

1. `每日首件`
2. `维修订单`

目的：

1. 把“操作靠近结果”的规则落到质量任务页。
2. 在第一批骨架稳定后接入执行型页面，降低返工风险。

### 11.3 第三批：剩余收口页

覆盖：

1. `报废统计`
2. `不良分析`

目的：

1. 平移前两批已经验证过的骨架和交互语言。
2. 完成质量业务页签层面的统一工作台收口。

## 12. 第一批页面的具体要求

### 12.1 质量数据

第一批中，`质量数据` 应承担“统一工作台骨架样板页”的角色。

要求：

1. 顶部控制台更紧凑规整。
2. 汇总卡与趋势区层级更清晰。
3. 当前多块内容区的节奏需要重新梳理，避免看起来像把很多卡片简单堆起来。

### 12.2 质量趋势

第一批中，`质量趋势` 应承担“分析看板型样板页”的角色。

要求：

1. 首屏结论优先。
2. 控制台压缩，图表主视觉提升。
3. 图表与维度对比区域形成固定阅读顺序。

## 13. 实现边界

本轮 spec 的第一批实现允许修改：

1. 页面布局层级
2. 控制台排布
3. 摘要/结论带顺序
4. 图表与表格主体区节奏
5. 与之匹配的页面测试

本轮第一批不允许修改：

1. 接口契约
2. 权限控制
3. 导出业务逻辑
4. 页签权限装配
5. 其他未纳入本批页面的大范围 UI 抽象

## 14. 验收标准

### 14.1 骨架层面

完成后应满足：

1. `质量数据` 与 `质量趋势` 首屏都能体现同一工作台骨架。
2. 控制台、结论带、主体区节奏一致。
3. 页面切换后用户能明显感受到“是同一模块里的两张工作台”。

### 14.2 页面层面

完成后应满足：

1. `质量数据` 的汇总/趋势/表格层级清晰。
2. `质量趋势` 的图表主视觉明显提升。
3. 两页的首屏都不再是“控件先行、结论后置”。

### 14.3 交互层面

完成后应满足：

1. 控制台位置、主操作位置稳定。
2. 空态与错误态语义一致。
3. 现有筛选、导出与刷新行为不回退。

## 15. 风险与控制

### 风险 1：过度统一导致页面失去业务个性

控制方式：

1. 只统一骨架和交互规则。
2. 主体区允许按任务型 / 汇总型 / 看板型差异化展开。

### 风险 2：一次性覆盖过多页面导致返工

控制方式：

1. 坚持三批推进。
2. 第一批先用代表页验证方向。

### 风险 3：把质量模块改版误做成全局组件治理

控制方式：

1. 只在本轮任务明确需要时复用现有基础件。
2. 不为了抽象而抽象，不顺带做全局 UI 体系重构。

## 16. 最终预期

本轮完成后，质量模块的 6 个业务页签将逐步从“多个功能页的并列集合”升级为：

1. 同一质量工作台语言下的多类型页面
2. 控制台与结论带统一
3. 主体区因业务而异
4. 切页时不再反复重新学习

的质量业务工作台体系。

## 17. 迁移说明

- 无迁移，直接替换
