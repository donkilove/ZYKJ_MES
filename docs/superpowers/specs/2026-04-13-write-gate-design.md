# 后端写链路回归门禁设计

- 日期：2026-04-13
- 设计对象：`backend/` 写链路回归与门禁体系
- 设计结论：落地采用“方案 B：分层写链路门禁”，长期目标收敛到“方案 A：全量写链路统一门禁”。
- 迁移口径：无迁移，直接替换。

## 1. 背景与目标

当前后端读链路、Detail 链路、权限池与 40 并发 P95 已经过多轮专项收敛，核心读路径具备较高覆盖率与较稳定的性能结果。但从总链路视角看，写链路与导出/流式链路仍缺少可重复、可放行、可审计的真实验证闭环。

现状问题不是“没有场景资产”，而是“已有资产尚未形成正式门禁”：

1. 仓库中已存在 `tools/perf/scenarios/write_operations_40_scan.json`。
2. 场景角色域与 token pool 元数据已补齐。
3. 写链路样本、复位、性能阈值与正式执行频率仍未形成统一设计。

本次设计目标如下：

1. 为后端全部写链路建立统一治理视角，不承认永久脱管链路。
2. 第一阶段即以“性能优先”为导向，将核心及尽量全量的写链路纳入并发、时延与成功率门槛。
3. 通过分层设计降低一次性落地风险，让体系能够真正跑起来，而不是停留在场景文件层。
4. 明确样本资产、复位、失败归因、周期运行与 evidence 留痕口径。
5. 为后续向统一全量写链路门禁演进预留清晰路径。

## 2. 设计原则

### 2.1 全量纳管，不永久豁免

所有 `POST`、`PUT`、`PATCH`、`DELETE` 链路最终都要进入写链路门禁体系。允许分阶段实施，不允许把某类链路长期停留在“以后再说”。

### 2.2 性能优先，但不只看性能

本轮设计以性能优先为前提，要求写链路在正式门禁中具备并发与时延门槛。但写链路不同于读链路，不能只看 `P95`，还必须同时看：

1. 功能成功率
2. 错误率与错误结构
3. 状态一致性
4. 样本可复位性

### 2.3 分层落地，统一收口

落地形态采用“方案 B：分层写链路门禁”，即通过链路分层来匹配不同副作用强度和不同阈值；长期目标再收敛到“方案 A：全量写链路统一门禁”。

### 2.4 样本先行，拒绝碰运气

任何写链路要进入正式门禁，都必须明确：

1. 前置依赖样本
2. 执行后的期望状态
3. 复位方式
4. 失败后的残留处理策略

没有这四项的写链路，不得进入正式门禁。

### 2.5 结果必须可比较、可审计

每次执行都要能回答：

1. 这次跑了哪些写链路
2. 每条链路属于哪一层
3. 哪些通过、哪些失败、失败为什么
4. 是否污染样本
5. 是否满足放行条件

## 3. 分层模型

### 3.1 L1：核心状态流转写链路

定义：

1. 直接推进主业务流程的写操作
2. 失败会影响真实业务状态正确性的写操作
3. 对用户感知最强、故障代价最高的写操作

典型示例：

1. 生产订单状态流转
2. 报工结束
3. 首件提交
4. 维修完成
5. 权限下发
6. 计划启停

要求：

1. 第一阶段必须纳入正式门禁
2. 阈值最严格
3. 必须有明确状态一致性校验
4. 必须有可重复样本准备与复位机制

### 3.2 L2：主数据维护写链路

定义：

1. 主数据的增删改操作
2. 会影响后续读链路与业务流程，但不直接推动核心流转的写操作

典型示例：

1. 产品新增、更新、生命周期切换
2. 工艺阶段/工序/模板维护
3. 供应商维护
4. 设备台账、规则、参数维护

要求：

1. 第一阶段纳入正式门禁
2. 性能阈值可略宽于 L1
3. 核心关注成功率、权限正确性和复位能力
4. 必须避免污染共享基线样本

### 3.3 L3：高副作用管理写链路

定义：

1. 副作用范围大
2. 易造成环境污染
3. 复位成本高
4. 具有导入、发布、回滚、归档、批量处理等特征

典型示例：

1. 导入
2. 发布
3. 回滚
4. 归档
5. 批量启停/批量处理

要求：

1. 第一阶段纳入治理视野
2. 采用受控执行模式
3. 并发强度可低于 L1/L2
4. 必须使用隔离样本或隔离环境状态
5. 不允许长期停留在非正式状态

## 4. 指标与通过标准

### 4.1 统一指标

所有层级统一看以下 5 类指标：

1. 功能成功率
2. 错误率与错误码结构
3. `P95 / P99` 时延
4. 状态一致性
5. 样本可复位性

### 4.2 功能成功率

写链路不能只要求“出现过成功请求”，而应要求在目标并发下稳定返回预期成功状态。

要求：

1. 成功状态码必须由场景显式声明
2. 不接受“部分请求 200，整体仍视为通过”的模糊结论
3. 放行时应同时输出场景级成功率与套件级成功率

### 4.3 错误率与错误结构

写链路的错误率必须拆分结构，而不是只给一个总数。至少输出以下分类：

1. `401`：认证问题
2. `403`：权限问题
3. `404`：样本不存在或路径错误
4. `409`：状态冲突
5. `422`：参数或契约问题
6. `5xx`：服务异常

要求：

1. 每次失败都能归到明确类别
2. 不允许把样本问题与性能问题混在一起讨论
3. 套件放行时必须附错误结构摘要

### 4.4 时延指标

写链路既然以性能优先为前提，就必须具备正式时延门槛。

设计口径：

1. `L1` 最严格
2. `L2` 次之
3. `L3` 采用受控放宽阈值
4. 长期目标再逐步向统一阈值收敛

本设计不在 spec 中直接固化数值阈值，而是在实施阶段形成配置化门限。原因是不同模块写链路副作用差异较大，阈值应可调、可演进，而不是写死在说明文档里。

### 4.5 状态一致性

写链路不能只看 HTTP 响应成功，还要验证：

1. 数据是否真正落库
2. 状态是否真正流转
3. 关联副作用是否符合预期
4. 下游查询是否反映变更结果

状态一致性应成为场景级通过标准的一部分，而不是额外“人工心证”。

### 4.6 样本可复位性

如果某条写链路执行后无法恢复到可复跑状态，则不能认定为正式纳入门禁。

要求：

1. 每次执行都记录复位结果
2. 复位失败应单独计入失败原因
3. 套件级放行必须输出复位成功率

### 4.7 双层通过标准

写链路门禁采用双层判定：

1. 场景级通过：
   - 成功率达标
   - 错误结构可接受
   - 时延达标
   - 状态一致性通过
   - 复位通过
2. 套件级通过：
   - 模块整体通过
   - 层级整体通过
   - 受控豁免项已记录且未突破上限

## 5. 样本资产设计

### 5.1 三层样本模型

#### 5.1.1 基线样本

用于提供长期存在、稳定可引用的底座数据。

包括：

1. 固定角色与账号池
2. 固定产品、工艺、供应商、设备等主数据
3. 固定管理员池与业务角色池

原则：

1. 不直接承接高副作用写操作
2. 只作为被引用底座

#### 5.1.2 场景样本

由具体场景在执行前显式生成，用于承接常规写操作。

包括：

1. 订单
2. 工序实例
3. 首件记录
4. 维修单
5. 导出任务

原则：

1. 执行前可显式准备
2. 执行后可重建或回滚
3. 不依赖“数据库里刚好有数据”

#### 5.1.3 一次性副本样本

用于承接高副作用场景，避免污染共享样本。

原则：

1. 执行前创建
2. 执行后销毁或重建
3. 只服务单轮执行

### 5.2 样本准入规则

每条写链路场景必须显式声明：

1. 依赖哪些基线样本
2. 需要创建哪些场景样本
3. 执行后预期状态是什么
4. 采用哪种复位方式

缺任一项即不准进入正式门禁。

## 6. 复位策略

### 6.1 可逆复位

适用于状态切换类写链路。

方式：

1. 按预定义路径把状态切回初始值
2. 验证切回后状态一致性

### 6.2 重建复位

适用于增删改幅度较大的主数据或业务单据。

方式：

1. 删除污染样本
2. 按标准模板重建干净样本

### 6.3 隔离复位

适用于 L3 高副作用场景。

方式：

1. 在隔离样本池中执行
2. 不直接触碰共享样本
3. 失败后销毁隔离副本

### 6.4 复位失败处理

复位失败时必须：

1. 记录污染对象
2. 标记本轮结果不可作为正式放行依据
3. 输出人工介入建议

## 7. 运行形态

### 7.1 PR / 日常回归模式

用途：

1. 快速检测受影响模块是否回归

范围：

1. 只跑受影响模块的 `L1`
2. 跑与本次修改直接相关的 `L2`

特点：

1. 反馈优先
2. 不追求全量

### 7.2 合并前正式门禁模式

用途：

1. 作为代码放行前的正式依据

范围：

1. 全模块 `L1`
2. 全模块 `L2`
3. 受控范围内的 `L3`

特点：

1. 输出正式 evidence
2. 有明确通过/不通过判定

### 7.3 周期性全量巡检模式

用途：

1. 弥补日常回归的覆盖盲区
2. 为长期目标 `方案 A` 持续铺路

范围：

1. 尽量全量写链路
2. 包含更多 `L3`

特点：

1. 更像系统体检
2. 不要求每次开发都阻塞在这里

## 8. 证据与留痕

每次门禁执行至少输出以下 4 类证据：

### 8.1 场景清单快照

包含：

1. 场景名称
2. 模块
3. 层级
4. token pool
5. 依赖样本

### 8.2 结果摘要

包含：

1. 成功率
2. 错误率
3. 错误码分布
4. `P95 / P99`
5. 复位成功率

### 8.3 失败归因

每个失败场景必须归因到以下之一：

1. 权限问题
2. 样本问题
3. 契约问题
4. 性能问题
5. 状态一致性问题
6. 环境阻塞

### 8.4 豁免与阻塞清单

必须记录：

1. 哪些链路本轮未执行
2. 未执行原因
3. 纳管前提
4. 预计下一步动作

## 9. 实施边界

本设计覆盖：

1. 写链路分层治理
2. 样本与复位策略
3. 门禁运行模式
4. evidence 结构

本设计不在本轮直接覆盖：

1. 具体阈值数值的最终拍板
2. 脚本实现细节
3. CI 编排细节
4. Docker 或数据库隔离环境的具体部署方案

这些内容应在后续 implementation plan 中展开。

## 10. 长期演进路线

### 10.1 第一阶段

采用“方案 B：分层写链路门禁”：

1. 建立 `L1/L2/L3` 分层口径
2. 补齐样本与复位声明
3. 建立正式 evidence 模板
4. 让门禁真正可执行

### 10.2 第二阶段

逐步减少差异化阈值和执行频率差异：

1. 收敛高副作用场景
2. 提升 `L3` 稳定性
3. 缩小层间门槛差异

### 10.3 长期目标

收敛到“方案 A：全量写链路统一门禁”：

1. 全量写链路统一治理
2. 统一指标结构
3. 尽量统一执行口径
4. 门禁结果可直接作为后端放行基础设施

## 11. 最终结论

后端写链路回归门禁不应继续以“场景文件存在但未正式纳管”的状态演进。推荐以“方案 B：分层写链路门禁”作为当前落地形态，在性能优先前提下，将全部写链路纳入治理视野，通过 `L1/L2/L3` 分层匹配不同副作用强度、样本策略与执行频率，先建立真实可跑的门禁闭环；待样本与高副作用链路收敛后，再向“方案 A：全量写链路统一门禁”演进。
