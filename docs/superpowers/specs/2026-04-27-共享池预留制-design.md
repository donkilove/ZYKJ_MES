# 共享池预留制 —— 设计共识

## 0. 文档信息

- 日期：2026-04-27
- 来源：`04-流转逻辑缺陷与改进项.md` §2.5 讨论衍生
- 状态：代码已实现（2026-04-28）

---

## 1. 背景

当前模型为"子工单分配制"：工序 `visible_quantity` → 均分到各操作员子工单 `assigned_quantity` → 各操作员独立消耗分配量。

切换为"共享池预留制"：工序所有操作员共享一份 `visible_quantity`，开始首件时预留 1 件，保证先开始首件的人至少有一个产品用于结束生产。

---

## 2. 已确认口径

### 2.1 数据模型层

| 项 | 结论 |
|---|---|
| 子工单表 | 保留，去掉 `assigned_quantity`、`is_visible`，仅作为"谁干过这个工序"的参与凭证 |
| 子工单状态 | `pending` / `in_progress`（去掉 `done`） |
| 子工单创建 | 懒创建——操作员第一次点首件时创建 |
| 池子剩余 | 实时计算：`visible_quantity - completed_quantity - in_progress子工单数`，不持久化 |
| 并行实例 | 保留，改名"流水线模式"，改挂工序级（不挂子工单） |

### 2.2 业务流转层

| 项 | 结论 |
|---|---|
| 首件通过 | 操作员子工单 `pending→in_progress`，池子预留 1 件 |
| 结束生产 | 报工量 ≤ `visible_quantity - completed_quantity - 其他in_progress人数`，报工后子工单回 `pending` |
| 每次做都要首件 | 是 |
| 我的工单展示 | 查工序级：`visible_quantity > 0` + 操作员角色匹配 + 工序非 `completed`，不依赖子工单 |

### 2.3 辅助链路层

| 项 | 结论 |
|---|---|
| 代班授权 | 代班人在工序共享池操作，系统校验授权有效性；发起人可对不同目标/工序继续发起 |
| 代班撤销 | 新增，仅发起人可操作 |
| 改单准入 | 所有工序 `pending` 或 `completed` |
| 维修回流 | `visible_quantity` 增减逻辑不变，去掉均分到子工单的步骤 |
| 手工完工级联 | 老授权置 `consumed`（沿用 §2.6 结论） |

## 3. 代码实现要点（2026-04-28）

### 3.1 模型变更

| 文件 | 变更 |
|---|---|
| `production_sub_order.py` | 删除 `assigned_quantity`、`is_visible` 字段；去掉 cascade delete |
| `production_constants.py` | 删除 `SUB_ORDER_STATUS_DONE` 及关联引用 |
| `order_sub_order_pipeline_instance.py` | `sub_order_id` 改为可空（`ondelete="SET NULL"`） |

### 3.2 Schema 变更

| 文件 | 变更 |
|---|---|
| `schemas/production.py` | `ProductionSubOrderItem` 删 `assigned_quantity`/`is_visible`；`MyOrderItem` 删 `user_assigned_quantity` |

### 3.3 服务层关键变更

| 函数 | 变更 |
|---|---|
| `_lock_sub_order` | 子工单不存在时**懒创建**，不再抛异常 |
| `_create_initial_sub_orders_for_process` | 改为 no-op |
| `ensure_sub_orders_visible_quantity` | 改为 no-op（返回 False） |
| `_split_quantity_evenly` | 改为返回空列表 |
| `_backfill_historical_release_sub_orders` | 改为 no-op |
| `submit_first_article` | 用 `pool_remaining = visible - completed - in_progress_count` 替代 `sub_remaining`；不设置 `is_visible` |
| `end_production` | `max_producible = process_remaining - in_progress_count + 1`；子工单统一回 `pending`（不检查 `assigned_quantity`） |
| `_build_my_order_item` | 支持 `sub_order=None`（用 `pool_remaining`）；`can_first_article` 不依赖子工单 |
| `_my_order_quantity_summary` | 去掉 `user_assigned_quantity` 分支 |
| `_ensure_target_sub_order_exists` | 改为 no-op（懒创建兜底） |
| `complete_order_manually` | 子工单 `completed_quantity` 对 `visible_quantity` 取 max，不引用 `assigned_quantity` |
| `update_order` | 改单准入改为检查所有工序 `pending`/`completed`（不在生产中） |

### 3.4 新增 API

| 端点 | 说明 |
|---|---|
| `DELETE /orders/{order_id}/assist-authorizations/{id}` | 撤销代班，仅发起人可操作 |
| `PERM_PROD_ASSIST_AUTHORIZATIONS_CANCEL` | 新增权限 `production.assist_authorizations.cancel` |

### 3.5 流水线模式重命名（2026-04-28）

| 旧名 | 新名 |
|---|---|
| `OrderSubOrderPipelineInstance` | `ProcessPipelineInstance` |
| `mes_order_sub_order_pipeline_instance` | `mes_process_pipeline_instance` |
| `pipeline_sub_order_no` | `pipeline_instance_no` |
| 各类索引/约束名 | 对应替换 `order_sub_order_pipeline` → `process_pipeline` |

> **注意**：alembic 迁移文件未更新，数据库表需新增一条迁移脚本完成表重命名。

### 3.6 订单状态收口函数统一（2026-04-28）

- `_refresh_order_status`：改为数量比较版（`completed_quantity < visible_quantity`），替代原来的 `process.status == COMPLETED` 判断
- `_recompute_order_status_by_quantities`：改为委托 `_refresh_order_status(db, order=order)`

### 3.7 我的工单查询改为工序级（2026-04-28）

- own view：从 `select(ProductionSubOrder)` 改为 `select(ProductionOrderProcess)`，按角色匹配工序编码过滤，左关联子工单
