# 工艺看板页驾驶舱化重构设计

## 1. 背景

`frontend/lib/features/craft/presentation/craft_kanban_page.dart` 当前已经具备完整的查询、导出和趋势展示能力，但页面表达仍停留在“标题 + 筛选条 + 说明框 + 内容列表”的基础后台页形态。

从现状截图与源码结构看，当前页面主要存在三类问题：

1. 顶部筛选区缺少信息层级，控件只是横向平铺，观感松散。
2. 统计说明独占一整行，但没有和筛选或结果形成组织关系，压缩了首屏有效空间。
3. 真正高价值的数据结果被推到页面下方，用户进入页面后先看到的是操作控件，而不是结论。

本轮目标不是继续做小修小补，而是在不改变业务口径与接口行为的前提下，把页面重构为“结论先行”的工业风驾驶舱。

## 2. 目标

1. 将工艺看板重构为首屏即可读结论的驾驶舱页面。
2. 提升纵向与横向空间利用率，减少大段空白和松散排布。
3. 保留现有筛选、日期、导出、刷新与图表逻辑，不改变业务交互结果。
4. 在宽桌面与窄桌面下都保持稳定、整齐、无溢出。
5. 让页面视觉更贴近制造场景下的工业仪表盘，而不是普通表单页。

## 3. 非目标

1. 不修改后端接口、统计口径与导出契约。
2. 不调整异常阈值算法、图表数据来源或样本计算方式。
3. 不改动 `CraftService`、`ProductionService` 或图表数据模型。
4. 不顺带重构其他工艺页、质量页或全局 UI 体系。
5. 不在本轮引入新的状态管理方案或页面级查询按钮。

## 4. 现状问题

### 4.1 首屏只看到控件，几乎看不到结论

页面进入后首先呈现的是多组筛选控件、导出按钮和长文本说明，真正的数据结果需要继续下滚才能看到。这导致页面更像“操作入口”，而不是“数据看板”。

### 4.2 顶部筛选区缺少控制台感

当前筛选区虽然已经通过 `Wrap` 解决了窄桌面换行问题，但控件之间没有形成主次结构：

1. 产品、工段、工序三个核心筛选没有组成一个明确的主操作区。
2. 日期筛选和导出按钮分散在独立层次中，缺少完整的控制面板感。
3. 统计说明与筛选逻辑脱节，既没有提升理解效率，也占用了首屏空间。

### 4.3 工序详情卡纵向过高

每个工序卡内部当前采用“工时图在上、产能图在下”的纵向堆叠布局。只要工序有数据，单张卡片高度就会快速抬高，导致：

1. 首屏可见工序数量偏少。
2. 用户需要频繁滚动才能横向比较不同工序。
3. 整体页面更像报告流，而不是工作看板。

### 4.4 空态表达过轻

当筛选无数据或查询失败后，内容区大多退化为一条文本。这虽然功能上可用，但在页面整体重心上显得发空，也无法给用户足够明确的状态反馈。

## 5. 已确认方向

本轮设计已与用户确认以下方向：

1. 页面风格选择：`数据驾驶舱`
2. 首屏重心选择：`先看结论`
3. 视觉气质选择：`工业仪表盘`

因此，本轮采用“控制台 + 摘要卡 + 趋势主体 + 工序双图卡”的驾驶舱结构，不走左侧固定筛选面板，也不走全黑大屏样式。

## 6. 总体设计

页面最终结构调整为五层：

1. 头部区
2. 驾驶舱控制台
3. 摘要指标带
4. 趋势主体区
5. 状态区

整体原则如下：

1. 先结论，后细节。
2. 先总览，后分项。
3. 同类信息收进同一视觉分区。
4. 优先压缩无效空白，不压缩可读性。

## 7. 页面骨架

### 7.1 头部区

头部仍沿用 `MesRefreshPageHeader`，但增加副标题，形成更明确的页面语义。

建议副标题方向：

1. 快速识别异常工时与工序产能波动。
2. 结合产品、工段、工序和时间范围观察工艺执行状态。

作用：

1. 把页面从“普通列表页”提升为“分析页”。
2. 与下方驾驶舱控制台形成语义连续性。

### 7.2 驾驶舱控制台

当前筛选区与统计说明将重构为单一控制台卡片，内部再分为两层：

1. 主筛选层
2. 辅助动作层

主筛选层承载：

1. 产品筛选
2. 工段筛选
3. 工序筛选

辅助动作层承载：

1. 开始日期
2. 结束日期
3. 清除日期
4. 导出按钮
5. 统计规则提示

要求：

1. 视觉上收进同一张控制台容器，而不是继续拆成分离的两组控件。
2. 产品筛选作为最强主筛选，占更大宽度。
3. 导出动作在辅助动作层中保持最明显的操作等级。
4. 原长段统计说明压缩为控制台底部的辅助提示，不再独占整行。

### 7.3 摘要指标带

控制台下方紧接摘要卡带，优先提供当前筛选条件下的核心结论。

建议首批指标：

1. `工序数`：当前结果中有数据的工序数量。
2. `样本数`：全部工序样本总量。
3. `平均工时`：全部样本平均工时。
4. `平均产能`：全部样本平均产能。
5. `异常样本`：工时超过均值 130% 阈值的样本数。

说明：

1. 若异常样本为 0，仍显示该卡，但文案落为“无异常”或 `0`。
2. 指标值应由现有 `_metrics` 与样本列表即时聚合得到，不增加接口依赖。
3. 指标卡是本轮“结论先行”的核心区，不可省略。

### 7.4 趋势主体区

趋势主体区采用“总览在上，工序详情在下”的结构。

#### 7.4.1 总览区

保留现有“工序趋势对比（平均工时/产能）”表格，但作为驾驶舱总览卡处理，强调：

1. 标题
2. 当前排序逻辑
3. 数据对比关系

这一块仍是结果的第一层摘要，位于所有工序详情卡之前。

#### 7.4.2 工序详情卡区

工序详情卡由当前纵向堆叠结构改为“双栏图表卡”：

1. 卡片头部：工段/工序名、样本区间、样本数标签。
2. 卡片内容：左侧工时柱图，右侧产能折线图。

作用：

1. 大幅压缩单卡纵向高度。
2. 提升工时与产能的并行对照效率。
3. 让页面更符合仪表盘而非报告式阅读路径。

### 7.5 状态区

状态区分为空态与错误态两类。

#### 7.5.1 空态

空态不再只显示“暂无可统计数据”，而是重构为状态卡，至少包含：

1. 主文案
2. 辅助说明
3. 当前筛选语义提示

建议主文案方向：

1. 当前筛选下暂无已完工样本。
2. 可尝试调整产品、工段、工序或日期范围后重试。

#### 7.5.2 错误态

查询失败后仍保留现有 `SnackBar`，但内容区状态表达应与空态区分清楚，避免页面整体突然退化成一行字。

## 8. 视觉风格

### 8.1 整体基调

本轮采用“浅底工业仪表盘”，而不是传统黑底大屏。

原则：

1. 保留企业应用的可读性与稳定性。
2. 通过更强的边界、卡片层级和数据色提升工业感。
3. 不与现有系统的基础主题完全割裂。

### 8.2 控制台容器

控制台应具备明确的“操作面板”感，而不是普通表单组。

建议特征：

1. 整体容器带更清晰的描边和圆角。
2. 内部控件高度统一，节奏更紧凑。
3. 主筛选与辅助动作通过分区或轻背景差做出层次。

### 8.3 摘要卡

摘要卡遵循“数值优先，标题后退”的表达方式。

要求：

1. 指标值为卡片主视觉。
2. 指标标题缩小但保持清晰。
3. 异常类指标使用更显眼的强调色。
4. 普通指标使用偏冷静的工业蓝或中性色。

### 8.4 图表容器

图表不再裸放于卡片中，而是形成更完整的分析区块。

要求：

1. 工时图区与产能图区语义区分清楚。
2. 工时柱图继续保留异常红柱语义。
3. 常规工时柱颜色统一为更稳重的工业蓝。
4. 产能折线图用较高识别度的暖色，与工时图形成对照。

### 8.5 统计说明

现有长段统计说明不删除，但降级为辅助信息，不再作为首屏主内容。

建议处理方式：

1. 收进控制台底部说明条。
2. 或缩成带图标的规则提示块。

原则是保留规则、弱化视觉抢占。

## 9. 响应式策略

### 9.1 宽桌面

适用范围：约 `>= 1280`

布局要求：

1. 控制台尽量在一张面板内完成主筛选与辅助动作的完整展示。
2. 摘要卡优先单排展示。
3. 工序详情卡固定双栏图表。

目标：

1. 首屏尽量看到控制台、摘要卡和趋势主体入口。
2. 充分利用横向空间，减少滚动成本。

### 9.2 中等桌面

适用范围：约 `980 - 1279`

布局要求：

1. 控制台允许上下两层，但仍保持单卡组织。
2. 摘要卡自动换成两行。
3. 工序详情卡优先保持双栏并排，必要时缩短图表高度。

目标：

1. 继续保持驾驶舱结构。
2. 不因收窄而退回松散表单感。

### 9.3 窄桌面

适用范围：约 `<= 979`

布局要求：

1. 主筛选可转为纵向或 `2 + 1` 布局。
2. 日期与导出动作进入下一行。
3. 工序详情卡中的双图改为上下堆叠。

目标：

1. 保证无溢出。
2. 保证控件仍然可点、图表仍然可读。
3. 结构降级但不失序。

## 10. 交互策略

### 10.1 保留现有筛选刷新行为

本轮不新增查询按钮，继续保留：

1. 修改产品后立即刷新
2. 修改工段后立即刷新
3. 修改工序后立即刷新
4. 选定日期后立即刷新
5. 清除日期后立即刷新

原因：

1. 避免改动用户既有操作习惯。
2. 将本轮改造聚焦在布局与信息表达层。

### 10.2 强化忙碌反馈

虽然不改变业务交互行为，但要增强忙碌态反馈：

1. 控制台区在加载中体现禁用态一致性。
2. 导出按钮在导出中体现忙碌态。
3. 刷新时页面头部与控制台语义一致。

### 10.3 日期区稳定性

日期按钮与“清除日期”应形成稳定的动作组，不再因是否出现清除动作而导致整体布局跳跃过大。

### 10.4 空态与错误态稳定性

即使查询失败或筛选无结果，页面也要保留看板结构的基本稳定感，而不是让内容区突兀塌缩。

## 11. 实现边界

### 11.1 主要改动文件

本轮核心改动集中在：

1. `frontend/lib/features/craft/presentation/craft_kanban_page.dart`
2. `frontend/test/widgets/craft_kanban_page_test.dart`

如确有必要，可在当前页面附近新增极少量私有 widget 文件，但不作为本轮默认前提。

### 11.2 允许的实现动作

1. 重组页面布局层级。
2. 新增摘要指标聚合逻辑。
3. 提炼页面内部私有组件或私有方法。
4. 调整图表容器排版和卡片结构。
5. 优化空态与说明区表达。

### 11.3 不允许的扩散

1. 不顺手抽全局通用驾驶舱组件。
2. 不顺手重做 `MesCrudPageScaffold`。
3. 不重构服务层与数据模型。
4. 不把本轮演化成全局 UI 基础件治理任务。

## 12. 验收标准

### 12.1 页面结构

完成后应满足：

1. 首屏包含头部、控制台、摘要卡与趋势入口。
2. 统计说明不再独占一整行大框。
3. 工序详情卡默认采用双栏图表结构。
4. 整体页面空间利用率明显高于当前版本。

### 12.2 行为保持

完成后应满足：

1. 刷新、筛选、日期、导出逻辑与现有行为一致。
2. 无产品时仍展示空态并隐藏不必要筛选控件。
3. 查询失败时仍可见错误反馈。
4. 日期选择后仍显示清除日期入口。

### 12.3 响应式

完成后应满足：

1. 既有窄桌面无溢出约束继续成立。
2. 中等宽度下控件换行后仍保持清晰分组。
3. 宽桌面下能明显感受到“驾驶舱首屏”。

### 12.4 视觉结果

完成后应满足：

1. 页面更像工业分析面板，而不是普通筛选页。
2. 数据结论优先级高于控件与说明文案。
3. 页面层次清楚，主次分明。

## 13. 验证策略

本轮至少覆盖以下验证：

1. `flutter analyze lib/features/craft/presentation/craft_kanban_page.dart test/widgets/craft_kanban_page_test.dart`
2. `flutter test test/widgets/craft_kanban_page_test.dart`

测试重点：

1. 窄桌面下顶部控制台不溢出。
2. 摘要区与主要结构正常出现。
3. 导出、日期、空态、错误态原有行为不回退。

## 14. 风险与控制

### 风险 1：首屏信息密度过高

表现：

1. 看起来更紧凑，但读起来反而更乱。

控制方式：

1. 坚持“控制台、摘要卡、趋势区”三层分区。
2. 通过卡片层次而不是堆控件提升密度。

### 风险 2：只在宽桌面好看

表现：

1. 宽屏驾驶舱效果成立，窄桌面重新出现溢出或拥挤。

控制方式：

1. 保留并扩充现有窄桌面测试。
2. 对工序卡双图布局设置明确降级策略。

### 风险 3：实现过程中顺手扩散成全局 UI 改造

表现：

1. 页面还没做完，先开始抽象一堆基础件。

控制方式：

1. 严格限制在工艺看板页内收口。
2. 只有在当前页面内出现高复用价值的局部结构时，才做最小提炼。
