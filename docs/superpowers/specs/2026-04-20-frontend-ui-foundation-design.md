# 前端 UI 基础件体系设计

## 1. 背景

当前前端已经具备一定的视觉设计意识，但整体仍停留在“页面各自完成”的阶段，还没有形成可复用、可治理、可持续演进的统一 UI 体系。

已有代码里已经出现了一些统一化雏形，例如 CRUD 页头、列表区、分页条和表头风格，但这些能力主要停留在局部列表场景，没有继续上升为全局页面骨架和通用交互模式。因此，随着首页、设置页、消息中心和后续复杂业务页持续演进，页面风格、布局密度、信息层级和交互语义会越来越容易分叉。

本轮目标不是“全站立刻重画一遍”，而是先建立一套足够清晰的前端 UI 基础件体系，让后续页面有统一规则可依、有统一骨架可装、有明确边界可拆。

## 2. 现状问题

当前 UI 的主要问题不是单个页面难看，而是系统层的一致性不足。

### 2.1 全局主题过薄

应用入口目前主要统一了种子色、亮暗主题、字体族和视觉密度，但还没有把颜色语义、间距、圆角、边框、阴影、标题层级、状态色语义等设计规则沉淀成可复用 token。

### 2.2 页面骨架语言不一致

登录页、首页、设置页、消息中心都各自形成了局部风格，但这些页面在头部结构、内容分区、卡片语言、操作栏组织方式上缺少统一骨架，导致页面之间不像同一套产品语言自然生长出来的。

### 2.3 统一化范围只覆盖了部分 CRUD 场景

现有 `core/widgets` 已经覆盖了 CRUD 页头、表格区和分页区，但没有统一“页面头部 + 内容分区 + 筛选条 + 状态卡 + 空态/错态 + 详情区”这一层更贴近页面表达的模式组件。

### 2.4 大页面进一步放大风格漂移

当前仓库已有多处超大页面文件。页面同时承担布局、局部样式、交互状态和业务拼装时，视觉规则会自然分叉，维护成本也会快速上升。

## 3. 已确认路线

本轮采用路线 B：建立“设计 Token + 页面骨架组件 + 业务组件组装”的三层体系。为了让工程边界更清晰，目录结构上额外拆出一层 `primitives` 作为基础拼装桥接层，因此概念上是三层，工程落地上是四层：

1. `core/ui/foundation/`
2. `core/ui/primitives/`
3. `core/ui/patterns/`
4. 各业务模块内的 `feature widgets`

这四层并不是为了增加抽象，而是为了让样式规则、通用外壳和业务拼装各归其位，避免继续在页面文件里直接堆叠临时 `BoxDecoration`、`EdgeInsets` 和 `TextStyle`。

## 4. 目标

1. 建立统一的 UI 基础规则，让颜色、间距、圆角、文本层级和状态语义有单一来源。
2. 把“页面骨架语言”沉淀为可复用组件，而不是继续在每个页面重复手写布局。
3. 让业务页面主要负责组装和业务表达，减少直接处理底层样式细节。
4. 为后续页面拆分、测试、样式治理和一致性检查建立稳定边界。

## 5. 非目标

1. 本轮不做全站一次性视觉重做。
2. 本轮不引入完整外部设计系统或第三方 UI 框架迁移。
3. 本轮不把所有旧页面一次性迁移到新体系。
4. 本轮不在 `core/ui` 中承载任何业务模型、业务接口或业务文案。

## 6. 体系总结构

### 6.1 `foundation`

`foundation` 是设计规则层，只负责统一 token、主题接法和语义样式映射，不直接表达具体业务页面。

建议承载以下内容：

1. 颜色语义 token
2. 间距 token
3. 圆角 token
4. 文本层级 token
5. 阴影、描边、状态透明度 token
6. 亮暗主题映射和 `ThemeExtension` 接法

### 6.2 `primitives`

`primitives` 是小颗粒基础件层，负责把 `foundation` 中的 token 变成可复用的最小视觉单元。它们不承担页面级语义，只解决“统一样式的基础拼装问题”。

建议放置：

1. 通用容器壳层
2. 统一间距与分隔件
3. 状态标识小组件
4. 统一标签、小徽标、信息行等轻量件

### 6.3 `patterns`

`patterns` 是页面模式层，负责表达产品中反复出现的页面骨架和信息组织方式。它们是本轮体系的核心交付层。

这一层既要统一视觉，也要统一页面组织方式。

### 6.4 `feature widgets`

业务模块下的 `feature widgets` 只负责业务语义和业务组合，例如消息中心专属卡片、设置项面板、首页专属统计块等。它们可以组装 `patterns` 和 `primitives`，但不应反向让 `core/ui` 依赖业务层。

## 7. Token 体系

### 7.1 颜色语义

颜色不再按“页面自己挑一个看起来顺眼的颜色”组织，而是统一按语义组织。

建议至少沉淀以下语义：

1. `background`
2. `surface`
3. `surfaceSubtle`
4. `surfaceRaised`
5. `border`
6. `borderStrong`
7. `primary`
8. `primaryOn`
9. `secondary`
10. `textPrimary`
11. `textSecondary`
12. `textMuted`
13. `success`
14. `warning`
15. `danger`
16. `info`

所有页面卡片、区块背景、状态提示、标题和辅助文字优先从语义 token 取值，而不是直接写颜色字面量。

### 7.2 间距体系

建议统一为有限档位，例如：

1. `xxs`
2. `xs`
3. `sm`
4. `md`
5. `lg`
6. `xl`
7. `xxl`

页面内边距、卡片间距、模块间垂直节奏、筛选项间距统一使用这些 token，不再在页面里随意写 `EdgeInsets.symmetric(horizontal: 13)` 之类的离散值。

### 7.3 圆角体系

建议统一最少三档：

1. `sm`
2. `md`
3. `lg`

其中：

1. 输入框、轻量按钮优先使用 `sm`
2. 普通卡片、工具栏容器优先使用 `md`
3. 概览卡、顶部强调容器优先使用 `lg`

### 7.4 文本层级

建议以“页面层级”而不是“随手写字号”组织：

1. `pageTitle`
2. `sectionTitle`
3. `cardTitle`
4. `body`
5. `bodyStrong`
6. `caption`
7. `metric`

页面标题、区块标题、指标值、辅助文本都应走语义层级，而不是各页单独定义。

### 7.5 状态与装饰 token

建议补齐：

1. 阴影等级
2. 描边粗细
3. 禁用态透明度
4. Hover 和选中态叠层色

这些 token 统一后，明暗主题切换、交互反馈和卡片层级才会稳定。

### 7.6 主题接法

建议通过 `ThemeExtension` 或同等集中方式暴露 `MesTokens`，并由应用主题在亮色与暗色模式下分别映射具体值。

页面和组件只消费 token，不直接感知亮暗主题细节。

## 8. 基础件清单分层

### 8.1 `foundation` 目录建议

建议目录结构：

```text
frontend/lib/core/ui/foundation/
  mes_colors.dart
  mes_spacing.dart
  mes_radius.dart
  mes_typography.dart
  mes_elevation.dart
  mes_tokens.dart
  mes_theme_extensions.dart
```

### 8.2 `primitives` 目录建议

建议目录结构：

```text
frontend/lib/core/ui/primitives/
  mes_surface.dart
  mes_gap.dart
  mes_divider.dart
  mes_badge.dart
  mes_status_chip.dart
  mes_info_row.dart
  mes_icon_label.dart
```

这一层规则：

1. 只处理基础视觉拼装
2. 不出现页面级布局假设
3. 不依赖任何业务模型和业务枚举

### 8.3 `patterns` 目录建议

建议首批模式组件清单如下：

1. `MesPageHeader`
2. `MesSectionCard`
3. `MesToolbar`
4. `MesFilterBar`
5. `MesEmptyState`
6. `MesErrorState`
7. `MesPaginationBar`
8. `MesMetricCard`
9. `MesDetailPanel`

这些组件分别解决如下问题：

1. `MesPageHeader`
   统一页面标题、副标题、操作区、返回行为和页面头部间距。
2. `MesSectionCard`
   统一区块标题、说明、容器外观和内容边距。
3. `MesToolbar`
   统一工具条中的主操作、次操作、搜索与右侧动作布局。
4. `MesFilterBar`
   统一筛选区域的排布、折叠、重置和提交区位置。
5. `MesEmptyState`
   统一空态图文层级和推荐动作位置。
6. `MesErrorState`
   统一错误提示、重试动作和信息密度。
7. `MesPaginationBar`
   统一页码、条数和翻页动作语义。
8. `MesMetricCard`
   统一指标卡的数值层级、说明文案和趋势信息表达。
9. `MesDetailPanel`
   统一详情侧栏、说明区和信息行展示方式。

### 8.4 `feature widgets` 目录建议

业务模块内部只保留“有业务含义、对其他模块不可泛化”的组件，例如：

1. 设置页的主题模式项、布局偏好项
2. 首页的生产态势概览块
3. 消息中心的消息分类摘要卡

这些组件可以依赖 `patterns` 和 `primitives`，但不得把业务依赖倒灌回 `core/ui`。

## 9. 页面装配原则

页面改造后的推荐装配顺序应为：

1. 页面使用 `MesPageHeader` 建立统一头部
2. 主内容区按 `MesSectionCard` 或 `MesDetailPanel` 切分
3. 有筛选需求的页面优先使用 `MesFilterBar`
4. 有概览数据的页面优先使用 `MesMetricCard`
5. 空数据和错误态统一使用 `MesEmptyState`、`MesErrorState`
6. 页面局部差异放在业务模块自己的 `feature widgets`

页面不应继续把大量临时样式直接写在 build 方法里。若页面需要重复写三次以上同类布局或同类装饰，应上提到 `primitives` 或 `patterns`。

## 10. 迁移顺序与实施策略

### 10.1 总体原则

本轮不做全站重写，按“先地基、再样板、后扩散”的顺序推进。

### 10.2 第一阶段：补齐地基

先建立 `foundation` 和最小可用的 `primitives`，把颜色、间距、圆角、标题层级和基础容器外观稳定下来。

### 10.3 第二阶段：建立首批页面模式件

优先落地：

1. `MesPageHeader`
2. `MesSectionCard`
3. `MesToolbar`
4. `MesFilterBar`
5. `MesEmptyState`
6. `MesErrorState`

这一步完成后，页面骨架语言才真正开始统一。

### 10.4 第三阶段：试点页面迁移

建议试点页按以下顺序推进：

1. 设置页
2. 首页
3. 消息中心

原因如下：

1. 设置页结构相对清晰，适合验证 token 和 section card 是否顺手。
2. 首页能验证指标卡、页面头部和概览区语言是否统一。
3. 消息中心能验证复杂业务页在新骨架下的可拆分性与可维护性。

### 10.5 第四阶段：批量迁移

试点稳定后，再把列表页、详情页和复杂表单页逐步迁移到同一体系。

### 10.6 不做的事情

迁移过程中不为了“追求统一”去顺手重构无关业务逻辑，也不把所有老组件一次性替换掉。优先替换真正影响统一性的页面骨架和核心展示层。

## 11. 验收标准

### 11.1 视觉一致性

试点页应满足：

1. 页面标题层级一致
2. 区块容器外观一致
3. 页面主间距节奏一致
4. 空态、错态和分页区语义一致
5. 明暗主题切换后无明显样式断层

### 11.2 工程一致性

试点页迁移后应满足：

1. 页面 build 方法中显著减少直接写样式字面量
2. `core/ui` 不依赖业务模块
3. 页面骨架和业务组件职责分离
4. 新增页面优先组装 `patterns`

### 11.3 质量门禁

至少执行：

1. `flutter analyze`
2. 与新增基础件相关的 widget tests
3. 试点页关键路径的 `integration_test`

## 12. 治理与长期演进规则

### 12.1 新增组件的归属判断

新增组件前先判断它属于哪一层：

1. 若只是样式规则，进入 `foundation`
2. 若是无业务语义的小颗粒拼装件，进入 `primitives`
3. 若是跨页面复用的骨架或模式，进入 `patterns`
4. 若只服务单一业务模块，留在该模块的 `feature widgets`

### 12.2 `core/ui` 的边界

`core/ui` 中禁止出现：

1. 业务接口调用
2. 业务枚举判断
3. 业务模型依赖
4. 专属于某一模块的文案语义

### 12.3 页面文件拆分规则

当页面同时承担以下任意两类以上职责时，应主动拆分：

1. 页面骨架布局
2. 复杂筛选区
3. 列表或表格展示
4. 详情区或侧边栏
5. 大量局部状态切换

拆分优先按职责边界进行，而不是继续堆在同一个超大文件里。

### 12.4 样式书写约束

业务页面中禁止长期保留大量临时：

1. `BoxDecoration`
2. `EdgeInsets`
3. `TextStyle`
4. 颜色字面量

若某一写法将在多个页面重复出现，应优先沉淀为 token、primitive 或 pattern。

## 13. 风险与控制

### 风险 1：抽象过度，基础件变得难用

控制方式：

1. 首批只做当前页面真正重复出现的模式
2. 不为了抽象而抽象
3. 以设置页、首页、消息中心作为试点验证可用性

### 风险 2：基础件边界不清，业务逻辑回流到 `core/ui`

控制方式：

1. 明确 `core/ui` 不得依赖业务层
2. 新增组件按四层归属规则收口
3. 评审时优先检查边界而不是只看视觉结果

### 风险 3：旧页面迁移节奏失控

控制方式：

1. 先试点后扩散
2. 不做全站同步重构
3. 每轮只迁移与当前目标直接相关的页面

## 14. 实施后的预期效果

体系落地后，前端将从“每页单独写样式”升级为“统一 token 定规则、统一 pattern 立骨架、业务组件专注表达”的开发模式。

这会带来三类直接收益：

1. UI 更统一，页面更像同一套产品
2. 页面更容易拆，复杂页面维护压力下降
3. 后续新增页面可以优先组装，而不是从零开始设计结构和样式

## 15. 迁移说明

- 无迁移，直接替换
