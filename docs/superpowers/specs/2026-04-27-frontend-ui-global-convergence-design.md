# 前端全站 UI 一致性与布局合理性收敛设计

## 1. 背景

仓库前端已经具备完整的 UI 基础件体系，但全站还没有完成统一迁移，存在新旧双轨并存的情况。

通过本轮审计，已经能够确认：

1. `frontend/lib/core/ui/foundation/` 已经沉淀了完整的 token 体系，包括 `MesTokens`、`MesColors`、`MesSpacing`、`MesRadius`、`MesTypography`、`MesTheme`，并通过 `ThemeExtension` 接入主题。
2. `frontend/lib/core/ui/primitives/` 已经沉淀了 `MesGap`、`MesInfoRow`、`MesStatusChip`、`MesSurface` 等小颗粒基础件。
3. `frontend/lib/core/ui/patterns/` 已经沉淀了 13 个页面级模式件，覆盖页头、筛选区、工具条、内容卡、表头、分页条、空态、错态、内嵌横幅、详情面板、列表详情壳层、指标卡、CRUD 页面壳层。
4. `frontend/lib/core/widgets/` 中仍有 6 个旧版共享件，与新版 patterns 共存。
5. 已有多份模块级 UI 收敛设计与实施计划已经落地，包括用户、产品、设备、生产、质量等模块的全模块收敛。
6. 用户模块的用户管理页是当前已经迁移到新基础件体系的代表性页面，但仍同时引用了 `locked_form_dialog`、`simple_pagination_bar` 等旧件，并存在 `LegacyLegacyUserManagementPage` 双重前缀命名等历史痕迹。

本轮目标是把已有的 UI 基础件体系从“分模块陆续推广”推进到“全站一致性收口”，并对布局合理性做对应修正。

## 2. 现状问题

### 2.1 体系已建立，但全站未完成统一迁移

`core/ui/foundation`、`core/ui/primitives`、`core/ui/patterns` 已经具备完整能力，但部分模块仍直接引用 `core/widgets` 旧件、自行手写 `BoxDecoration` / `EdgeInsets` / `TextStyle`，导致同一类视觉节点在不同页面呈现不一致。

### 2.2 模块级抽取深度差异显著

不同 feature 模块在“页面外壳 + 头部 + 筛选 + 操作 + 列表 + 详情”这条链路上的抽取程度差异较大：

1. 产品模块抽取最为充分。
2. 用户模块有独立的 `shared/` 子目录沉淀复用件。
3. 生产、设备等模块的页面级抽取仍较少，存在较多页面内直接拼接结构的情况。

这种差异在视觉层面表现为页面骨架语言不统一，在维护层面表现为同类问题需要在多处分别修。

### 2.3 浮层与状态界面参差

弹窗、抽屉、侧栏详情、空态、加载态、错误态在不同页面有不同的容器、间距与底部操作区表达，主页面统一后浮层依然容易“出戏”。

### 2.4 用户管理页基线尚未完全收口

用户管理页虽然作为本轮约定的视觉基线，但其自身仍有遗留：

1. 同时引用了新旧组件。
2. 类名带有双重 `Legacy` 前缀。
3. `shared/` 中有部分能力本可上升至模块外公共层。

需要先把基线本身收口，再以它为参考对全站做收敛。

## 3. 已确认路线

本轮采用“**两层交付**”路线：

1. 第一层：以用户管理页为视觉基线，对全站所有可见界面完成第一轮统一收敛。
2. 第二层：对关键高频页面与最突出的布局问题做第二轮深打磨。

工程执行上，本路线展开为四个连续阶段（审计、第一轮全覆盖、第二轮打磨、收口与文档），详见 §10。

不引入新的设计体系，不更换 UI 风格，不重构与一致性无关的业务逻辑，不为未来需求提前抽象。

## 4. 目标

1. 在不更换当前 UI 风格的前提下，把已有 `core/ui` 体系推广到全站可见界面。
2. 让页头、筛选区、操作条、内容容器、列表分页、状态反馈、浮层、响应式行为在全站具备一致的骨架语言。
3. 修正布局合理性问题，包括留白节奏、信息层级、工具条换行、密度失衡、双栏切换等。
4. 把用户管理页本身的遗留收口，使其真正具备“可被全站映射”的基线资格。
5. 形成一份可被后续工作直接引用的全站 UI 基线说明文档。

## 5. 非目标

1. 不更换主题、品牌、色板和字体族。
2. 不引入新的设计系统或第三方 UI 库。
3. 不强行把所有业务页面改成与用户管理页同构。
4. 不为追求统一去抹掉业务上确有必要的布局差异。
5. 不承诺每个页面都做像素级精修；限时内先完成第一轮全覆盖再做第二轮重点打磨。
6. 不处理与 UI 一致性无关的业务缺陷与功能改动。
7. 不调整后端 API、数据模型与权限边界。

## 6. 范围

按用户确认的 C 口径，本轮覆盖前端**全部可见界面**：

1. 主路由页面与二级详情页。
2. 登录页、设置页、独立页与 misc 页。
3. 弹窗、抽屉、侧栏详情。
4. 空状态、加载态、错误态。
5. 页面内对视觉一致性有决定作用的容器与工具条。

模块覆盖至少包括：auth、craft、equipment、message、misc、plugin_host、product、production、quality、settings、shell、time_sync、user。

## 7. 视觉基线：用户管理页 8 类基线

本轮以用户管理页作为视觉基线，从中抽取 8 类骨架基线作为全站对齐目标。基线的目的是让全站“结构上像”，而不是“表面上像”。

### 7.1 页面头部基线

统一标题区组成：标题、次级说明、右侧主操作或刷新入口、与下方筛选区的距离。所有页面都需要先呈现“标题层 + 动作层”，再进入业务内容。

### 7.2 筛选区基线

统一查询条件区表现：控件排列节奏、单行多行换行规则、查询/重置/导出动作位置优先级、筛选区与列表区之间的视觉断层。本基线作为第一优先级收口对象。

### 7.3 操作条基线

统一新增、批量操作、次级操作的组织方式：主操作优先级、次级动作右置或收口、多操作堆叠间距与换行、与筛选区/表格区的连接关系。

### 7.4 内容容器基线

统一卡片、表格容器、详情容器壳层：外边距、内边距、圆角、边框、分区留白、区块标题与内容关系、大区块垂直节奏。

### 7.5 列表与分页基线

统一列表区信息组织：表格上下沿留白、空数据占位、分页条位置与密度、与列表容器的连接、统计信息保留尺度。

### 7.6 状态反馈基线

统一页面级反馈与局部反馈：成功/失败提示条、警示信息、加载骨架与占位、错误态、空状态层级与语气。覆盖空态、加载态、错误态。

### 7.7 浮层基线

统一弹窗、抽屉、侧栏详情：标题区、内容分组、底部操作区位置与优先级、表单密度、关闭/取消/确认顺序。

### 7.8 响应式与密度基线

统一不同宽度下的退化规则：宽页面双栏阈值、中等宽度切单栏、工具条换行、表单从并排转纵向、高密度业务页面允许保留的例外边界。

## 8. 抽取与对齐策略

每个页面会被分配三种处理策略之一：

1. **直接套壳**：原有内容基本可用，只需统一外层骨架与间距。
2. **局部重排**：工具条、筛选区、列表区顺序或分组调整。
3. **结构重组**：仅在页面明显混乱、不重组无法对齐基线时使用。

策略选择基于审计阶段输出的全站差异映射表，避免对所有页面采用相同改造强度。

## 9. 体系定位与公共层使用

本轮不新建独立设计系统，公共能力优先复用已有四层结构：

1. `core/ui/foundation/` 维持现状，仅在确实需要补足 token 时小幅扩充。
2. `core/ui/primitives/` 维持现状，仅在确实重复出现的小颗粒模式时扩充。
3. `core/ui/patterns/` 是本轮主要落点：所有页面的页头、筛选、工具条、内容卡、列表表头、分页、空错状态、内嵌横幅、详情面板、列表详情壳层、CRUD 壳层都对齐到 patterns。
4. `core/widgets/` 旧件不在本轮做大规模物理删除，但所有新引用必须切换到 patterns；旧件仅在残留页面中保留至自然替换。
5. 业务模块内的 `feature widgets` 主要负责组装与业务表达，不再承担底层样式细节。

## 10. 9 小时四阶段执行方案

### 阶段 0：全站审计与基线映射（约 1.5 小时）

输出一份全站差异映射，包括：

1. 页面、浮层、状态界面清单。
2. 不一致点分类：页头、筛选区、操作条、容器、列表、分页、反馈、浮层、响应式。
3. 直接对齐项、保留差异项、需要轻量重组项。
4. 用户管理页自身收口项。

### 阶段 1：全站第一轮统一（约 4.5 小时）

按以下优先顺序推进：

1. 公共壳层与页面骨架。
2. 筛选区与操作条。
3. 列表容器与分页。
4. 弹窗、抽屉、侧栏详情。
5. 空状态、加载态、错误态。

标准是“全站不再明显出戏”，不追求每页都精修。

### 阶段 2：重点页第二轮打磨（约 2 小时）

集中处理：高使用频率页、视觉断层最明显页、布局复杂且影响效率的页。重点修留白节奏、信息层级、工具条换行、双栏与单栏切换、浮层表单密度。

### 阶段 3：收口与文档（约 1 小时）

1. 关键页面与状态界面统一复查。
2. 输出一份项目内 UI 基线文档到 `docs/`。
3. 跑前端验证并写 evidence。

## 11. 验证方式

### 11.1 代码级验证

1. `flutter analyze`。
2. `flutter test`。
3. 受影响高频页面的相关 widget / integration test 补跑。

### 11.2 页面级验证

1. 启动前端，真实查看关键页面与浮层。
2. 检查统一项是否生效：页头层级、筛选区密度、操作条位置、容器留白、分页与表格关系、空错加载状态、弹窗抽屉操作区一致性。

### 11.3 留痕

1. 任务日志写入 `evidence/`。
2. 最终基线补入 `docs/`。
3. 工具不可用或验证降级时，明确记录影响范围。

## 12. 风险边界

1. 不因为统一抹掉业务必需的布局差异。
2. 不把所有页面强改成与用户管理页同构。
3. 不为了抽象而抽象，模式需稳定重复出现才上升为公共层。
4. 不把与 UI 一致性无关的业务问题混入本轮。
5. 当前工作区有 `frontend/windows/flutter/generated_plugin_registrant.cc`、`frontend/windows/flutter/generated_plugins.cmake` 两个未提交生成文件改动，本轮全程避开。
6. 9 小时是上限承诺；若个别页面牵涉过深，优先完成第一轮收口，第二轮再细化，不在限时内做大重构。

## 13. 交付物

1. 全站 UI 一致性第一轮收敛结果。
2. 重点页面第二轮打磨结果。
3. 必要的轻量公共壳层 / patterns 调整。
4. 一份 UI 基线文档，说明用户管理页如何作为全站参考。
5. 验证结果：分析与测试命令、页面级检查结论。
6. evidence 留痕。

## 14. 迁移口径

无迁移，直接替换。

## 15. 后续步骤

1. 本设计经用户审阅后，进入实施计划阶段。
2. 实施计划阶段通过 `writing-plans` 技能产出可执行的分阶段实施计划。
3. 实施阶段按阶段 0 到阶段 3 顺序推进，每阶段结束有 evidence 留痕与可见交付。
