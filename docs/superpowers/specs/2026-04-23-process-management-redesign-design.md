# 工序管理页紧凑工作台重构设计

## 1. 背景

`frontend/lib/features/craft/presentation/process_management_page.dart` 是当前前端中典型的历史大页之一。页面同时承担以下多类职责：

1. 页面生命周期与数据加载
2. 工段与工序列表展示
3. 搜索、筛选与 jump 承接
4. 页面提示与错误展示
5. 工段/工序弹窗与删除确认
6. 页面级布局与大量局部样式

当前实现虽然功能可用，但已经明显进入“单文件全包”的维护状态。继续在现有结构上叠加功能，会同时放大两个问题：

1. 页面结构难以理解与维护
2. 页面骨架风格难以继续向统一 UI 体系靠拢

在第一轮设计确认后，原本的方向是“三栏工作台 + 右侧聚焦工序详情”。但经用户进一步确认，本轮更优先的目标不是保留更多并列上下文，而是 **节省横向空间**。同时，右侧固定详情卡片本身的业务价值被判断为偏低，因此本轮设计改为：

**默认只看工序列表，工段作为辅助入口，不再保留固定详情卡片。**

## 2. 目标

1. 将 `process_management_page.dart` 从超大单文件重构为职责清晰的页面组合。
2. 让页面真正接入现有 `core/ui` 统一骨架，而不是仅局部替换几个卡片。
3. 将页面重构为“默认工序主视图 + 工段辅助入口”的紧凑工作台，减少横向空间浪费。
4. 在不改变接口契约、权限语义和交互结果的前提下，提升可维护性与统一度。

## 3. 非目标

1. 不修改后端接口和数据契约。
2. 不改变工段/工序增删改查业务规则。
3. 不改变 jump 承接语义、权限判断结果和关键交互路径。
4. 不在本轮引入新的全局状态管理方案或额外 controller/service 架构。
5. 不顺手重做整套工艺模块视觉。
6. 不再保留固定右侧“聚焦工序详情”卡片。

## 4. 现状问题

### 4.1 文件职责混杂

当前页面将装配、状态、列表、提示、弹窗和布局全部堆叠在同一个文件中，导致：

1. 单个变更的影响面过大
2. 局部功能难以独立理解
3. 页面结构难以复用到其他历史大页

### 4.2 页面骨架未统一

当前工艺模块已有 `CraftPageHeader`、`CraftPageShell` 等统一件，但 `process_management_page.dart` 仍主要依赖旧式 `CrudPageHeader`、手写 `Card`、局部 `DataTable` 和临时布局拼装。页面整体更像“历史后台页”，而不是统一 UI 体系下的紧凑工作台。

### 4.3 横向空间使用效率偏低

旧方案中工段、工序和详情长期并排，虽然信息同时可见，但对当前业务场景来说：

1. 默认高频操作更集中在工序上
2. 工段更多承担辅助查看和配置入口
3. 右侧详情卡片长期占据宽度，却没有足够高频且必要的信息价值

因此，继续维持三栏并排不符合本轮“更紧凑”的优化方向。

## 5. 已确认方向

本轮采用“激进重构型”，但严格限制在展示层和页面结构层，遵守以下边界：

1. 做 UI 骨架统一
2. 做页面拆分减重
3. 不改业务行为
4. 不改接口语义
5. 默认只显示工序主视图
6. 工段改为辅助入口
7. 去掉固定详情卡片

换句话说，这次允许做较大规模的页面结构重组，但不允许借重构之名修改业务逻辑。

## 6. 总体架构

建议将当前页面重构为三层：

### 6.1 页面壳层

负责接收：

1. `session`
2. `onLogout`
3. `canWrite`
4. `processId`
5. `jumpRequestId`

并将这些顶层输入交给页面状态编排层和统一页面骨架。

### 6.2 页面状态编排层

负责：

1. 数据加载
2. jump 承接
3. 搜索与筛选
4. 当前视图切换状态
5. 页面反馈消息
6. 工段/工序动作分发

这一层依然驻留在页面附近，不上升为新的全局状态管理或业务服务层。

### 6.3 业务区块层

拆成若干独立 widgets，只负责单一区块的渲染与局部交互：

1. 页头
2. 反馈区
3. 视图切换区
4. 工段面板
5. 工序面板
6. 工段弹窗
7. 工序弹窗

## 7. 文件拆分方案

建议拆分为以下文件：

### 7.1 页面入口

- `frontend/lib/features/craft/presentation/process_management_page.dart`

职责：

1. 页面入口
2. 连接统一骨架
3. 连接状态编排层
4. 保留少量生命周期入口

### 7.2 页面 widgets

- `frontend/lib/features/craft/presentation/widgets/process_management_page_header.dart`
  - 工艺模块语义页头，内部接 `MesPageHeader`
- `frontend/lib/features/craft/presentation/widgets/process_management_feedback_banner.dart`
  - 跳转定位、加载失败、权限提示统一出口
- `frontend/lib/features/craft/presentation/widgets/process_management_view_switch.dart`
  - 视图切换按钮区，负责“工序列表 / 工段列表”切换
- `frontend/lib/features/craft/presentation/widgets/process_stage_panel.dart`
  - 工段搜索、工段列表、工段操作
- `frontend/lib/features/craft/presentation/widgets/process_item_panel.dart`
  - 工序搜索、工段筛选、工序表格、工序操作
- `frontend/lib/features/craft/presentation/widgets/process_stage_dialog.dart`
  - 新建/编辑工段弹窗
- `frontend/lib/features/craft/presentation/widgets/process_item_dialog.dart`
  - 新建/编辑工序弹窗
- `frontend/lib/features/craft/presentation/widgets/process_delete_dialogs.dart`
  - 删除确认弹窗
- `frontend/lib/features/craft/presentation/widgets/process_management_models.dart`
  - 页面内部 view model / action enum
- `frontend/lib/features/craft/presentation/widgets/process_management_state.dart`
  - 页面状态编排层

说明：

原先计划中的 `process_focus_panel.dart` 不再纳入本轮范围。

## 8. 页面最终骨架

页面骨架采用“页头 + 反馈区 + 视图切换 + 单主视图区”的紧凑工作台：

### 8.1 顶部页头

使用工艺模块自己的 `process_management_page_header.dart`，而不是直接保留旧 `CrudPageHeader`。

要求：

1. 标题和副标题接统一骨架语言
2. 刷新、新建工段、新建工序等页面级动作集中在页头动作区

### 8.2 顶部反馈区

独立于主视图区存在，专门显示：

1. jump 定位结果
2. 数据加载失败
3. 权限与不可执行提示

这类信息不再散落在局部卡片中。

### 8.3 视图切换区

在主视图区上方提供轻量切换按钮：

1. `工序列表`
2. `工段列表`

要求：

1. 默认激活 `工序列表`
2. 切换按钮本身接统一骨架语言
3. 不改变页面级操作区位置

### 8.4 主视图区

一次只显示一个主区块：

1. 默认显示 `工序列表`
2. 用户切换后显示 `工段列表`
3. 不再长期并列显示工段、工序与详情

### 8.5 弹窗区

新建/编辑工段、新建/编辑工序、删除确认全部拆出主页面文件，由主页面只负责触发和接收结果。

## 9. 交互边界

### 9.1 默认行为

1. 页面首次进入时默认显示 `工序列表`
2. 页面大多数高频操作都围绕工序主视图展开

### 9.2 工段视图

1. `工段列表` 是辅助视图，不再与工序列表长期并排
2. 用户可主动切换到工段视图查看、编辑或删除工段

### 9.3 jump 承接

1. 外部 jump 命中工序时，页面必须自动切回 `工序列表`
2. 命中的工序需要在工序列表中被定位或高亮
3. jump 提示继续在顶部反馈区承接

### 9.4 从工段操作返回

1. 若用户在工段视图中完成会自然导向工序的操作（例如新建工序、跳转命中工序），页面应自动回到 `工序列表`
2. 不允许用户被困在错误视图中

### 9.5 不再承接的内容

1. 不再提供固定右侧详情面板
2. 若需要表达极少量聚焦信息，只允许放在：
   - 反馈区
   - 列表高亮
   - 行内提示

## 10. 状态下沉边界

### 10.1 保留在主页面入口层

仅保留：

1. widget 参数
2. 生命周期触发
3. 页面骨架装配

### 10.2 下沉到页面状态编排层

建议集中管理以下跨区块联动状态：

1. `_loading`
2. `_message`
3. `_jumpNotice`
4. `_stages`
5. `_processes`
6. `_stageKeyword`
7. `_processKeyword`
8. `_processStageFilter`
9. `_activeView`
10. `_focusedProcessId`
11. `_lastHandledJumpRequestId`

### 10.3 仅留在局部区块中的状态

仅保留局部、不会跨区块联动的临时展示态：

1. 搜索框 controller
2. 面板内部展开/折叠
3. hover / selected 等纯展示态
4. 弹窗表单临时输入值

### 10.4 不纳入本轮的下沉范围

本轮不引入：

1. 独立 controller/service
2. 全局状态管理
3. 事件总线
4. 新的数据访问抽象

原因：本轮目标是页面结构重构，不是状态架构升级。

## 11. 统一骨架接法

### 11.1 统一的部分

本轮应统一：

1. 页面头部
2. 顶部反馈区
3. 视图切换区
4. 主视图区容器语言
5. 主要操作区节奏

### 11.2 保留业务个性的部分

本轮不要求把工艺页改得和普通 CRUD 页面完全一样。应保留：

1. 工艺模块自身业务语义
2. 工序作为主工作区的业务优先级
3. 工段作为辅助入口的操作逻辑

目标是“统一骨架语言”，不是“抹平业务特征”。

## 12. 验收标准

### 12.1 页面主结构

完成后应满足：

1. 右侧“聚焦工序详情”整块被移除
2. 主内容区不再三栏并排
3. 顶部存在明确的视图切换入口：
   - `工序列表`
   - `工段列表`
4. 默认进入页面时直接落在 `工序列表`

### 12.2 交互行为

完成后应满足：

1. 外部 jump 命中工序时，页面自动切到 `工序列表`
2. 目标工序在列表里能被定位或高亮
3. 切到 `工段列表` 时，不丢失页面级操作和反馈信息
4. 从工段相关操作返回后，不会把用户困在错误视图里

### 12.3 风格统一

完成后应满足：

1. 页头仍然接统一骨架
2. 反馈区仍然独立存在
3. 工序视图和工段视图虽然切换显示，但容器语言一致
4. 页面整体比当前更紧凑，不再浪费横向空间

### 12.4 维护性

完成后应满足：

1. 主页面文件明显变薄
2. 工序视图、工段视图、切换区、反馈区、弹窗各自独立
3. 后续优化时只需改单一区块文件，不必回到主页面翻大段代码

### 12.5 保持不变的部分

以下内容在验收时必须保持不变：

1. 工段/工序增删改查行为
2. 接口契约
3. 权限判断
4. jump 语义
5. 主壳和消息中心跳到工序管理页的关键路径

## 13. 风险与控制

### 风险 1：切换式视图导致上下文丢失

表现：

1. 用户频繁来回切换
2. jump 后仍落在错误视图

控制方式：

1. 默认主视图固定为工序
2. jump 命中工序时强制切回工序视图
3. 工段只作为辅助入口

### 风险 2：拆分过度

表现：

1. 文件增多但边界不清
2. 理解成本反而上升

控制方式：

1. 每个文件只保留单一职责
2. 不引入额外全局状态和 controller 层

### 风险 3：重构过程中误伤业务行为

控制方式：

1. 明确“只改结构和骨架，不改业务规则”
2. 后续 implementation plan 中要求对关键交互做回归验证

## 14. 最终预期

本轮完成后，`process_management_page.dart` 将成为一张：

1. 默认工序主视图
2. 工段辅助入口
3. 风格统一
4. 更节省横向空间
5. 能继续长期维护

的紧凑工艺工作台页面。
