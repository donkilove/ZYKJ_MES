# 串口助手第二阶段设计

- 日期：2026-04-24
- 主题：先打通插件宿主承载，再把串口助手第一页做成可用工作台
- 状态：待用户审阅

## 1. 背景

当前 `plugin_host` 宿主已经具备以下基础能力：

1. 能扫描 `plugins/` 目录并识别 `serial_assistant`
2. 主壳左侧已经有“插件中心”入口
3. 插件中心左侧已经能显示“串口助手”
4. 插件中心右侧已经有基础工作区结构

当前 `serial_assistant` 插件已经具备以下最小能力：

1. 插件 `manifest.json`
2. `launcher.py`
3. Python 串口桥接层
4. 本地 HTTP 服务
5. 基础 Web 页面
6. `vendor/pyserial`

但当前用户可见问题仍然很明显：

1. 点击左侧插件项后，右侧仍停留在占位态，而不是直接打开插件
2. 宿主没有统一的启动中 / 异常态工作区
3. 串口助手页面还没有形成清晰、可长期使用的第一屏工作台

本轮的目标不是扩平台，而是把“宿主能打开插件”和“串口助手能完成一次真实收发”做成一个完整闭环。

## 2. 目标与非目标

### 2.1 目标

1. 点击左侧 `串口助手` 后，右侧立即进入插件启动流程
2. 宿主在插件未就绪前显示统一的启动面板
3. 插件就绪后，右侧自动切到真实插件页面
4. 启动失败或超时后，右侧显示统一异常面板
5. 串口助手第一页重构为“三段工作台”：
   - 连接区
   - 发送区
   - 接收日志区
6. 支持一次完整的最小闭环：
   - 刷新端口
   - 打开端口
   - 发送内容
   - 读取一次
   - 关闭端口

### 2.2 非目标

1. 本轮不做十六进制模式
2. 本轮不做多实例
3. 本轮不做多页签
4. 本轮不做脚本宏、命令模板、历史记录
5. 本轮不做高级串口参数面板（数据位、停止位、校验位等）
6. 本轮不做插件安装器、卸载器、市场能力
7. 本轮不扩展除 `serial_assistant` 以外的插件体验

## 3. 方案选择

本轮采用：

**目标采用方案 B：做出完整闭环**

但实施顺序采用：

**节奏按方案 A：先宿主，后插件**

原因：

1. 你当前最强烈的痛点是“插件没有真正打开”
2. 宿主状态流先稳定后，插件页面怎么调都不会再退回“占位卡”
3. 这样既能保证第二阶段最终交付是完整闭环，又能控制实现风险

## 4. 宿主交互设计

### 4.1 左侧插件列表

左侧插件列表不再是纯“选中器”，而是“打开入口”。

规则：

1. 点击左侧 `串口助手` = 立即触发打开流程
2. 不再停留在简介卡或详情占位态
3. 如果插件已在运行，再次点击左侧项，不创建新实例，只把焦点切回当前运行中的工作区

### 4.2 单实例约束

第二阶段采用：

**单插件单实例**

规则：

1. `serial_assistant` 只允许同时存在一个运行实例
2. 不支持多开
3. 不支持多个串口助手页签

这样能避免：

1. 串口占用冲突
2. 状态同步复杂化
3. 宿主生命周期过早膨胀

### 4.3 工作区状态流

右侧工作区只保留 4 种宿主状态：

1. `空态`
2. `启动中`
3. `运行中`
4. `异常`

#### 空态

仅在还未打开任何插件时显示。

#### 启动中

点击左侧 `串口助手` 后立即进入。

展示内容：

1. 标题：`正在启动串口助手`
2. 说明：宿主正在拉起插件进程并等待页面就绪
3. 状态步骤：
   - 已定位插件目录
   - 已准备 Python 解释器
   - 已启动插件进程
   - 正在等待插件返回页面地址
4. 底部动作：
   - `取消启动`
   - `查看日志`

#### 运行中

收到 `ready` 后立即切换。

规则：

1. 宿主不再显示简介卡
2. 宿主不再解释插件内部业务
3. 右侧直接承载插件页面

#### 异常

以下情况进入异常态：

1. 插件启动失败
2. 等待 `ready` 超时
3. 插件进程退出
4. 心跳检查失败

展示内容：

1. 标题：`串口助手启动失败` 或 `串口助手连接中断`
2. 原因摘要
3. 诊断信息：
   - 插件目录
   - 解释器路径
   - 最近错误信息
4. 动作：
   - `重试`
   - `查看日志`
   - `关闭插件`

### 4.4 生命周期动作

#### 关闭插件

规则：

1. 点击“关闭插件” = 结束插件进程
2. 右侧返回空态
3. 左侧状态回到未启动

#### 重启插件

规则：

1. 点击“重启插件” = 先停旧进程
2. 再启动新进程
3. 右侧重新进入启动中
4. `ready` 后再进入运行中

## 5. 串口助手第一页设计

第二阶段第一页采用“三段工作台”。

### 5.1 连接区

放在最上方，优先回答“是否已连接”。

包含：

1. 端口下拉
2. 波特率输入或下拉
3. `刷新端口`
4. `打开`
5. `关闭`
6. 状态文字：
   - 未连接
   - 正在连接
   - 已连接 `<port> @ <baudrate>`
   - 连接失败

设计原则：

1. 状态必须清楚
2. 不把连接状态埋进日志
3. 第一步先服务“快速连接”

### 5.2 发送区

放在中间，服务“快速发送”。

包含：

1. 多行输入框
2. `发送`
3. `清空发送框`

第一版只支持文本发送，不扩十六进制。

### 5.3 接收日志区

放在下方，给足面积。

包含：

1. 大面积日志显示区
2. `读取一次`
3. `清空日志`

日志至少区分三类：

1. 系统状态
2. 发送记录
3. 接收记录

## 6. 文件边界

### 6.1 宿主侧文件

优先修改以下文件：

- `frontend/lib/features/plugin_host/presentation/plugin_host_controller.dart`
  - 左侧点击后直接打开
  - 单实例生命周期管理
  - 启动中 / 运行中 / 异常态切换
- `frontend/lib/features/plugin_host/presentation/plugin_host_page.dart`
  - 保持只做页面装配
- `frontend/lib/features/plugin_host/presentation/widgets/plugin_host_workspace.dart`
  - 去掉简介卡主路径
  - 接入启动面板、异常面板、运行态 WebView
- `frontend/lib/features/plugin_host/presentation/widgets/plugin_host_sidebar.dart`
  - 列表项点击语义从“选中”改为“打开/聚焦”
- `frontend/test/widgets/plugin_host_page_test.dart`
  - 覆盖宿主状态流与交互路径

### 6.2 插件侧文件

优先修改以下文件：

- `plugins/serial_assistant/web/index.html`
  - 重排为三段工作台
- `plugins/serial_assistant/web/app.js`
  - 收敛连接区、发送区、日志区逻辑
- `plugins/serial_assistant/app/server.py`
  - 确保 Web 页面所需接口稳定
- `plugins/serial_assistant/app/serial_bridge.py`
  - 保持最小串口能力稳定
- `plugins/serial_assistant/tests/test_serial_bridge.py`
  - 保持真实回环测试

## 7. 实施顺序

推荐顺序：

1. 先改宿主交互
2. 先补宿主 widget 测试
3. 再整理串口助手 Web 页面
4. 最后补 Python 侧接口细节

这样做的目标是：

1. 尽快消除当前“点击后还是占位态”的体验问题
2. 先让打开链路稳定
3. 再把插件页面打磨成可用工作台

## 8. 测试策略

### 8.1 宿主测试

至少覆盖：

1. 点击左侧插件项后进入 `启动中`
2. 收到 `ready` 后进入 `运行中`
3. 启动失败后进入 `异常`
4. 已运行时再次点击左侧项不会新开实例
5. 关闭后回到空态

### 8.2 Python 测试

至少保留：

1. `loop://` 回环收发测试

### 8.3 静态检查

至少执行：

1. `flutter analyze`

### 8.4 人工验证

至少验证：

1. 打开插件中心
2. 点击左侧 `串口助手`
3. 右侧先出现宿主启动面板
4. 随后切到真实插件页面
5. 完成一次打开、发送、读取、关闭

## 9. 验收标准

以下 5 条同时满足即视为第二阶段完成：

1. 左侧点击 `串口助手` 后，右侧不再停在简介占位态
2. 宿主先显示启动面板，再切到真实插件页面
3. 插件第一页是连接区、发送区、接收日志区
4. 能完成一次真实串口收发闭环
5. 启动失败时能看到统一异常面板

## 10. 风险与取舍

1. 第二阶段仍坚持单实例，意味着暂时不能多开串口助手
2. 第一屏不做高级参数，意味着高级串口场景继续延期
3. 先宿主后插件会让前半段看起来“功能变化不多”，但这是为了尽快稳定打开链路
