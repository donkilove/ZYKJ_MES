# 消息模块复审

## 1. 当前总体判断

- 消息模块已从“基础版可用”提升到“主链路基本可用但未完全闭环”。
- 上一轮重点问题中，页签跳转、批量已读、仅看待处理、详情预览、统计口径一致性都已有明显改善。
- 但仍未达到完整验收，主要缺口在精确上下文跳转、已读状态实时同步前端消费、公告/规则能力与更完整的异常提示语义。

## 2. 已满足项

- 已有独立一级侧边栏“消息”入口，并展示未读角标。
- 后端已提供消息列表、未读数、摘要统计、单条已读、全部已读、批量已读、WebSocket 连接能力。
- 前端消息中心已支持：关键字/状态/分类/优先级/来源模块/时间范围筛选、仅看待处理、刷新、重置、全部已读、批量已读。
- 消息中心在宽屏下已有固定详情预览区，窄屏可弹窗查看详情。
- 列表与统计已统一以 `active` 消息为口径。
- 主壳层已支持“一级页面 + 页签编码”的消息跳转承接。

## 3. 部分满足项

- 跳转已能到模块/页签，但仍不能携带对象级上下文精确落位。
- WS 与轮询共同存在，但实时一致性更多是最终一致而非即时一致。
- 已读状态变化后端会推送事件，但前端未完整消费该事件。
- 详情预览已有，但字段仍不够完整。
- 公告当前更多只是消息类型的一种展示，不等于公告发布/范围控制已实现。

## 4. 未满足项

- 未实现基于 `target_route_payload_json` 的精确业务上下文跳转。
- 未实现公告发布页面、公告范围控制、规则配置页面与权限。
- 未实现来源对象“无权限访问”与“已删除/归档”两类原因的完整细分链路。
- 未见基于 `expires_at` 的完整过期、保留清理与统计联动实现。
- 部分来源消息仍未补齐跳转目标。

## 5. 相比上一轮已修复项

- 已修复“未实现批量标记已读”。
- 已修复“未实现仅看待处理筛选”。
- 已修复“无页面内详情预览区”。
- 已修复“列表与未读统计口径不一致”。
- 已修复质量模块错误页签编码问题。
- 已明显改善“主壳层只支持一级菜单，不支持页签联动”。

## 6. 仍遗留的关键问题

- 跳转未到上下文，只到模块/tab。
- 前端未完整消费 `message_read_state_changed`，多端实时同步仍未闭环。
- 公告/规则能力基本仍为空白。
