# 品质模块需求说明（最终可评审稿）

## 1. 文档信息

- 模块名称：品质模块
- 文档版本：V1.0
- 文档状态：可评审
- 目标：完成每日首件、品质统计、不良与报废分析、维修闭环与质量趋势等核心能力建设，满足 MES 场景下品质结果查询、异常追溯、闭环处理和趋势分析需求。

---

## 2. 本期范围

本模块本期包含以下主页面：

1. 每日首件页面
2. 品质数据页面
3. 报废统计页面
4. 维修订单页面

本模块本期包含以下子页面或增强页面：

1. 首件处置页面（建议新增）
2. 报废详情页面（建议新增）
3. 维修订单详情页面（建议新增）
4. 不良分析 / 质量趋势页面（建议新增）

### 2.1 模块边界说明

- **以上页面均纳入本期规划范围。**
- 品质模块负责首件记录、品质统计、不良分析、报废统计、维修结果和质量趋势等品质业务数据的归口定义与验收。
- 品质模块中涉及订单执行、首件提交、报工、工艺模板、产品主数据、用户组织关系等内容时，默认作为**引用信息或跨模块联动能力**使用，不在本模块内做主数据维护。
- 首件提交动作发生在生产模块执行链路中，品质模块负责首件记录查询、首件结果分析和后续处置。
- 报废统计和维修订单从业务归属上纳入品质模块范围；若系统入口暂时仍保留在生产模块，也应视为品质模块能力的一部分，后续可保留双入口或统一入口。
- 产品主数据来源于产品模块。
- 工段、工序、工艺模板来源于工艺模块。
- 用户、角色、权限来源于用户模块。

### 2.2 本期建议新增页面与功能

在现有系统能力基础上，本需求说明**额外纳入以下新增建议**：

- 新增**首件处置页面**，支持首件不通过后的判定、复检、复判、处置留痕。
- 新增**不良分析 / 质量趋势页面**，支持按产品、工序、人员、时间维度分析不良和质量趋势。
- 新增**报废详情页面**，支持查看报废来源、关联订单、关联维修处理和结果。
- 新增**维修订单详情页面**，支持查看维修原因、回流分配、处理结果和完整追溯。
- 在每日首件页面补充**结果筛选、产品筛选、工序筛选、操作员筛选、导出**能力。
- 在品质数据页面补充**产品维度统计、趋势图、导出**能力。

## 4. 核心对象定义

### 4.1 首件记录

首件记录用于记录某订单、某工序、某操作员在某校验日期下提交的首件检验结果，是品质模块最基础的品质事件数据。

### 4.2 品质统计

品质统计用于对首件结果、不良结果、通过率、人员和工序表现进行汇总分析，是品质管理和复盘的核心数据集。

### 4.3 不良现象

不良现象用于记录生产过程中发现的异常类型、数量和分布情况，是报废统计、维修订单和质量趋势分析的基础输入。

### 4.4 报废统计

报废统计用于汇总异常处理过程中形成的报废数量、报废原因和处理进度，是品质结果域的重要指标对象。

### 4.5 维修订单

维修订单用于承载由不良或异常引发的送修处理，记录维修数量、原因、回流分配、报废结果和完成状态。

### 4.6 回流分配

回流分配用于记录维修完成后，修复成功的数量回流到哪个工序继续生产，是品质闭环与生产状态联动的关键对象。

### 4.7 质量趋势

质量趋势用于在一定时间范围内，按产品、工序、人员等维度观察质量表现变化情况，例如首件通过率、不良数量、报废率、维修占比等。

## 5. 每日首件页面

## 5.1 页面目标

用于按日查看首件检验记录、当日校验码、来源和结果，支持品质人员快速核对首件执行情况，并为后续品质分析提供查询入口。

## 5.2 页面功能

### 5.2.1 查询条件

支持以下筛选条件：

- 查询日期
- 关键字搜索（订单号 / 产品 / 工序 / 操作员）
- 首件结果筛选（通过 / 不通过，建议新增）
- 产品筛选（建议新增）
- 工序筛选（建议新增）
- 操作员筛选（建议新增）

### 5.2.2 顶部操作区

支持以下按钮：

- 查询
- 刷新
- 导出首件记录（建议新增）

### 5.2.3 信息概览区

至少展示以下内容：

- 查询日期
- 当日校验码
- 校验码来源
- 当前查询总数

### 5.2.4 列表字段

| 表头 | 说明 |
| --- | --- |
| 提交时间 | 首件提交时间 |
| 订单号 | 所属订单 |
| 产品 | 产品名称 |
| 工序 | 工序名称 + 工序编码 |
| 操作员 | 提交首件的操作员 |
| 结果 | 通过 / 不通过 |
| 校验日期 | 首件校验日期 |
| 备注 | 首件备注 |
| 操作 | 查看详情 / 进入处置（建议新增） |

### 5.2.5 页面业务规则

- 默认按提交时间倒序展示。
- 支持分页查询。
- 校验码来源需明确展示为“系统记录 / 默认值 / 无”。
- 无首件记录时展示空状态，不报错。
- 若查询日期不是当天且无系统记录校验码，则允许展示为“无”。

| 字段 | 类型 | 必填 | 说明 |

- 完成维修后需更新相关订单和工序状态。
- 完成维修属于关键动作，必须写入事件日志或处理日志。

## 13. 通用业务规则

## 13.1 首件规则

- 首件提交动作由生产模块执行链路触发。
- 品质模块必须能查询所有首件记录。
- 首件记录至少应包含订单、工序、操作员、校验日期、校验码、结果、备注、提交时间。
- 应支持首件不通过后的处置闭环。

## 13.2 统计规则

- 统计口径应在页面内清晰说明。
- 总览、工序、人员、产品、趋势分析应基于一致的数据口径。
- 统计结果和导出结果必须一致。

## 13.3 不良与报废规则

- 不良现象应可追溯到报工记录、工序和操作员。
- 报废统计结果必须与维修完成结果口径一致。
- 已纳入维修处理的报废记录，需能查看关联维修信息。

## 13.4 维修闭环规则

- 维修完成必须满足数量守恒：报废 + 回流 = 送修数量。
- 回流目标工序必须合法。
- 维修完成后应同步更新订单和工序状态。

## 13.5 日志与追溯规则

- 首件处置、报废处理、维修完成等关键动作均需留痕。
- 页面应支持从汇总页跳转到详情页查看完整处理链路。

## 16. 评审确认结论

以下内容建议在本版评审中固化：

1. 品质模块本期以“首件记录 + 品质统计 + 报废统计 + 维修闭环”为核心边界。
2. 报废统计和维修订单虽然可暂时保留生产模块入口，但业务归口和验收归口属于品质模块。
3. 首件提交入口继续保留在生产执行链路中，品质模块主要负责查询、统计、处置和闭环。
4. 首件处置页面、不良分析 / 质量趋势页面、报废详情页面、维修订单详情页面作为建议新增项纳入本期规划。
5. 品质统计页面建议补齐产品维度、趋势分析和导出能力，避免长期停留在基础汇总状态。
6. 品质模块必须支持关键结果的可追溯，不能只停留在列表和总数展示层。
