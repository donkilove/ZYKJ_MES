# 业务缺页实施清单 V2（供 Claude 执行）

- 编写日期：2026-03-14
- 适用范围：当前 MES 项目未上线阶段，优先补齐业务缺页与页面纳管问题
- 输出目标：为后续开发提供一份可直接拆任务、排优先级、按包实施的执行清单
- 实施原则：先纳管已实现能力，再补真实缺页；先补业务闭环，再补分析增强；尽量复用现有接口、权限、页面结构与目录机制

---

## 1. 结论摘要

当前项目里所谓“缺页”分成两类：

1. **页面已存在，但未进入正式页面目录 / 默认可见 / 权限快照体系**。
2. **页面确实未实现，需新增前后端模型、接口、权限和页面。**

从投入产出比看，建议优先处理第一类，因为：

- 开发量更小；
- 风险更低；
- 更容易快速提升系统完成度；
- 能尽快把已有代码价值释放出来。

- 模板/产品维度增强筛选
- 更完整的跳转链路
- 发布前影响提示联动

## 6. 推荐开发顺序

1. A1 产品版本管理页正式纳管
2. A2 工艺引用分析页正式纳管
3. A3 品质模块页签目录对齐
4. A4 并行实例追踪页正式入口化
5. B1 设备规则 / 运行参数页
6. B2 品质不良分析页

推荐理由：

- 前 3 项几乎都是“目录、权限、默认可见、页签对齐”工作，投入最小；
- 第 4 项利用已有页面和接口，属于低风险增强；
- 第 5、6 项才是真正的新增业务页，应放在前面收益释放后再做。

## 7. Claude 执行建议

### 7.1 第一批建议一次性交付

- A1 产品版本管理页正式纳管
- A2 工艺引用分析页正式纳管
- A3 品质模块页签目录对齐

原因：

- 三者都偏“页面目录与权限体系对齐”；
- 容易一次性改顺；
- 对现有代码侵入较小。

### 7.2 第二批建议单独交付

- A4 并行实例追踪页正式入口化

原因：

- 涉及页面模式切换（详情子页 -> 正式页签）；
- 可能需要补后端检索接口；
- 需要较明确的 UI 形态确认。

### 7.3 第三批建议单独设计后开发

- B1 设备规则 / 运行参数页
- B2 品质不良分析页

原因：

- 都是新增业务页；
- 要先统一字段口径、权限边界、接口返回结构；
- 不宜边做边猜。

## 8. 最终建议

如果目标是“尽快把未上线项目的功能完成度做高”，建议采取如下策略：

1. **先释放已有能力**：A1、A2、A3
2. **再补业务闭环显著缺口**：A4
3. **最后做新能力建设**：B1、B2

这样既能快速提高页面完备度，也能避免一开始就陷入高成本新功能开发。
