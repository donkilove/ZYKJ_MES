# 消息模块需求说明（最终可评审稿）

## 1. 文档信息

- 模块名称：消息模块
- 文档版本：V1.0
- 文档状态：可评审
- 目标：完成站内消息中心、未读管理、实时推送、公告通知与业务跳转能力建设，满足 MES 场景下待办提醒、事件通知、公告触达和跨模块消息追踪需求。

---

## 2. 本期范围

本模块本期包含以下主页面：

1. 消息中心页面

本模块本期包含以下子页面或增强页面：

1. 消息详情页（建议新增）
2. 公告发布页面（建议新增）
3. 消息规则 / 订阅配置页面（建议新增）

### 2.1 模块边界说明

- **消息中心页面纳入本期上线范围。**
- 消息模块以独立一级侧边栏“消息”入口进入，视觉风格、布局结构和交互规范需与现有一级模块保持一致。
- 消息模块负责站内消息聚合、收件人状态管理、未读统计、实时提醒和业务跳转，不负责维护来源业务主数据。
- 用户、生产、品质、设备、产品、工艺等模块可作为消息来源；消息模块仅承接消息生成、投递、展示和追溯。
- 本模块本期实时通道采用 **WebSocket**，客户端需提供断线重连能力，并以 HTTP 轮询作为兜底方案。
- 本模块本期默认只做**站内消息**，不包含短信、邮件、企业微信、钉钉等外部渠道投递。
- 本模块本期不替代原业务页面中的审批、处理、编辑能力；消息中心负责提醒、聚合和跳转承接。

### 2.2 本期建议新增页面与功能

在现有系统能力基础上，本需求说明**额外纳入以下新增建议**：

- 新增侧边栏未读角标，便于用户在任意模块下感知新消息。
- 支持按**待处理 / 通知 / 公告**分类展示消息。
- 支持从消息直接跳转至来源业务页面，并保留来源对象编号与摘要信息。
- 支持单条已读、全部已读和批量已读能力。
- 支持面向角色、用户或条件范围发布站内公告。
- 支持高优先级消息置顶或高亮展示。
- 支持消息失效状态和保留期限控制（建议新增）。

## 4. 核心对象定义

### 4.1 站内消息

站内消息是消息模块的核心业务对象，用于承载由系统事件、业务动作或公告发布形成的消息内容，包括标题、摘要、正文、分类、优先级和来源信息。

### 4.2 消息收件记录

消息收件记录用于表示某条消息投递给某个用户后的状态，记录是否已读、已读时间、送达时间、失效状态和客户端处理状态。

### 4.3 消息分类

消息分类用于区分不同业务语义，建议至少支持：待处理提醒、普通通知、系统公告、异常预警、高优先级提醒。

### 4.4 来源对象

来源对象用于表示消息来自哪个业务实体，例如注册申请、代班审批单、生产订单、维修单、保养工单等，以便在消息中展示来源编号并支持跳转追溯。

### 4.5 跳转目标

跳转目标用于表达点击消息后应打开的页面、模块、页签或业务详情上下文，是消息中心与源业务联动的关键对象。

### 4.6 实时推送事件

实时推送事件用于向客户端通知“有新消息”“未读数变化”“消息状态变化”等轻量事件，不直接替代消息详情查询接口。

### 4.7 公告

公告用于表达面向单人、多角色或全员范围发布的系统通知，支持发布时间、生效状态、优先级、失效时间和发布人追溯。

## 5. 核心流程概述

本模块本期需支持以下核心流程：

1. 业务事件触发消息
2. 消息生成与收件人解析
3. 站内消息持久化
4. WebSocket 实时推送提醒
5. 客户端未读数刷新与消息列表拉取
6. 用户查看消息并标记已读
7. 用户跳转来源业务页面
8. 消息失效、归档与保留清理

### 5.1 主链路说明

- 来源模块发生关键事件后，由消息服务生成标准化消息。
- 系统根据角色、用户、业务上下文或规则确定收件人范围。
- 消息和收件记录落库后，服务端向在线客户端推送轻量事件。
- 客户端收到推送后，刷新未读数和消息列表；若实时通道不可用，则按轮询策略兜底。
- 用户查看消息详情、打开来源页面或手动执行“标记已读”后，系统更新该用户对应的已读状态。
- 若来源对象已失效、无权限访问或已删除，消息仍保留历史摘要，但跳转能力需明确提示不可用原因。

### 5.2 兜底与异常规则

- WebSocket 连接断开时，客户端应自动重连；重连失败时使用 HTTP 轮询刷新未读数与列表。
- 消息重复触发时需具备去重或幂等机制，避免同一用户收到明显重复提醒。
- 推送失败不应影响来源业务主流程提交；失败消息需保留投递日志或状态，便于后续排查。

## 6. 消息中心页面

## 6.1 页面目标

用于集中展示用户可见的待办消息、业务通知和系统公告，并提供未读管理、消息筛选、详情查看和来源业务跳转能力。

## 6.2 页面组成

建议由以下 4 个区域组成：

1. 顶部概览区
2. 查询筛选区
3. 消息列表区
4. 消息详情预览区（建议补充）

### 6.2.1 查询条件

支持以下筛选条件：

- 关键字搜索（标题 / 摘要 / 来源编号）
- 状态筛选（全部 / 未读 / 已读）
- 分类筛选（待处理 / 通知 / 公告 / 预警）
- 优先级筛选（普通 / 重要 / 紧急）
- 来源模块筛选（用户 / 生产 / 品质 / 设备 / 产品 / 工艺）
- 时间范围筛选（推送时间）
- 仅看待处理（建议新增）

### 6.2.2 顶部操作区

支持以下按钮或操作：

- 刷新
- 全部标记已读
- 重置筛选
- 批量标记已读（建议新增）
- 发布公告（建议新增，仅有权限用户可见）

### 6.2.3 信息概览区

至少展示以下内容：

- 全部消息数
- 未读消息数
- 待处理消息数
- 高优先级消息数

### 6.2.4 列表字段

| 表头 | 说明 |
| --- | --- |
| 状态 | 未读 / 已读 / 已失效 |
| 标题 | 消息标题 |
| 分类 | 待处理 / 通知 / 公告 / 预警 |
| 优先级 | 普通 / 重要 / 紧急 |
| 来源模块 | 用户 / 生产 / 品质 / 设备 / 产品 / 工艺 |
| 来源对象 | 业务对象编号或名称 |
| 摘要 | 消息摘要信息 |
| 推送时间 | 消息生成或送达时间 |
| 已读时间 | 当前用户已读时间 |
| 操作 | 查看详情 / 标记已读 / 跳转业务 |

### 6.2.5 详情预览字段

建议展示以下内容：

- 标题
- 正文
- 来源模块
- 来源对象编号
- 触发时间
- 推送时间
- 当前状态
- 跳转入口
- 处理建议或附加说明

### 6.2.6 页面业务规则

- 默认按推送时间倒序展示最新消息。
- 未读消息需在列表中高亮展示，并参与侧边栏角标统计。
- 用户查看详情或执行跳转后，系统可自动将该消息标记为已读；同时保留手动标记入口。
- “全部标记已读”仅影响当前用户可见消息，不影响其他收件人状态。
- 高优先级消息可置顶或高亮，但不得破坏同优先级下的时间顺序。
- 若当前用户已无来源页面访问权限，则消息仍可查看摘要，但跳转按钮需禁用并明确提示原因。
- 若来源对象已被删除或归档，消息需保留历史记录，并提示“来源对象不可访问”。

### 6.2.7 实时提醒规则

- 用户登录成功后，客户端应建立消息实时连接。
- 收到新消息事件后，应更新侧边栏未读角标和当前列表数据。
- 若用户当前不在消息页面，也应能看到侧边栏未读数实时变化。
- 应用从后台恢复或页面重新激活时，应主动重连并补拉未读数。
- 实时通道不可用时，应降级为固定周期轮询，避免未读状态长期不一致。

### 6.2.8 侧边栏入口规则

- 消息模块需作为独立一级侧边栏项展示，入口名称为“消息”。
- 入口图标、选中态、悬停态、间距和排版风格需与其他一级模块保持一致。
- 若存在未读消息，应在侧边栏入口上展示未读角标；无未读时不强制显示角标。

## 7. 消息详情页（建议新增）

## 7.1 页面目标

用于查看单条消息的完整内容、来源上下文、处理说明和跳转入口，增强消息追溯与操作承接能力。

## 7.2 页面内容

至少展示以下内容：

- 消息标题
- 消息正文
- 消息分类与优先级
- 来源模块与来源对象编号
- 推送时间 / 已读时间
- 当前收件状态
- 跳转业务按钮
- 关闭或返回按钮

## 7.3 页面业务规则

- 打开详情页后，默认将当前用户该条消息标记为已读。
- 若消息存在来源业务页，则应支持跳转；若无来源页，则仅保留历史展示。
- 详情页不承担源业务处理逻辑，仅承接查看与跳转。

## 8. 公告发布与消息规则

## 8.1 公告发布页面（建议新增）

用于面向指定角色、指定用户或全员范围发布站内公告。

### 8.1.1 建议表单字段

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| 公告标题 | 输入框 | 是 | 标题简洁明确 |
| 公告正文 | 富文本或多行输入 | 是 | 公告完整内容 |
| 公告范围 | 单选 / 多选 | 是 | 全员 / 指定角色 / 指定用户 |
| 优先级 | 下拉选择 | 是 | 普通 / 重要 / 紧急 |
| 生效时间 | 日期时间 | 否 | 默认立即生效 |
| 失效时间 | 日期时间 | 否 | 到期自动失效 |
| 状态 | 单选 | 否 | 默认启用 |

### 8.1.2 业务规则

- 公告发布后需自动生成对应收件记录。
- 已失效公告默认不再进入未读统计，但历史记录仍可追溯。
- 无发布权限用户不得看到“发布公告”入口。

## 8.2 消息规则 / 订阅配置页面（建议新增）

用于维护业务事件到消息的映射规则、收件范围和保留策略。

### 8.2.1 建议配置项

- 事件编码
- 来源模块
- 是否启用
- 收件人范围规则
- 优先级默认值
- 保留天数
- 是否实时推送

## 9. 权限与角色规则

本模块本期建议至少包含以下权限能力：

- 页面访问：消息模块入口 / 消息中心页面
- 查看本人消息
- 标记单条已读
- 全部标记已读
- 查看消息详情
- 跳转来源业务
- 发布公告（建议新增）
- 停用 / 撤回公告（建议新增）
- 配置消息规则（建议新增）

### 9.1 特殊规则

- 普通用户默认仅能查看投递给自己的消息。
- 高权限角色可根据授权查看更广范围的公告或管理型消息，但不应默认查看所有个人消息。
- 发布公告与配置规则权限应独立控制，不应默认赋予全部管理员角色。

---

## 10. 跨模块联动范围

本模块本期建议优先接入以下消息来源：

1. 用户模块：注册审批待处理、账号停用、强制下线提醒
2. 生产模块：代班审批待处理、关键订单事件、异常维修流转
3. 品质模块：首件不通过、维修结果回传、报废处理结果
4. 设备模块：保养到期、工单逾期、自动生成工单结果
5. 产品 / 工艺模块：版本发布、模板发布、关键变更通知（建议新增）

### 10.1 联动规则

- 消息应保留来源模块标识、来源对象 ID 和可跳转目标。
- 来源模块提交失败时，不应伪造消息成功状态。
- 来源业务撤销、归档或删除时，应同步调整消息跳转提示或失效状态。

---

## 11. 非功能要求

- 消息模块页面需同时适配桌面端和移动端常见尺寸。
- 实时推送应具备断线重连、重复事件抑制和弱网兜底能力。
- 消息列表与未读数需具备一致性，不允许长期出现角标与列表数据明显不符。
- 所有消息操作需保留基本审计痕迹，便于排查误推送、漏推送和权限异常。
- 消息模块前端风格需与系统现有模块保持统一，不得使用完全割裂的独立视觉语言。

---

## 12. 本期不做

- 短信、邮件、企业微信、钉钉等外部渠道投递
- 复杂可视化规则编排器
- 消息撤回后客户端强制删除历史内容
- 离线系统级推送（移动端原生通知）
- 以消息中心替代原业务页面中的真实审批或处理逻辑

---

## 13. 验收重点

建议重点关注以下验收项：

1. 消息模块是否以独立侧边栏一级入口呈现，且风格与其他模块一致。
2. 未读角标、消息列表、已读状态是否保持一致。
3. WebSocket 实时推送、断线重连和轮询兜底是否有效。
4. 消息跳转是否正确承接到来源业务页面，并在无权限或对象失效时给出明确提示。
5. 公告、待办和普通通知是否在展示和权限上得到清晰区分。
