# 消息模块开发计划（供 Claude 执行）

- 编写日期：2026-03-14
- 对照文档：`docs/功能规划V1/消息模块/消息模块需求说明.md`
- 现状检查：`docs/功能规划V1/消息模块/检查结果-20260314.md`
- 目标：在现有仓库基础上，落地“独立一级侧边栏消息模块 + 站内消息中心 + 未读管理 + WebSocket 实时提醒 + HTTP 轮询兜底”的第一版可用能力
- 实施原则：最小可用、分批落地、先骨架再实时、先个人收件箱再公告扩展、尽量复用现有权限/页面目录/日志结构

---

## 1. 开发目标与交付边界

### 1.1 第一版必须交付

- 独立一级侧边栏入口：`message`
- 消息中心页面：列表、筛选、未读/已读、单条已读、全部已读
- 全局未读角标：侧边栏“消息”入口展示未读数
- 消息主表 + 消息收件记录表
- HTTP API：列表、未读数、单条已读、全部已读
- WebSocket 实时提醒：仅推送轻量事件，不直接推送完整消息详情
- HTTP 轮询兜底：WebSocket 断开或失败时，仍能刷新未读数
- 首批消息来源接入：注册审批待处理、代班审批待处理、关键生产事件、设备保养到期/自动生成结果
- 基础权限接入：模块入口、页面访问、查看消息、标记已读

### 1.2 第一版明确不做

- 短信、邮件、企业微信、钉钉等外部渠道
- 富文本公告编辑器
- 复杂规则编排器
- 消息撤回后的客户端历史清理
- 移动端系统通知推送
- 非必要的大范围前端壳层重构

### 1.3 建议第二版再做

- 公告发布与停用管理页面
- 批量已读更复杂操作
- 精确跳转到业务详情内部子状态
- 消息模板、规则配置、保留期限可视化配置
- 多角色/多范围高级投递规则

---

## 2. 实施策略

### 2.1 总体策略

- 先打通“数据结构 -> 接口 -> 页面 -> 角标 -> 实时提醒”主链路，再补高级来源与公告能力。
- WebSocket 只负责通知客户端“有变化”，消息详情仍走 HTTP 拉取，避免权限泄漏与断线补偿复杂化。
- 前端全局状态尽量挂在 `mes_client/lib/pages/main_shell_page.dart`，这样无论用户在哪个模块，侧边栏未读角标都能同步更新。
- 权限体系严格复用现有 page catalog + authz snapshot 机制，不另做一套消息入口控制逻辑。

### 2.2 推荐分批

- A 批：权限与页面骨架 + 数据模型 + 基础 API
- B 批：前端页面 + 全局角标 + HTTP 轮询兜底
- C 批：WebSocket 实时提醒
- D 批：首批消息来源接入与联调
- E 批：公告能力与第二阶段增强（可选，建议后置）

## 3. 目标架构

### 3.1 数据模型建议

建议至少新增两张表：

1. `message`
2. `message_recipient`

#### 3.1.1 `message` 建议字段

- `id`
- `message_type`：`todo / notice / announcement / warning`
- `priority`：`normal / important / urgent`
- `title`
- `summary`
- `content`
- `source_module`
- `source_type`
- `source_id`
- `source_code`
- `target_page_code`
- `target_tab_code`
- `target_route_payload_json`
- `dedupe_key`
- `status`：`active / expired / archived`
- `published_at`
- `expires_at`
- `created_by_user_id`
- `created_at`
- `updated_at`

#### 3.1.2 `message_recipient` 建议字段

- `id`
- `message_id`
- `recipient_user_id`
- `delivery_status`：`pending / delivered / failed`
- `delivered_at`
- `is_read`
- `read_at`
- `is_deleted`
- `last_push_at`
- `created_at`
- `updated_at`

#### 3.1.3 关键索引建议

- `message(message_type, priority, published_at desc)`
- `message(source_module, source_type, source_id)`
- `message(dedupe_key)` 唯一索引或受控唯一索引
- `message_recipient(recipient_user_id, is_read, created_at desc)`
- `message_recipient(message_id, recipient_user_id)` 唯一索引

### 3.2 WebSocket 事件建议

建议仅推送以下轻量事件：

- `message_created`
- `message_updated`
- `message_read`
- `unread_count_changed`

推荐 payload：

```json
{
  "event": "unread_count_changed",
  "user_id": 12,
  "unread_count": 5,
  "latest_message_id": 103,
  "occurred_at": "2026-03-14T10:20:30Z"
}
```

### 3.3 前端状态建议

- 全局维护：`unreadCount`、`socketConnected`、`lastRefreshAt`
- 消息页维护：筛选条件、列表、分页、加载状态、错误状态
- 当收到 WebSocket 事件时：
  - 更新 `unreadCount`
  - 若当前在消息页，则静默刷新列表第一页
  - 若当前不在消息页，仅更新角标即可

## 4. 分批开发任务

## 4.1 A 批：模块骨架、权限、模型、API（第一优先）

### 4.1.1 后端页面目录与权限任务

需要修改的文件：

- `backend/app/core/page_catalog.py`
- `backend/app/core/authz_catalog.py`
- `backend/app/core/authz_hierarchy_catalog.py`
- `backend/app/services/authz_snapshot_service.py`

具体任务：

- 新增一级页面 `message`
- 在页面目录中设置 `page_type = sidebar`
- 设置合理 `sort_order`，建议放在 `user` 后、`product` 前，或根据产品预期调整
- 新增模块定义：`message`
- 新增消息模块入口权限：`module.message.access`
- 新增页面访问权限：`page.message.view`
- 新增能力权限，建议至少包括：
  - `feature.message.center.view`
  - `feature.message.read.manage`
  - `feature.message.announcement.manage`（若第一版不做公告，可先预留）
- 新增动作权限，建议至少包括：
  - `message.messages.list`
  - `message.messages.unread_count`
  - `message.messages.read`
  - `message.messages.read_all`

### 4.1.2 后端模型与迁移任务

需要新增的文件：

- `backend/app/models/message.py`
- `backend/app/models/message_recipient.py`
- `backend/alembic/versions/<new_revision>_add_message_tables.py`

具体任务：

- 定义消息主表和收件记录表
- 建立外键关系
- 增加必要索引
- 迁移脚本中明确创建顺序和回滚顺序
- 如仓库有模型导出汇总文件，记得同步注册

### 4.1.3 后端 Schema / Service / API 任务

需要新增的文件：

- `backend/app/schemas/message.py`
- `backend/app/services/message_service.py`
- `backend/app/api/v1/endpoints/messages.py`

需要修改的文件：

- `backend/app/api/v1/api.py`

建议接口：

- `GET /messages`
- `GET /messages/unread-count`
- `POST /messages/{message_id}/read`
- `POST /messages/read-all`

建议查询参数：

- `page`
- `page_size`
- `keyword`
- `status`
- `message_type`
- `priority`
- `source_module`
- `start_time`
- `end_time`

服务层任务：

- 消息列表查询
- 未读数统计
- 单条已读
- 全部已读
- 将数据库实体转换为前端所需 DTO
- 为后续来源模块接入预留创建消息方法，如：
  - `create_message(...)`
  - `create_message_for_users(...)`
  - `create_message_from_event(...)`

### 4.1.4 A 批验收门槛

- 权限目录可查询到 `message` 模块
- 页面目录接口返回 `message` 一级入口
- 消息表与收件表迁移成功
- 列表、未读数、单条已读、全部已读接口可用
- 无消息时返回空列表和 `0` 未读数，不报错

## 4.2 B 批：前端页面、角标、轮询兜底

### 4.2.1 前端目录与页面任务

需要新增的文件：

- `mes_client/lib/models/message_models.dart`
- `mes_client/lib/services/message_service.dart`
- `mes_client/lib/pages/message_center_page.dart`

需要修改的文件：

- `mes_client/lib/pages/main_shell_page.dart`
- `mes_client/lib/models/page_catalog_models.dart`
- `mes_client/lib/pages/function_permission_config_page.dart`
- `mes_client/lib/models/authz_models.dart`

具体任务：

- 在前端兜底目录中新增 `message`
- 在功能权限配置模块名称映射中新增 `message: 消息管理` 或 `消息中心`
- 在 `main_shell_page.dart` 中：
  - 新增 `_messagePageCode`
  - 配置图标，如 `Icons.notifications_rounded`
  - 在 `_buildContent` 中挂载 `MessageCenterPage`
  - 增加全局未读数状态
  - 在侧边栏入口展示 badge
- 新建 `MessageCenterPage`，布局风格对齐现有页面，如 `audit_log_page.dart`

### 4.2.2 页面功能任务

消息中心第一版建议实现：

- 列表查询
- 关键词筛选
- 状态筛选（全部 / 未读 / 已读）
- 分类筛选
- 优先级筛选
- 刷新
- 单条标记已读
- 全部标记已读
- 空状态、加载态、错误态

UI 风格要求：

- 与现有模块保持一致
- 不单独做新设计系统
- 顶部条件区 + 列表区风格参考现有管理页
- 侧边栏项风格必须与其他一级模块完全一致，仅新增未读角标

### 4.2.3 轮询兜底任务

在 `main_shell_page.dart` 中增加消息轮询逻辑：

- 默认 20~30 秒拉取一次未读数
- 应用从后台恢复时主动刷新一次
- 登录后初始化刷新一次
- 如果当前在消息页，可视情况同步刷新第一页列表

### 4.2.4 B 批验收门槛

- 侧边栏出现独立“消息”一级入口
- 角标能显示后端返回未读数
- 消息页可以完成查询、已读、全部已读
- 没有 WebSocket 时，未读数仍可通过轮询更新

## 4.3 C 批：WebSocket 实时提醒

### 4.3.1 后端任务

建议新增：

- `backend/app/api/v1/endpoints/message_ws.py` 或直接挂在 `messages.py`
- `backend/app/services/message_push_service.py`
- `backend/app/services/message_connection_manager.py`

建议能力：

- 建立连接：基于当前登录态校验用户身份
- 保存在线连接：按 `user_id` 维护连接池
- 推送轻量事件给指定用户
- 断开连接时清理连接池

### 4.3.2 前端任务

需要修改：

- `mes_client/pubspec.yaml`
- `mes_client/lib/pages/main_shell_page.dart`

建议新增：

- `mes_client/lib/services/message_ws_service.dart`

具体任务：

- 引入 WebSocket 客户端依赖
- 在主壳层建立连接
- 处理连接成功、断开、异常、重连
- 收到 `unread_count_changed` 后刷新角标
- 收到 `message_created` 后，若当前在消息页则静默刷新列表

### 4.3.3 连接策略建议

- 首次进入主壳层时连接
- 断线后指数退避重连
- 应用恢复前台时主动重连
- 用户登出时主动关闭连接

### 4.3.4 C 批验收门槛

- 新消息创建后，目标用户无需手动刷新即可看到未读数变化
- 断线后可自动恢复连接
- WebSocket 不可用时，轮询仍可工作

## 4.4 D 批：消息来源接入

### 4.4.1 首批接入清单

#### 用户模块

- 注册审批待处理
- 账号被停用提醒（可选）
- 强制下线提醒（可选）

#### 生产模块

- 代班审批待处理
- 关键订单事件：如手工完工、异常送修、维修完成、关键状态变化

#### 设备模块

- 保养到期提醒
- 自动生成工单结果
- 工单逾期提醒（可选）

### 4.4.2 接入方式建议

- 不要在各业务模块里直接拼装前端显示文案到处散落。
- 建议统一通过 `message_service.py` 暴露创建方法。
- 各业务模块只传：来源对象、消息类型、收件人、标题、摘要、跳转目标。

### 4.4.3 去重策略建议

- 使用 `dedupe_key` 避免同一业务事件重复创建相同消息
- 建议格式：`<source_module>:<source_type>:<source_id>:<event_code>:<recipient_user_id>`

### 4.4.4 D 批验收门槛

- 注册审批能触发管理员消息
- 代班审批待处理能触发对应审批人消息
- 至少 1 类生产事件能进入消息中心
- 至少 1 类设备事件能进入消息中心

## 7. 开发中的关键约束

### 7.1 不要做的事

- 不要一开始就把公告、规则配置、消息模板全部混在同一批完成
- 不要直接把生产事件日志表当作消息主表使用
- 不要在 WebSocket 中直接推送完整消息正文和敏感业务详情
- 不要绕开现有权限快照机制做前端硬编码入口显示
- 不要在没有 `dedupe_key` 设计的情况下批量接入多消息来源

### 7.2 必须保持的约束

- 侧边栏风格必须与现有一级模块一致
- 所有入口显示必须受权限控制
- 已读状态必须按“用户维度”记录，不能做全局消息级已读
- WebSocket 必须有轮询兜底
- 新增文本文件必须保持 UTF-8 无 BOM、LF 行尾

## 8. 测试与验证计划

### 8.1 后端建议测试

- 消息列表查询测试
- 未读数统计测试
- 单条已读测试
- 全部已读测试
- 同一消息重复创建去重测试
- 权限不足访问消息接口返回 403 测试
- WebSocket 鉴权与事件推送测试（如仓库已有测试基础，可补）

### 8.2 前端建议测试

- 消息模型解析测试
- 消息服务接口测试
- 未读数更新测试
- 角标显示逻辑测试
- 消息页状态筛选测试
- WebSocket 事件触发刷新测试

### 8.3 手工验收场景

- 用户登录后看到“消息”一级入口
- 无消息时页面展示空状态
- 创建一条测试消息后，侧边栏未读数增加
- 打开消息页后能看到最新消息
- 标记单条已读后，列表状态和角标同步变化
- 全部已读后，角标归零
- WebSocket 断开后，轮询仍可刷新未读数
- 无权限用户看不到消息入口或无法访问消息页面

### 8.4 仓库强制验证

完成代码或文档变更后，必须执行并报告：

- `python backend/scripts/check_chinese_mojibake.py`
