# 产品模块需求实现检查结果

- 检查日期：2026-03-14（第六次更新）
- 对照文档：`docs/功能规划V1/产品模块/产品模块需求说明.md`（V1.0）
- 检查范围：后端（FastAPI）+ 前端（Flutter）+ 关键测试
- 检查方式：需求条款逐条对照 + 代码与测试复核

---

## 总体结论

产品模块的主流程能力已经可用（产品创建/编辑、版本新建与复制、版本生效、参数维护、参数历史、版本对比与回滚），但仍存在多项与需求文档不一致的 P0 级缺口。

**当前状态判断**：可用于内部联调，不建议按《需求说明 V1.0》直接验收通过。

主要风险集中在：删除语义、查询口径、导出能力、参数与版本语义一致性、Link 格式校验、审计日志完备性。

---

## 一、后端检查

### 1.1 数据模型与核心语义

| 需求项 | 实现情况 | 说明 |
|---|---|---|
| 产品名称全局唯一 | ✅ 已实现 | `mes_product.name` 唯一约束 + 服务层校验 |
| 产品分类固定枚举（贴片/DTU/套件） | ⚠️ 部分实现 | 有字段，但后端无强枚举约束 |
| 产品状态（启用/停用） | ✅ 已实现 | `active/inactive` |
| 创建后自动生成 `V1.0` 草稿 | ✅ 已实现 | 新建后自动创建初始草稿版本 |
| 同一产品同一时刻仅一个草稿版本 | ✅ 已实现 | 新建版本/复制版本均有拦截 |
| 同一产品同一时刻仅一个生效版本 | ✅ 已实现 | 生效时会替换旧生效版本 |
| 参数与版本绑定维护 | ⚠️ 部分实现 | 当前参数表按 `product_id` 存储，版本通过 `snapshot_json` 快照承载，语义可追溯但非“版本维度实体化参数” |
| Link 类型格式校验（http/https/UNC/C:\） | ❌ 缺失 | 后端仅校验类型，不校验格式 |
| 产品删除为逻辑删除 | ❌ 缺失 | 当前为物理删除（`db.delete(product)`） |

## 三、与需求不一致的关键问题（按优先级）

### 3.1 P0（必须补齐）

1. 产品删除语义不符：应逻辑删除，当前为物理删除。
2. 导出能力缺失：产品列表导出、版本参数导出、参数查询导出均未实现。
3. 产品详情能力缺失：产品管理页无“查看详情”与详情抽屉。
4. 参数编辑缺“参数说明”写入链路（前后端模型与接口未贯通）。
5. Link 类型参数缺后端格式校验，存在脏数据写入风险。
6. 产品管理筛选不完整：缺“生效版本状态”“更新时间范围”。

### 3.2 P0-中/低（建议本期补齐）

1. 列表默认排序应改为“更新时间倒序”。
2. 参数管理页缺“参数分组筛选”“最后修改时间范围筛选”。
3. 参数历史缺“变更类型”字段。
4. 产品关键动作审计日志建议补齐统一记录。

### 3.3 待评审确认

1. 参数按版本“快照承载”是否可作为“绑定版本”的可接受实现。
2. 产品分类与参数分组是否要求后端强枚举约束（而非仅前端约束）。
