# 项目规则与指挥官模式统一手册

## 1. 文档定位与优先级

- 本手册是“项目规则 + 指挥官模式”的统一阅读入口，用于减少规则分散带来的理解成本。
- 本手册用于聚合与导读，不替代原始规范、模板与工具清单。
- 权威优先级（高到低）：
  1. `AGENTS.md`（全局强制规则）
  2. `docs/commander/指挥官工作流程.md`（流程与角色闭环）
  3. `docs/commander/commander_tooling_governance.md`（工具触发、门禁、降级细则）
  4. `docs/commander/指挥官任务日志模板.md` 与 `docs/commander/指挥官工具化验证模板.md`（留痕模板）
  5. `docs/opencode_tooling_bundle.md` 与 `docs/host_tooling_bundle.md`（工具索引）
- 冲突处理：若本手册与 `AGENTS.md` 存在冲突，一律以 `AGENTS.md` 为准。

## 2. 快速上手（执行顺序）

1. 先读 `AGENTS.md`，确认硬性约束、语言要求、测试基线与交付口径。
2. 按 `docs/commander/指挥官工作流程.md` 拆分原子任务，明确执行子 agent 与验证子 agent。
3. 参考 `docs/commander/commander_tooling_governance.md` 选择分类、触发工具与验证门禁。
4. 记录到 `docs/commander/指挥官任务日志模板.md`，命中工具化场景时补 `docs/commander/指挥官工具化验证模板.md`。
5. 交付时统一说明完成项、验证结果、残余风险、迁移口径（若无迁移写“无迁移，直接替换”）。

## 3. 强制规则总览

- 全程中文沟通、中文注释、中文文档；新文件 UTF-8（无 BOM）。
- 编码前必须做 Sequential Thinking；若工具不可用，需记录降级原因、触发时间、替代动作与风险。
- 指挥官模式下主 agent 只做拆解、派发、汇总和判定，不直接承担业务实现与最终验证。
- 每个原子任务至少“执行子 agent -> 独立验证子 agent”闭环；失败必须重派修复并复检。
- 子 agent 用完即关；完成当前原子任务、回执被接收或被判废后，不保留空闲子 agent 常驻。
- 若子 agent 回执出现越位口吻、冒用主 agent 话术或直接面向用户交付，主 agent 必须直接丢弃或重派，不得原样暴露给用户。
- 默认测试基线：后端 `pytest`；前端 `flutter test` + `integration_test`。
- `desktop_tests/flaui/` 仅历史保留/按需 fallback，且必须串行执行。
- 工具不可用时不得中断等待，需即时降级并留痕补偿。

## 4. 指挥官角色与闭环

- 主 agent：拆解目标、定义验收、调度子 agent、维护任务队列、汇总结论。
- 调研子 agent：补齐上下文、依赖关系、风险与范围边界。
- 执行子 agent：按边界实现改动并提交自测结果。
- 验证子 agent：独立复核并给出通过/不通过结论。
- 标准闭环：
  1. 启动：明确目标/范围/非目标/验收标准。
  2. 循环：调研（按需）-> 执行 -> 独立验证 -> 失败重派。
  3. 收尾：统一交付变更、验证、阻塞、风险与迁移说明。

## 5. 任务分类与默认工具

- CAT-01 后端模型/接口/迁移：默认 `postgres MCP` + `Bruno` + `openapi-validate`。
- CAT-02 前后端契约同步：默认 `openapi-validate`，页面联动补 `Playwright`/`flutter-ui`。
- CAT-03 Flutter 页面改造：默认 `flutter test` + `integration_test`，按需 `flutter-ui`。
- CAT-04 权限/可见性联动：默认覆盖“接口授权 + 页面显隐”双口径。
- CAT-05 本地联调与启动：默认 `http-probe`，按需 `mitmproxy/Fiddler` 与 `postgres MCP`。
- CAT-06 中文与编码一致性：需至少一条真实显示验证。
- CAT-07 接口复现与抓包：需保留请求样本与链路证据。
- CAT-08 发布前审计：默认 `Syft` + `Trivy`，协作项补 `gh`。

说明：

- `openapi-validate`、`http-probe`、`flutter-ui` 在项目中是 `tools/project_toolkit.py` 暴露的包装命令（非独立 MCP）。
- `FlaUInspect` 已弃用，仅历史保留，不是默认工具链。

## 6. 验证门禁与失败重试

- 进入“通过”前至少满足：
  1. 已完成任务分类与触发依据记录。
  2. 执行与验证角色分离。
  3. 关键验证真实执行（不是只列建议命令）。
  4. evidence 能串起“触发 -> 执行 -> 验证 -> 重试 -> 收口”。
  5. 迁移口径明确（或声明无迁移）。
- 验证不通过处理：
  1. 记录失败现象与根因判断。
  2. 重新派发执行子 agent 修复。
  3. 由新的验证子 agent 复检。
  4. 重复直至通过或进入硬阻塞收口。

## 7. 留痕与 evidence 要求

- 指挥官任务必须在开始阶段建立/更新 `evidence/` 任务日志。
- 至少留痕：任务拆分、子 agent 输出摘要、验证命令、失败重试、阻塞与最终结论。
- 证据记录至少包含：证据编号、来源、适用结论、形成时间、责任归属。
- 只读调研若无法直接写 evidence，主 agent 可代记并注明代记责任与时间。
- 推荐模板：
  - `docs/commander/指挥官任务日志模板.md`
  - `docs/commander/指挥官工具化验证模板.md`

## 8. 工具降级与硬阻塞

- 常见降级：`Sequential Thinking`、`Task`、`Serena`、`Context7`、`postgres MCP`、`openapi-validate`、`Playwright`、`flutter-ui`、`gh` 等不可用。
- 降级最低要求：写明不可用原因、影响范围、替代工具/流程、补偿措施、残余风险。
- 硬阻塞（如权限/凭证缺失、外部服务不可达）时：
  1. 先完成可推进项。
  2. 最终统一汇总阻塞项、影响与建议动作。
  3. 不采用“无限等待用户回复”策略。

## 9. 交付与迁移口径

- 交付内容建议包含：完成项、关键改动、验证结果、重试收口、残余风险、阻塞说明。
- 默认策略为结果导向与可审计；不交付占位方案。
- 迁移口径必须显式声明：
  - 有迁移：给出具体迁移与回滚步骤。
  - 无迁移：写“无迁移，直接替换”。
