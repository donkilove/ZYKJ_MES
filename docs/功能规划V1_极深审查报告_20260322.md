# 功能规划 V1 需求极深审查报告（2026-03-22）

## 1. 任务说明

- 审查目标：严格对照 `docs/功能规划V1` 下 7 个模块需求说明，极深检查 `backend/` 与 `frontend/` 的实现完整性、契约一致性、权限闭环、测试覆盖与功能可交付性。
- 执行方式：
  - 7 个模块调研子 agent 独立静态审查
  - 7 个模块独立复查子 agent 反向验证
  - 1 个系统级验证子 agent 执行后端集成测试、前端静态检查与前端测试
- 需求基线：
  - `docs/功能规划V1/用户模块/用户模块需求说明.md`
  - `docs/功能规划V1/产品模块/产品模块需求说明.md`
  - `docs/功能规划V1/工艺模块/工艺模块需求说明.md`
  - `docs/功能规划V1/设备模块/设备模块需求说明.md`
  - `docs/功能规划V1/品质模块/品质模块需求说明.md`
  - `docs/功能规划V1/生产模块/生产模块需求说明.md`
  - `docs/功能规划V1/消息模块/消息模块需求说明.md`
- 限制说明：本轮以仓库当前代码、现有测试与需求文档为准；未进行人工页面联调，不把历史审计报告直接视为本轮结论。

## 2. 系统级验证结果

- 后端：在仓库虚拟环境 `.venv\Scripts\python.exe` 下执行 7 个模块关键集成测试，结果 `Ran 37 tests ... OK`。
- 前端：在 `frontend/` 下执行 `flutter analyze lib test`，结果 `No issues found!`。
- 前端：在 `frontend/` 下执行 `flutter test`，结果 `All tests passed!`，共通过 142 项。
- 环境限制：系统默认 `python` 缺少 `fastapi`、`sqlalchemy` 依赖，默认解释器无法直接运行后端测试；在仓库推荐环境下测试通过。
- 审计解释：测试通过仅证明“当前代码可运行且现有回归通过”，**不能等同于“严格满足需求说明”**。

## 3. 总体结论

- 最终结论：**本轮需求极深审查不通过。**
- 结论依据：7 个模块经“调研子 agent + 独立复查子 agent”双闭环后，均存在不同程度的结构性缺口，且多个模块被复查进一步下调为“明显偏差”或“高风险未通过”。
- 总体判断：
  - 代码基础与测试基础明显优于早期状态；
  - 但需求对照层面仍存在大量“契约被改写、前后端闭环不完整、权限粒度未落地、统计口径偏移、消息/跳转/审计链路未收口”的问题；
  - 因此当前不能向业务方宣称“已完整满足 docs 需求说明”。

## 5. 模块深审要点

### 5.1 用户模块

- 首次登录强制改密后，后端已注销当前会话，但前端仍保留旧 token 并直接进入主壳，不符合“立即失效并重新登录”要求。
  - 代表证据：`frontend/lib/pages/force_change_password_page.dart`、`frontend/lib/main.dart`、`backend/app/api/v1/endpoints/me.py`
- 用户导出已在后端定义独立权限，但前端按钮未按该能力码做显隐或可用控制，只能依赖后端 403。
  - 代表证据：`frontend/lib/pages/user_management_page.dart`、`backend/app/api/v1/endpoints/users.py`
- 登录日志/在线会话已在后端拆成细粒度权限，但前端固定同时加载两类接口且固定展示两个 Tab。
  - 代表证据：`frontend/lib/pages/login_session_page.dart`、`backend/app/api/v1/endpoints/sessions.py`
- 复查新增问题：编辑用户弹窗可直接修改密码，与需求中“重置密码”为独立流程的定义冲突。
  - 代表证据：`frontend/lib/pages/user_management_page.dart`、`backend/app/services/user_service.py`

### 5.2 产品模块

- 产品“启用/停用”语义被改写为“停用 + 版本生效恢复”，不再是独立生命周期动作。
  - 代表证据：`frontend/lib/pages/product_management_page.dart`、`backend/app/services/product_service.py`
- 需求要求生效前校验“产品状态为启用”，但当前后端允许停用产品直接生效版本并自动恢复激活状态。
  - 代表证据：`backend/app/services/product_service.py`
- 参数变更历史未区分新增/编辑/删除，仅保留统一的 `edit` 类型，追溯粒度不达标。
  - 代表证据：`backend/app/services/product_service.py`、`frontend/lib/pages/product_parameter_management_page.dart`
- 复查新增问题：编辑产品不会同步参数/版本快照；参数查询页权限链路断裂，页面可见不等于接口可用。
  - 代表证据：`backend/app/api/v1/endpoints/products.py`、`frontend/lib/pages/product_parameter_query_page.dart`

### 5.3 工艺模块

- 模板页前端虽有多维筛选，但服务端公开契约只支持极少字段，当前实现依赖本地过滤和大页拉取补偿，不具备严格契约完整性。
  - 代表证据：`backend/app/api/v1/endpoints/craft.py`、`backend/app/services/craft_service.py`、`frontend/lib/pages/process_configuration_page.dart`
- 新建产品默认套版缺少需求要求的策略开关；只要系统母版存在就自动克隆默认模板。
  - 代表证据：`backend/app/services/product_service.py`
- 发布流程缺少生命周期限制与步骤连续性校验；新建模板还支持“直接发布”，绕过独立发布门禁。
  - 代表证据：`backend/app/services/craft_service.py`、`frontend/lib/pages/process_configuration_page.dart`
- 复查新增问题：导入允许覆盖已发布历史；自动套版生成的模板未保留母版来源版本信息。
  - 代表证据：`backend/app/services/craft_service.py`、`backend/app/services/product_service.py`

### 5.4 设备模块

- 设备详情聚合接口只校验台账查看权限，却直接返回关联计划、未完成工单、最近记录，可能绕过执行与记录维度的细粒度权限控制。
  - 代表证据：`backend/app/api/v1/endpoints/equipment.py`、`backend/app/services/equipment_service.py`
- 自动生成保养工单虽有汇总日志，但缺少计划级明细留痕、失败统计与持久化排障数据，不满足需求对可追踪性的要求。
  - 代表证据：`backend/app/services/maintenance_scheduler_service.py`
- 保养记录主快照已落库，但“执行工段快照”等字段仍依赖工单回查，详情页展示并非完全自足。
  - 代表证据：`backend/app/models/maintenance_record.py`、`backend/app/api/v1/endpoints/equipment.py`
- 复查新增问题：设备/项目/计划表单均未直接暴露状态字段；关键动作审计留痕分布不均衡。
  - 代表证据：`backend/app/schemas/equipment.py`、`frontend/lib/pages/maintenance_plan_page.dart`

### 5.5 品质模块

- 品质入口已经建立，但报废统计与维修订单仍主要复用生产页面、生产服务、生产动作权限与生产命名空间。
  - 代表证据：`frontend/lib/pages/quality_page.dart`、`frontend/lib/services/production_service.dart`、`backend/app/api/v1/endpoints/quality.py`
- 质量趋势与产品统计先按首件记录收缩 scope，再聚合不良/报废/维修数据；无首件记录的真实质量事件会被统计为 0。
  - 代表证据：`backend/app/services/quality_service.py`
- 维修完成后消息跳转目标仍写死为生产模块，而不是品质模块。
  - 代表证据：`backend/app/services/production_repair_service.py`
- 复查新增问题：质量页未把路由载荷透传给报废/维修页；质量命名空间下缺少导出与完成维修的完整合同。
  - 代表证据：`frontend/lib/pages/quality_page.dart`、`backend/app/api/v1/endpoints/quality.py`

### 5.6 生产模块

- 并行实例虽有建模、列表和追踪页，但未进入首件/报工的真实执行约束链路，无法证明满足“不可重复实例、不可越序执行”的核心需求。
  - 代表证据：`backend/app/services/production_order_service.py`、`backend/app/services/production_execution_service.py`
- 订单导出缺少“当前工序”等需求字段，且直接导出原始状态码，不符合业务文案。
  - 代表证据：`backend/app/services/production_order_service.py`
- 多处状态和提示文案与需求口径不一致，如“已完成/生产完成”“待执行/待生产”等。
  - 代表证据：`frontend/lib/models/production_models.dart`、`frontend/lib/pages/production_order_management_page.dart`
- 复查修正：报废统计后端关键词实际已覆盖工序名，问题主要在前端提示文案与测试缺失，而不是后端完全不支持。
  - 代表证据：`backend/app/services/production_repair_service.py`、`frontend/lib/pages/production_scrap_statistics_page.dart`

### 5.7 消息模块

- 消息主干能力已成型，但推送失败后仅标记 `failed`，缺少自动重试、补偿任务与完整投递闭环。
  - 代表证据：`backend/app/services/message_service.py`
- 收件记录已有状态与时间戳，但未持久化失败原因，也没有独立投递日志表。
  - 代表证据：`backend/app/models/message_recipient.py`、`backend/alembic/versions/e5f6a7b8c9d0_add_message_tables.py`
- 事件级去重依赖应用层 `dedupe_key` 先查后插，缺少数据库唯一约束；前端分页状态存在但没有翻页 UI。
  - 代表证据：`backend/app/services/message_service.py`、`backend/app/models/message.py`、`frontend/lib/pages/message_center_page.dart`
- 复查新增问题：WebSocket 重复事件抑制未实现；公告指定用户范围只拉前 200 个用户；消息投递与状态变化缺少完整审计闭环。
  - 代表证据：`frontend/lib/services/message_ws_service.dart`、`frontend/lib/pages/message_center_page.dart`

## 6. 跨模块共性问题

### 6.1 测试通过与需求满足存在明显错位

- 运行验证显示后端 37 项集成测试、前端分析与 142 项测试全部通过。
- 但多个高风险缺口仍然存在，说明当前测试更多证明“现有实现按当前假设运行”，不足以证明“需求逐条满足”。

### 6.2 权限模型与前端可见性/行为闭环不一致

- 用户模块导出权限、登录日志/在线会话权限。
- 设备详情聚合权限过宽。
- 品质模块继续借用生产动作权限。
- 产品查询/导出能力的可见性与接口权限不完全对齐。

### 6.3 契约被实现层改写

- 产品启用逻辑被版本生效替代。
- 工艺导入/发布/自动套版逻辑与需求前提不一致。
- 消息投递与去重机制低于需求闭环要求。
- 生产并行实例停留在台账层，没有进入执行约束层。

### 6.4 统计口径与深链路由存在隐性偏差

- 品质统计被首件 scope 裁剪。
- 消息跳转未完全按模块归口。
- 质量页路由载荷未透传。
- 工艺模板版本与生产订单模板版本语义不够统一。

## 7. 整改批次建议

### 批次一：阻断上线问题

1. 用户：修正首次改密后重登链路；拆分编辑与重置密码；补齐导出与登录会话细粒度权限前端闭环。
2. 产品：恢复独立启用/停用语义；禁止停用产品直接生效；修正参数历史与参数快照一致性。
3. 工艺：补齐发布门禁、导入默认草稿规则、看板导出契约；禁止新建模板绕过发布流程。
4. 设备：修复设备详情越权聚合；补齐自动生成明细日志与统计。
5. 品质：将报废/维修真正收口到品质模块；修正趋势/产品统计口径；修正维修消息跳转。
6. 生产：将并行实例纳入首件/报工约束链路；修正订单导出字段与状态值输出。
7. 消息：补失败重试与失败原因持久化；为事件级去重增加库约束；补齐分页与来源失效闭环。

### 批次二：契约与页面一致性

1. 修正筛选项、字段、按钮、导出、文案与需求口径不一致的问题。
2. 收敛页面跳转、路由载荷、对象失效提示和模块归口。
3. 清理实现中超出需求但未正式入库的“隐式业务规则”。

### 批次三：测试与观测补强

1. 按模块补充接口级与页面级回归，覆盖高风险规则负例。
2. 为权限粒度、统计口径、消息一致性、并行实例执行规则建立专项回归。
3. 补齐自动任务、消息投递、关键状态流转的审计与结构化留痕。
