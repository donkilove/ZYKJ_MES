# 生产订单流转业务逻辑说明（实现级，重点：数量流转）

> 说明：本文以当前代码实现为准，重点对应 `src/controller/order_controller.py`、`src/service/order/order_service_legacy_full.py`、`src/impl/order/order_impl_legacy_full.py`。其中 `src/impl/order_impl.py`、`src/service/order_service.py` 只是薄封装。

## 1. 一句话结论

这套系统不是“订单数量一路减到 0”这么简单，而是把数量拆成了 5 层口径一起算：

1. 订单总量 `orders.quantity`
2. 工序目标量 `process_target`
3. 工序可生产量 `available_quantity`
4. 用户可生产量 `user_available_quantity`
5. 返修/报废/回流修正量

所以，任何页面上看到的“可生产数量”，本质上都是运行态重新计算出来的结果，不是单表字段直取。

---

## 2. 主链路

```text
创建订单
  -> 生成订单 + 工序 + process_status
  -> 创建首工序子订单
  -> 首工序开始首件
  -> 结束生产 / 报工
  -> 写 production_records / process_status / order_sub_orders
  -> 需要时创建 repair_orders
  -> 分配下一工序子订单
  -> 返修完成后回写回流/报废/补充数量
  -> 重新计算后续工序可生产量
```

当前 UI 主路径是：

`OrderController.end_production_with_assignment()` -> `OrderService.end_production_with_assignment()` -> `OrderImpl.end_production()` / `create_repair_order()` / `assign_next_process_sub_order()`

历史兼容路径是：

`OrderController.end_production_with_flow()` -> `OrderService.end_production_with_flow()` -> `OrderImpl.end_production()`

---

## 3. 核心表和字段

| 对象 | 字段 | 作用 |
|---|---|---|
| `orders` | `quantity` | 订单原始计划数量 |
| `orders` | `produced_quantity` | 订单级辅助展示量，不是最终门控主口径 |
| `orders` | `status` | 待生产 / 生产中 / 生产完成 |
| `order_processes` | `status` | 工序状态 |
| `order_processes` | `visible_quantity` | 下游可见数量 |
| `process_status` | `product_quantity` | 工序累计完成量，核心实绩口径 |
| `process_status` | `ongoing_users` | 当前工序进行中的用户列表 |
| `order_sub_orders` | `assigned_quantity` | 分配给这个用户/工序的数量 |
| `order_sub_orders` | `completed_quantity` | 这个用户已完成的累计数量 |
| `order_sub_orders` | `status` | 待生产 / 生产中 / 进行中 / 已完成 |
| `order_sub_orders` | `is_visible` | 是否在查询页显示 |
| `repair_orders` | `repair_quantity` | 送修数量 |
| `repair_orders` | `repaired_quantity` | 维修后可回流数量 |
| `repair_orders` | `scrap_quantity` | 报废数量 |
| `repair_orders` | `scrap_replenished` | 报废是否已补充 |
| `repair_return_routes` | `return_quantity` | 回流到目标工序的数量 |

## 4. 状态体系

### 4.1 订单状态

- `待生产`
- `生产中`
- `生产完成`

### 4.2 工序状态

- `待生产`
- `进行中`
- `生产中`
- `生产完成`
- `已完成`

### 4.3 子订单状态

- `待生产`
- `生产中`
- `进行中`
- `已完成`

其中：

- `生产中` / `进行中` 表示用户正在这张子订单上作业
- `待生产` 表示当前可见，但还没消耗
- `已完成` 通常会配合 `is_visible = FALSE`

## 5. 数量公式

### 5.1 过程 key 规则

所有工序数量映射都优先按 `process_step_id`，没有时才退化为 `process_name`。

```text
process_key = id:{process_step_id}   // 优先
process_key = name:{process_name}    // 兜底
```

这就是为什么同名工序、改名工序、老数据兼容时不会直接乱套。

### 5.2 工序目标数量 `process_target`

#### 首工序

```text
首工序目标量
= 订单总量
- 首工序未补充报废量
+ 其他工序已补充报废量
+ 回流入量
- 回流出量
```

代码对应：`_get_first_process_target()` / `_compute_process_target_map_from_context()`

#### 非首工序

```text
当前工序目标量
= 上一工序已完成量
- 当前工序已完成报废量
+ 回流入量
- 回流出量
```

代码对应：`_get_process_target_quantity()`

### 5.3 工序可生产数量 `available_quantity`

```text
available_quantity
= max(process_target - process_completed - pending_repair, 0)
```

其中：

- `process_completed` = `process_status.product_quantity`
- `pending_repair` = 当前工序下未完成维修单的 `repair_quantity` 汇总

代码对应：`get_available_quantity_for_process()`

### 5.4 用户可生产数量 `user_available_quantity`

```text
user_available_quantity
= max(available_quantity - 其他活跃操作员数, 0)
```

如果当前用户已经在生产中，且算出来结果小于 1，则强制给 1。

活跃操作员来源有两套：

1. `order_sub_orders.status IN ('生产中', '进行中')`
2. `process_status.ongoing_users`

代码对应：`get_user_available_quantity()`

### 5.5 生产报工数量

用户输入的 `production_quantity` 不是直接落库数量，真正写入工序的有效量是：

```text
effective_quantity = production_quantity - defect_quantity
```

要求：

- `defect_quantity <= production_quantity`
- `effective_quantity > 0`
- `effective_quantity <= user_available_quantity`

代码对应：`OrderService.end_production_with_assignment()`

### 5.6 子订单数量

子订单有两个核心字段：

- `assigned_quantity`：分给这个用户/工序的可用额度
- `completed_quantity`：这个用户已经实际完成了多少

它们不是同一个概念。

## 6. 订单创建时数量怎么落

`add_order_with_processes()` 会一次性做完这些事：

1. 插入 `orders`
2. 插入 `order_processes`
3. 插入对应 `process_status`
4. 首工序的 `visible_quantity = 订单总量`
5. 其余工序的 `visible_quantity = 0`
6. 自动调用 `create_sub_orders_for_order()`

### 6.1 首工序子订单

首工序会给该角色下的每个用户建一张子订单：

- `assigned_quantity = 订单总量`
- `completed_quantity = 0`
- `status = 待生产`
- `is_visible = TRUE`

### 6.2 为什么只有首工序先有量

因为下游工序的数量不是“预先猜”，而是靠上游真实完成量驱动。

---

## 7. 开始生产时发生什么

### 7.1 当前主路径入口

`OrderController.start_production()` -> `OrderService.start_production()`

### 7.2 非流水线门禁

如果不是流水线工序，必须满足：

```text
当前工序可生产数量 > 0
```

### 7.3 流水线门禁

如果开启流水线：

- 必须满足上工序状态允许开始
- 必须满足当前链路上至少有一个工序还有可生产数量

### 7.4 并发门禁

系统还会检查：

```text
当前活跃任务数 < 用户并发上限
```

并发上限来自用户表的 `max_parallel_sub_orders`。

### 7.5 真正落库

`OrderImpl.start_production()` 会：

- 把订单工序改成 `进行中`
- 把订单改成 `生产中`
- 把 `process_status.ongoing_users` 追加当前用户
- 确保这个用户有一张 `order_sub_orders`

如果用户已存在子订单：

- 直接把它更新为 `生产中`
- `assigned_quantity` 取原值和当前可生产量的较大值

如果用户没有子订单：

- 新建一张
- `assigned_quantity = 当前可生产量`
- `completed_quantity = 0`
- `status = 生产中`

## 8. 结束生产时发生什么

### 8.1 当前主路径入口

`OrderController.end_production_with_assignment()` -> `OrderService.end_production_with_assignment()`

### 8.2 结束前的校验顺序

1. 解析目标工序
2. 检查烧录类工序的离线数量门禁
3. 检查流水线结束门禁
4. 检查工序权限
5. 汇总 `defect_items`
6. 计算 `effective_quantity`
7. 检查 `effective_quantity <= user_available_quantity`

### 8.3 真正写入的数量

真正写入 `production_records` 和工序累计量的是 `effective_quantity`，不是原始输入量。

也就是说：

```text
记录生产 = 合格数量
维修单 = 不良数量
```

### 8.4 impl 层会做什么

`OrderImpl.end_production()` 会：

- 往 `production_records` 插一条
- 累加 `process_status.product_quantity`
- 更新当前工序状态
- 更新当前用户子订单的 `completed_quantity`
- 如果不是最后工序，会把下一工序的 `visible_quantity` 改成当前已完成量
- 如果下一工序还没有子订单，会给该角色的所有用户创建子订单

### 8.5 service 层会再修正什么

`OrderService.end_production_with_assignment()` 会在 impl 落库后，再按“本轮可生产额度”重写当前用户子订单：

- 如果这次已经吃满本轮额度，子订单变成 `已完成 + 隐藏`
- 如果还没吃满，子订单回到 `待生产 + 可见`

它判断的核心是：

```text
user_available_before
user_available_after
production_quantity
```

### 8.6 返修单怎么生成

只要 `defect_quantity > 0`：

- 创建 `repair_orders`
- 写入不良现象明细
- 维修单 `repair_quantity = defect_quantity`

### 8.7 订单何时算完成

`check_order_completion()` 的判断不是看订单表总量，而是看最后一道工序：

```text
last_process_product_quantity >= order_target_quantity
```

其中：

```text
order_target_quantity = orders.quantity - 全订单未补充报废量
```

所以，订单完成的本质是：

- 最后一工序达标
- 且有效订单目标量已经被满足

## 9. 下一工序怎么分配

`assign_next_process_sub_order()` 会按当前工序的下一道工序重新计算分配数量。

### 9.1 计算规则

```text
next_available_quantity = get_available_quantity_for_process(next_process)
```

### 9.2 已有子订单时

如果下一工序已经有子订单：

- 直接更新 `assigned_quantity`
- 若数量 > 0，把已完成的行重新显示成 `待生产`

### 9.3 没有子订单时

如果还没有：

- 给该角色的每个用户各建一张
- `assigned_quantity = next_available_quantity`
- `status = 待生产`
- `is_visible = TRUE`

这意味着下一工序的“开工额度”永远跟着上游实时完成量走。

## 10. 查询页的数量口径

### 10.1 角色订单列表

角色视图里显示的 `quantity` 不一定是原始订单总量：

- 首工序显示订单总量
- 非首工序显示上一工序完成量 `visible_quantity`

### 10.2 子订单查询

`get_sub_orders_by_role()` / `get_sub_orders_by_role_page()` 不是纯读：

1. 先恢复可能被隐藏、但又重新有量的子订单
2. 再补齐新用户需要的子订单
3. 再批量计算 `available_quantity`
4. 再批量计算 `user_available_quantity`
5. 再决定哪些行显示、哪些行隐藏

### 10.3 什么时候会把 `已完成` 恢复成 `待生产`

如果某条隐藏子订单对应的工序重新有了可生产数量，查询时会被恢复：

- `status = 待生产`
- `is_visible = TRUE`

这一步在查询路径里就可能发生，所以它不是完全无副作用的只读接口。

## 11. 返修、报废、回流怎么改数量

### 11.1 维修单创建

`create_repair_order()` 会先写：

- `repair_quantity`
- 不良现象明细

然后等维修完成再真正影响可生产数量。

### 11.2 维修完成

`complete_repair_order()` / `update_completed_repair_order()` 会校验：

- `cause_items` 合计必须等于 `repair_quantity`
- `scrap_quantity` 由 `is_scrap = TRUE` 的条目汇总

### 11.3 报废是否补充

`scrap_replenished` 的意思是：

- 报废是否已经通过补料/回流/其他方式被补回

它会影响后续 `repair_orders` 的统计和工序目标量重算。

### 11.4 回流路线

`repair_return_routes` 记录的是：

- 从哪个工序出去
- 回到哪个工序
- 回多少

它会进入数量公式里的 `return_in` / `return_out`。

### 11.5 为什么修复后第一工序可生产数量会变

因为首工序目标量不是固定常量，而是：

```text
订单总量
- 未补充报废
+ 其他工序已补充报废
+ 回流入量
- 回流出量
```

所以维修完成后，第一工序可生产数量可能：

- 不变
- 回升
- 或者因为回流路线/报废补充变化而重新计算

### 11.6 实际回归验证

仓库里已有分支回归覆盖：

- 无报废时可恢复可生产数量
- 报废未补充时数量变小
- 报废已补充时数量回升
- 历史脚本分支 09 已专门修正了“已修复回流 + 报废补充重复计入”的旧口径

## 12. 流水线模式的特殊规则

流水线模式不是单独一套数量体系，它只是把门禁更严格。

### 12.1 开启条件

- 至少选择 2 道工序
- 选中的第一道工序当前可生产数量必须 > 0

### 12.2 开始首件门禁

- 必须等上工序达到允许状态
- 同时当前链路上至少有一道工序存在可生产数量

### 12.3 结束门禁

- 必须等上工序允许结束

### 12.4 数量公式不变

流水线不会改变：

- `process_target`
- `available_quantity`
- `user_available_quantity`

它只改变“能不能开始/能不能结束”。

## 13. 最容易混淆的点

1. `orders.quantity` 不是最终实绩，只是原始计划量。
2. `orders.produced_quantity` 是辅助展示口径，不要当唯一真相。
3. `process_status.product_quantity` 才是工序累计完成主口径。
4. `order_sub_orders.completed_quantity` 是单个用户口径，不等于工序总量。
5. `available_quantity` 会被维修未完成数量锁住。
6. `user_available_quantity` 会被其他活跃用户再压一层。
7. 查询页会触发恢复隐藏子订单，所以不是纯读。

## 15. 最后总结

这套系统的数量流转核心可以概括成一句话：

> 先算工序目标量，再扣当前已完成量，再扣待返修量，再按用户并发切分，最后把结果反写回子订单、下一工序和返修链路。
