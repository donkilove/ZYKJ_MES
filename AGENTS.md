# 项目规则与指挥官流程统一手册

## 1. 文档定位

- 本文件是 `ZYKJ_MES` 项目内“项目规则 + 指挥官流程 + 工具治理 + 留痕模板”的唯一权威规则源。
- 本文件用于直接指导任务执行、验收、留痕与交付，不再保留旧版导读结构。
- 与项目规则、指挥官模式、工具门禁、留痕模板相关的旧文档体系已停用并移除，不再作为并列规则源。
- 本文件默认覆盖以下主题：
  - 项目执行规则
  - 指挥官模式与角色闭环
  - 工具触发、验证门禁与降级代偿
  - 任务日志与工具化验证模板
  - 规则执行所需的辅助工具索引入口
- 迁移说明：无数据迁移，文档入口已直接切换至当前 `AGENTS.md`。

## 2. 适用范围与优先级

### 2.1 适用范围

- 本手册适用于仓库全部目录，除非子目录另有更严格的局部约束。
- 本手册重点约束“任务拆解、执行、验证、留痕、交付”五个环节。
- 只要任务涉及项目规则、指挥官流程、工具门禁、留痕模板或规则文档本身，均按本手册执行。

### 2.2 优先级

1. 系统级与开发者级强制指令。
2. 用户显式指令。
3. 本文件中的强制约束、工作流、工具治理与模板细则。
4. 支撑文档与辅助说明文档。

### 2.3 冲突处理

- 若用户显式指令与本手册冲突，优先执行用户指令，并在前置说明与任务日志中记录偏差原因。
- 若本手册对同一事项在不同位置存在表述差异，按“更严格、更新、可执行”的口径处理。
- 若本手册不同章节之间出现冲突，以“强制约束 > 工作流 > 工具治理 > 模板 > 附录”的顺序判定。

## 3. 核心原则

1. 强制优先：规则一旦命中，默认必须执行，不以个人偏好覆盖。
2. 结果导向：以可交付、可验证、可审计为完成标准，不交付占位方案。
3. 可追溯：关键输入、假设、证据、验证、失败重试与迁移口径必须留痕。
4. 最小变更：在满足目标前提下控制改动边界，避免无关扩散。
5. 工具优先：研究、实现、验证优先使用既定工具链，不凭空主观判断。
6. 先验证后通过：无独立验证结论，不得标记完成。
7. 默认不中断：除硬阻塞外，不因普通实现分歧暂停等待。

## 4. 强制约束

### 4.1 通用执行约束

- 所有沟通、代码注释、文档与新增文件默认使用中文，编码为 UTF-8（无 BOM）。
- 每次任务开始与结束都要更新 `evidence/` 或等效项目日志。
- 每次对外回复开头提供“前置说明”；若默认强制优先使用的 Docker 宿主 MCP 工具（统一口径为 `MCP_DOCKER`）存在缺失、未配置、不可达、权限不足，或受更高优先级指令约束无法使用，必须在前置说明中写明缺失工具、原因、替代工具与影响范围；若存在外部调用，在末尾补“工具调用简报”。
- 编码前必须完成 `MCP_DOCKER Sequential Thinking` 分析；若所需 `MCP_DOCKER` 工具不可用，必须记录降级原因、触发时间、替代动作、影响范围与残余风险，并在 `evidence/` 中同步留痕。
- 工作流状态统一使用 `update_plan` 或 `TodoWrite` 维护；若工具不可用，允许在任务日志中维护步骤、状态、验收标准与更新时间。
- 仅运行安全命令，禁止破坏性清理、泄露密钥、令牌或内部敏感链接。
- 默认采取破坏性改动并拒绝向后兼容，但必须在交付中明确迁移口径；若无迁移需求，统一写“无迁移，直接替换”。

### 4.2 测试与质量基线

- 后端默认测试基线：`pytest`。
- 前端默认测试基线：`flutter test` 与 `integration_test`。
- `desktop_tests/flaui/` 仅作为历史保留或按需 fallback，不是默认测试主线，且必须串行执行。
- 构建、编译、静态检查必须零报错；关键路径应有真实验证证据。
- 若任务命中供应链或发布前审计，需保留 SBOM、扫描摘要或等效证据。

### 4.3 用户模块与同步收敛

- 只要命中用户模块行为、入口、筛选、状态流转、文案边角分支等变更，必须在同一轮任务内同步收敛：
  - 后端/API
  - Flutter
  - `integration_test`
  - `evidence/`
  - 执行命令口径
- 禁止只改一端后将其余部分留待下轮补齐。

### 4.4 安全与退避

- 遇 HTTP `429` 固定退避 20 秒。
- 遇 HTTP `5xx` 或超时，退避 2 秒后最多一次重试。
- 若重试后仍失败，需提供保守离线结论，并明确局限与下一步建议。
- 默认仅维持最低安全基线，不额外引入超出任务目标的安全设计。

## 5. 指挥官模式

### 5.1 触发条件

以下任一条件成立时，必须按指挥官模式执行：

1. 用户明确要求使用“指挥官模式”。
2. 任务需要拆成两个及以上可独立验收的原子任务。
3. 任务要求执行与验证角色显式分离。
4. 任务涉及规则整改、流程重构、跨模块收敛或多轮失败重试闭环。

### 5.2 角色定义

- 主 agent：拆解目标、定义验收、维护任务队列、派发子 agent、汇总结果、判定通过或不通过。
- 调研子 agent：补齐上下文、依赖、范围边界、风险与现有实现。
- 执行子 agent：在明确边界内完成实现、修复、整理或补充。
- 验证子 agent：独立复核并真实执行验证，给出通过或不通过结论。

### 5.3 强制门禁

- 主 agent 不直接承担业务实现，不直接兼任最终验证。
- 每个原子任务至少经过“执行子 agent -> 独立验证子 agent”闭环。
- 验证失败后，必须重新派发执行子 agent 修复，再由新的验证子 agent 复检。
- 子 agent 完成当前原子任务或回执被判废后，必须立即关闭，不得空闲常驻。
- 若子 agent 回执出现越位口吻、角色冒用或面向用户的成稿式交付，主 agent 必须丢弃或重派。

### 5.4 连续执行规则

- 默认不中断，不因命名偏好、局部方案选择或普通歧义停下等待用户。
- 仅在破坏性操作、权限或凭证缺失、外部环境不可用、目标本身无法判定时，才允许收口为硬阻塞。
- 若某一原子任务阻塞但其他任务可推进，应先清空可推进队列，再统一汇总阻塞影响。

## 6. 标准工作流

### 6.1 最小闭环

1. Research：完成背景检索、边界确认、约束识别。
2. Plan：维护计划、原子任务、验收标准、失败重试口径。
3. Implement：由执行子 agent 在边界内完成修改。
4. Verify：由独立验证子 agent 真实执行检查并出具结论。
5. Deliver：汇总变更、验证、阻塞、风险与迁移说明。

### 6.2 启动阶段要求

主 agent 在启动阶段必须完成：

1. 提炼目标、范围、非目标与成功标准。
2. 盘点相关代码、文档、测试、依赖与已有证据。
3. 判断是否存在前置硬阻塞。
4. 拆分为若干可独立验收的原子任务。
5. 为每个原子任务定义输入、输出、验证方式与完成判定。

### 6.3 原子任务拆分标准

- 单一目标：一个原子任务只解决一个明确问题。
- 单一验收：必须存在清晰通过标准。
- 单一主要输出物：聚焦一组紧密相关文件或一个独立行为闭环。
- 可独立验证：可以单独测试、检查或证明完成。
- 可定位失败：失败时能清晰回溯到具体原子任务。

### 6.4 收尾阶段要求

全部原子任务完成后，主 agent 统一输出：

- 完成内容
- 影响范围
- 验证清单与结果
- 失败重试与收口情况
- 阻塞项与影响
- 迁移说明；若无迁移需求，写“无迁移，直接替换”

## 7. 工具治理

### 7.1 默认工具口径

- 默认强制优先使用 Docker 提供的宿主 MCP 工具，统一口径为 `MCP_DOCKER`；若所需 `MCP_DOCKER` 工具缺失、未配置、不可达、权限不足，或受更高优先级指令约束无法使用，方可进入降级与补偿流程。
- `MCP_DOCKER Sequential Thinking`：任务拆解、边界澄清、失败重试前复盘。
- `update_plan` 或 `TodoWrite`：维护步骤、状态与验收标准。
- `MCP_DOCKER ast-grep`：代码与文档结构定位、引用追踪。
- `MCP_DOCKER Context7`：外部官方文档与规范补证。
- `MCP_DOCKER Playwright`：页面联调、交互验证与页面证据采集。
- `MCP_DOCKER database-server`：数据库连接、查询与抽检。
- `MCP_DOCKER OpenAPI Toolkit`：契约读取、OpenAPI 校验、接口补证。
- `MCP_DOCKER Git / GitHub`：仓库状态、差异、远端协作与 GitHub 结果核对。
- `MCP_DOCKER Fetch`：在前述 `MCP_DOCKER` 能力不足时补充网页抓取。
- `MCP_DOCKER Filesystem`：用于目录访问、本地文件读取与受控写入；若当前会话受更高优先级宿主文件工具约束，则按更高优先级执行。
- `MCP_DOCKER Memory`：用于跨任务稳定偏好、模块关系与长期上下文补充，但不能替代 `AGENTS.md`、`evidence/` 与仓库文档。
- 宿主安全命令与宿主文件工具：仅作为本地补偿/降级手段，用于安全命令执行、本地文本检索、文件读写、测试与验证。

### 7.3 验证门禁

| 门禁 | 检查项          | 通过标准                            |
|----|--------------|---------------------------------|
| G1 | 任务分类已判定      | 已映射到主分类，且有触发依据                  |
| G2 | 工具触发有依据      | 已记录默认触发、补充触发或降级原因               |
| G3 | 执行与验证分离      | 存在独立验证子 agent 或等效降级补偿           |

| G6 | 降级已代偿        | 不可用工具、影响范围、替代动作与残余风险齐全          |

### 7.4 工具降级规则

- 所需 `MCP_DOCKER` 工具缺失、未配置、不可达、权限不足，或受更高优先级指令约束无法使用：先在前置说明写明缺失工具、原因、替代工具与影响范围，再在 `evidence/` 中同步记录降级与补偿。
- `MCP_DOCKER Sequential Thinking` 不可用：以书面拆解代替，并写入 `evidence/`。
- `MCP_DOCKER ast-grep` 不可用：退回宿主安全文本检索、宿主文件工具与最小 patch 编辑，并记录无法做结构化/语义级定位的影响。
- `MCP_DOCKER Context7`、`MCP_DOCKER Fetch` 不可用：使用仓库既有文档、离线经验结论或宿主安全网页检索补偿，并标注时效与不确定性。
- `MCP_DOCKER Filesystem` 不可用：退回宿主文件工具，并明确当次目录访问范围与写权限边界；若当前会话本就受更高优先级文件工具约束，则继续按该约束执行。
- `MCP_DOCKER Memory` 不可用：退回 `AGENTS.md`、`evidence/` 与仓库文档补齐上下文；正式规则、过程留痕与交付结论不得由 Memory 替代。
- `MCP_DOCKER database-server`、`MCP_DOCKER Playwright`、`MCP_DOCKER OpenAPI Toolkit`、`MCP_DOCKER Git / GitHub` 等验证类工具不可用：退回宿主安全命令、宿主文件工具、本地包装命令或人工步骤脚本，但必须说明未完成的真实验证边界。

## 8. 质量与交付标准

### 8.1 质量门槛

- 构建、编译、静态检查需零报错。
- 单元、集成、契约、E2E、性能与回归验证应覆盖关键路径与异常分支。
- 关键任务应生成覆盖率、扫描摘要、性能结果或等效质量证据。
- 构建与交付流程需可重复、可审计、可回滚。

### 8.2 交付要求

- 明确完成项、未完成项、验证结果、残余风险、阻塞与迁移口径。
- 不交付占位方案、空模板或 `NotImplemented`。
- 若存在迁移：给出切换与回滚步骤。
- 若无迁移：统一写“无迁移，直接替换”。

## 9. 留痕与 evidence 要求

### 9.1 必留痕内容

- 用户原始目标与关键约束
- 任务拆分与排序依据
- 原子任务输入、输出与验收标准
- 子 agent 输出摘要
- 验证命令与验证结果
- 失败轮次与修复记录
- 降级、硬阻塞与代偿措施
- 最终交付结论与迁移说明

### 9.2 证据字段

证据记录至少包含：

- 证据编号
- 来源
- 形成时间
- 适用结论
- 记录责任

### 9.3 evidence 代记

- 若只读调研子 agent 无法直接写入 `evidence/`，允许由主 agent 代记。
- 代记时必须注明：代记责任人、代记时间、原始来源、适用结论。
