## 5. 当前 UI 判断
- 结论：
  - 当前前端“可用且局部精致”，但还没有形成真正统一的 UI 系统。
- 主要证据：
  - 全局主题过薄：
    - `main.dart` 只统一了种子色、字体、明暗与密度，没有统一卡片、按钮、输入框、边框半径、间距、页面标题层级、状态色语义
  - 页面各自造风格：
    - 登录页 `login_page.dart` 有完整的渐变、标签、信息卡语言
    - 首页 `home_dashboard_header.dart` 又有另一套渐变头图与 chip 语言
    - 设置页 `software_settings_page.dart` 偏“表单设置页”风格
    - 消息中心 `message_center_page.dart` 更接近“密集业务后台”风格
    - 这些页面都不难看，但彼此不像同一个系统里长出来的
  - 统一化范围不完整：
    - `core/widgets` 已经开始统一 CRUD 页头、列表区、分页和表头
    - 但没有扩展成更高层的页面骨架、筛选条、状态卡、统计卡、详情区、空态/错态组件
  - 文件体量偏大，放大了风格漂移：
    - `message_center_page.dart` 约 `1698` 行
    - `process_management_page.dart` 约 `1620` 行
    - `process_configuration_page.dart` 约 `2769` 行
    - 当页面既管业务又管布局又管局部样式时，统一性会越来越难维持

## 6. 可选统一化路线
- 路线 A：只补设计 Token 和全局 Theme
  - 做法：抽色板、圆角、阴影、间距、按钮尺寸、输入框风格、标题层级
  - 优点：改动最小，见效快
  - 缺点：只能统一“表面”，页面结构和信息密度问题还在
- 路线 B：建立“页面骨架 + 业务组件”双层系统
  - 做法：
    - 底层统一 `Theme/DesignTokens`
    - 中层统一 `PageScaffold / SectionCard / PageHeader / FilterBar / MetricCard / EmptyState / ErrorState / DetailPanel / ActionBar`
    - 业务页只负责组装这些骨架件
  - 优点：
    - 既统一视觉，也统一页面组织方式
    - 最适合当前仓库已经存在的 `core/widgets` 演进方向
    - 后续页面拆分与测试也更顺
  - 缺点：需要一轮有节制的中等重构
- 路线 C：按模块各自演进，不做统一框架，只做“最佳实践约束”
  - 做法：保留现有页面自由度，只给编码规范与样式约定
  - 优点：改动最轻
  - 缺点：长期最容易继续发散，统一成本会越来越高

## 7. 当前推荐
- 推荐路线：
  - 选 `路线 B`
- 推荐原因：
  - 你们已经不是“完全没有基础件”的状态，仓库里已有 CRUD 统一件雏形
  - 当前最大问题不是单个页面丑，而是页面“骨架语言”不一致
  - 只补 Theme 不够，因为登录页、首页、消息中心、设置页的布局范式差异很大
  - 直接全量重做又太重，最稳的是：
    - 先补 Token
    - 再抽通用骨架
    - 最后分批迁移页面

## 8. 优先落点建议
- 第一优先：
  - 统一 `PageHeader + SectionCard + ActionBar + FilterBar + EmptyState + ErrorState`
- 第二优先：
  - 统一“统计卡/概览卡”的视觉语言，让首页、消息中心、后续报表页说同一种设计语言
- 第三优先：
  - 统一表单与列表页面的间距、标题、按钮区和分页区
- 第四优先：
  - 再去处理更大型页面的拆解，例如工艺与消息模块

## 9. 对用户问题的直接结论
- “现在 UI 怎么样”：
  - 比很多纯工具后台要好，已经有一些主动设计感，不是毛坯
  - 但系统级统一度还不够，核心问题是“局部好看、整体不够像一家人”
- “有没有更统一的方法来构建”：
  - 有，最合适的是建立“设计 Token + 页面骨架组件 + 业务组件组装”的三层方法，而不是继续每个页面单独写样式
