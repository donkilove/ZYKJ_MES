## 4. 用户模块现状勘察
- 入口壳层：
  - `frontend/lib/features/user/presentation/user_page.dart`
  - 现状是 `TabBar + TabBarView` 壳层，负责装配 `user_management / registration_approval / role_management / audit_log / account_settings / login_session / function_permission_config`
  - 仍属于旧式 Material 壳层，尚未接入新的 `core/ui/patterns`
- 适合作为第二波试点的页面：
  - `frontend/lib/features/user/presentation/user_management_page.dart`
    - 当前类名仍为 `LegacyLegacyUserManagementPage`
    - 已部分使用 `CrudPageHeader / SimplePaginationBar / user_filter_toolbar / user_data_table`
    - 更接近“可继续收敛为统一 CRUD 骨架”的状态
  - `frontend/lib/features/user/presentation/registration_approval_page.dart`
    - 已部分使用 `CrudPageHeader / CrudListTableSection / SimplePaginationBar`
    - 结构也属于典型的筛选 + 列表 + 操作流
- 成本较高但价值也高的页面：
  - `frontend/lib/features/user/presentation/account_settings_page.dart`
    - 更偏个人中心 / 信息卡 / 设置组合页
    - 视觉和信息组织改造价值高，但不适合作为第二波第一刀

## 6. 当前判断
- 推荐优先级：
  1. `user_management_page.dart`
  2. `registration_approval_page.dart`
  3. `role_management_page.dart` / `audit_log_page.dart`
  4. `user_page.dart` 壳层
  5. `account_settings_page.dart` / `login_session_page.dart`
- 原因：
  - 第一组最贴近已完成的 UI 基础件能力边界
  - 可以先把 `MesCrudPageScaffold / MesListDetailShell` 一类上层业务骨架打稳
  - 之后再收口用户模块 tab 壳层和偏个性化的信息卡页面，风险更低
