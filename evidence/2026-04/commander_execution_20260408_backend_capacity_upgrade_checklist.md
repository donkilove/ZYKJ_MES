## 10. 待用户决策项

1. Docker 落地口径
   - 选项：仅提供单机 `docker compose` 基线；或同时预留未来多副本/反向代理扩展位。
   - 推荐：先做单机 `docker compose` 基线，但文件结构按后续多副本可扩展方式组织。
2. Web worker 起始规模
   - 选项：保守 `2 worker`；平衡 `3-4 worker`；激进 `4+ worker`。
   - 推荐：先按 `4 worker` 设计，再用压测复核。
3. 数据库连接池预算
   - 选项：保守池；平衡池；激进池。
   - 推荐：先按“每 worker `pool_size=10`、`max_overflow=10`”设计，但同步校验 PostgreSQL `max_connections` 预算。
4. 是否引入 Redis
   - 选项：不引入，先做进程内缓存；引入 Redis 承担权限缓存、在线状态与节流标记。
   - 推荐：若目标是稳定到 `80-150 人`，建议引入 Redis；若只想先冲到 `50-80 人`，可先不引入。
5. 后台循环拆分方式
   - 选项：拆成单独 worker 容器；拆成 cron / one-shot job；继续留在 Web 内但限制单实例。
   - 推荐：拆成单独 worker 容器，避免未来多 worker / 多副本重复执行。
6. 压测门禁强度
   - 选项：仅压关键接口；或把登录、鉴权、典型列表、统计页全部纳入发布门禁。
   - 推荐：至少纳入登录、鉴权轻接口、典型列表页 3 类场景。
7. 目标优先级
   - 选项：先把承载抬到 `50-80 人` 再二次迭代；或直接按 `80-150 人` 目标一次做完。
   - 推荐：直接按 `80-150 人` 目标一次做完，避免两轮重复改 Docker、连接池和后台任务结构。
