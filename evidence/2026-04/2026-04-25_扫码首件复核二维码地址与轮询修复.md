## 2. 任务目标、范围与非目标

### 任务目标

1. 操作员页二维码内容应为微信可直接打开的后端局域网网页地址。
2. 后端直接提供手机扫码复核页面，不依赖前端 Web 主机。
3. 轮询扫码复核状态时，不得因为状态接口返回空 `review_url` 导致二维码消失。

### 任务范围

1. 后端手机扫码页与扫码复核 URL 生成。
2. Flutter 操作员页二维码地址生成与轮询状态合并。
3. 后端、Widget 与集成路径验证。
4. evidence 留痕与中文提交。

### 非目标

1. 不扩展手机端扫码复核业务流程，只提供最小登录与复核提交页面。
2. 不引入额外独立 Web 部署依赖。

## 5. 子 agent 输出摘要

- 调研摘要：后端创建会话返回相对 `review_url`；状态轮询返回空 `review_url` 会覆盖页面持有的旧会话，导致 3 秒后二维码消失。用户进一步确认手机只能访问后端所在局域网，因此扫码页必须由后端直接承载。
- 执行摘要：后端新增 `app.web.first_article_review_page` 与静态 HTML 手机页，`/first-article-review` 由 FastAPI 直接返回；创建/刷新扫码复核会话时由 API 层按请求主机生成后端完整 URL。Flutter 操作员页改为优先使用后端完整 URL，并在状态轮询中保留已存在的 `reviewUrl`，避免二维码 3 秒后消失。后续跟进中又补充了后端 `public_base_url` 配置与 `localhost/127.0.0.1` 自动替换局域网 IP 的逻辑，避免二维码继续显示回环地址。本轮进一步新增扫码复核手机页专用 7 天登录口径、运行中的静态扫码页 `localStorage` 持久化与 Flutter 版手机页自动恢复/切换账号对齐逻辑。
- 验证摘要：后端手机页单测通过；Flutter widget 与 Windows integration_test 通过；后端数据库集成测试因本机 PostgreSQL 不可达未完成。后续针对“127.0.0.1 误用”补充的后端单测与 Flutter widget 回归也已通过。进一步在 Docker 运行态中确认 `public_base_url` 已生效，扫码地址稳定为宿主机局域网地址，且 worker 恢复正常。最新跟进中已确认手机端登录失败不是账号密码错误，而是超长手机 `User-Agent` 写入登录终端字段时超长；修复后用超长手机 UA 实测登录返回 `200`。本轮 7 天免登录相关的后端与 Flutter 自动恢复测试也已全部通过。
- 最新排障结论：代码与 Docker 公开地址生成均正常，但宿主机 `192.168.1.54:8000` 的局域网入站访问被环境层阻断，当前会话也无管理员权限修改 Windows 防火墙规则。
