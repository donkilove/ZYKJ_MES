## 4. 当前判断

- 若要“完全解耦”且“仍显示在主程序内部”，前端就不能加载插件自带 Flutter 组件，也不应直接嵌入 Python UI 框架窗口句柄作为第一阶段方案。
- 当前更像“宿主内嵌的独立应用”模型，而不是“宿主渲染插件页面”模型。
- 需要在以下方向中收敛：
  - 内嵌 Web 应用：插件自带本地 Web UI，宿主用内嵌 WebView 打开
  - 内嵌原生窗口：插件创建原生子窗口，由宿主承载窗口句柄
  - 远程画布/流式界面：插件输出渲染流，宿主显示
- 从实现复杂度、跨语言隔离和后续插件开发成本看，当前最值得优先讨论的是“本地 Web UI + 宿主 WebView”。
- 时效性核对结论：
  - 当前 `webview_flutter` 官方包页面未把 Windows 列为直接支持平台
  - Windows 侧存在基于 WebView2 的实现路径，例如 `webview_windows`、`webview_all_windows`
  - 因此设计上应把“宿主内嵌 WebView 承载插件本地 Web UI”视为可行方向，但具体包选型仍需在实施前再做一次稳定性复核
  - Python 官方当前仍提供 Windows embeddable package，可随主程序分发私有 Python 运行时
  - 该嵌入式分发不包含 pip，第三方依赖更适合由主程序/插件安装包一并固化分发，而不是到用户机器上临时安装
